RBDoom3 BFG Path Tracing Experiment
===================================

Purpose
-------

This branch is an experimental attempt to evolve the Vulkan/NVRHI renderer in
RBDoom3 BFG toward a Quake II RTX style brute-force path traced renderer. The
target hardware is currently an RTX 4090 class GPU, so the early goal is image
quality and learning rather than broad hardware support or high performance.

The current work is intentionally staged as "RT smoke" debug functionality. It
proves pieces of the path tracing pipeline in small, testable steps before
replacing the main renderer.

This file is the broad chronology/status document: what the branch is trying to
do, what has been tried, what works, what remains risky, and how it is tested.
The `pathtrace_scene_producer_*` documents are retained as detailed intervention
logs and historical build records; durable architecture, CVar, and testing
guidance should be kept here and in the module/feature notes below.
For a file-by-file map of the extracted PathTrace* modules and feature-specific
"where to touch" guidance, see:

    docs/pathtrace_modules.txt

For persistent geometry guardrails, see:

    docs/pathtrace_geometry_universe.md

For persistent material guardrails, see:

    docs/pathtrace_material_universe.md

For the current plan to refactor the monolithic PathTracePrimaryPass /
PathTraceSmokeDispatch execution layer into explicit frame resources, primary
surface, ReSTIR, debug, and future denoise/reflection contracts, see:

    docs/pathtrace_core_refactor_plan.txt
    docs/pathtrace_core_refactor_tasks/00_agent_readme.txt

The early refactor sequence must include a real path-tracer core settings pass
before deeper ReSTIR work: samples per pixel, max path depth, diffuse/specular
bounce limits, first reflection mode, and accumulation reset policy belong to
the base integrator, not only to ReSTIR debug modes.

The same refactor sequence must avoid locking in early reverse-engineered
material and geometry assumptions as final contracts. The current
PathTraceDoomMaterialClassifier can remain a stopgap, but the material universe
needs room for the newer idTech4 material documentation and later RBDoom3/mod
PBR roles such as metallic/roughness, AO, height/parallax, and displacement.
Likewise, source3 portal-area residency is the validated BVH baseline, and the
geometry contract must make current/previous frame vertex or transform data a
near-term addition for motion vectors.

Current State
-------------

The renderer has a debug-only path tracing smoke pass behind r_pathTracing.
It traces primary camera rays through an NVRHI ray tracing pipeline and displays
the RT output as the 3D scene layer while leaving the normal Doom HUD, GUI, and
menus drawn on top.

Mode 18 now skips the normal 3D raster view by default after the RT output has
been produced, while still allowing later GUI/HUD/menu draw commands to rasterize
on top. This is controlled by r_pathTracingSkipRaster3D. Leaving the full 3D
raster path running underneath the path traced output roughly doubled frame cost
in testing and also kept raster-side hitches in play, making RT/PT performance
and stability work harder to interpret. Set r_pathTracingSkipRaster3D to 0 only
for side-by-side regression checks or when explicitly hunting a raster side
effect that the path traced mode still depends on.

PathTracePrimaryPass used to own nearly all of the prototype. It has now been
reduced to a small frame-level shell: feature detection, per-frame entry,
output sizing, and present/blit glue. The scene build, resource lifetime,
dispatch, readback, material processing, diagnostics, acceleration helpers,
skinning bridge, GUI admission, and emissive inventory are split across
PathTrace* modules under neo/renderer/NVRHI.

Current validated feature set:

- Primary camera rays through NVRHI ray tracing.
- Static and dynamic BLAS/TLAS with separate geometry buffers.
- Persistent static-world geometry cache.
- Per-frame dynamic geometry capture.
- RT-only CPU skinning for skinned MD5 characters.
- CPU-side skinned/deforming surface identity and current-to-previous range
  matching for the RT-only CPU-skinned bridge. The previous skinned vertex copy
  is retained for diagnostics and feeds the off-by-default previous-position
  scaffold buffer when that scaffold is enabled.
- Off-by-default PT GPU skinning scaffold. `r_pathTracingGpuSkinning 1` builds
  diagnostic skinned source/current-output/previous-position/dispatch buffers
  and current/previous joint-matrix buffers from the existing CPU-skinned
  bridge, but the normal BLAS and shader path still consume the existing dynamic
  CPU-skinned output. Mode 2 is reserved for later validated GPU output use and
  is not active yet.
- With `r_pathTracingGpuSkinning 1`, skinned primary-surface records can now
  look up the retained previous CPU-skinned position for the same dynamic
  triangle and barycentric point. This only fills the primary-surface
  previous-position/object-motion fields for validated skinned hits; it does
  not change BLAS input, shading, material classification, raster GPU skinning,
  or the future GPU compute skinning plan.
- Routed rigid instances now carry current and previous object-to-world rows in
  their shader-visible route records when a consecutive previous observation is
  available. Primary-surface packing can reconstruct a previous world-space hit
  position from the current rigid triangle barycentrics and previous transform
  without duplicating rigid vertex buffers.
- Bindless material texture table using Texture.Load.
- Manual bilinear Texture.Load filtering.
- Doom diffuse YCoCg decode.
- Alpha any-hit for cutout/perforated materials.
- Albedo, normal, and specular texture discovery.
- Persistent RT smoke material universe records with stable material indexes,
  cached material facts, emissive facts, and an active material table now built
  from the universe by default.
- Debug selected-light direct lighting with RT shadow rays.
- Mode 14 normal-map lighting using Doom/RBDoom's DXT5-style normal decode.
- Mode 14 simple Blinn-style specular from Doom spec maps.
- Mode 14 unshadowed emissive/glow material preview from safe ambient/additive
  stages.
- Mode 18 toy one-bounce path tracing with selected Doom lights, texture
  sampling, force-ambient control, simple secondary rays, temporal
  accumulation, and a first extracted native NEE shader module. Primary
  selected-light direct lighting still uses the legacy full selected-light loop
  for comparison, while secondary selected-light NEE can run off, one-sample
  bounded uniform sampling, or the legacy full loop through
  `r_pathTracingSecondaryNeeMode`. Secondary/reflection Doom analytic-light NEE
  has a separate explicit gate, `r_pathTracingSecondaryAnalyticNeeMode`; mode 0
  disables it, mode 1 evaluates a bounded number of uniformly sampled analytic
  light proposals from the current uploaded candidate set, and mode 2 keeps the
  legacy full analytic-light loop as the reference. The bounded proposal count
  is controlled by `r_pathTracingSecondaryAnalyticNeeSamples`; it does not
  reduce the authored/uploaded analytic-light universe. The native NEE records
  reserve source identity, proposal PDF, target-weight, visibility, MIS, and
  path-event fields for later RTXDI/ReSTIR PT extension without importing NVIDIA
  sample code. They are RTXDI-shaped local records, not RTXDI-equivalent records
  yet: selected point lights use a discrete proposal PDF, leave `targetPdf`
  unset until a real target-function evaluator exists, and do not claim a finite
  solid-angle PDF.
  Doom analytic samples now have replay-safe helper boundaries:
  `ValidateSmokeDoomAnalyticLightSampleForReplay`,
  `EvaluateSmokeDoomAnalyticLightSampleForSurface`, and
  `SmokeNeeAnalyticTargetWeight`. These helpers let future reservoir final
  shading evaluate one already-selected analytic sample without reselecting or
  looping the light buffer. The replay validation now checks both renderer
  light identity and Doom entity identity (`renderLightIndex` plus
  `entityNumber`) before accepting an analytic sample. Current analytic NEE
  radiance is explicitly marked as solid-angle-scaled local radiance, so later
  ReSTIR code must not divide it by `solidAnglePdf` again.
- Mode 19 emissive triangle inventory visualization. The CPU inventory is a
  broad material/triangle candidate list for future ReSTIR/PT work, while the
  debug view samples emissive texture luminance and skips skinned/deformed
  character dissolve overlays.
- Mode 20 single-frame reservoir direct emissive lighting from the current
  light-candidate contract. Spatial and temporal reuse are still off.
- Host-side RTXDI ReSTIR PT scaffolding. A separate packed PT reservoir buffer
  and RTXDI_PTParameters constant buffer are allocated/updated from
  rtxdi::ReSTIRPTContext. The smoke ray tracing shader now has the Step 3
  RTXDI reservoir ABI binding (`RTXDI_PTParameters` at b28,
  `RWStructuredBuffer<RTXDI_PackedPTReservoir>` at u29, and
  `RTXDI_PT_RESERVOIR_BUFFER` before including `Rtxdi/PT/Reservoir.hlsli`),
  and local Step 4 RAB type declarations under
  `neo/shaders/builtin/pathtracing/RtxdiBridge`. The Step 5 shader bridge can
  now build `RAB_Surface` and `RAB_Material` records from an actual
  `PathTraceSmokePayload` hit, using the same smoke material, albedo, normal,
  opacity, emissive, and specular helpers as the existing modes. The
  `RAB_GetGBufferSurface` and `RAB_GetGBufferMaterial` callbacks now read
  current/previous primary-surface buffers through the named
  PathTracePrimarySurface contract. The record is versioned and mirrored on CPU
  and shader sides at 176 bytes: header/version/valid flags, world position plus
  view depth, geometric and shading normals, view direction, diffuse albedo,
  alpha cutoff, roughness, F0/specular, emissive radiance, material/surface
  flags, instance/primitive IDs, an object/entity ID placeholder,
  height/parallax/displacement placeholder, and previous-position/object-motion
  placeholder. The object/entity ID is deliberately not finalized yet; the
  correct Doom renderer/game identity source must be chosen before object-motion
  rejection or denoiser inputs depend on it. The internal PT convention for
  motion-vector conversion is current pixel to previous pixel, in internal PT
  render-resolution pixels, using unjittered current/previous camera state; any
  external consumer that expects another basis needs an explicit conversion
  shim. Mode 29 validates same-pixel history, and mode 30 validates temporary
  current-world-position to previous screen-space reprojection for camera
  motion. Routed rigid instances can now provide previous world positions for
  primary-surface object motion when their previous transform is continuous,
  but final denoiser-facing vector export and static-world disocclusion policy
  are still future work. The Step 6 RNG bridge is also present and wraps RTXDI's
  own random sampler initialization and `RTXDI_GetNextRandom` behind local
  `RAB_RandomSamplerState` callbacks so future path replay can preserve seed
  plus consumed random index. Step 7 adds conservative opaque-diffuse BRDF
  callbacks over `RAB_Surface`: cosine-hemisphere sampling, matching solid-angle
  PDF, Lambert BRDF evaluation, and BRDF*cos/PDF throughput. Unsupported
  translucent/special surfaces currently fail closed instead of pretending to
  have a complete BSDF. This is a deliberate validation simplification only:
  the RAB material/history records still preserve roughness, specular F0,
  opacity, alpha cutoff, material flags, and glass/window fallback flags so
  glossy reflection, mirror-like reflection, and glass/transmission behavior can
  be added later without redefining the whole bridge contract. Step 8 adds first light bridge callbacks over the
  existing smoke light data: current-frame emissive triangle loading/sampling,
  current-frame Doom analytic sphere loading/sampling, light target weighting,
  light direction/distance helpers, conservative visibility that only verifies
  valid surface/sample data, and neutral MIS callback placeholders. The Doom
  analytic RAB path uses the same Doom influence radius and quadratic falloff
  convention as mode 18 native analytic NEE, so ReSTIR-side proposals do not
  survive outside the authored light volume. There is still no traced RAB
  visibility, no environment light bridge, and no
  previous-frame light buffer. Step 9 adds a compile-validated `RAB_PathTrace`
  skeleton around the smoke TLAS path: it samples the current opaque-diffuse RAB
  BRDF, sets RTXDI continuation-ray state, traces the engine `SmokeScene`,
  converts hits back through `RAB_BuildSurfaceFromSmokePayload`, records hit
  surfaces, and records emissive hits when present. It is deliberately capped to
  a tiny bridge bounce depth and does not run NEE, environment hits, temporal
  replay, spatial reuse, or a dispatched RTXDI shader pass yet.
- Mode 21 solid drawSurf mirror bounds boxes and mode 22 wireframe drawSurf
  mirror bounds boxes. These are replacement diagnostic views for inspecting
  captured bounds/BVH/source issues, not overlays on mode 18/20.
- Mode 23 traces only the experimental rigid TLAS route instances. It is a
  validation view for local-space rigid BLAS plus TLAS transforms, not the
  normal path-traced render path.
- Mode 24 traces the old fallback scene and routed rigid instances separately,
  then color-codes their hit-distance overlap. It is the validation step before
  removing rigid triangles from the dynamic fallback.
- Mode 25 traces the normal fallback scene plus routed rigid TLAS instances
  with a simple material/normal lighting view. It is the first routed-rigid
  lighting validation path and does not change mode 18/20.
- Source 3 scene producer scaffold: existing drawSurf-seeded static cache plus
  mirrored dynamic-frame drawSurfs. It preserves the old shader-compatible
  static/dynamic buffer contract while exposing mesh identity and per-frame
  instance identity for diagnostics.
- Source3 area-selected static preload and rigid residency. Static world
  geometry can be preloaded from the current Doom area plus portal-neighbor
  areas, and routed rigid instances can stay resident from cached mesh/instance
  records instead of disappearing whenever raster visibility stops submitting
  them.
- Default source3 routed rigid BLAS/TLAS path for local-space rigid mesh
  candidates. It uploads per-mesh vertex/index buffers, builds per-mesh BLAS,
  routes instances through TLAS in modes 18/20/23/24/25, and removes
  routed-ready rigid triangles from the dynamic fallback. The old drawSurf-only
  BVH path remains available with `r_pathTracingSceneSource 0` for emergency
  comparison/testing, but it is no longer the normal PT scene stack.
- Rigid emissive-card promotion for stable object-attached signage/glow cards,
  used mainly to keep rigid prop emissives resident with their parent object.
  The gate is conservative and currently excludes `_add + 0200` default-RGBA
  variants plus the known game-logic-driven `swinglight` family.
- Doom renderer light diagnostics and a first analytic sphere-light bridge.
  Authored `idLight` entities are linked back to renderer `lightDef` handles,
  dumped with stable identity/game-state fields, filtered by the same
  current-area plus portal-neighbor selection used for PT light residency, and
  uploaded through a separate analytic light buffer when enabled. The first
  shader path treats every eligible point light as a diagnostic sphere emitter;
  it preserves evaluated RGB and game animation/on-off state, but does not yet
  classify Doom light textures into rectangle/disk/projected/area light types.
- GUI/SWF texture and surface diagnostics behind opt-in CVars.
- Placeholder smoke particle dithering.

RT scene residency and Doom portal areas:

- The current source3 BVH strategy deliberately separates freshness from
  residency. The final raster drawSurf stream is still the best source of
  fresh transforms, material overrides, evaluated shader registers, and
  skinned/deforming geometry. It is not a sufficient residency rule for path
  tracing, because Doom's raster frontend can stop submitting fully occluded or
  off-camera surfaces even though those surfaces must still cast shadows or emit
  light for RT.
- Static world geometry is therefore allowed to preload from Doom renderer
  area/portal membership instead of waiting for `viewDef->drawSurfs` to seed the
  static cache. Rigid routed geometry can similarly stay resident from cached
  mesh/instance records when its last known area remains in the selected area
  set.
- The selected area set is based on renderer portal areas from
  `PointInArea`, `NumPortalsInArea`, and `GetPortal`, not raw BSP leaf/internal
  cells and not the temporary `PathTraceSceneUniverse` inventory as a live scene
  source. `PathTraceSceneUniverse` is used only for static-map diagnostics and
  area-selected static-world preload.
- Portal walks seed from all valid camera-area probes instead of trusting only
  `viewDef->areaNum`: the renderer area, initial view-area origin, exact view
  origin, and short +/- view-axis offsets are inserted at depth 0 when valid.
  This keeps source3 static preload, rigid residency, Doom analytic lights, and
  light-universe area filtering stable when the camera is on a portal border or
  tiny portal-adjacent area. The failure this fixes was not a missing portal
  edge or ignored depth CVar: `viewDef->areaNum` could be valid but point at a
  sliver/connector area on a portal boundary, causing the PT walk to spend
  several hops reaching the room the player visually occupied. At one Mars
  City border this made rigid residents and Doom analytic lights in the next
  visible portal behave as if four walks had been consumed before selection
  reached them. Multi-seeding gives all plausible camera areas depth 0, so
  authored portal topology still matters but border classification no longer
  burns the traversal budget by itself.
- Early current-area-plus-one-step testing was too shallow. Mars City-style
  spaces can look like a single room to the player while spanning several Doom
  renderer portal areas; visible adjacent geometry or emissives may require
  multiple portal hops. Depth 4 is now the default source3 portal traversal
  depth.
- `PS_BLOCK_VIEW` portal edges are reported in diagnostics but currently do not
  stop source3 static preload, rigid residency, or light-area traversal. This is
  intentional while validating RT residency: Doom's raster visibility/blocking
  state is useful diagnostic input, but using it as hard authority reproduced
  raster-like BVH gaps. This policy may need refinement later once the portal
  taxonomy is better understood.
- Source3 with depth-4 portal traversal is now the normal PT default. If a later
  map proves severely game-breaking, the old behavior can be forced from a
  launch file by setting source3/residency/route CVars back to 0.

Current module/refactor state:

- PathTracePrimaryPass is intentionally small and still owns most persistent
  scene/build m_smoke* handles/counters, but output-sized PT textures,
  accumulation/readback resources, smoke reservoirs, ReSTIR PT packed
  reservoirs, primary-surface history buffers, frame settings, reset reasons,
  and readback queue bookkeeping now live behind PathTraceFrameResources.
  Static-world geometry cache ownership has moved behind
  PathTraceGeometryUniverse as the first persistent-geometry scaffold.
- PathTraceSmokeSceneBuild owns the high-level per-frame scene build sequence.
- PathTraceSmokeResources owns NVRHI smoke shader/pipeline creation,
  scene-resource creation/reset/commit, and forwards output-sized resize
  validation to PathTraceFrameResources.
- PathTracePrimarySurface defines the CPU/shader primary-surface record and
  shader helpers used by the current smoke raygen, ReSTIR PT RAB screen
  callbacks, and mode 29/30 history/reprojection validation views.
- PathTraceSceneInputs defines the CPU-side scene/material/light package
  committed after a successful smoke scene build. It records TLAS/BLAS and
  buffer handles, geometry/material/light counts, split signatures, source3
  portal-policy facts, current/previous history placeholders, skinned previous
  CPU-bridge diagnostics, and upload/timing estimates without changing GPU
  layouts or shader dispatch behavior.
- PathTraceSmokeDispatch owns constants, resource transitions, dispatch, and
  accumulation/readback use of the frame resources. It still runs the same
  smoke raygen path; this refactor did not split shader dispatches.
- Scene capture, material classification, texture discovery, texture registry,
  persistent material universe records, active material table construction,
  the static geometry universe scaffold, drawSurf mirror capture, instance
  universe diagnostics, skinning, GUI surface admission, emissive candidates,
  diagnostics, acceleration helpers, light selection, and geometry helpers are
  extracted into dedicated PathTrace* modules.
- The current split is still conservative. A future PathTraceSmokeState or
  PathTraceSmokeScene object may be useful, but that should be a deliberate
  architecture change rather than another mechanical extraction.

Refactor chronology:

1. The first phase proved the RT smoke path in one place, mostly inside
   PathTracePrimaryPass, so feature work could move quickly.
2. Geometry, skinning, GUI admission, material classification, texture registry,
   dynamic material state, emissive candidates, diagnostics, scene capture,
   light selection, smoke dispatch, readback, resources, acceleration helpers,
   CVars, and metadata registration were then extracted into PathTrace* modules.
3. The last broad extraction moved BuildRayTracingSmokeTestScene into
   PathTraceSmokeSceneBuild.cpp and left PathTracePrimaryPass as the pass shell.
4. docs/pathtrace_modules.txt was added after the extraction pass as the detailed
   file/feature map. This file remains the broader chronology and status guide.
5. The material universe then became the default producer for the active RT
   smoke material table, with optional legacy comparison/fallback validation.
   Its persistent records store stable material facts and universe indexes only;
   frame-local descriptor indexes and visible texture bindings stay in
   PathTraceDynamicMaterialState.
6. PathTraceGeometryUniverse was added as the first persistent-geometry
   ownership boundary. It currently owns the static-world cache vectors and
   stats while preserving the existing scene-capture/build behavior.
7. The next phase should be targeted architecture/performance work, especially
   persistent geometry records and GPU buffer ownership, not broad refactoring
   for line count.
8. Mode 18 was changed to behave more like a real PT integration point: the RT
   output owns the 3D scene layer, and the old 3D raster path is skipped by
   default while GUI/HUD/menu raster remains active. This removed the cost of
   rendering both paths at once and isolated a repeatable hitch to the skipped
   raster tail rather than the RT smoke build/dispatch path.
9. The generated-GUI frontEndGeo/layer-offset experiment in GuiModel.cpp was
   reverted. That experiment made Doom's painter-order GUI quads look like real
   scene geometry, which broke in-world monitor/video panels, caused black or
   overlapping screen layers, and triggered severe cinematic/GUI hitches in
   raster and PT routes. GuiModel.cpp should keep the original behavior:
   generated GUI drawSurfs have frontEndGeo = NULL.
10. FFmpeg/Bink decoder choice, upstream/master, Debug-vs-Release, and RETAIL
    were checked as possible explanations for the area-specific performance
    drop. Codec choice did not change the result. A clean latest upstream source
    bisect from v1.6.0 to babd04b3 tested clear; the final "first bad" commit
    was README-only and the direct upstream tip build was also clear. The
    decisive switch was Debug versus Release: fresh Release builds, including a
    generic non-retail Release control, fix the raster stall, while the legacy
    Debug executable reproduces the broken performance. Disabling
    r_useValidationLayers restored raster Debug to the 120 FPS cap, while PT
    remained much slower in Debug/RelWithDebInfo. The working theory is now two
    overlapping debug costs: NVRHI validation for raster, and debug
    runtime/codegen overhead in the CPU-heavy PT scene-build path. It is not
    upstream renderer regression, PT GPU cost, codec choice, or ID_RETAIL.
11. PathTraceDrawSurfCapture and PathTraceInstanceUniverse were added as the
    live-scene mirror for source3. The important split is mesh identity without
    transform versus instance identity with transform/material/entity metadata.
    PathTraceSceneUniverse remains static-map diagnostics/preload only; it is
    not the primary scene representation. Modes 21 and 22 were added as PT-native
    solid/wireframe bounds-box diagnostics after raster/debug-line and post-blit
    overlay attempts proved too fragile for useful spatial inspection.
12. PathTraceGeometryUniverse gained a rigid BLAS GPU scaffold for
    eligible source3 rigid mesh candidates. The scaffold builds local-space
    mesh input buffers, can optionally submit standalone BLAS builds, and was
    then extended into explicitly gated routed rigid TLAS integration for modes
    18/20/23/24/25.
13. Source3 static preload and rigid residency moved the BVH away from pure
    raster-visible drawSurf residency. Static-world cache population and cached
    routed rigid instances now use Doom renderer area/portal selection when the
    relevant gates are enabled. The key validation lesson is that current+1
    portal depth is too shallow for Mars City-style areas; source3 now uses
    depth 4 by default for static preload, rigid residency, and light-area
    diagnostics.
14. Source3, depth-4 static preload, rigid residency, rigid BLAS builds, rigid
    TLAS routing, dynamic rigid removal, and safe rigid emissive-card promotion
    became the default PT scene stack. The original drawSurf-only BVH is kept as
    `r_pathTracingSceneSource 0` for emergency comparison rather than as the
    default path.
15. Doom renderer lights gained a diagnostics-first identity bridge and then a
    first render-affecting analytic sphere-light path. `PathTraceDoomLights`
    maps renderer light indexes back to `idLight` game entities through
    `GetLightDefHandle`, records entity/material/color/state/area/portal-depth
    fields, and builds a separate GPU analytic-light buffer. The shader samples
    those candidates in mode 14/18/20-style lighting without adding sphere
    meshes to the BVH. `PathTraceDoomLights.cpp` owns the candidate radius
    policy through `r_pathTracingAnalyticSphereLightRadiusScale`,
    `r_pathTracingAnalyticSphereLightRadiusMin`, and
    `r_pathTracingAnalyticSphereLightRadiusMax`. The first render path was then
    changed from a hard center-point shadow ray to one-sample solid-angle sphere
    sampling, with the default intensity scale raised to 4.0 and
    `r_pathTracingAnalyticSphereLightRadiusScale` raised to 0.08 after visual
    testing. Initial in-game validation confirmed
    the candidates appear, obey RGB color, follow light animations/on-off game
    logic, and survive a level-load transition after adding game-state and
    render-world map-change guards.

Implemented debug CVars:

- r_pathTracing
  Enables the experimental RT path.

- r_pathTracingDebugWidth / r_pathTracingDebugHeight
  Set the RT debug output resolution. Tested at 320x180 and 1280x720. This is
  the PT internal render size; it is intentionally independent from the
  application/window back-buffer size. Changing one dimension first can produce
  a temporary odd aspect ratio, but aspect ratio itself is not expected to cause
  device removal. The dangerous transition is replacing the PT output,
  accumulation, readback, smoke reservoir, ReSTIR PT reservoir, or primary
  history resources while the previous frame's RT dispatch/blit/copy can still
  be in flight. `ResizeRayTracingSmokeOutput` delegates that destructive
  replacement to PathTraceFrameResources, which waits for the GPU to idle and
  runs NVRHI garbage collection before replacing existing output-sized PT
  resources. Keep that fence at the frame-resource owner; do not move it to the
  app/window resize path, because app resolution changes do not drive PT output
  size.

- r_pathTracingMode20TestPreset
  One-shot console helper that applies the common mode 20 source3 routed-rigid
  test stack, then resets itself to 0:

      1 = mode 20 + source3 + rigid route + dynamic rigid removal
      2 = preset 1 plus light-area diagnostics
      3 = preset 2 plus render-affecting light-area filter apply
      4 = preset 2 for BVH validation with light-area diagnostics but no
          render-affecting light-area filter apply

  The source3 routed stack is now already default-on. This helper remains a
  convenience/debug shortcut for switching to mode 20 and enabling extra
  diagnostics. All presets use portal depth 4, keep source3/routed rigid/static
  preload/rigid residency enabled, and ensure
  `r_pathTracingRigidRouteMaxInstances >= 256`. Presets 2, 3, and 4 also enable
  `r_pathTracingLightUniverseChurn 1`; preset 3 enables render-affecting
  light-area filter apply.

- r_pathTracingMode18TestPreset
  One-shot console helper that applies the same source3 routed-rigid/static
  preload/rigid residency stack to mode 18, then resets itself to 0:

      1 = mode 18 + source3 + rigid route + dynamic rigid removal
      4 = preset 1 plus portal depth 4 and rigid route max 256 for current
          BVH validation

  The source3 routed stack is now already default-on. This helper remains a
  convenience/debug shortcut for switching to mode 18 and ensuring depth-4
  source3/routed rigid/static preload/rigid residency state.
  Routed rigid hits still use the compact rigid route payload. Interactive
  condition-register emissive cards that need exact game-logic toggling should
  stay out of rigid emissive-card promotion until the route has a safe
  per-frame material-state channel.

- r_pathTracingClassDump
  One-shot dump of sampled surface classification reasons.

- r_pathTracingTextureProbeIndex
  Selects the material-table index to bind for debug texture probing.
  -1 means first safe texture.

- r_pathTracingTextureProbeReset
  Releases the latched texture probe and selects again.

- r_pathTracingTextureProbeDump
  One-shot dump of the currently latched texture probe and safe candidates.

- r_pathTracingTextureTableLimit / r_pathTracingTextureTableStart
  Selects a window of safe captured diffuse textures for debug mode 8. The
  active slot count currently supports up to 2048 descriptors for broader albedo
  coverage while texture sampling remains experimental.

- r_pathTracingTextureSampleMethod
  Selects the mode 8 texture fetch path:
  0 = disabled/fallback material color
  1 = SampleLevel diagnostic path
  2 = Texture.Load stable default

- r_pathTracingTextureFilter
  Selects filtering for Texture.Load sampling:
  0 = point Load
  1 = manual bilinear Load

- r_pathTracingTextureDecode
  Enables decode of Doom texture encodings in the RT texture path. Defaults to
  1. Currently this decodes diffuse YCoCg textures before albedo/lighting use.

- r_pathTracingForceTextureCodeUse
  Opt-in experiment for Doom texture filename usage codes. Defaults to 0. When
  set to 1, RT smoke material discovery treats four-digit filename codes near
  the end of image names as usage hints: 0300 = normal, 1000 = hard alpha/cutout,
  0100 = diffuse/YCoCg payload, 0200 = default RGBA, and 0000 = specular. These
  hints are used as tie-breakers before the older stage-lighting discovery
  paths, not as final material semantics. The code is generated from image usage
  and cube-file enums, so image->GetOpts().colorFormat and the material stage
  remain better truth sources when they are available.

- r_pathTracingUseNormalMaps
  Enables sampled normal maps in debug mode 14 selected point-light direct
  lighting. Defaults to 1. The normal map is decoded in the shader and applied
  through a per-hit tangent basis derived from triangle position/UV gradients.
  The decode follows the existing raster G-buffer convention: normal X is read
  from texture alpha, normal Y from green, and normal Z is reconstructed. This
  is an experimental bridge toward the later Doom material-to-PBR path;
  roughness/specular conversion is still separate future work.

- r_pathTracingNormalMapFlipGreen
  Diagnostic toggle for Doom 3 legacy normal-map convention checks. Defaults to
  0. When enabled, the RT smoke shader flips sampled normal-map green/Y before
  tangent-space reconstruction. Use this to A/B modes 26/27 after the ReSTIR PT
  initial-reservoir path showed valid reservoirs but rough normal-map shading.
  This is not a complete Doom normal fix: the current RT hit path still builds
  TBN from triangle UV gradients instead of the original idTech4 vertex
  tangent/sign basis used by legacy renderbump assets.

- r_pathTracingUseSpecularMaps
  Enables sampled specular maps in debug mode 14 selected point-light direct
  lighting and in mode 18 when `r_pathTracingToyFakePBRSpecular 1` is active.
  Defaults to 1. Mode 14 still uses a simple Blinn-style diagnostic highlight
  using Doom's specular color map. Mode 18 can opt into the rbdoom3 legacy
  specmap-to-PBR approximation described below.

- r_pathTracingUseEmissiveMaps
  Enables debug-only emissive/glow material contribution in mode 14. Defaults
  to 1. The RT smoke path looks for safe ambient/additive or glow/signage-like
  material stages, binds their texture when available, and adds the result as
  an unshadowed emissive term. This is a visibility/material-capture diagnostic,
  not light-source sampling yet. GUI/screen, particle, glass/window, animated,
  view-dependent stage textures, and skinned/deformed hits are rejected for now
  to avoid broad white false positives and game-logic-driven overlays.

- r_pathTracingToyMaxRayDistance
  Maximum secondary/direct ray length for mode 18 toy path tracing. Defaults to
  1024 Doom units. This helps reduce leaks from holes in the current
  visible-surface TLAS.

- r_pathTracingToyLightScale / r_pathTracingToyEmissiveScale
  Separate multipliers for mode 18 selected-light and emissive material
  contribution. Current useful defaults are 0.3 for lights and 4.0 for
  emissives.

- r_pathTracingToyLightTraceCap
  Caps the number of selected point lights that mode 18 traces for each direct
  lighting evaluation. Defaults to 8, max 32. This protects the toy PT path
  from brute-forcing too many shadow rays per pixel while ReSTIR PT work is
  still being staged. The cap is applied on the CPU by reducing the mode 18
  light-selection request count; the shader-facing LightInfo layout is
  unchanged.

- r_pathTracingToyFakePBRSpecular
  Opt-out shader experiment. Defaults to 1. When enabled, mode 18, ReSTIR PT
  bridge modes 26/27/28/29/30/31/32/33, and path-tracer visualizer modes
  34/35/36/37 also enable specular-map sampling and interpret Doom legacy
  specular maps using the same rbdoom3/Kennedith98-style fake-PBR idea as the
  raster interaction shaders: derive approximate F0 and roughness from the
  gamma-space specular texture. For ReSTIR PT this roughness and F0 are stored
  in the RAB material and primary-surface history records so temporal/spatial
  validation sees the converted Doom material state instead of treating every
  legacy surface as roughness 1.0. This is useful for a quick visual check of
  less-flat PT diffuse/specular response, but it is not a full Doom material
  conversion. It does not detect packed `_rmao` maps, does not add glossy bounce
  sampling, and does not solve alpha/blend/decal/special-stage classification.

- r_pathTracingToyAccumulation / r_pathTracingToyAccumMaxFrames
  Enables mode 18 temporal accumulation across stable camera frames and caps
  the accumulated frame count. Accumulation resets on camera/FOV/output-size
  changes and on relevant lighting, ambient, and toy-path CVar changes.

- r_pathTracingSamplesPerPixel
  Explicit mode 18 samples per pixel. Defaults to 1 and currently clamps to 1..4
  because the smoke raygen is still a monolithic validation dispatch. Values
  above 1 jitter additional primary rays per pixel and multiply secondary/direct
  ray work, so use `r_pathTracingIntegratorDump 1` before raising resolution and
  SPP together.

- r_pathTracingMaxPathDepth
  Maximum mode 18 path depth including the primary hit. Defaults to 2, which
  preserves the previous one-secondary-bounce toy behavior. Setting this to 1
  disables secondary diffuse/reflection bounces even if their lobe limits are
  nonzero.

- r_pathTracingDiffuseBounceLimit / r_pathTracingSpecularBounceLimit /
  r_pathTracingTransmissionBounceLimit
  Explicit per-lobe mode 18 bounce limits. Defaults are diffuse=1, specular=0,
  transmission=0, preserving the previous diffuse-only toy path. Transmission is
  reserved and currently fails closed; glass/liquid policy is not implemented in
  this task.

- r_pathTracingReflectionMode
  First opt-in reflection baseline for mode 18. Defaults to 0/off. Mode 1 traces
  one mirror-ish validation bounce only on low-roughness/high-F0 surfaces when
  `r_pathTracingSpecularBounceLimit > 0`; mode 2 relaxes the roughness gate for
  rougher validation. It is crude and denoiser-free by design.

Reflection/specular reservation:

- The current reflection path is only a validation bounce in mode 18. Future
  reflection work should have separate output/history names and carry roughness,
  F0/specular, geometric normal, shading normal, hit distance, reflection
  confidence, current/previous surface identity, and motion-vector or previous
  position input. Do not fold reflection history into diffuse ReSTIR history.
- Half-resolution or checkerboard reflection execution is a valid future policy,
  but it should be an explicit pass setting after pass splitting, not an
  implicit side effect of debug mode 18.

- r_pathTracingRussianRouletteDepth
  Reserved depth where Russian roulette may start. Defaults to 0, disabled. The
  current conservative shader path only reports the configured state in debug
  mode 37; it does not randomly terminate paths yet.

- r_pathTracingNextEventEstimation
  Enables the existing direct-light work from selected lights, emissive/Doom
  analytic light candidates, and the ReSTIR PT bridge path tracer. Defaults to
  1 to preserve current mode 18 behavior. Setting it to 0 leaves ambient and
  emissive terms but suppresses direct-light visibility work in the new
  integrator contract.

- r_pathTracingSecondaryNeeMode
  Selects secondary selected-light NEE behavior in mode 18 when
  `r_pathTracingNextEventEstimation` is enabled: 0 disables secondary
  selected-light NEE as a diagnostic throttle, 1 samples one uniformly selected
  point light and compensates by `selectedLightCount`, and 2 runs the legacy
  full selected-light loop. The default is 1. Doom analytic-light NEE remains
  controlled by the analytic-specific secondary gate below and is not folded
  into this selected-light sampler, but it now uses the same native
  `SmokeNeeLightSample` and `SmokeNeeResult` helper boundary inside
  `pathtrace_nee.hlsli`.

- r_pathTracingSecondaryNeeVisibility
  Diagnostic visibility gate for sampled secondary selected-light NEE. Defaults
  to 1, which traces the selected sample's shadow visibility ray. Set to 0 only
  to isolate sampled-light selection and weighting from shadow-ray visibility
  pressure; it shades selected samples as visible and is not a final lighting
  mode.

- r_pathTracingSecondaryAnalyticNeeMode
  Selects secondary/reflection Doom analytic-light NEE behavior in mode 18 when
  `r_pathTracingNextEventEstimation` and analytic-light candidates are enabled:
  0 disables secondary analytic NEE, 1 evaluates bounded uniformly sampled
  analytic-light proposals, and 2 runs the legacy full analytic-light loop. The
  default is 1 because testing with large analytic-light counts showed major
  performance gains without obvious image loss. Primary-hit analytic direct
  lighting still uses the legacy full loop for comparison.

- r_pathTracingSecondaryAnalyticNeeSamples
  Number of Doom analytic-light proposals evaluated per secondary/reflection
  NEE vertex when `r_pathTracingSecondaryAnalyticNeeMode` is 1. It clamps to
  0-8 and defaults to 4. This is a per-vertex proposal bound; the eligible
  analytic light set remains the uploaded Doom analytic-light candidate buffer
  after normal safety/upload caps.

- r_pathTracingIntegratorDump
  One-shot dump of the path tracer core settings and rough ray-budget estimate:
  output resolution, SPP, max depth, diffuse/specular/transmission limits,
  reflection mode, Russian-roulette depth, NEE state, secondary NEE mode and
  sampled visibility gate, selected/analytic light trial counts, estimated rays
  per pixel, and estimated dispatch rays. This is a safety signal for
  device-removal risk; it is intentionally approximate rather than a physical
  ray counter.

Ray dispatch safety diagnostics:

- r_pathTracingSafetyDump
  One-shot pre-dispatch safety dump. It reports output size, debug mode, SPP,
  depth/bounce/reflection/Russian-roulette/NEE settings, secondary NEE mode and
  sampled visibility gate, selected/analytic/emissive/light-candidate counts,
  material table entries, active descriptor table size, static/dynamic/
  rigid-route buffer capacities, TLAS/BLAS presence, primary-history byte
  count, smoke/ReSTIR reservoir byte counts, binding-set presence, and the
  current safety kill-mask.

- r_pathTracingDispatchTileEnable
  Diagnostic device-removal mitigation path. When set, the smoke/path tracing
  raygen uses the same TLAS, bindings, clears, and command list, but splits
  `DispatchRays` into sequential screen-space rectangles. Default is 0, so the
  normal full-screen dispatch path is unchanged.

- r_pathTracingDispatchTileWidth / r_pathTracingDispatchTileHeight
  Tiled-dispatch rectangle size in pixels. The last row/column dispatches a
  smaller partial tile when the output size is not evenly divisible. Shader-side
  output, primary-history, reservoir, reprojection, and random-seed coordinates
  use absolute full-frame pixels rather than tile-local coordinates.

- r_pathTracingDispatchTileDump
  One-shot tiled-dispatch plan dump. It reports output size, tile size, tile
  count, estimated rays per pixel, estimated rays per tile, and estimated rays
  for the full frame.

- r_pathTracingDisableAnyHitAlpha
  Diagnostic kill switch for any-hit alpha/translucent/glass/particle rejection.
  Shadow self-ignore still runs; only the alpha/translucent rejection path is
  bypassed.

- r_pathTracingDisableSelectedLightLoop
  Diagnostic kill switch for legacy selected point-light loops and selected-light
  sprite proxies.

- r_pathTracingDisableAnalyticLightLoop
  Diagnostic kill switch for Doom analytic light loops and ReSTIR/RAB analytic
  light sampling. This is independent from the upload/build CVars.

- r_pathTracingDisableEmissiveTriangleSampling
  Diagnostic kill switch for emissive triangle sampling in mode 20 and ReSTIR PT
  validation paths.

- r_pathTracingDisableDiffuseSecondaryRay
  Diagnostic kill switch for diffuse secondary/continuation rays in mode 18 and
  the RAB path-tracer bridge.

- r_pathTracingDisableReflectionRay
  Diagnostic kill switch for mode 18 reflection/specular secondary rays.

- r_pathTracingDisablePrimarySurfaceHistory
  Diagnostic kill switch for primary-surface history writes, clears, copies, and
  previous-camera history validity. Re-enabling it forces the history path to
  clear before it becomes valid again.

- r_pathTracingDisableReservoirWrites
  Diagnostic kill switch for shader-side smoke and ReSTIR PT reservoir writes.
  Re-enabling leaves the smoke reservoir clear request pending.

- r_pathTracingDisableRestirVisibilityRay
  Diagnostic kill switch for the ReSTIR PT preview visibility ray even when a
  mode or `r_pathTracingRestirPTPreviewVisibility` requests it.

Glass/transmission reservation:

- `r_pathTracingTransmissionBounceLimit` is reserved and currently clamps/fails
  closed. Do not treat the existing portal/window/object-glass visibility
  approximations as final glass transport.
- Future glass work needs explicit opacity, alpha cutoff, default IOR, tint or
  transmission color, thickness approximation, nested-surface policy, and Doom
  window/liquid class flags. Ask before finalizing Doom-specific glass or liquid
  behavior.

Denoise/upscale/Ray Reconstruction reservation:

- Required future pass outputs/inputs are color, albedo, normal, roughness,
  depth, motion vectors, hit distance, diffuse/specular separation, exposure,
  and history reset reasons. Primary-surface history already reserves several
  material/motion fields, but camera-only reprojection is not enough for DLSS/RR.
- Do not integrate DLSS/RR or a final denoiser against camera-only motion. The
  geometry universe must provide current/previous transforms or vertex data for
  surfaces whose history is reused.
- Tiled ray dispatch remains a future stability lever for long GPU dispatches,
  but it is intentionally not part of this reservation task.

- r_pathTracingEmissiveInventoryDump
  One-shot dump of the current emissive triangle inventory: total/captured
  static/dynamic/routed-rigid candidate counts, unique material count, area,
  approximate luminance, candidate weight, estimated radiance, per-material
  summary rows, emissive texture slot/dimensions, and a short triangle sample

- r_pathTracingRigidRouteEmissiveDump
  One-shot dump focused on routed rigid emissive append diagnostics. It uses
  the emissive inventory dump path and adds route instance count, seen/cache
  split, routed rigid emissive instance split, triangle considered/captured/cap
  counts, invalid/non-emissive counts, area, and weighted luminance.

- r_pathTracingEmissiveInventoryMaxTriangles
  Caps the CPU/GPU emissive triangle records captured for future light-sampling
  infrastructure. The default is 4096. This also caps the merged persistent
  static + current dynamic light-candidate list. This is intentionally an
  inventory for later ReSTIR/PT integration, not a hand-rolled area-light
  sampler.

- r_pathTracingLightUniverseDump
  One-shot dump of persistent emissive light-universe stats: persistent static
  triangle count, static seen/new/updated/missing this frame, semi-static
  dynamic count, dynamic seen/promoted/updated/missing/aged this frame, current
  frame-local dynamic triangle count, merged upload count, area-membership
  diagnostics, and generation. Static emissives are retained by stable-ish
  identity across frames. The area diagnostics report current area, total areas,
  known/unknown static and dynamic candidate area membership, and how many
  merged candidates are in the current, selected, connected, disconnected, or
  unknown area sets. Selected areas are the current camera area plus open
  portals up to `r_pathTracingScenePortalSteps`. This is diagnostic only; it
  does not filter mode 20 candidates yet.

- r_pathTracingLightUniverseChurn
  Defaults to 0. When enabled, tracks frame-to-frame changes in the uploaded
  RT smoke emissive light set. `r_pathTracingLightUniverseDump 1` then reports
  previous/current/stayed/entered/left counts, matching weight totals, and top
  entered/left samples by weight. This is for mode 20 portal-edge stability
  diagnostics; leave it enabled while moving before taking a dump so the
  previous active set is meaningful.

- r_pathTracingLightUniversePersistDynamic
  Defaults to 1. Allows stable rigid-entity emissives to be promoted into
  semi-static light-universe persistence. Skinned/deformed, particle, and
  other non-rigid dynamic emissives remain frame-local.

- r_pathTracingLightUniverseInjectMissingDynamic
  Defaults to 0. When enabled, promoted semi-static dynamic emissives that are
  missing from the current capture are injected back into the light list. This
  remains off by default because the matching dynamic occluder/object geometry
  is not persistent yet; injecting only the light record can create inaccurate
  ghost lighting.

- r_pathTracingLightUniverseDynamicMinSeenFrames
  Defaults to 2. Number of frames a rigid-entity emissive must be seen before
  it is promoted into the semi-static light universe.

- r_pathTracingLightUniverseDynamicMaxMissingFrames
  Defaults to 90. Number of frames a promoted semi-static dynamic emissive may
  be missing from the current capture before it ages out.

- r_pathTracingLightAreaPortalSteps
  Defaults to 4. Portal traversal depth for RT smoke emissive light-area
  selection diagnostics. This is separate from the scene-universe diagnostic
  portal depth so mode 20 light selection can be tuned independently.
  `PS_BLOCK_VIEW` edges are counted for diagnostics but do not block traversal,
  matching source3 static preload and rigid residency.

- r_pathTracingLightAreaFilter
  Defaults to 0. Diagnostic-only light-area selector gate. When enabled, the
  dump reports what would be selected by area plus overflow rules, but uploaded
  mode 20 candidate buffers remain unchanged.

- r_pathTracingLightAreaFilterApply
  Defaults to 0. Experimental render-affecting gate for the light-area selector.
  When enabled, mode 20 uploads selected-area emissive candidates plus the
  highest-weight connected overflow candidates up to
  `r_pathTracingLightAreaOverflowMax`. Disconnected and unknown-area candidates
  are dropped. Keep this off unless testing area-filter impact.

- r_pathTracingLightAreaOverflowMax
  Defaults to 512. Connected-area overflow budget for light-area selection. The
  high default is intentionally greedy for Doom 3 content: keep all selected
  portal-area candidates plus most or all connected overflow candidates, and
  mainly drop disconnected/unknown-area candidates. When
  `r_pathTracingLightAreaFilterApply 1` is enabled, this budget is applied to
  the uploaded mode 20 candidate list.

- r_pathTracingReservoirDump
  One-shot dump of current ReSTIR/PT reservoir ownership state: screen-sized
  record count/bytes, scene and dispatch signatures, reset/clear counters,
  emissive triangle counts, and light-candidate counts. Reservoir history
  currently resets on output size, camera/FOV, material table,
  emissive-candidate, light-candidate, or static-BLAS signature changes. The
  current reservoir is written by mode 20; previous and spatial scratch
  reservoirs are still reset/cleared ownership scaffolding.

- r_pathTracingSceneInputsDump
  One-shot dump of the committed RtPathTraceSceneInputs package after the next
  successful scene build. It reports scene source, debug mode, output size,
  geometry/material/light signatures, geometry counts and history placeholders,
  material table path and active texture count, emissive/Doom analytic light
  counts, skinned previous-bridge and GPU-skinning scaffold counts, source3
  view/current area plus portal step/area/edge facts, and CPU upload/timing
  estimates. This is a diagnostics contract only; it does not change dispatch
  routing or shader-visible buffer layouts.

- r_pathTracingGpuSkinning
  Off-by-default PT GPU skinning scaffold gate. `0` keeps only the existing
  CPU-skinned bridge. `1` creates diagnostic skinned source/current-output/
  previous-position/dispatch/current-joint/previous-joint buffers from the CPU
  bridge and exposes their counts in `r_pathTracingSceneInputsDump`. `2` is
  reserved for a later validated switch to GPU output. No current mode consumes
  the scaffold buffers for rendering yet.

- r_pathTracingPortalTransitionDump
  Diagnostic logging for suspected PT scene-resource lifetime hazards. When a
  committed scene transition changes source, portal area/set counts, selected
  geometry/material/light counts, scene signatures, or core resource handles,
  this prints old/new portal facts, geometry/material/light counts, upload byte
  estimates, descriptor-table create/write flags, binding-set changes, buffer
  handle changes, and static/dynamic BLAS handle changes. It is intended for
  portal-threshold corruption/device-removal diagnosis and does not change
  rendering behavior. Values are:
  `0` off, `1` print the next portal-area transition and auto-disable,
  `2` print every portal-area transition, `3` print every broad scene/resource
  transition. Use `1` for normal in-game testing; `3` is expected to be noisy.

- r_pathTracingWaitForIdleOnPortalChange
  Diagnostic synchronization gate. When nonzero and the same PT scene
  transition signature changes, `PathTraceSmokeResources` waits for the NVRHI
  device to idle immediately before replacing the committed PT scene resources.
  This was used to confirm the portal-threshold texture corruption/device
  removal failure as an in-flight scene-resource lifetime hazard. The normal
  runtime fix is `r_pathTracingSceneRetireFrames`, not this full-device idle.
  This path is silent unless transition dumping is also enabled. Keep it
  disabled for normal testing and performance work.

- r_pathTracingSceneRetireFrames
  Frame-lagged lifetime for replaced committed PT scene packages. The default
  is `6`, clamped to `0..32`. Nonzero values keep old scene buffers,
  static/dynamic BLAS handles, the TLAS handle, binding set, descriptor table,
  and active texture handles alive after replacement so portal transitions do
  not need a normal `waitForIdle()`. Setting this to `0` restores immediate
  replacement for A/B testing. This is the confirmed fix for the high-GPU-load
  portal-transition corruption/device-removal bug: heavy ray workloads only
  made the hazard easier to hit by keeping older frames in flight longer.

- r_pathTracingSceneRetireDump
  Retirement diagnostics: `0` off, `1` print the next retire/release event and
  auto-disable, `2` print every event. Logs include current frame, retire frame,
  queue size, released count, old/new area facts, transition flags, diagnostic
  wait usage, and which old package handle groups were retained.

- r_pathTracingDebugMode 20
  Single-frame direct lighting debug view from the emissive candidate contract.
  It samples one emissive triangle candidate per pixel, writes the current
  reservoir buffer, traces a visibility ray, and shades the hit with that
  sample. This is intentionally noisy and does not perform temporal reuse,
  spatial reuse, MIS, or final light-sampler weighting yet. Static emissive
  candidates now come from a persistent light universe. Rigid-entity emissives
  can be promoted to semi-static persistence; other dynamic emissives remain
  current-frame only. The result can still have screen-space-like gaps until an
  area has been seen and promoted.

  Resolved stability note: the earlier transient texture corruption followed by
  device removal was not ultimately fixed by tiling, texture-sampling changes,
  NEE extraction, or lower light counts. The decisive diagnostic was
  `r_pathTracingWaitForIdleOnPortalChange 1`, which removed the failure by
  forcing a full sync before committed PT scene resources were replaced during
  portal/scene transitions. The production fix is frame-lagged scene-resource
  retirement with `r_pathTracingSceneRetireFrames 6`, which keeps the old scene
  package alive for later GPU work while allowing
  `r_pathTracingWaitForIdleOnPortalChange` to remain disabled. High ray
  workload, high internal resolution, and FPS below the frame cap made the bug
  easier to trigger because more work remained in flight across portal
  transitions. `r_pathTracingToyLightTraceCap` is still useful to keep the toy
  mode manageable, but it should not be treated as the root-cause fix for this
  device-removal class.

  Useful first test:

      r_pathTracingTextureTableLimit 2048
      r_pathTracingDebugMode 20
      r_pathTracingReservoirTwoSidedEmissives 1
      r_pathTracingLightUniversePersistDynamic 1
      r_pathTracingLightUniverseInjectMissingDynamic 0
      r_pathTracingLightUniverseDynamicMinSeenFrames 2
      r_pathTracingLightUniverseDynamicMaxMissingFrames 90
      r_pathTracingLightUniverseDump 1
      r_pathTracingReservoirDump 1

- r_pathTracingSceneSource2RigidEntities

Runtime emissive state:

- Stable material metadata means "this material can be emissive"; it is not
  enough to decide whether a particular surface is emitting this frame. Doom
  game/render logic can drive material stage condition/color registers per draw
  surface, including switchable fluorescent lights that dim as they turn off.
- PathTraceSceneCapture stamps captured triangles with the evaluated
  draw-surface emissive state from drawSurf->shaderRegisters. The emissive
  inventory and modes 18/19/20 honor that per-surface state, so switchable
  light surfaces no longer glow just because their material has an emissive
  stage in the declaration.
- Routed rigid emissive-card promotion is only safe for stable object-attached
  signage/glow cards. The known `swinglight` material family is temporarily
  excluded because it is game-logic/register driven and must remain on the
  dynamic drawSurf path until routed rigid records have a safe per-frame
  material-state channel. Do not use material name exclusions as the final
  model; the intended replacement is a detector for stage condition/color
  dependencies on shader parms, globals, time, sound, or other dynamic
  expression inputs.
- The `_add + 0200` classifier exclusion is separate from runtime material
  state. It prevents generated/default-RGBA additive variants from being treated
  as real translucent/emissive signage candidates.
- This is an important ReSTIR/PT contract: Material universe owns stable
  capabilities/facts, while the per-frame capture path owns runtime facts such
  as active emissive, dimming, and on/off state.
- See docs/pathtrace_interactive_surfaces.md for the deeper material-stage,
  shader parm, and evaluated register model behind switchable lights and
  red/green door panels.

Level-load safety:

- The Doom analytic light bridge only scans `gameLocal.entities` while
  `gameLocal.GameState()` is `GAMESTATE_ACTIVE`. Map shutdown/startup frames
  can otherwise expose stale entity pointers while the renderer is still
  drawing loading or transition frames.
- `PathTraceSmokeSceneBuild` treats render-world map name/timestamp changes as
  a scene-cache reset boundary even when Doom reuses the same render-world
  object. The reset clears PT scene buffers, BLAS handles, geometry/light
  universes, accumulation, and binding state, but deliberately preserves
  output-sized resources such as reservoir buffers and descriptor tables. A
  full scene-resource reset at this point caused a black screen with "failed to
  create RT smoke binding set" because the reservoir buffers had just been
  cleared after output resize.

PT output resize stability:

- A repeatable hard crash/device-removal during `r_pathTracingDebugWidth` /
  `r_pathTracingDebugHeight` changes was traced to PT internal output-size
  resource lifetime, not to application-window resizing and not to the transient
  aspect ratio created when width and height are changed one CVar at a time.
  The crash happened after the console printed the new PT output resolution,
  then the driver/device was removed.
- Earlier attempts to fix app back-buffer resize were misleading because the PT
  output is not tied to the native back buffer. Gating the debug blit, skipping
  PT/raster 3D for native resize frames, and invalidating the cached LDR
  framebuffer were still reasonable presentation safeguards, but they did not
  address the failing path.
- The actual resize safety policy now lives in PathTraceFrameResources and is
  called by `PathTraceSmokeResources::ResizeRayTracingSmokeOutput`: before
  replacing any existing PT output, accumulation, readback, smoke reservoir,
  ReSTIR PT reservoir, or primary-surface history handle, the code calls
  `waitForIdle()` and `runGarbageCollection()`. This prevents old output-sized
  resources from being released while the previous frame's ray dispatch, output
  blit, readback copy, or descriptor use is still pending on the GPU. The
  frame-resource diagnostics report the wait reason, reset reasons, output
  resolution, resource byte counts, descriptor/AS commits, and readback queue
  activity through `r_pathTracingReservoirDump`.
- Future PT output-resolution work should continue to treat width/height as an
  internal render-scale pair. It is acceptable for users to enter width and
  height separately; the intermediate frame may look stretched, but it must not
  replace GPU resources without an idle/deferred-destruction boundary.

Geometry and BVH Work Completed
-------------------------------

The pass now builds real ray tracing acceleration structures from visible Doom
draw surfaces:

- Captures visible draw surfaces from the current view.
- Builds BLAS/TLAS using NVRHI ray tracing APIs.
- Rebuilds dynamic-candidate geometry each frame.
- Maintains a persistent static-world geometry cache.
- Owns that cache through PathTraceGeometryUniverse, currently as a behavior-
  preserving scaffold around the existing static surface keys, vertices,
  indexes, triangle classes, and triangle material IDs.
- Rebuilds the static-world BLAS only when newly seen static surfaces are added.
- Rebuilds the TLAS every frame.
- Splits static-world and dynamic-candidate geometry into separate buffers.
- Instances static and dynamic BLASes in one TLAS.
- Closest-hit shader fetches from the correct static or dynamic buffers based
  on InstanceID.

The static-world cache is populated independently of the center-ray anchor,
using stable surface keys. This avoids obvious instability from Doom draw-surface
ordering changes.


Surface Classification
----------------------

Captured surfaces are classified into debug buckets:

- static-world
- rigid-entity
- skinned/deformed
- particle/alpha
- unknown

This is used for logging, debug visualization, and the static/dynamic BLAS split.
Entity-owned static models with static caches and no joints are treated as
rigid-entity rather than fully dynamic.

The classifier has been exercised in game. It broadly behaves as expected:

- World geometry is usually green/static.
- Doors, crates, and entity-owned rigid objects are dynamic/rigid.
- Skinned characters are correctly identified.
- Smoke cards and translucent particle-like surfaces are particle/alpha.

Some edge cases remain, such as occasional static-looking pieces classified as
rigid/dynamic. This is acceptable for now because correctness is favored over
over-aggressive static caching.

Translucent surfaces now have a second debug-only subtype classifier used by
mode 12:

- decal/grime
- object glass
- smoke/particle
- signage/glow
- portal/window
- GUI/screen
- unknown

This classifier is intentionally diagnostic. It helped identify Doom 3's heavy
use of translucent cards for dirt, grime, logos, warning labels, windows, smoke,
and light/glow surfaces. It should not be treated as final material semantics.
Some categories, especially glow/signage and object glass, need a more complete
material and lighting model before they can be handled correctly.


Skinned Geometry
----------------

Initially, skinned characters appeared in T-pose because the RT capture was
using base/static mesh data while the raster renderer was using GPU skinning.

Current behavior:

- With r_useGPUSkinning enabled, RT smoke now has an RT-only CPU skinning path
  for MD5/skinned surfaces.
- It uses current joint data and the weighted joint transform logic from the
  MD5 model path.
- Scene capture now records explicit skinned/deforming surface keys and dynamic
  ranges, matches current records to the previous frame by key, retains a
  skinned-only CPU copy of the previous RT CPU-skinned vertex output, and reports
  conservative match/invalid reason counts through `r_pathTracingSceneInputsDump`.
- With `r_pathTracingGpuSkinning 1`, the retained previous copy is also uploaded
  as a shader-visible previous-position buffer plus a dispatch/range map. The
  closest-hit output is unchanged, but primary-surface packing can mark validated
  skinned hits with `RT_PRIMARY_SURFACE_HAS_OBJECT_MOTION` and
  `RT_PRIMARY_SURFACE_HAS_PREVIOUS_POSITION` and store the previous world-space
  hit position in `previousPositionOrMotion.xyz`.
- `r_pathTracingGpuSkinning` defaults to `0`. Value `1` builds diagnostic
  source/output/previous-position/dispatch/current-joint/previous-joint buffers
  from those same records so the future GPU-skinning contract can be validated
  without changing rendering. Compute dispatch remains inactive in this
  scaffold, and `skinnedGpuSkinningAvailable` stays false.
- This does not change the main raster r_useGPUSkinning setting.
- Characters now animate correctly in the RT debug output.

This path is not optimized. It is a correctness bridge so the RT path can trace
posed skinned geometry while the renderer continues using GPU skinning normally.

Material Debug Work
-------------------

The RT smoke path now tracks material identity per triangle:

- Each captured surface gets a stable material ID from the idMaterial name.
- Per-triangle material IDs are stored for static and dynamic geometry.
- Debug mode 6 colors hits by material ID.
- A compact material table exists for captured materials.
- Per-triangle material table indexes are stored.
- Debug mode 7 shades from the material table debug color.

This provides the CPU/GPU plumbing needed for real material data, without yet
implementing the full Doom material system in RT.


Texture Probe Work
------------------

Debug mode 8 is an early sampled diffuse texture probe.

The current implementation:

- Discovers plausible diffuse/base-color images from captured materials.
- Avoids calling idMaterial::GetEditorImage() during RT discovery.
- Builds an NVRHI bindless Texture_SRV descriptor table.
- Reuses the descriptor table instead of recreating it every scene build.
- Writes a conservative active window of safe Doom diffuse textures.
- Stores descriptor indexes in the RT material table for captured materials.
- Fetches bindless texture data in the ray generation shader.
- Falls back to material debug colors when a material has no bound descriptor.
- Keeps texture dump logging for controlled testing.

Important history:

- Fixed Texture2D slots were tried first and caused device-removed crashes.
- A private GPU texture copy path was tried and hit Doom/NVRHI state assertions.
- Switching to bindless descriptor tables compiled and ran.
- The remaining repeatable crash was traced to texture discovery calling
  GetEditorImage(), which can load missing editor images on demand.
- Removing the editor-image fallback eliminated the repeatable crash in the
  tested map location.
- A later device-removed crash near animated SWF/GUI terminals was narrowed to
  SampleLevel from the RT shader. It reproduced even when sampling the fallback
  white texture and with bindless disabled.
- Texture.Load is stable in the same animated terminal cases so far. It is now
  the default mode 8 fetch path. SampleLevel remains available as a diagnostic.
- Texture.Load now has manual bilinear filtering behind
  r_pathTracingTextureFilter. This improves the debug texture probe without
  using sampler-state based SampleLevel in the ray tracing shader.
- Diffuse textures using Doom's YCoCg DXT5 encoding are now tagged in the RT
  material payload and decoded in the shader when r_pathTracingTextureDecode is
  enabled. Without this, TD_DIFFUSE images appear with incorrect green/orange
  colors and exaggerated block compression artifacts.

Mode 8 is still only a probe. Texture.Load does not yet implement mip selection
or full Doom material behavior.

Doom BFG generated .bimage names append a storage hint as #__%02d%02d, where
the first two digits are textureUsage_t and the second two are cubeFiles_t.
Observed 2D material texture codes are:

- #__0000 = TD_SPECULAR / CFM_DEFAULT. Specular/control texture. Its alpha is
  not material opacity.
- #__0100 = TD_DIFFUSE / CFM_YCOCG_DXT5. This is a CoCg_Y diffuse payload:
  sampled alpha stores Y/luminance for color reconstruction, not transparency.
  Never use this alpha as material alpha in the general case.
- #__0200 = TD_DEFAULT / CFM_DEFAULT. This is ordinary RGBA/default storage. If
  an alpha channel is present and the material stage/blend uses it, it should be
  treated as real alpha.
- #__0300 = TD_BUMP / CFM_NORMAL_DXT5. Normal-map payload. The raster convention
  reads normal X from alpha, normal Y from green, and reconstructs Z.
- #__1000 = TD_COVERAGE / CFM_GREEN_ALPHA. Coverage/cutout data for clipping or
  stencil-style visibility.

The safest RT material rule is: first trust image->GetOpts().colorFormat, then
the material stage lighting/blend/alpha-test state, and use filenames or paths
only as tie-breakers. In particular, CFM_YCOCG_DXT5 means alpha is occupied by
color reconstruction, CFM_DEFAULT means alpha can be real material data, and
CFM_GREEN_ALPHA/TD_COVERAGE is the dedicated coverage path.

Classic Doom 3 decal intent also survives strongly in paths and material
stages. Textures under /decals/ that are #__0100/CFM_YCOCG_DXT5 generally do
not have real alpha; they are blend decals using decoded RGB as the effect.
Lighten/additive decals tend to use black as the no-effect key, while
darken/filter decals tend to use white as the no-effect key. Their visibility
should be inferred from decoded RGB and blend state, not from the YCoCg alpha
channel. #__0200 decal/default textures with alpha are different: because they
are CFM_DEFAULT RGBA, their alpha can be real if the stage uses it.

Exception: `_add...#__0200` / `_add...0200` generated texture variants are
explicitly banned from the translucent/signage/glow classifier path for the PT
prototype. The classifier records this as `hasAddDefault0200Texture`; overlay,
emissive, alpha, subtype, and rigid emissive-card promotion consumers reject
that flag so broad `_add` name matching does not turn default-RGBA add variants
into translucent/routed emissive cards.

The active texture table was expanded to 2048 descriptors after diagnostics
showed the earlier conservative limits left many valid captured materials on
fallback debug colors. With Texture.Load as the default fetch path, larger
descriptor windows cover longer traversal tests without the earlier SWF/GUI
terminal crash pattern.


Alpha, Albedo, and Lighting Debug Modes
---------------------------------------

Alpha-test exploration now exists in debug-only form:

- Material discovery looks for dedicated alpha/coverage images on perforated
  materials when available.
- Translucent glass/signage/glow-like cards can also opt into alpha any-hit
  when the material exposes an explicit alpha-test stage, which catches cutout
  cards such as glasses lenses without treating all blended translucency as
  clipped geometry.
- Ambient blended decal/grime materials can provide their safe ambient image as
  RT albedo when no diffuse stage exists. The any-hit path also masks their
  blend-mode no-effect texels: white-keyed filter decals drop near-white texels,
  while black-keyed filter decals drop near-black texels. Retained filter decal
  texels are stochastically thinned as a temporary overlay approximation so
  door/wall surfaces can still show through in the one-hit debug path.
- Packed YCoCg perforated materials that reuse the diffuse image as their
  coverage stage can derive the RT alpha test from decoded diffuse luminance
  instead of the raw alpha channel. Reflective eyewear cutouts such as glasses
  and goggles use an inverted dark-key variant of that coverage test, plus a
  dark neutral placeholder albedo while keeping their normal/specular maps,
  since their original look depends on reflection/cube-map behavior that the
  debug path does not implement yet.
- Mode 9 visualizes alpha pass/fail using stable Texture.Load fetches.
- An any-hit shader is wired for alpha-tested materials and calls IgnoreHit()
  when sampled alpha is below the material cutoff.
- Opaque materials stay on the normal closest-hit path.

This has worked well for fan grates and perforated floors. Some Doom character
materials expose dissolve/blood-style masks; these were initially over-applied,
then narrowed so normal character albedo is usable again.

Albedo and simple lighting exploration:

- Mode 10 shades hits with sampled diffuse/albedo textures and falls back to
  material debug colors when no safe texture descriptor exists.
- Mode 11 overlays translucent/decal-like surfaces on top of the albedo view for
  inspection.
- Mode 12 colors translucent subtype classifications.
- Mode 13 applies a simple fixed-direction Lambert light using sampled albedo
  and the interpolated/geometric RT normal.
- Mode 14 extends mode 13 with debug-only shadow rays from the primary hit
  toward up to 32 selected visible Doom point lights. The default selector
  ranks lights by estimated camera influence from evaluated light color,
  r_lightScale, and radius falloff; r_pathTracingLightSelection 0 restores the
  original nearest-camera ranking. Mode 14 still uses simple radius-based debug
  attenuation and falls back to the old fixed directional light if no point
  light is available. The shadow rays preserve the existing alpha any-hit path
  so cutout blockers can be ignored when their alpha tests reject the hit.
  Mode 14 can apply sampled normal maps when r_pathTracingUseNormalMaps is 1.
  The normal decode intentionally matches the raster G-buffer shader: X comes
  from the texture alpha channel, Y from green, and Z is reconstructed. A
  previous direct .xyz normal decode made most surfaces face away from lights
  and killed the selected-light response. Mode 14 can also add a simple
  Blinn-style direct specular highlight from the sampled Doom specular map when
  r_pathTracingUseSpecularMaps is 1, plus an unshadowed emissive/glow preview
  from safe ambient/additive material stages when r_pathTracingUseEmissiveMaps
  is 1. This is diagnostic shading, not final PBR or light-source sampling.
- Mode 15 visualizes the dominant selected light at each primary hit. It uses
  the same selected-light payload and alpha-aware shadow rays as mode 14, but
  shades by light slot instead of final albedo lighting so selection and
  occlusion can be inspected directly.

Particle/smoke placeholder handling:

- Smoke/particle translucent subtype surfaces can use stable alpha dithering
  when r_pathTracingSmokeParticleDither is 1. Shadow rays ignore these smoke
  particles so broad cards do not erase direct-light shadows.
- r_pathTracingSmokeParticleAlphaScale scales the dither survival probability.
  r_pathTracingSmokeParticleEdgeFade applies a simple UV edge fade to reduce the
  visible rectangle silhouette. This remains a placeholder hack; it can still
  look noisy and card-like.
- Portal/window translucent subtype surfaces can use a similar stable
  stochastic transparency placeholder when r_pathTracingPortalWindowStochastic
  is 1. This targets large level windows only; small object glass remains on the
  existing path until a real glass material exists. Portal/window materials with
  no safe diffuse texture use their constant ambient/diffuse stage color as a
  temporary fallback tint instead of the material hash debug color, and fall
  back to a neutral blue-gray tint if the stage color is not constant. Plain
  portal/window fallback materials are kept out of the decal overlay thinning
  path and receive a small minimum stochastic opacity so indoor portal glass
  remains visible even when it has no grime/albedo mask.
- Object-glass translucent subtype materials, such as small props and vending
  machine glass doors, use the same stochastic glass placeholder at a lower
  minimum opacity. This is still a debug approximation for reflect-cube glass,
  not a physically correct glass/reflection implementation.
- Selected animated light shaders can draw a debug-only screen/ray-space sprite
  proxy in mode 14. This is meant to restore visibility for strobe/flare-like
  light visuals whose Doom material is a light shader rather than a captured
  surface. It uses the evaluated light shader color, so table/Parm-driven
  pulsing is reflected, but it is not sampled as a real path-traced emitter.

GUI/SWF panel placeholder findings:

- In-world Doom GUI panels are not ordinary textured screen meshes. The raster
  path calls the GUI/SWF system, emits many 2D painter-order draw commands into
  a synthetic GUI view entity, and then rasterizes those commands onto or around
  the panel. Generated GUI drawSurfs normally have no frontEndGeo, and that is
  important: the generated painter-order rectangles are not the physical monitor
  mesh.
- The generated GUI surfaces do not belong to a normal render model. Validation
  therefore must exempt GUI draw surfaces from the usual entity hModel check;
  otherwise r_pathTracingGuiDump reports many GUI surfaces with captured=0 and
  no material metadata.
- Directly tracing the generated GUI quads is only a proof-of-life placeholder.
  The UI content becomes visible, including animated/SWF content and individual
  text/image layers, but ray tracing interprets Doom's 2D painter-order
  rectangles as real 3D occluders. Layer offsets can make one panel look better
  while breaking another: background layers may flip to the front, large
  rectangles can hide nearby world geometry or monitor shells, and round
  screens can need additional alpha clipping. Some circular panels happened to
  look better in one offset configuration while rectangular terminals became
  worse.
- The frame-local generated-GUI frontEndGeo/layer-offset experiment has been
  removed. GuiModel.cpp is back to its original behavior, with generated GUI
  draw surfaces not exposing synthetic frontEndGeo to PT capture.
- This was not cosmetic. The bad GuiModel.cpp experiment affected the shared
  raster path too: in-world video/GUI screens could become black or show only
  leaking edges, layer panels could overlap the monitor shell, and Bink/GUI
  update work could explode into multi-hundred-millisecond hitches. Reverting
  GuiModel.cpp fixed the visible screen breakage and removed the repeatable
  hitch route. Do not reintroduce generated-GUI frontEndGeo as a normal path.
- The likely correct long-term design is to keep Doom's GUI/SWF raster path for
  panel contents: rasterize each live panel offscreen into a texture, identify
  the actual physical screen/panel surface, keep the generated painter-order GUI
  quads out of the TLAS, and have the RT material sample the composited panel
  texture on the physical surface. Interaction/raycasting should stay with the
  original GUI system. This preserves Doom's animation, text, alpha, clipping,
  and painter order while RT only sees a single physical screen material.

Known limitations:

- Normal maps can still look wrong on some props. Doom 3 legacy normal maps use
  a green/Y convention that can differ from modern defaults, now testable with
  `r_pathTracingNormalMapFlipGreen`, and original assets were baked against
  idTech4's pre-MikkTSpace tangent basis. The RT debug path still recomputes
  tangent space in shader and normalizes handedness, which can erase the engine
  vertex tangent/sign convention used by legacy renderbump assets. A proper fix
  should preserve/import the engine tangent basis or add a narrowly scoped
  legacy-vs-modern material path, rather than hard-flipping all normals globally.
- Custom-shader deferred callback render entities are skipped by default in RT
  smoke capture. This avoids a confirmed pickup/delete crash on an item disk
  without hiding normal animated/skinned entities that use model callbacks.

Material table performance:

- The material side is now mostly through the material universe for easy
  single-threaded wins: persistent records/facts, universe-derived active table,
  cached active-table builds, and stable material/universe indexes are in place.
- RT smoke caches the universe-derived active material table by default with
  r_pathTracingMaterialCache. The cached-table hit path keeps static materials
  first and rebuilds per-triangle material indexes from stable material IDs so
  cached static triangle material-index buffers remain valid across dynamic
  material changes.
- Per-material texture metadata is cached independently of the active material
  table. This avoids repeatedly scanning material stages for already discovered
  materials.
- PathTraceMaterialUniverse owns persistent per-material records, stable
  universe indexes, material facts, and emissive facts.
  PathTraceDynamicMaterialState now builds the active shader-facing table from
  those records by default while preserving the legacy compact row order needed
  by cached static material-index buffers.
- Timing logs report materialTablePath=universe/legacy/legacyFallback,
  materialUniverse record counters, and optional universeTableCompare counters.
  In testing, the universe table path reduced single-threaded material build
  cost without visible material corruption after the row-order compatibility fix.
- Diagnostic texture coverage stats are skipped unless smoke logging is
  enabled.

Geometry universe status:

- PathTraceGeometryUniverse owns the persistent static-world cache vectors and
  exposes stats/generation tracking. It also owns static surface record lookup,
  per-frame lifecycle bookkeeping, append completion, and opt-in validation for
  the existing static cache data.
- PathTraceSmokeSceneBuild still uses the same shader-facing static arrays as
  before. PathTraceSceneCapture builds static geometry through the universe
  rather than directly pushing surface keys itself, but no shader-visible buffer
  contract change has been made.
- Static records now track last/previous seen frames, seen/new/gone flags,
  dirty/history validity, and explicit current/previous vertex/index/triangle
  ranges. For unchanged static world surfaces, consecutive-frame history maps
  the previous range to the same cached range.
- Existing diagnostics report static record lifecycle and validation through
  the static BLAS summary: seen/new/gone, history/previous-range, dirty, and
  validate=total/range/duplicate-key/history/key-vector. User testing in mode
  18 reported no crashes or visible regressions, with validate=0/0/0/0/0.
- Static lookup is backed by a key-to-record map so per-surface touches do not
  scale as O(n). Heavy validation, including duplicate-key scans, is gated by
  r_pathTracingGeometryUniverseValidate.
- Persistent geometry records are for static or future semi-static geometry with
  stable identity. One-frame effects such as muzzle flashes, particles, sparks,
  and temporary overlays should stay out of this model unless given a deliberate
  separate identity/history design.
- The geometry universe must be designed from here onward to retain one previous
  frame of geometry/history data. DLSS Ray Reconstruction and similar temporal
  reconstruction paths need accurate motion vectors; camera-only motion is not
  sufficient for dynamic objects, animated/skinned meshes, moving rigid
  instances, reflected/refracted surfaces, or disocclusion handling.
- The long-term target is geometry-derived motion: for each persistent surface
  or triangle range, the RT path should be able to access current and previous
  world positions or equivalent current/previous local vertices plus
  current/previous transforms. This lets the shader project both positions and
  produce correct screen-space/depth motion instead of guessing from object
  velocity or frame-local draw order.
- This requirement should shape persistent record identity now. Surface records
  need stable identities, previous/current range mapping, transform history,
  invalidation/disocclusion flags, and enough per-vertex or per-triangle history
  to support static, rigid, and eventually skinned geometry. Avoid designs that
  only keep the current frame's packed buffers and throw away the previous
  geometry universe.
- The next useful geometry step is to cleanly package current/previous ranges
  into explicit range structs and add a focused diagnostic dump for validation
  failures. After that, persistent GPU buffers, dirty ranges, previous-frame
  buffers, and stronger static/dynamic separation can be introduced.

Modes 13 and 14 are not Doom lighting. They are proofs that the RT path can
combine geometry, normals, alpha any-hit, material table data, bindless albedo,
a lighting equation, and now a single direct visibility query in debug
visualizations.


Known Blockers and Issues
-------------------------

Texture system integration is the main active blocker.

Current safe approach:

- Only use already available fast-path diffuse or explicit diffuse-stage images.
- Do not use editor image fallback for RT texture discovery.
- Do not force-load missing images during RT scene capture.
- Do not copy raster-owned textures into private RT textures until resource
  state ownership is better understood.
- Use Texture.Load for RT texture debug mode 8. Treat SampleLevel as unsafe near
  animated SWF/GUI terminals until the sampler/state interaction is understood.
- Skip custom-shader deferred callback render entities by default. A repeatable
  crash on disk pickup was fixed by r_pathTracingSkipCallbackEntities 1,
  strongly suggesting callback-driven item shell geometry can become invalid
  during pickup/delete transitions.

Open issues:

- Modes 8, 10, and 13 use the experimental bindless table, not a final global
  material texture system.
- Material stages are much richer than a single diffuse texture.
- Doom materials can have generated images, videos, blend stages, alpha tests,
  deform stages, lights, and special render behavior.
- Normal/specular texture discovery exists and is folded into mode 14 direct
  lighting, and mode 18 has an opt-out `r_pathTracingToyFakePBRSpecular` path
  that reuses rbdoom3's legacy specmap-to-PBR roughness/F0 approximation for a
  basic visual test. These paths are still diagnostic. The final renderer still
  needs a deliberate Doom material facts layer and PT/PBR interpretation,
  including packed RMAO detection, blend/decal/alpha semantics, roughness/F0
  storage, and denoiser-facing material data. A roughness buffer will likely
  matter for NVIDIA ray reconstruction/denoising work later.
- Alpha any-hit exists for simple cutout/perforated materials, but blended
  translucent surfaces are not handled physically yet.
- Smoke particle dithering is a visibility hack, not correct translucency.
- GUI/SWF panels are not solved. The direct generated-GUI-surface path is an
  opt-in diagnostic only and can occlude unrelated geometry or layer panels
  incorrectly. A rasterized offscreen panel texture projected onto the physical
  panel surface is the intended future direction.
- Mode 18 has a crude one-bounce toy path tracer with accumulation, selected
  point lights, texture sampling, and emissive visibility. It is deliberately
  simple and does not include MIS, denoising, ReSTIR, or emissive area-light
  sampling yet. `r_pathTracingToyFakePBRSpecular 1` enables a basic
  specmap-derived F0/roughness direct-specular test in this mode, but the bounce
  path remains diffuse-only. Mode 14 remains the selected point-light
  direct-shadow diagnostic path.
- Mode 18 previously brute-forced every selected point light for both the
  primary direct-light pass and the one-bounce direct-light pass. At
  r_pathTracingLightCount 32 this can mean up to 64 shadow rays per pixel,
  plus primary/bounce rays, and those shadow rays still execute any-hit
  alpha/material code. That ray workload made the portal-transition
  resource-lifetime bug much easier to hit by increasing in-flight GPU work,
  but it was not the root cause of the texture corruption/device-removal class.
  `r_pathTracingSceneRetireFrames` now defaults to 6 and keeps replaced scene
  packages alive; `r_pathTracingToyLightTraceCap` defaults to 8 and remains a
  useful toy-mode workload guard before constants/shader dispatch.
- Mode 13 is a fixed-light debug shader, not real Doom light integration.
- Performance is not optimized. CPU skinning and per-frame dynamic rebuilds are
  acceptable for now but not final.

Build/config notes from the GUI/video and raster-stall investigations:

- Debug can be configured either as FFMPEG=ON/BINKDEC=OFF or
  FFMPEG=OFF/BINKDEC=ON. The latter uses bundled libbinkdec, which can be slow
  in unoptimized Debug builds, but switching to FFmpeg did not change the
  observed monitor/area behavior after GuiModel.cpp was restored.
- RETAIL=ON only adds ID_RETAIL. It strips most non-shipping console commands,
  hides some dev UI, changes a few defaults, and enforces more shipping-like
  behavior. It does not imply Release/RelWithDebInfo optimization in the Visual
  Studio multi-config generator.
- RETAIL=ON was an early suspect because the first fresh clean RT build was
  retail-like, but it is not the proven performance switch. A fresh generic
  Release non-retail build also ran at the 120 FPS cap.
- PATH_TRACING_DEV adds ID_PATH_TRACING_DEV while keeping the stable PT Release
  build lane. Use this for PT-specific diagnostics, commands, and cvars without
  relying on Debug builds for performance testing.
- OPTICK=ON is available through win64-pt-dev-optick for profiling. Keep Optick
  off for final performance sanity checks unless the profile capture itself is
  the point of the test. The PT build/dispatch code keeps optional GPU marker
  scopes for diagnosis, but `r_pathTracingOptickGpuMarkers` defaults to 0
  because enabling those PT-specific native command-object marker calls caused
  level-entry crashes during the diagnostics cleanup pass.
- The legacy Visual Studio Debug compile path is broken for PT validation. It
  can be used only as a compile/link sanity check, and the generated project may
  be stale until `cmake -S neo -B build` is rerun after NVRHI source additions.
  Do not copy `build\Debug\RBDoom3BFG.exe` into the prepared game folder for
  normal PT testing. Use the preset Release/PT lanes instead. Debug executables
  are about 30 MB here, while Release executables are about 14-15 MB. That
  difference is expected from /Od, /Zi, /RTC1, debug CRT, and incremental debug
  linking; it is not a source-code size clue. The old Debug executable
  reproduced the broken frame-pacing/performance collapse.
- `r_useValidationLayers` defaults to 1, enabling NVRHI validation. Turning it
  off at startup restored raster Debug performance to the 120 FPS cap. Use
  validation for targeted resource/state/binding debugging, not performance
  comparisons.
- PT is still much slower in Debug/RelWithDebInfo than Release because the
  prototype does heavy CPU staging every frame: Doom surface capture, vertex
  transforms, material metadata/table work, emissive inventory, buffer upload
  setup, and dynamic BLAS/TLAS preparation. Debug CRT allocation checks,
  disabled/reduced inlining, `/Od`, `/RTC1`, and debug-friendly codegen make
  that path non-representative. RelWithDebInfo in this generated project is
  optimized but still uses `MultiThreadedDebugDLL` and weaker inlining than
  Release.
- Clean upstream/master from v1.6.0 through babd04b3 was bisected and tested
  clear with matching generated renderprogs2. Do not assume the PT/raster stall
  comes from upstream unless clean upstream starts reproducing again.

The host side now builds a CPU ReSTIR PT pass plan for modes 26-33 before the
legacy smoke raygen dispatch. The plan names the active initial, temporal,

RTXDI ReSTIRPTContext resampling mode, and records the RTXDI reservoir buffer

Mode 26 is a gated RTXDI ReSTIR PT initial-sampling debug path. It traces the
normal primary ray, converts supported current hits into RAB surfaces, calls
RTXDI `GenerateInitialSamples`, and stores the returned `RTXDI_PTReservoir` into
the initial path-tracer output slice from `RTXDI_PTParameters`. The local
`RAB_PathTrace` bridge now records one conservative NEE sample from the current
RAB light list per visited path vertex before continuing the BRDF path, because
the original BRDF-only probe was too sparse to show visible reservoirs in
normal Doom scenes. The NEE picker uses the existing weighted emissive-triangle
distribution and uniform analytic-light proposals, then streams the candidates
through a small local RIS selector. It no longer keeps the highest-scoring
analytic trial while recording the raw uniform draw PDF; the RIS estimator is
folded into `radianceOverPdf` with a matching scalar effective `neePdf` for the
path-tracer metadata. Doom analytic RAB samples reject points outside
`doomRadiusAndArea.x` and apply the same quadratic influence falloff as mode 18
native analytic NEE. A single uniform pick from the full light list was too
weak for validation. This is not a
roughness/PBR problem: mode 26 is currently diffuse-only with roughness 1.0,
which is valid enough for this reservoir visibility check. Misses, unsupported
translucent/special surfaces, and invalid hits store empty reservoirs. The
visible output is a sparse heat/debug view for selected useful reservoirs; it
is not final ReSTIR PT shading.
Temporal reuse, spatial reuse, previous-frame surface history, environment
sampling, and dispatched ReSTIR PT resampling passes are still off. Modes 18
and 20 remain the stable toy PT and single-frame direct reservoir paths.

Mode 26 must use the same source3 scene membership baseline as the playable PT
diagnostic modes. It now participates in rigid TLAS routing, rigid residency,
and routed rigid emissive-card inventory; otherwise its ReSTIR PT validation can
silently test against a culled/fallback scene that differs from modes 18/20.

Temporary mode 26 diagnostic fallback colors:

Post-fix validation from user testing: mode 26 changed from mostly blue to
mostly green with noisy white signal on emissive panels and only some remaining
blue in areas likely affected by strong normal maps. In the current temporary

Mode 27 keeps the same initial-sampling reservoir write path as mode 26 but
removes the diagnostic color key and previews initial reservoir shading using
`max(TargetFunction, 0) * max(WeightSum, 0)`, tonemapped into the smoke output.
Invalid pixels fall back to dim albedo or black misses. This is an
initial-only reservoir shading preview, not temporal/spatial ReSTIR PT. It was
added before temporal resampling because the real Step 11 prerequisites,
especially previous primary-surface data and motion vectors, are not exposed to
the smoke shader yet.

User validation: mode 27 works fairly well and runs well performance-wise, with
no serious stability or gameplay issues reported. The remaining visible issues
are rough normal handling and a few fully black objects. Treat those as localized
RAB material/normal/unsupported-surface cases, not as evidence that initial
reservoir generation or the source3 scene/light membership is failing.
Follow-up normal investigation found two likely Doom-specific causes: the green
normal-map channel convention may need inversion for legacy assets, and original
Doom 3 content was baked against idTech4's custom tangent basis rather than
MikkTSpace. `r_pathTracingNormalMapFlipGreen` was added as a diagnostic toggle
for the first issue. The second issue needs a later geometry/material payload
change so PT can use the engine's original tangent and bitangent sign instead
of reconstructing TBN from triangle UV gradients in the hit shader. Mode 27 RAB
surface setup also now constrains decoded shading normals to the oriented
geometric-normal hemisphere before BRDF evaluation, so an over-tilted or
wrong-convention normal map should not turn a reservoir-lit surface fully black.

Mode 28 is the paired initial-reservoir visibility preview. It keeps mode 27's
initial reservoir generation and shading preview, then multiplies the preview
contribution by one conservative visibility ray from the current surface to the
selected reservoir target. For NEE reservoirs it decodes the packed RTXDI light
reference, resamples the current RAB light, respects non-shadow-casting Doom
analytic lights, and ignores the selected static emissive triangle for direct
emissive visibility rays so the light does not immediately shadow itself. For
non-NEE reservoirs it falls back to the reservoir translated world position.
This mode is intentionally a validation bridge for reservoir visibility and
shadowing; it does not enable temporal reuse, spatial reuse, final RTXDI shading
passes, or full path replay.
User validation: mode 28 showed no gameplay/stability issues. The visible
shadows are expected to be 100% sharp in this mode because the preview uses one
binary visibility ray to the selected reservoir sample; soft penumbrae require
many light samples over time/spatial reuse and later final shading/denoising.

Mode 29 is the primary-surface history validation view. The RT shader now writes
an output-sized current primary-surface record for each raygen pixel and the
backend copies current to previous after dispatch so the next frame can read
stable previous records. The versioned PathTracePrimarySurface record stores
world position/depth, geometry and shading normals, view direction, material
identity, surface flags/class, diffuse albedo, opacity, alpha cutoff,
roughness/F0, emissive radiance, instance/primitive IDs, an object/entity ID
placeholder, a height/parallax/displacement placeholder, and a future
previous-position/object-motion placeholder. When
`r_pathTracingToyFakePBRSpecular 1` is enabled, modes 26/27/28/29/30/31/32/33
sample the Doom specular texture and store the converted fake-PBR roughness/F0
in that record; modes 34/35/36/37 use the same material policy for path-tracer
core visualizers. Mode 29 colors the current-vs-previous same-pixel comparison:
blue means current valid but no previous history yet, red means previous valid
but current invalid, yellow means material mismatch, purple means normal
mismatch, orange means roughness mismatch, and green means a conservative
material/normal/roughness match. Reprojection rejection and no-object-motion
status constants are reserved for later object-motion validation; they are not
part of the normal mode 29 success path. This is still same-pixel validation
with no motion vectors or reprojection, so camera motion is expected to create
mismatches. A previous test build produced a full-screen blue result during
camera motion because the primary-surface history clear was tied to
`m_smokeReservoirNeedsClear`, whose dispatch signature changes with camera
origin/direction/FOV. That was a history-lifetime bug, not a real reprojection
diagnostic: primary history now has its own clear flag and is only cleared on
hard history resets such as output/history resource replacement.
Post-fix user validation: stationary views return the expected green-tinted
history match, and moving-camera views no longer collapse to full-screen blue.
Remaining purple normal mismatches and orange roughness mismatches during
motion are expected same-pixel history artifacts until motion vectors and
reprojection are implemented.

Mode 30 is the temporary screen-space reprojection validation view for ReSTIR PT
history. The backend stores the camera basis/tangents that produced the
previous primary-surface history buffer. The shader projects each current
primary hit's world position into that previous camera, samples
`PrimarySurfaceHistoryPrevious` at the resulting previous pixel, and then uses
the same material/normal/roughness color key as mode 29. This should turn many
static-world camera-motion mismatches back toward green before real temporal
ReSTIR is enabled. It is deliberately not a final motion-vector solution:
animated/skinned meshes, moving rigid props, doors/platforms, particles,
translucency, and disocclusions still need vertex/object motion or explicit
invalidation before temporal reuse can be trusted.
User validation: mode 30 is mostly green during camera motion, which confirms
that static-world current-position to previous-camera reprojection is finding
the previous primary history in normal gameplay views. The remaining purple and
orange outlines around moving objects and silhouettes look like an edge-detect
overlay; that is expected for this temporary screen-space method because moving
objects, disocclusions, and subpixel edge changes do not have vertex/object
motion yet. A later visual check noticed that one debug color in this mode had
swapped red/blue compared with an earlier expectation; the user confirmed the
mode still performs its validation role, so treat this as visualizer color drift
rather than a functional regression unless a future cleanup explicitly
normalizes the palette.

Mode 31 is the first temporal-reservoir diagnostic. It switches the RTXDI
ReSTIR PT context to temporal mode for that debug view so the packed reservoir
buffer indices ping-pong through `initialPathTracerOutputBufferIndex`,
`temporalResamplingInputBufferIndex`, and `temporalResamplingOutputBufferIndex`.
The shader still runs inside the existing raygen path: it writes the current
initial reservoir, uses the mode 30 previous-pixel reprojection as the temporal
candidate, rejects missing or mismatched previous surfaces before RTXDI
resampling, then calls `RTXDI_PTTemporalResampling` and writes the temporal
output reservoir. The color key remains diagnostic only: green means a valid
temporal reservoir path, blue means no usable previous surface/reservoir,
yellow/purple/orange are the material/normal/roughness rejection colors from
mode 29/30, and red means invalid current or temporal output. Mode 31 does not
feed final shading, does not enable spatial reuse, and does not solve moving
object motion vectors.
User validation: mode 31 looks broadly similar to mode 30 in normal motion.
Violent camera movement can produce blue sections around screen edges until
history catches up. That is expected because those pixels reproject offscreen or
into disoccluded areas and therefore have no usable previous primary surface or
reservoir yet.

Mode 32 is the temporal-reservoir shading preview. It uses the same temporal
context and conservative previous-surface gates as mode 31, but instead of only
visualizing validity it shades the temporal output reservoir using the same
`TargetFunction * WeightSum` preview style as mode 27. The extra conservative
visibility ray used by mode 28 is now behind
`r_pathTracingRestirPTPreviewVisibility`; keep it at 0 for normal mode 32
testing and enable it only for A/B visibility checks. The current initial
reservoir is generated and stored before temporal resampling, then the temporal
output reservoir is written to `temporalResamplingOutputBufferIndex` and
previewed. This is still a debug view: it does not feed production final
shading, does not enable spatial reuse, and inherits mode 30/31
edge/disocclusion limits until real vertex/object motion vectors exist.

Mode 32 is also currently very expensive. User validation reports it works at
roughly 1280x720 but going much beyond that can device-remove the GPU. Treat
this as a TDR/workload-pressure limit of the current debug implementation, not
as evidence that the ReSTIR PT data path is unusable. The debug path runs a
full-screen raygen that builds the current initial PT reservoir and then invokes
RTXDI temporal resampling/replay work per pixel; full-resolution presentation is
expected to rely on later denoising/upscaling rather than brute-force full-res
mode 32/33. `r_pathTracingRestirPTPreviewMaxPixels` exists as an experimental
safety cap for the temporal preview family, but the practical validated guidance
is still to keep mode 32/33 at or below 720p for now.

Mode 33 is the temporal light-source attribution diagnostic. It uses the same
temporal reservoir generation path as mode 32, but colors the selected temporal
reservoir by the source that supplied the NEE sample: cyan/blue means a Doom
analytic sphere light, orange means an emissive triangle, green means a non-NEE
or indirect reservoir sample, and red means an invalid/unknown packed light.
The normal mode 30/31 rejection colors still apply before a valid temporal
reservoir exists. Modes 26 through 33 use
`r_pathTracingRestirPTAnalyticLightCandidates 1` as a default-on convenience
path for Doom analytic light candidate collection and shader-side analytic-light
enablement when the global analytic-light CVar is off. Set
`r_pathTracingAnalyticLightCandidates 0` and
`r_pathTracingRestirPTAnalyticLightCandidates 0` to test emissive-only ReSTIR
behavior. The same ReSTIR mode range also uses the routed rigid scene path and
routed rigid emissive-card inventory, so temporal diagnostics see the same
source3 membership baseline as the earlier initial-reservoir views.

Clear optimization path for mode 32/33 / future final shading:
- move ReSTIR PT stages out of the monolithic debug raygen into explicit
  pass-sized dispatches so initial sampling, temporal resampling, visibility,
  and final shading can be profiled and throttled independently;
- keep the visibility/shadow ray optional until a final shading pass has a
  denoiser/upscaler target;
- add half-resolution, checkerboard, or fixed internal-resolution execution for
  temporal PT, then upscale to the game output;
- reduce per-pixel replay cost by tightening initial sample count, bounce depth,
  and hybrid-shift/reconnection settings once correctness is established;
- replace the temporary camera-only reprojection with vertex/object motion
  vectors so fewer pixels churn through rejection/fallback paths;
- profile GPU pass timing before raising resolution, because driver device
  removal here is consistent with timeout pressure from long RT dispatches.

Rejected temporal-resolve shortcut: a follow-up, uncommitted mode 33 prototype tried to
reuse `SmokeAccumulation` as previous resolved color, read it at the reprojected
previous pixel, write current output to `SmokeOutput`, and copy output back to
accumulation after dispatch. User validation reported that it mostly picked up
emissives with little regular lighting, showed orange outlines, and
device-removed after roughly 30 seconds. That prototype was removed from the
active mode list; active mode 33 is now the light-source attribution diagnostic
described above, not a temporal resolve. Do not reintroduce the resolved-color
shortcut as the next denoiser step; the next temporal resolve should use an
explicit history resource/pass design with clear ownership, lower internal
resolution/checkerboard execution, and GPU timing around each stage.

Mode 20 reservoir scope is intentionally narrower than mode 18 lighting. Mode
20 samples `SmokeEmissiveTriangles` / `SmokeLightCandidates`; regular Doom
scene lights selected for mode 18 do not contribute to the reservoir yet.
Therefore some light seen in mode 18 can be absent in mode 20. Routed rigid
emissive candidates move with their routed object transform, but remaining
edge-of-screen flicker can still occur when an emissive candidate source depends
on current visible drawSurfs. Persistent/off-screen emissive light records and

CPU-hotspot triage checklist:

- Check scene build timing before blaming the raygen: capture/classification,
  material discovery, emissive/light candidate generation, buffer creation,
  buffer upload, BLAS/TLAS submit, and descriptor/binding rebuilds can all
  dominate the frame before the RT dispatch is submitted.
- Check whether skinned/deformed geometry, material table validation, texture
  lookup, portal walks, readback, or debug overlays ran on the frame being
  profiled.
- Check whether a resource resize/replacement forced idle/garbage collection or
  invalidated descriptors/history.

Device-removal triage checklist:

- Record build lane first. Release/PT dev Release is the only meaningful lane
  for performance conclusions; Debug/validation runs are correctness traces.
- Record the last timing line, debug mode, output resolution, estimated rays per
  pixel, ReSTIR pass label, visibility/reflection/spatial/temporal toggles,
  readback activity, and whether output-sized resources resized that frame.
- Capture `r_pathTracingSafetyDump 1` before changing workload assumptions. If
  the device removal happens at modest ray counts, treat invalid indexing,
  resource lifetime, stale descriptors, barriers, or pathological shader control
  flow as the first suspects.
- If the failure correlates with sustained single-dispatch GPU time rather than
  invalid resource access, enable `r_pathTracingDispatchTileEnable 1` and capture
  `r_pathTracingDispatchTileDump 1`. Start with 512x512 tiles, then reduce to
  256x256 or 128x128 if needed. Tiling is diagnostic dispatch shaping only; it
  does not change scene selection, TLAS/BLAS inputs, or shading quality.
- Recommended bisect sequence for mode 18/20/26-33 device removal:
  disable readback and Optick GPU markers, dump pass timing and safety metadata,
  then toggle in this order until the failure disappears:
  `r_pathTracingDisableAnyHitAlpha 1`,
  `r_pathTracingDisableReflectionRay 1`,
  `r_pathTracingDisableDiffuseSecondaryRay 1`,
  `r_pathTracingDisableSelectedLightLoop 1`,
  `r_pathTracingDisableAnalyticLightLoop 1`,
  `r_pathTracingDisableEmissiveTriangleSampling 1`,
  `r_pathTracingDisableRestirVisibilityRay 1`,
  `r_pathTracingDisableReservoirWrites 1`,
  `r_pathTracingDisablePrimarySurfaceHistory 1`.
  Once a switch changes stability, re-enable earlier switches one at a time to
  separate a bad index/resource path from pure workload fan-out.
- Do not stop at "Wait for Frame", present, or average FPS until CPU staging,
  upload/readback pressure, resource lifetime, and named pass timing have been
  checked.

If a hitch appears only with r_pathTracingSkipRaster3D 0, treat it as a raster
tail issue unless another RT/PT diagnostic contradicts that. Skipping 3D raster
removed the GUI/cinematic hitch during extended testing after GuiModel.cpp was
restored to original behavior, while also producing a large FPS gain.

Recommended Next Steps
----------------------

Near term:

1. Grow PathTraceGeometryUniverse into a real persistent geometry system.
   - Static surface key lookup and append policy now live in the universe.
   - Expand the persistent surface records so they can own or reference static
     ranges, material IDs, BLAS/build signatures, and dirty state.
   - Reserve a previous-frame/history model early: current and previous
     transforms, stable range mapping, and previous vertex/triangle data where
     transforms alone are insufficient.
   - Treat geometry-derived motion vectors as a core requirement for future
     DLSS Ray Reconstruction support, not as a late denoiser integration detail.
   - Keep the shader-facing buffers and material-index compatibility unchanged
     until a deliberate resource contract change is made.
   - Prefer persistent GPU buffers and dirty-range uploads over more
     per-frame CPU micro-optimizations when the behavior is clear.

2. Keep material-universe work focused on validation and missing facts rather
   than another broad rewrite.
   - The universe-derived active table is the default and is cached.
   - Use r_pathTracingMaterialUniverseTableValidate when changing material table
     behavior; a clean frame reports universeTableCompare=1/0.
   - Add new persistent material facts only when they remove repeated per-frame
     work or feed future PT/ReSTIR data.

3. Prepare emissive inventory data for NVIDIA ReSTIR PT integration without
   duplicating the sampler itself.
   - Keep mode 19 as a broad CPU material/triangle inventory plus texel-tight
     debug view.
   - Add metadata that future reservoir sampling will need: area, luminance
     proxy, and transform stability. Emissive texture slot/dimensions and
     triangle UV bounds are already recorded per candidate. Candidate records
     also carry centroid UV, estimated radiance/luminance, area*luminance
     weight, normalized placeholder inventory PDF, area PDF, material universe
     index, material id, and a 64-bit stable-ish triangle identity hash; the
     shader has a helper for centroid emissive texture evaluation.
     The current CPU estimated radiance/luminance is only the material emissive
     color proxy, so many Doom emissive stages report white `(1,1,1)`. The
     stored candidate weight is `area * estimatedLuminance` and is useful as a
     placeholder, but final ReSTIR/PT light preparation should replace or
     augment it in a GPU pass that samples `emissiveSlot` at `centroidUV` or
     over `uvBounds`.
   - Current dumps show sane material variety in clustered emissive scenes:
     static/dynamic split, real texture slots/sizes, and skinned/game-logic
     dissolve overlays excluded.
   - Continue excluding game-logic/skinned dissolve emissive overlays from
     light candidates.
   - Reservoir current/previous/spatial buffers now exist as owned resources.
     They have a reset signature covering camera/output, material table,
     emissive candidate records, light-candidate records, and static BLAS
     identity, and are cleared when that signature changes. They are not yet
     bound into a sampling shader.
   - A persistent light universe now keeps static emissive candidates across
     frames by stable-ish identity, tracks stable rigid-entity emissives as
     semi-static dynamic records, and merges current-frame dynamic emissives
     with the persistent static list. Missing dynamic records are not injected
     by default because matching dynamic occluder geometry is not persistent
     yet. Mode 20 samples that merged list, but spatial and temporal reuse are
     still intentionally off.
   - Q2RTX's Ray Reconstruction reference fork gives the right direction for
     off-camera emissives: static emissive BSP surfaces become light polygon
     records, are assigned to BSP clusters, and are sampled through per-cluster
     light lists instead of through the current screen-visible emissive set.
     Dynamic/model lights are injected into those cluster lists each frame, and
     temporal ReSTIR history uses previous-to-current light identity remapping
     so missing dynamic lights invalidate history. For Doom 3 BFG, this argues
     for future area/portal/cluster-style light membership and identity
     remapping, not blindly retaining stale dynamic emissive candidates.

4. Start a deliberate Doom material to RT/PBR conversion.
   - Base color from albedo.
   - Shading normal from normal map.
   - Specular/F0 from Doom spec map.
   - Roughness approximation for denoising/ray reconstruction.

6. Investigate why SampleLevel in RT removes the device near animated SWF/GUI

7. Start separating decal/blend/translucent behavior from opaque albedo. The

8. Keep a visible-portal-chain augmentation in reserve for rare portal layouts

The user also keeps a Q2RTX Ray Reconstruction reference fork here:

    E:\prog\references\Q2RTX-rayreconstruction-master

Use it for architecture study around DLSS Ray Reconstruction, motion-vector
outputs, previous-frame geometry/transform history, and temporal reconstruction
contracts. The relevant design lesson for this branch is that the geometry
universe should preserve previous-frame geometry state so RT shaders can compute
motion from current vs previous hit positions. Do not rely on camera-only motion
or a frame-local geometry rebuild if Ray Reconstruction is a goal.

For emissive light sampling, the relevant design lesson is that Q2RTX does not
depend on emissive surfaces being on screen. Static emissive BSP surfaces are
promoted into light polygon records and assigned to cluster/PVS light lists.
ReSTIR samples the list for the shaded point's cluster. Dynamic/model emissive
lights are transformed and injected into those same lists per frame, with
previous-to-current light id remapping for temporal reuse and invalidation when
a dynamic light disappears.

Keep reference checkouts outside the RBDoom3 BFG repo. Use them as reference
material for reservoir/light-sampling architecture, motion-vector architecture,
and manuals; do not revive the removed in-tree RTXPT/Donut attempt unless doing
a deliberate clean integration.


Notes for Future Contributors
-----------------------------

This work is debug-only and intentionally conservative. If something looks
overly defensive, it is probably because a previous experiment crashed the device
or produced silently wrong shader data.

Avoid these traps unless you are intentionally revisiting them:

- Do not call GetEditorImage() from RT texture discovery during frame capture.
  It can load missing images and caused repeatable device-removed crashes.

- Do not assume a BLAS built from one vertex/index buffer can be reused with
  different shader-side attribute buffers. BLAS geometry and closest-hit fetches
  must refer to matching data.

- Do not merge static and dynamic triangle metadata casually. InstanceID and
  per-instance buffer selection are important.

- Be cautious with Doom draw-surface order. It is not stable enough to use as a
  persistent identity.

- Be cautious with large single surfaces and transient draw surfaces. The capture
  code has validation for null data, stale CPU geometry, invalid indexes,
  limits, and zero-area triangles for a reason.

The current code is not final architecture. It is a working ladder: each rung is
there because it made the next ray tracing feature testable in game.
