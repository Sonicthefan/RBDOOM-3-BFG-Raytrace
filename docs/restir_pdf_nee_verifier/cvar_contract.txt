PDF + NEE Replacement CVar Contract
===================================

Purpose
-------

Reserve a small CVar surface for the replacement current-frame producer.
These CVars must not control temporal accumulation.

The old PDFNEE light-mode matrix, ReGIR recipe, and portal-candidate CVars are
discarded as implementation routes. They were built before the working RLU
path existed and must not become the normal producer.


Default-Functionality Rule
--------------------------

One enable CVar must activate the replacement module's intended current-frame
estimator with useful defaults:

    default source = current RLU dense domain
    default source policy = current accepted RLU producer policy
    default output = useful current-frame lighting or clean-compatible current
        reservoir page
    diagnostics = optional only

The normal path must not require a recipe of verifier light modes, ReGIR grid
settings, portal toggles, mode 56, view 72, clean-side debug toggles, or legacy
domain switches.


Active Prefix
-------------

Use this prefix unless a manager explicitly changes it:

    r_pathTracingRestirPdfNeeVerifier*


Required Active CVar
--------------------

r_pathTracingRestirPdfNeeVerifierEnable

    0 = disabled
    1 = run the replacement current-frame RLU producer with default behavior

    Default: 0


Allowed Diagnostic CVars
------------------------

Diagnostic CVars may expose:

    numeric PDF/weight views
    selected dense RLU light identity
    invalid-state classifier
    one-shot route dump
    sample-count stress, if the default remains useful
    forced visibility for calibration only

Diagnostic CVars must not be required to make the basic estimator function.
They must not switch the default producer away from full-domain uniform RLU
sampling. Typed-range, bounded per-range, payload-weighted, portal, and ReGIR
sampling policies require separate manager-approved proof tasks before they can
exist as diagnostics or alternatives.

After PDFNEE-RLU-02 has been accepted, later RLU tasks may use a source-policy
CVar for comparing:

    full-domain uniform RLU baseline
    an explicitly normalized typed/range proposal distribution

If a typed/range policy is proven and made default, the full-domain uniform policy
must remain available as a diagnostic comparison. The normal route must still
be activated by r_pathTracingRestirPdfNeeVerifierEnable 1 and must not require
a multi-CVar recipe.


Discarded Legacy CVars And Recipes
----------------------------------

Do not use these as implementation requirements or proof paths:

    r_pathTracingRestirPdfNeeVerifierLightMode 8
    r_pathTracingRestirPdfNeeVerifierLightMode 9
    r_pathTracingRestirPdfNeeVerifierDomain as the default source selector
    r_pathTracingRestirPdfNeeVerifierAnalyticPortalProposal
    r_pathTracingRestirPdfNeeVerifierAnalyticPortalSteps
    r_pathTracingRestirPdfNeeVerifierAnalyticProposalMaxLights
    r_pathTracingRestirPdfNeeVerifierAnalyticDefensiveSamples
    r_pathTracingRestirPdfNeeVerifierAnalyticPortalDump
    r_pathTracingReGIREnable plus ReGIR grid/cache CVars as a producer recipe
    r_pathTracingCleanRtxdiDiExternalPdfNeeCurrent as an old analytic-only
        identity handoff

If these names still exist in code, treat them as old debug/failure evidence
only. They are not accepted NEE functionality and must not be used for the
replacement module's normal path.


Forbidden CVar Behavior
-----------------------

Replacement CVars must not:

    require multiple auxiliary CVars for default functionality
    make debug/status views the default useful output
    enable temporal reuse
    enable spatial reuse
    enable best-light sampling
    enable denoiser/confidence/gradient paths
    route through r_pathTracingDebugMode 56
    reuse r_pathTracingRestirPTDiDebugView as proof
    require ReGIR, portal pools, split-domain arrays, or old NEE records
    change the known-good clean RTXDI DI / RLU temporal path


Required Runtime Shape
----------------------

For estimator proof:

    r_pathTracingRestirPdfNeeVerifierEnable 1

No additional CVar should be required to assemble the normal RLU current-frame
producer path. Extra CVars are only for diagnostics after the default path is
already useful.

The normal default uses:

    r_pathTracingRestirPdfNeeVerifierSamples 32
    r_pathTracingRestirPdfNeeVerifierSourcePolicy 1

Lower sample counts are diagnostic/quality controls only. Complex scenes may
need 64 proposals to make the current-frame estimator representative; 64 is the
supported diagnostic ceiling.

Source policy:

    0 = PDFNEE-RLU-02 full dense current RLU uniform baseline
    1 = PDFNEE-RLU-04 range-stratified current RLU proposal distribution

Policy 1 is the normal default after PDFNEE-RLU-04. Policy 0 must remain
available as the baseline comparison.

The normal visibility setting is:

    r_pathTracingRestirPdfNeeVerifierVisibility 1

This means "store selected-light visibility separately", not "fold candidate
visibility into targetPdf". Candidate streaming must compute targetPdf from:

    RAB_GetLightSampleTargetPdfForSurface(lightSample, surface)

Selected-light replay must call RAB_GetSelectedNeeVisibility and store the
result with RTXDI_StoreVisibilityInDIReservoir. The replacement PDFNEE/RLU
producer forces selected strict shadow visibility in its own constants when
r_pathTracingRestirPdfNeeVerifierVisibility is 1, so the normal one-CVar path
does not require a separate r_pathTracingRestirPTVisibilityPolicy recipe.
r_pathTracingRestirPTVisibilityPolicy remains a diagnostic control for other
ReSTIR PT paths and for proposal-stream strictness experiments.

For non-emissive selected samples, strict visibility must ignore the originating
RAB surface identity. This prevents source-surface self-occlusion from being
misreported as an RLU selected-light failure.

If selected-light replay produces visibility 0, the current reservoir remains
valid. Store zero visibility and zero final contribution; do not convert that
case into an empty-reservoir error.

If proposal streaming produces no positive-target candidate for a pixel, the
normal one-CVar producer output is zero contribution. Do not paint this as an
error color in the default output; invalid-state colors belong in an explicit
diagnostic classifier.

The PDFNEE-RLU-02 source PDF policy is:

    proposal domain = dense current lightIndex range [0, currentRluLightCount)
    sourcePdf = 1.0 / currentRluLightCount
    invSourcePdf = currentRluLightCount

Do not read RLU payload sourcePdf/sourceWeight as the proposal PDF for the
first producer.

The superseded PDFNEE-RLU-03 typed policy used:

    choose class with explicit classProbability
    choose uniformly inside that class
    sourcePdf = classProbability / classCount
    invSourcePdf = classCount / classProbability

The current PDFNEE-RLU-04 policy is not fixed 50/50. It streams a bounded
sample count from each non-empty raw current RLU typed range and computes:

    sourcePdf = rangeSampleCount / (rangeCount * totalProposalSamples)
    invSourcePdf = (rangeCount * totalProposalSamples) / rangeSampleCount

Here rangeSampleCount is the number of streamed proposals for that range after
bounded sampling, rangeCount is the current RLU typed range size, and
totalProposalSamples is the total streamed proposal count for the pixel. Do not
substitute payload sourcePdf/sourceWeight, light power, ReGIR cell weights, or
portal weights for this proposal PDF.

If code or CVar help still describes policy 1 as 50/50, treat that text as
stale unless the implementation is changed to match it.

The current reservoir M contract must match the working RTXDI initial path:

    finalize with the streamed proposal count
    store M = 1 in the produced current reservoir

Do not leave the current reservoir's stored M equal to the number of streamed
proposals; temporal consumers use M as part of the reservoir contract.
