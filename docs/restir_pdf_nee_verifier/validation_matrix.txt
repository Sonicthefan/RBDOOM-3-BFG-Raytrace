PDF + NEE Replacement Validation Matrix
=======================================

All runtime validation is performed in:

    E:/prog/rbdoom-3-BFG-prebuilt


Current Active Validation
-------------------------

The old PDFNEE/ReGIR validation rungs are discarded as implementation gates.
They failed to prove useful current-frame lighting and were built before the
current working RLU path existed.

Do not claim success from:

    old PDFNEE lightMode 8
    old PDFNEE lightMode 9
    old PDFNEE/ReGIR debug views
    portal candidate pools
    mode 56
    view 72
    non-crashing builds
    status colors
    old rbdoom NEE records or contribution output

The active validation target is:

    RLU dense current light domain as source of truth
    active RRX RAB load/sample helpers over that RLU domain
    clean RTXDI DI compatible current reservoir page as output
    clean RTXDI DI view 12 temporal path as the consumer proof

Default-functionality rule:

    one enable CVar must activate the normal producer path
    default source is RLU
    default source policy is the current accepted RLU producer policy
    default output is useful lighting or a clean current reservoir page
    extra CVars are diagnostics only


Rung RLU-V-00: RLU/Clean DI Contract Audit
------------------------------------------

Purpose:

    Prove the replacement module is being designed against the working RLU and
    clean RTXDI DI contracts, not against old split-domain, PDFNEE, portal, or
    ReGIR assumptions.

Expected:

    RLU current payload count and typed ranges are named
    active RRX RAB load/sample path is named
    clean RTXDI DI current/temporal/previous reservoir pages are named
    selected reservoir light identity matches the dense RLU consumer domain
    first producer source policy is stated as sourcePdf =
        1/currentRluLightCount and invSourcePdf = currentRluLightCount

Pass:

    worker can state the exact producer-consumer boundary before editing

Fail:

    worker proposes ReGIR, portal pools, full-map brute force, split-domain
    fallback, or old NEE records as the default source


Rung RLU-V-01: Default Current-Frame Producer
---------------------------------------------

Purpose:

    Prove that the new producer does useful current-frame light sampling with
    default CVars.

Expected:

    one enable CVar activates the normal path
    no view/mode/debug recipe is required
    light proposals come from current RLU dense domain
    selected light identity is a dense RLU current index
    sourcePdf is 1/currentRluLightCount for the first producer
    invSourcePdf is currentRluLightCount for the first producer
    payload sourcePdf/sourceWeight is ignored by the first producer
    sourcePdf, solidAnglePdf, targetPdf, reflectedRadiance, visibility, and
        reservoir weight are stated

Pass:

    raw current-frame output or current reservoir page is useful without
    enabling diagnostic views

Fail:

    output is grey/black/status-only
    normal function requires several auxiliary CVars
    debug views are treated as success proof
    any old PDFNEE/NEE path supplies the result


Rung RLU-V-02: Clean RTXDI DI Consume
-------------------------------------

Purpose:

    Prove the current-frame producer can feed the known-good clean RTXDI DI
    temporal consumer.

Expected:

    producer writes the clean-compatible current reservoir page
    clean RTXDI DI consumes it without rebuilding a different light domain
    temporal off preserves current-frame mean
    temporal on visibly reduces noise without mean drift

Pass:

    user visual testing confirms noisy-to-less-noisy accumulation on real Doom
    lighting through the RLU domain

Fail:

    mean brightness changes to hide the error
    old PDFNEE/ReGIR views are used as proof
    spatial, best-light, mode 56, RRX, denoiser, or fallback paths are required


Rung RLU-V-03: Range-Stratified RLU Proposal Distribution
---------------------------------------------------------

Purpose:

    Improve current-frame sample usefulness while preserving the proven
    PDFNEE-RLU-02 reservoir, replay, targetPdf, visibility, and clean consume
    contract.

Expected:

    proposal ranges are only current RLU typed ranges
    selected light identity remains dense current RLU lightIndex
    rangeSampleCount is stated for every streamed range
    totalProposalSamples is stated
    sourcePdf = rangeSampleCount / (rangeCount * totalProposalSamples)
    invSourcePdf = (rangeCount * totalProposalSamples) / rangeSampleCount
    sourcePdf is normalized over the streamed proposal distribution
    payload sourcePdf/sourceWeight, ReGIR weights, portal distance, and alias
        tables are not used

Pass:

    one-class scenes state the active range PDF
    multi-range scenes state a normalized sourcePdf table
    temporal off preserves current-frame mean against PDFNEE-RLU-02 baseline
    clean RTXDI DI view 12 consumes typed reservoirs without mean drift
    user visual testing shows equal or better sample usefulness/noise behavior

Fail:

    sourcePdf cannot be derived from rangeSampleCount, rangeCount, and
        totalProposalSamples
    sourcePolicy 1 is documented or reported as fixed 50/50 while the shader
        uses rangeSampleCount/totalProposalSamples
    selected identity is not dense current RLU lightIndex
    targetPdf, visibility, final contribution, temporal, or reservoir
        finalization changes are mixed into the proposal-distribution task
    ReGIR, portal, payload-weighted, alias-table, or broad suppression behavior
        is introduced


Rung RLU-V-04: Clean View-12 Analytic Parity
--------------------------------------------

Purpose:

    Prove that PDFNEE/RLU current-frame Doom analytic lighting matches the
    known-good clean RTXDI DI view 12 current-domain analytic load/replay path.

Expected:

    PDFNEE/RLU and clean view 12 use the same RLU domain
    domain 0 analytic-only is compared only to domain 0 analytic-only
    domain 2 mixed RLU is compared only to domain 2 mixed RLU
    compare the same dense current RLU lightIndex
    compare the same RLU record sourceIndex
    compare direct DoomAnalyticLights payload values when clean view 12 uses
        that sourceIndex for current-domain analytics
    compare PDFNEE and clean RAB_LightInfo fields
    compare sample UV, solidAnglePdf, targetPdf, reflectedRadiance, visibility,
        and finalContribution for the same surface

Pass:

    PDFNEE analytic lighting is not missing, brighter, dimmer, or differently
        shaped relative to clean RTXDI DI view 12 except for expected stochastic
        sample choice/noise
    any remaining difference is traced to a named proposal/sample event, not a
        different light payload interpretation

Fail:

    PDFNEE domain 0 is compared against clean view-12 domain 2, or the reverse
    legacy PDFNEE verifier views are used as the visual baseline
    temporal accumulation is tuned before current analytic parity passes
    active RRX unified-record loading rejects or reshapes a current Doom
        analytic that clean view 12 accepts through DoomAnalyticLights
    fallback albedo/beauty, broad light suppression, denoising, or history
        clamping is used to hide the mismatch


Proof Standard
--------------

Required visual behavior:

    current-frame only before consume
    raw flat diffuse direct-light estimate or clean current reservoir page
    one sampled candidate per pixel unless a rung explicitly raises samples
    noisy but correct mean
    no albedo or beauty fallback
    no old NEE output
    no temporal smoothing until the clean consume rung
    no denoiser

Required numeric evidence:

    lightSelectionPdf or invSourcePdf
    solidAnglePdf
    targetPdf
    reservoir inverse PDF / weight
    reflectedRadiance
    finalContribution


Hard Stop
---------

Stop immediately if:

    source PDF cannot be stated
    source PDF does not sum to 1.0 for the proposal domain
    first producer uses typed-range, bounded per-range, payload-weighted,
        portal, ReGIR, or payload sourcePdf/sourceWeight sampling before
        PDFNEE-RLU-02 full-domain uniform is accepted
    PDFNEE-RLU-04 uses any non-uniform policy without a normalized
        rangeSampleCount / rangeCount / totalProposalSamples source PDF
    selected light identity is not a dense RLU current-domain index
    PDFNEE analytic current lighting does not match clean RTXDI DI view 12 for
        the same current RLU analytic domain
    targetPdf and final contribution use different light samples
    solidAnglePdf is missing
    output uses fallback albedo/beauty
    old NEE records or old NEE contribution output are used
    temporal is required to make the current-frame estimator look correct
    broad light suppression is proposed as the fix
