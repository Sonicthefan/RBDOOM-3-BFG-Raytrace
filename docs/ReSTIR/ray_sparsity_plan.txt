ReSTIR PT Ray Sparsity Plan
Date: 2026-05-14
Project: rbdoom-3-bfg-rt path tracing

Purpose
-------
Ray sparsity is a separate pressure-release path from resolution scaling.
Resolution scaling lowers the reservoir domain size. Ray sparsity keeps the
reservoir domain dimensions stable, but updates only an interleaved subset of
reservoir pixels each frame and reconstructs consumers from active
representatives.

Current Scope
-------------
The direct-lighting implementation is default-off and limited to the staged
ReSTIR direct lighting path:

- Mode 50 spatial reservoir shading.
- Mode 51 spatial source attribution.
- Mode 56 combined direct + GI preview.
- Mode 18 when `r_pathTracingRestirPTDirectLighting 1` is enabled.

GI modes 53-56 now have a separate default-off sparse update path. They use the
staged initial-GI producer/consumer split, keep the GI reservoir domain at full
resolution, and reconstruct inactive visible pixels from active full-resolution
GI representatives.

Sparse updates do not apply to standalone initial-reservoir modes 26-28 yet
because those modes still generate and consume their reservoirs in one dispatch.

Control
-------
`r_pathTracingRestirPTRaySparsity`

Historical note: this direct-side control is scaffolding over raw DI/direct
diagnostics. Do not read it as evidence that DI/direct temporal or spatial
reservoir reuse is complete. Future DI reuse should duplicate/adapt the
accepted GI reservoir path.

- `1`: disabled.
- `2..8`: legacy direct-domain sparse update values.

`r_pathTracingRestirPTGiRaySparsity`

- `1`: disabled.
- `2..8`: update roughly one out of N full-resolution GI reservoir pixels in the
  staged mode 53-56 producer pass.

The active phase advances with the ReSTIR PT frame index. The direct-side
temporal wording below is historical; it is not evidence that DI temporal reuse
is complete.

Implementation Contract
-----------------------
For legacy direct-lighting producer dispatches:

1. Treat the old direct-domain dispatch and neighbor language as scaffolding.
2. DI/direct is raw reservoir output on this branch.
3. Future DI sparsity should be designed after DI is rebuilt from the accepted
   GI reservoir path.

For staged GI initial producer dispatches:

1. Sparse producer dispatches now launch only `ceil(outputWidth / rate)` columns
   by `outputHeight` rows.
2. Sparse dispatch pixels map back to the same active full-resolution GI pixels
   used by the original diagonal `1/rate` pattern.
3. Active GI pixels generate initial indirect reservoirs normally.
4. Visible modes 53-56 keep their full-resolution primary hit for material
   context, but load the nearest active GI initial reservoir representative.

The old inactive-pixel early returns remain as safety checks, but real sparse
producer dispatch should avoid launching those inactive lanes in the first
place. In mode 56, `r_pathTracingRestirPTPrimarySurfacePrepass 1` now supplies a
full-resolution primary surface pass, so GI, direct, and final mode-56 passes
can all consume primary history even when GI sparsity is active.

Representative lookup is now constant-time for both sparse direct and sparse
GI. Earlier shader code searched a local `[-8, 8]` x/y window to find an active
representative, which could cost up to 289 active-pixel tests per consumer
pixel. Because mode 56 is still a full-resolution visible consumer, that
consumer-side search could eat much or all of the producer savings. The current
mapping solves the existing active-pixel pattern directly along the same row:

```
((pixel.x + pixel.y * 3u + phase) % rate) == 0u
```

This keeps sparsity deterministic while avoiding a per-pixel loop in the
consumer/reconstruction path.

This is intentionally different from downscaling:

- Primary output dimensions stay full size.
- The ReSTIR direct reservoir dimensions stay whatever
  `r_pathTracingRestirPTDirectResolutionScale` selected.
- Sparse reconstruction happens by representative lookup inside that domain.

Known Limits
------------
- Sparse direct lighting is a diagnostic/rough-preview feature, not a final
  denoiser contract.
- The representative search is local and deterministic, not a true
  reconstruction filter.
- GI modes 53-56 use a full-resolution sparse representative lookup, not the
  direct-lighting scaled-domain lookup.
- Sparse dispatch is still a `1/rate` row pattern. It does not yet provide a
  `1/rate^2` block/checkerboard producer domain.
- Sparse temporal reuse still depends on the existing motion-vector and
  NEE-reuse safety rules. Keep
  `r_pathTracingRestirPTTemporalAnalyticNeeReuse 0` unless the light-domain
  remap/state validation work is explicitly resumed.
- User validation of mode 56 reported visible image changes from sparsity but
  no meaningful performance improvement. Treat sparsity as a
  visual/reconstruction control for now. User Nsight validation reports the
  bad cases are GPU-bound below 60 FPS, with `vkCmdTraceRays` dominating frame
  time but RT core utilization only around 5-10%. The likely bottleneck is
  shader/location-sensitive work inside the raygen path rather than raw
  traversal ray count.
- The first sparsity tests were run before constant-time representative lookup.
  Retest direct and GI sparsity after that change before concluding that
  producer-ray reduction has no possible performance value.
- Mode 56 emits four `vkCmdTraceRays` events in pass order. User Nsight
  validation reports events 2 and 3, the direct temporal and direct spatial
  producers, dominate frame time by far. That explains why sparsity can visibly
  change reconstruction without providing meaningful performance relief. Event
  1, the GI initial producer, also roughly doubles in bad locations but remains
  lower than events 2 and 3, so location sensitivity is shared but the direct
  producer passes remain the primary target.

Suggested Validation
--------------------
Start conservative:

```
r_pathTracingDebugMode 50
r_pathTracingRestirPTDirectResolutionScale 1
r_pathTracingRestirPTRaySparsity 2
r_pathTracingRestirPTSpatialSamples 1
r_pathTracingRestirPTSpatialRadius 16
r_pathTracingRestirPTPassDump 1
```

Then compare:

- Mode 50 with sparsity 1 vs 2.
- Mode 51 with sparsity 1 vs 2 to confirm source colors still identify Doom
  analytic lights as cyan/blue and emissives as orange.
- Mode 18 with `r_pathTracingRestirPTDirectLighting 1` at `samplesPerPixel 1`.

Then test GI separately:

```
r_pathTracingDebugMode 54
r_pathTracingRestirPTGiRaySparsity 2
r_pathTracingRestirPTPassDump 1
```

- Mode 54 with GI sparsity 1 vs 2.
- Mode 55 with GI sparsity 1 vs 2 to confirm source attribution still shows
  Doom analytic lights as cyan/blue and emissives as orange.
- Mode 56 with direct and GI sparsity tested independently, then together, to
  confirm the combined preview still tracks mode 50 direct lighting and mode 54
  indirect/GI lighting.

Do not combine high sparsity, low direct resolution scale, and large spatial
sample counts until the low-cost direct and GI cases are stable.
