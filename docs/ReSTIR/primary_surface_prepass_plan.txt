ReSTIR PT Primary Surface Prepass Plan
Date: 2026-05-15
Project: rbdoom-3-bfg-rt path tracing

Purpose
-------
Mode 56 currently runs several raygen passes that each trace their own primary
camera ray before doing ReSTIR work. User Nsight captures showed mode 56 emits
four `vkCmdTraceRays` events:

1. GI initial producer.
2. Legacy-named direct temporal producer.
3. Legacy-named direct spatial producer.
4. Final combined consumer.

Events 2 and 3 use legacy direct producer labels. Those labels are scaffolding,
not proof that DI/direct has temporal/spatial reservoir reuse. Ray sparsity
changes the image but does not materially reduce frame time.

This document is now historical for the direct side. Future DI work should
duplicate/adapt the accepted GI reservoir path rather than continue from the
old direct temporal/spatial label names.

Recent Failed Shortcut
----------------------
An earlier uncommitted experiment tried to add a mode 56 primary-surface
producer and let later dispatches consume it directly inside the monolithic
shared raygen path. It produced a light-blue output, meaning the PT output clear
color survived because the final pass did not write valid output.

Do not revive that exact shortcut. The corrected version must keep pass identity
explicit, default off, and keep mode 18 untouched unless the user opts into the
experimental ReSTIR path.

Current Useful Contracts
------------------------
The branch already has the key pieces:

- Full-resolution `PrimarySurfaceHistoryCurrent` and
  `PrimarySurfaceHistoryPrevious` buffers.
- `PathTracePrimarySurface.hlsli` packing/unpacking helpers.
- Direct-domain to full-resolution mapping helpers:
  `PathTraceFullPixelToRestirDirectPixel`,
  `PathTraceRestirDirectPixelToRepresentativeFullPixel`, and
  `PathTracePrimarySurfaceLoadPixel`.
- Legacy-named direct producer wrappers for modes 50-51, mode 56, and mode 18
  when `r_pathTracingRestirPTDirectLighting 1` is enabled. Treat their
  temporal/spatial naming as scaffolding only.
- A staged GI initial producer for modes 53-56 that already traces full
  resolution before the legacy-named direct producer wrappers in mode 56.

Implementation Rule
-------------------
This path is now default on for mode-56 testing because its main current value
is avoiding the large combined monolithic shader load:

```
r_pathTracingRestirPTPrimarySurfacePrepass 1
```

Set it to `0` only to test the older legacy fallback. Stable mode 18 with
`r_pathTracingRestirPTDirectLighting 0` must not change.

The shader dispatch identity should be explicit:

- `0`: normal full-output dispatch.
- `1`: direct-domain producer that traces its own primary ray.
- `2`: full-resolution primary-surface producer only.
- `3`: direct-domain producer that consumes full-resolution primary-surface
  history instead of tracing primary.

Do not add a new constant-buffer field for this first slice. Reuse the existing
`RestirPTDirectInfo.z` dispatch-mode lane so CPU and shader layout stays
unchanged.

Pass Graph
----------
Modes 50-51 and mode 18 ReSTIR direct hybrid:

1. Optional full-resolution primary-surface producer dispatch.
2. Legacy-named direct temporal producer dispatch, consuming primary history.
3. Direct spatial producer dispatch, consuming primary history.
4. Full-resolution visible consumer.

Mode 56:

1. Optional full-resolution primary-surface producer dispatch through the
   standalone `pathtrace_primary_surface_producer.rt.hlsl` library.
2. GI initial producer dispatch. Active sparse GI pixels may still trace their
   own primary payload for this first slice; inactive sparse GI pixels reject
   before primary work.
3. Direct temporal producer dispatch, consuming primary history.
4. Direct spatial producer dispatch, consuming primary history.
5. Full-resolution combined consumer, consuming primary history through
   `EvaluateRestirPTCombinedDirectGiPreviewFromSurface`.

This intentionally adds an explicit Pass A when the CVar is enabled. It is the
first step toward the real pass graph, not a final performance win by itself.

Shader Rules
------------
Direct producer dispatches that consume history must:

- Keep their dispatch dimensions in the direct reservoir domain.
- Map direct-domain pixels to representative full-resolution pixels through the
  existing helpers.
- Load `PrimarySurfaceHistoryCurrent` through the same domain-aware RAB surface
  path used by spatial neighbor lookups.
- Skip primary `TraceRay` and skip primary-surface writes.
- Only run producer logic that is valid from a `RAB_Surface`; they must not use
  an uninitialized `PathTraceSmokePayload`.

Full-output visible consumers may keep tracing primary rays in this first slice
except mode 56, which has an explicit surface-based combined preview path. Mode
56 only consumes GI-produced current primary history when GI sparsity is 1; if
GI sparsity is active, direct producers and the final consumer fall back to
their normal primary traces so sparse GI does not leave holes in the primary
surface history they read.

Validation
----------
Build from:

```
E:\prog\rbdoom-3-bfg-rt\neo
cmake --build --preset win64-pt-dev-release
```

Deploy:

```
E:\prog\rbdoom-3-bfg-rt\build-win64-pt-dev\Release\RBDoom3BFG.exe
  -> E:\prog\rbdoom-3-BFG-prebuilt\RBDoom3BFG.exe

E:\prog\rbdoom-3-bfg-rt\base\renderprogs2
  -> E:\prog\rbdoom-3-BFG-prebuilt\base\renderprogs2
```

Start with:

```
r_pathTracingDebugMode 56
r_pathTracingRestirPTPrimarySurfacePrepass 0
r_pathTracingRestirPTPassDump 1
```

Then enable:

```
r_pathTracingRestirPTPrimarySurfacePrepass 1
r_pathTracingRestirPTPassDump 1
```

Expected behavior:

- With the CVar off, pass dumps and visuals match the current working baseline.
- With the CVar on, mode 56 still writes a full image, not the blue clear color.
- The pass dump reports primary prepass use and direct producers consuming
  primary history.
- Mode 18 without `r_pathTracingRestirPTDirectLighting` is unchanged.
- If a perf gain exists, it should show first in events 2 and 3.

Validation result, 2026-05-15:

- The path works visually.
- It did not produce a meaningful performance improvement.
- Keep the CVar as a diagnostic switch, but do not treat primary-ray reuse as
  the main mode 56 bottleneck anymore.
- A standalone primary-surface producer shader library now exists:
  `pathtrace_primary_surface_producer.rt.hlsl`.
- When the CVar is enabled, the prepass uses that standalone library instead of
  re-entering the temporal ReSTIR shader.
- The GI initial producer can now consume `PrimarySurfaceHistoryCurrent` from
  the standalone prepass instead of tracing a fresh primary payload for active
  GI pixels. Pass dumps report this as `giConsumesPrimary=1`.
- Mode 56 can now use `pathtrace_restir_combined_resolve.rt.hlsl` as a
  standalone final resolve when the prepass is active. Pass dumps report this
  as `finalResolve=1`.
- The standalone producer records current material/normal/albedo/specular/
  emissive surface context, but does not yet pack object-motion previous
  positions. Preserve this limitation until the real Pass A owns guide-buffer
  and motion-vector exports explicitly.
- The standalone producer now mirrors the monolithic stochastic any-hit
  rejection rules for GUI alpha, smoke particles, glass/portal fallback,
  additive decals, and filter decals. This fixes the first-order
  translucent/discard behavior needed by `primaryPrepass=1`, but it is not a
  replacement for full layered translucent composition.

Failure signals:

- Light-blue output: final visible pass did not write the output texture.
- All-invalid mode 50/51/56: direct-domain producer consumed the wrong surface
  coordinate domain or stale history.
- Mode 18 baseline changes with the CVar on but ReSTIR direct lighting off:
  guard scope leaked into the stable path.

Follow-up
---------
If this works, the next step is not to keep adding hidden modes to the shared
raygen. Move toward explicit pass libraries or compile-time pass entry points:

- primary-surface producer,
- legacy-named direct temporal producer,
- legacy-named direct spatial producer,
- GI initial producer,
- final visible consumer.

Only after this prepass is stable should final consumers be converted to load
primary surfaces instead of retracing them.
