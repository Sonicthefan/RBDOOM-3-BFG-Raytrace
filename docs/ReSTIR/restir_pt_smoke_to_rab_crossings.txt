ReSTIR PT pathtrace_smoke to RAB Crossings
Date: 2026-05-18

Purpose
-------
Map the specific places where rbdoom's `pathtrace_smoke.rt.hlsl` supplies data
into the local RAB bridge, and name the corresponding NVIDIA RTXDI / FullSample
source for the same contract.

This is narrower than `restir_pt_external_rab_inputs.txt`. That broader file
lists all external RAB inputs. This file is for route visualization and repair
planning:

    pathtrace_smoke.rt.hlsl -> rbdoom RAB_* file
    corresponding NVIDIA RTXDI / FullSample supplier or RAB path

Rules for this file
-------------------
1. The main section only includes edges where rbdoom data comes from
   `pathtrace_smoke.rt.hlsl` or a helper included by it before/around the RAB
   bridge.

2. Plain rbdoom `RAB_* -> RAB_*` edges are not listed in the main section when
   the NVIDIA sample has the same kind of internal RAB cluster. Those edges are
   implementation wiring, not the primary external supply problem.

3. If rbdoom is `pathtrace_smoke.rt.hlsl -> RAB_*` but NVIDIA routes the same
   logical need through a RAB-to-RAB helper, this file still lists it in the main
   section and names the NVIDIA helper. That is an external-supply mismatch on
   the rbdoom side.

4. rbdoom `RAB_* -> RAB_*` edges where NVIDIA's design is materially different
   are listed at the end under "Lower Priority Internal RAB Design Differences".


Main Crossings: pathtrace_smoke.rt.hlsl -> rbdoom RAB
=====================================================

1. Surface Payload Construction
-------------------------------
rbdoom route:
    pathtrace_smoke.rt.hlsl
        PathTraceSmokePayload
        RAB_BuildSurfaceFromSmokePayload(...)
        RAB_BuildMaterialFromSmokePayload(...)
    -> RtxdiBridge\RAB_SurfaceCore.hlsli
        RAB_Surface fields consumed by:
            RAB_IsSurfaceValid
            RAB_GetSurfaceWorldPos
            RAB_GetSurfaceNormal
            RAB_GetSurfaceGeoNormal
            RAB_GetSurfaceViewDir
            RAB_GetSurfaceLinearDepth
            RAB_GetMaterial
            RAB_GetSurfaceRoughness

rbdoom supplied fields:
    valid
    worldPos
    linearDepth
    geometryNormal
    shadingNormal
    viewDir
    materialId
    materialIndex
    instanceId
    primitiveIndex
    surfaceClass
    triangle flags
    RAB_Material

rbdoom source details:
    `RAB_BuildSurfaceFromSmokePayload` reconstructs the RAB surface directly
    from the path tracing payload, ray origin/direction, Doom material index,
    normal map decode, and local surface class.

NVIDIA corresponding route:
    Samples\FullSample\Shaders\LightingPasses\RtxdiApplicationBridge\RAB_Surface.hlsli
        RAB_Surface
        RAB_GetGBufferSurface(pixelPosition, previousFrame)
        RAB_IsSurfaceValid
        RAB_GetSurfaceWorldPos
        RAB_GetSurfaceNormal
        RAB_GetSurfaceGeoNormal
        RAB_GetSurfaceViewDir
        RAB_GetSurfaceLinearDepth

    Samples\FullSample\Shaders\LightingPasses\RtxdiApplicationBridge\PathTracer\RAB_PathTracer.hlsli
        LoadSurfaceFromRayPayload(...)
        getGeometryFromHit(...)
        sampleGeometryMaterial(...)

NVIDIA detail:
    The FullSample's primary surface path is G-buffer-backed. Its continuation
    surface path is ray-payload-backed, but the payload is decoded through
    `getGeometryFromHit` and `sampleGeometryMaterial` using the sample scene
    geometry/material buffers. rbdoom does the equivalent surface reconstruction
    in `pathtrace_smoke.rt.hlsl` itself, then hands a ready `RAB_Surface` to the
    RAB cluster.

current risk:
    This route is valid but local. Any error in Doom payload identity, material
    decoding, normal map handling, or surface class enters every downstream RAB
    callback as if it were reference G-buffer data.

    history buffer and exposes it through the same RAB callback name.

current risk:
    This is one of the most important non-reference substitutions. It is shaped
    like NVIDIA's RAB API, but the supplier is a custom PT prepass/history
    record, not the FullSample G-buffer stack.

Main Exclusions: Internal RAB Cluster Edges With Matching NVIDIA Shape
======================================================================

These are intentionally not in the main list:

    rbdoom RAB_LightTarget -> RAB_Brdf
    rbdoom RAB_LightTarget -> RAB_LightSamplingCore
    rbdoom RAB_Visibility -> RAB_LightSamplingCore
    rbdoom RAB_MISCallbacks -> RAB_LightSamplingCore
    rbdoom PathTracer\RAB_PathTracer -> RAB_Brdf
    rbdoom PathTracer\RAB_PathTracer -> RAB_LightInfoRuntime
    rbdoom PathTracer\RAB_PathTracer -> RAB_LightSamplingCore
    rbdoom PathTracer\RAB_PathTracer -> RAB_MISCallbacks
    rbdoom PathTracer\RAB_PathTracer -> RAB_Visibility

Reason:
    NVIDIA also has RAB callbacks calling other RAB callbacks for BRDF, target
    pdf, light sampling, MIS, visibility, and path tracing. Those edges matter
    for code navigation, but they are not the primary external data supply
    problem this file is meant to expose.


Lower Priority Internal RAB Design Differences
==============================================

These are rbdoom `RAB_* -> RAB_*` edges where the NVIDIA design is materially
different. They are lower priority than `pathtrace_smoke.rt.hlsl -> RAB_*`
because they do not identify the original external data supplier, but they are
still useful when debugging why the local bridge differs from the reference.

1. rbdoom RAB_Brdf split vs NVIDIA RAB_Surface BSDF implementation
------------------------------------------------------------------
rbdoom internal route:
    RtxdiBridge\RAB_Brdf.hlsli
        includes RAB_SurfaceCore.hlsli
        includes RAB_RandomSamplerState.hlsli
        implements cosine hemisphere diffuse sampling
        evaluates Lambert diffuse only

NVIDIA design:
    Samples\FullSample\Shaders\LightingPasses\RtxdiApplicationBridge\RAB_Surface.hlsli
        carries material fields and BRDF/BSDF behavior in the surface bridge
        implements diffuse, GGX, and delta BSDF branches

why lower priority:
    The missing external data is already called out in the material crossing:
    rbdoom supplies roughness/F0 but the internal callback ignores the full
    reference BSDF design.


2. rbdoom RAB_LightTarget split vs NVIDIA RAB_LightSampling target helpers
-------------------------------------------------------------------------
rbdoom internal route:
    RtxdiBridge\RAB_LightTarget.hlsli
        includes RAB_Brdf.hlsli
        includes RAB_LightSamplingCore.hlsli
        owns RAB_GetLightSampleTargetPdfForSurface
        owns RAB_GetPTSampleTargetPdfForSurface
        owns RAB_GetReflectedBsdfRadianceForSurface

NVIDIA design:
    Samples\FullSample\Shaders\LightingPasses\RtxdiApplicationBridge\RAB_LightSampling.hlsli
        owns target pdf helper shape
