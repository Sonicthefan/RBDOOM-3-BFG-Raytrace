ReSTIR PT Resolution Scaling and Reconstruction Plan
Date: 2026-05-14
Project: rbdoom-3-bfg-rt path tracing

Purpose
-------
This note is a new-session outline for moving the current rough ReSTIR direct-lighting work toward the resolution hierarchy used by practical path-traced renderers.

For the next architecture slice, see:

    docs/ReSTIR/real_pass_structure_plan.txt

That plan makes the current direction explicit: keep mode 56 as a debug
preview, then build a real full-resolution primary-surface pass, lower/sparse
lighting-domain producer passes, and a full-resolution resolve before adding
reflections.

The important direction is:

1. Keep the primary visibility and reconstruction anchors full resolution.
2. Let expensive lighting domains run at variable resolution.
3. Reconstruct/composite those lower-resolution lighting results back into the full-resolution path-traced image.
4. Keep the design compatible with later DLSS Ray Reconstruction or a dedicated denoiser.

Current Baseline
----------------
Current ReSTIR state is raw DI/direct reservoir output plus the developing
GI/indirect reservoir path. Do not treat DI/direct as having completed
temporal/spatial reservoir processing.

Relevant modes and controls:

- Mode 18 is the stable base path-traced renderer.
- Mode 50 is the current raw direct/DI reservoir preview.
- Mode 51 is the direct/DI source-attribution diagnostic.
- Mode 18 can opt into ReSTIR direct lighting with:
  - r_pathTracingDebugMode 18
  - r_pathTracingRestirPTDirectLighting 1
  - r_pathTracingSamplesPerPixel 1

The older staged-direct flow described here is obsolete as implementation
status. Future DI work should duplicate/adapt the accepted GI reservoir path
after GI temporal/spatial reuse is visually accepted.

Observed behavior:

- DI/direct is raw reservoir output and should not be judged as a completed
  ReSTIR DI implementation.
- Sample count and full-resolution reservoir passes are expensive.
- Device removal has historically appeared when raising PT resolution too far, especially above roughly 720p-class testing on this branch.

Resolution Hierarchy
--------------------
Use this hierarchy as the guiding contract:

Full resolution:

- Primary hit/visibility.
- Depth.
- Geometric normal.
- Shading normal.
- Material id / material index.
- Albedo, specular, roughness, and fake-PBR converted material terms.
- Motion vectors.
- Surface identity, instance identity, primitive identity.
- Validity/source masks.

Variable resolution:

- ReSTIR direct-light reservoirs.
- Future multi-bounce / indirect reservoirs.
- Future reflection reservoirs/rays.
- Future volumetric or particle lighting domains.

Full-resolution resolve:

- Final image composition stays full resolution.
- Lower-resolution lighting domains are sampled through explicit mapping/reconstruction.
- Final resolve must not pretend that scaled lighting reservoirs are full-resolution buffers.

Non-Negotiable Rules
--------------------
Do not downscale the primary reconstruction anchor in the first pass. Primary surfaces, motion vectors, depth, material identity, and roughness stay full resolution until the renderer has a deliberate alternate reconstruction path.

Do not let scaled reservoir dimensions leak into full-resolution output dimensions. Every producer and consumer must know which pixel domain it is operating in.

Do not bind full-resolution surface history as if it were the same dimensions as scaled reservoirs. If a shader reads full-resolution primary history from a scaled reservoir dispatch, it must map coordinates explicitly.

Do not make direct lighting scale imply future indirect/reflection scaling. Add the pattern once, then replicate it per domain later.

Do not conflate resolution scaling with ray sparsity. Scaling changes the
reservoir domain dimensions; sparsity keeps the domain dimensions stable and
updates only an interleaved subset of pixels. Track sparsity in
`docs/ReSTIR/ray_sparsity_plan.txt`.

Do not tie this to DLSS RR immediately. Keep the data contract compatible with DLSS RR, but first make the reservoir domain explicit and stable.

Do not use Debug builds for validation. Use the PT development Release lane.

First Implementation Scope
--------------------------
Implement variable resolution for ReSTIR direct lighting only.

Reserved but not implemented yet:

- Indirect / multi-bounce reservoir scaling.
- Reflection scaling.
- Volumetric scaling.
- DLSS Ray Reconstruction integration.
- Half-resolution reflection resolve.
- Checkerboard or interleaved sampling.

The first usable target is:

- Full-resolution mode 18 primary path remains intact.
- Future ReSTIR direct reservoir producers may run at a smaller direct-lighting
  resolution after DI is rebuilt from the accepted GI path.
- Mode 50/51 diagnostics currently expose raw direct/DI reservoir behavior.

Historical implementation notes, superseded by the current reservoir-stage gate:

- Older notes describe `r_pathTracingRestirPTDirectResolutionScale` and staged
  TemporalAndSpatial direct-lighting paths. Treat those as provisional
  scaffolding, not completed DI reservoir processing.
- The ReSTIR PT context and temporal/spatial producer dispatches use the scaled
  direct-lighting dimensions.
- Producer raygen maps each direct reservoir pixel to a representative
  full-resolution primary pixel for tracing and surface packing.
- RAB G-buffer callbacks map direct-domain neighbor pixels back to
  full-resolution primary-surface history.
- Mode 18, mode 50, and mode 51 final/visible consumers stay full-resolution and
  use nearest full-pixel to direct-reservoir lookup.
- Consumer shaders must not treat the scaled direct domain as their visible
  dispatch domain. The scaled-domain runtime flag is only allowed to affect
  producer-style dispatches; final consumers use full-resolution raygen/output
  and explicit full-to-direct reservoir lookup.
- Mode 32 remains full-resolution because it is still a single dispatch that
  both generates and shades temporal reservoirs.
- The packed reservoir UAV binding still has full-output capacity in this first
  slice to avoid binding-set churn; RTXDI reservoir parameters and dispatch
  dimensions restrict active reads/writes to the scaled direct domain.

Suggested CVars
---------------
Add direct-lighting scale first:

- r_pathTracingRestirPTDirectResolutionScale

Suggested range:

- 1.0 = full resolution.
- 0.75 = moderate reduction.
- 0.5 = half resolution.
- 0.25 = quarter resolution, useful for stress testing but likely visibly rough.

Reserve names for future domains, but do not allocate or wire them unless the slice explicitly expands:

- r_pathTracingRestirPTIndirectResolutionScale
- r_pathTracingRestirPTReflectionResolutionScale
- r_pathTracingRestirPTVolumetricResolutionScale

Clamp all scales to a sane minimum. Avoid zero-sized dispatches:

- Minimum dimension should be at least 1x1.
- Practical minimum scale should probably be 0.25 for interactive testing.

CPU Resource Model
------------------
Add a small helper for lighting-domain dimensions.

Possible helper names:

- PathTraceLightingResolution
- PathTraceReservoirResolution
- PathTraceRestirResolution

Responsibilities:

- Read full path-trace output size.
- Apply the direct-lighting scale.
- Round up carefully so small view sizes do not collapse.
- Store both full-resolution size and direct reservoir size.
- Detect changes in either full size or direct scale.

The RTXDI/ReSTIR context width and height must be the reservoir domain size, not the full output size, for scaled direct-light passes.

Reservoir buffers used by the direct ReSTIR pass must be allocated at direct reservoir size.

Current/previous validity for the ReSTIR context must include direct reservoir dimensions. A scale change is a context resize and must invalidate or recreate ReSTIR reservoir resources just like a full output resize.

Likely CPU files to inspect/edit:

- neo/renderer/NVRHI/PathTraceCVars.*
- neo/renderer/NVRHI/PathTraceFrameResources.*
- neo/renderer/NVRHI/PathTraceRestirPT.*
- neo/renderer/NVRHI/PathTraceRestirPTReservoirs.*
- neo/renderer/NVRHI/PathTraceSmokeDispatch.cpp
- Any existing path-trace output-size helper currently feeding ReSTIR dispatch dimensions.

Shader Domain Model
-------------------
Add shader-visible sizes for at least:

- Full PT output size.
- ReSTIR direct reservoir size.
- Direct-resolution scale or reciprocal mapping terms if useful.

Add explicit mapping helpers:

- uint2 FullPixelToRestirDirectPixel(uint2 fullPixel)
- uint2 RestirDirectPixelToRepresentativeFullPixel(uint2 restirPixel)
- float2 FullPixelToRestirDirectUv(uint2 fullPixel) if filtering/reconstruction is added.
- bool IsValidRestirDirectPixel(uint2 restirPixel)

The first pass can use nearest-neighbor mapping. Bilinear or neighborhood reconstruction should be a later quality step.

Mapping rules:

- Always clamp mapped pixels to valid bounds.
- Use pixel centers, not integer truncation without thinking about center alignment.
- Avoid frame-varying jitter in the mapping until history rules are explicit.

Likely shader files to inspect/edit:

- neo/shaders/builtin/pathtracing/pathtrace_smoke.rt.hlsl
- neo/shaders/builtin/pathtracing/pathtrace_smoke_restir_temporal.rt.hlsl
- neo/shaders/builtin/pathtracing/pathtrace_smoke_restir_spatial_reservoir.rt.hlsl
- neo/shaders/builtin/pathtracing/pathtrace_smoke_restir_spatial.rt.hlsl

Primary Surface History Policy
------------------------------
This is the main architectural hazard.

The existing ReSTIR bridge was originally built around a single output-size assumption. Once direct reservoirs become scaled, a shader cannot blindly use full-resolution primary-surface buffers as if they were reservoir-sized buffers.

Preferred eventual model:

- Full-resolution primary surface history remains the renderer's reconstruction anchor.
- Future ReSTIR direct producers should have a direct-resolution surface domain
  or a clean coordinate-mapped access path after DI is rebuilt from the accepted
  GI reservoir path.
- RAB_GetGBufferSurface and related callbacks are explicit about whether their pixel coordinate is full-resolution or direct-reservoir-resolution.

Acceptable first bridge:

- Scaled ReSTIR producer dispatches operate in direct reservoir dimensions.
- For each direct reservoir pixel, pick a representative full-resolution pixel using RestirDirectPixelToRepresentativeFullPixel.
- Use that mapped full-resolution pixel to read primary surface data and trace/evaluate the direct-light sample.
- Spatial neighbor queries in the scaled domain map each neighbor reservoir pixel back to its representative full-resolution surface before evaluating compatibility.

This is not final high-quality reconstruction, but it establishes the correct domain separation without shrinking the full-resolution primary anchor.

Avoid:

- Allocating scaled reservoirs but indexing full-resolution surface buffers with scaled coordinates by accident.
- Reusing full-resolution dispatch dimensions while pretending the result is scaled.
- Scaling mode 18 output or primary ray generation as part of this slice.

Dispatch Model
--------------
Direct ReSTIR dispatch dimensions should become:

- dispatchWidth = restirDirectWidth
- dispatchHeight = restirDirectHeight

Mode 18 hybrid final render stays:

- dispatchWidth = fullOutputWidth
- dispatchHeight = fullOutputHeight

Future mode 18 hybrid work would map:

- full pixel -> direct reservoir pixel -> accepted direct sample.

Mode 50 and 51 can either:

1. Display the direct-reservoir domain stretched/mapped to full output, or
2. Continue as direct-domain diagnostics with clear console/docs wording.

For developer sanity, prefer showing diagnostics in the full output window with explicit mapping so the user does not need to reason about a smaller viewport.

Reconstruction Plan
-------------------
First pass:

- Nearest direct reservoir lookup.
- No cross-bilateral filtering.
- No disocclusion-aware upsampling.
- No DLSS RR.

Second pass:

- Add optional 2x2 or small-radius reconstruction from direct reservoir pixels.
- Weight by depth, normal, material id, roughness, and motion validity from the full-resolution anchor.
- Use material/surface identity to reject obviously wrong neighbors.

Later:

- Feed the full-resolution anchor and lower-resolution lighting signal into DLSS Ray Reconstruction or another denoiser.
- Consider half-resolution or checkerboard reflection pass.
- Consider different reconstruction filters per lighting domain.

DLSS Ray Reconstruction Contract
--------------------------------
DLSS RR should eventually receive or derive stable full-resolution guide inputs:

- Full-resolution depth.
- Full-resolution motion vectors.
- Full-resolution normals.
- Full-resolution roughness.
- Full-resolution material/surface identity where available.
- Noisy lighting/radiance input.
- Exposure information.

Because DLSS RR expects strong guide buffers, do not sacrifice full-resolution primary data to make ReSTIR cheaper. Make lighting cheaper, then reconstruct/denoise against stable full-resolution data.

The first guide-buffer scaffold is now tracked in
`docs/ReSTIR/dlss_rr_bridge_plan.txt`. The standalone primary-surface producer
exports albedo, normal+roughness, depth, and hit-distance UAVs at full output
resolution while reusing the existing motion-vector textures.

Future Domain Expansion
-----------------------
After the real pass structure is stable:

1. Indirect / multi-bounce reservoirs.
   - Separate scale CVar.
   - Separate reservoir dimensions.
   - Explicit mapping from full-res primary hit to indirect reservoir pixel.
   - Different history validity rules from direct lighting.

2. Reflections.
   - Do not add before the full-resolution primary surface, lighting-domain
     producers, and resolve/composite pass are explicit.
   - Likely half-resolution by default.
   - Needs roughness-aware reconstruction.
   - Glass and mirror-like materials need special treatment.
   - Avoid destroying sharp reflections by forcing all reflection rays into one low-res bucket.

3. Volumetrics.
   - Can be much lower resolution.
   - Temporal accumulation and reprojection are mandatory before judging quality.

4. Final denoiser / DLSS RR integration.
   - Should happen before chasing perfect raw reservoir quality at high resolution.

Failure Signals
---------------
Treat these as likely domain mismatch bugs:

- Mode 50/51 suddenly all blue/green with no scene variation.
- Mode 18 hybrid becomes flat/unlit or exposure-insensitive.
- ReSTIR direct appears shifted relative to geometry edges.
- Lighting swims when camera is stationary.
- Reservoir reuse breaks only at scaled resolutions.
- Crashes occur only after changing the direct resolution scale CVar.
- Device removal happens immediately on scale changes.

Treat these as expected quality limitations of the first pass:

- Lower-resolution direct lighting looks blocky.
- Fine emissives are less stable at 0.5 or 0.25.
- Small lights may need better candidate/reconstruction rules.
- ReSTIR DI still looks similar to native direct lighting in many areas.

New Session First Slice
-----------------------
First coding slice status: implemented as the initial direct-domain bridge above.
Remaining follow-up work:

1. Validate 1.0 scale first, then 0.5.
2. If 0.5 is stable, add the first small reconstruction filter over nearby
   direct reservoir pixels.
3. Weight or reject reconstruction taps with full-resolution depth, normal,
   material/surface identity, and roughness.
4. Split mode 32 into producer/consumer form if temporal-only preview also needs
   scaled-domain testing.
5. Move packed reservoir allocation to direct-size capacity only when the
   binding-set/resource lifetime churn is handled deliberately.

Do not start with:

- DLSS RR integration.
- Reflection scaling.
- Indirect reservoir scaling.
- A large reconstruction filter.
- A broad rewrite of the RAB bridge.

The first goal is only to make ReSTIR direct lighting live in an explicit, independently sized pixel domain.
