Rough ReSTIR PT Preview Plan
Date: 2026-05-12

Goal
----
Build a deliberately rough, developer-facing ReSTIR PT lighting preview before
attempting production integration, spatial reuse, denoising, or reflection/
glass recursion. The preview should prove that current ReSTIR reservoirs can
select Doom analytic lights and emissive candidates, temporally reuse them, and
produce a visible lighting estimate through the existing path tracing material
contract.

Current Baseline
----------------
- Mode 18 is the stable/playable path tracer. It owns the real material path:
  fake-PBR conversion, roughness/F0 derivation, normal handling, samples per
  pixel, path depth, analytic-light NEE, reflections, and the current output
  integration behavior.
- Modes 26-28 are ReSTIR PT initial-reservoir diagnostics.
- Modes 29-30 are primary-surface history/reprojection diagnostics.
- Mode 31 calls RTXDI_PTTemporalResampling and validates temporal reservoirs.
- Mode 32 shades the temporal reservoir as a preview.
- Mode 33 colors temporal source attribution.
- Mode 50 is a debug-only raw direct/DI reservoir preview. Historical names may
  say TemporalAndSpatial, but DI/direct processing is not complete.
- Mode 51 colors direct/DI source attribution over the current raw reservoir
  output.
- Modes 53-55 are the first debug-only indirect/GI initial-reservoir views.
  They skip primary NEE in the RAB path tracer so the visible reservoir domain
  starts at secondary/indirect path candidates.
- Mode 56 combines raw direct/DI reservoir output with the completed
  indirect/GI initial reservoir from the mode 54 path.
- ReSTIR shader libraries are now split and force-specialized per mode, which
  makes iteration feasible again. Keep core mode 18 independent from ReSTIR
  debug-mode shader cost.

Rough Preview Definition
------------------------
The first rough test version is not "final ReSTIR PT." It is a constrained
lighting preview:
- Use the existing temporal reservoir generation path from mode 32.
- Output lighting from the selected temporal reservoir, not diagnostic colors.
- Keep modes 26, 31, and 33 available as diagnostics.
- Use Doom analytic-light and emissive candidates through the RAB bridge.
- Use the same material facts as mode 18 wherever practical: albedo, normal,
  roughness/F0, and fake-PBR conversions.
- Keep visibility optional and conservative until the visibility contract is
  split out of the debug raygen.
- Keep resolution conservative. Current device-remove behavior above roughly
  720p-class output is a known limitation, not a reason to widen the scope.

Immediate Implementation Steps
------------------------------
1. Inspect the mode 32 shader path and host routing.
2. Separate diagnostic color output from temporal lighting-preview output.
3. If needed, add a narrow CVar or mode label that makes "rough ReSTIR PT
   preview" explicit without changing mode 18 behavior.
4. Make mode 32 present a usable temporal-reservoir lighting estimate.
5. Preserve mode 33 attribution as the source-mix debug view.
6. Add or reuse low-noise console dumps for:
   - pass label and selected ReSTIR shader library,
   - temporal mode active/inactive,
   - preview visibility active/inactive,
   - output resolution and preview pixel cap.
7. Build with the PT dev-release lane and copy updated artifacts to the prepared
   game folder.

Do Not Do Yet
-------------
- Do not treat mode 50 as production spatial reuse; it is still a debug raygen
  preview with a shader-local neighbor-offset table.
- Do not merge ReSTIR debug libraries back into the core path tracer.
- Do not make the authored Doom light universe smaller to hide performance
  issues. Bound proposals per shading point instead.
- Do not treat debug color views as final lighting evidence.
- Do not fold reflection/glass/transmission history into diffuse ReSTIR history.
- Do not require high output resolution for this milestone.

Implementation Note
-------------------
Mode 32 should behave as the rough visible lighting preview. It should not tint
temporal rejection, missing previous surfaces, or invalid reservoirs with the
strong debug colors used by mode 31. Those colors are useful diagnostics, but
they make rough lighting tests look like edge-detection overlays when moving.
Mode 32 should fall back to near-black material context when temporal reuse
fails; use modes 31 and 33 to inspect rejection and source attribution.

If the RTXDI PT estimate is effectively black while the selected reservoir still
contains a packed NEE light, mode 32 may reconstruct that selected light sample
for preview only. This is acceptable for the rough milestone because it proves
candidate selection and material-aware direct lighting visibility, but it must
not be mistaken for the final ReSTIR PT estimator.

Temporal NEE Safety Note
------------------------
The validated rough-preview state keeps temporal reservoir reuse enabled but NEE
light-reservoir reuse disabled:

  r_pathTracingRestirPTTemporalReservoirReuse 1
  r_pathTracingRestirPTTemporalAnalyticNeeReuse 0

The CVar name is historical. In the current implementation it rejects prior-frame
NEE reservoirs from both Doom analytic lights and emissive/strobe sources before
temporal combine. This was required after a same-portal robot-train/strobe
sequence proved that current-frame NEE sampling was clean, but stale previous
NEE reservoirs could corrupt area lighting when the animated source stopped
offscreen. The long-term fix is stable light-domain identity and remapping; do
not treat the current guard as final quality behavior.

Success Criteria
----------------
- Mode 18 still launches normally and does not pay the full ReSTIR shader load
  cost.
- Mode 32, or the chosen rough-preview mode, shows actual temporal-reservoir
  lighting rather than flat attribution/debug color.
- Moving the camera does not immediately collapse to solid diagnostic color.
- Doom analytic lights and emissives can both contribute when their candidate
  gates are enabled.
- Known limitations are documented instead of hidden: spatial reuse is mode-50
  debug-only, light identity remap is fragile, visibility is conservative,
  resolution is limited, and there is no production final pass yet.

Spatial Follow-Up
-----------------
Modes 50-51 have legacy TemporalAndSpatial labels in the wrapper path, but this
branch should treat DI/direct as raw reservoir output. Do not use these modes as
evidence that direct temporal/spatial resampling is complete.

Mode 51 uses the same reservoir path as mode 50 but outputs source attribution
instead of lighting: cyan/blue for Doom analytic lights, orange for emissive
triangles, green for non-NEE/indirect reservoirs, and red for invalid/unknown
light metadata. Use it to compare source stability against mode 33 and to look
for spatial source bleeding across depth/object edges.

Modes 50-51 now use `r_pathTracingRestirPTDirectResolutionScale` for the staged
direct-lighting reservoir domain. Producer prepasses run at the scaled direct
size, while the visible mode 50/51 outputs remain full-resolution and use a
nearest full-pixel to direct-reservoir lookup. Scale 1.0 should match the
pre-scaling path; lower scales are expected to look rougher until a real
reconstruction filter is added.

For harder validation, set `r_pathTracingRestirPTSpatialDiagnosticView 2` in
mode 51. That view compares temporal and spatial packed sources and marks
changed Doom analytic/emissive NEE sources yellow when the selected spatial
source cannot reconstruct usable local rough-preview lighting at the current
surface, which catches attribution that does not agree with mode 50 shading.

The first mode 50/51 implementation intentionally avoids a new host descriptor
for RTXDI neighbor offsets. It uses a 32-entry shader-local offset table and sets
`neighborOffsetMask = 31`. Replace this with a real RTXDI neighbor-offset buffer
when spatial reuse graduates out of debug-only preview work.

Controls:

  r_pathTracingRestirPTSpatialSamples
  r_pathTracingRestirPTSpatialRadius

The spatial pass reuses `r_pathTracingRestirPTTemporalDepthThreshold` and
`r_pathTracingRestirPTTemporalNormalThreshold` for neighbor validation. Its
disocclusion boost sample count is deliberately tied to the normal spatial
sample count for now, avoiding a hidden multi-sample workload jump while
device-removal limits are still being characterized.

Indirect/GI Follow-Up
---------------------
Modes 53-55 establish the first indirect/GI reservoir surface without adding a
new pass graph yet:

- Mode 53: indirect/GI initial reservoir diagnostics.
- Mode 54: indirect/GI initial reservoir shading preview.
- Mode 55: indirect/GI path/source attribution.

These modes reuse the initial RTXDI PT reservoir buffer and shader library. The
RAB path tracer skips primary-surface NEE only in modes 53-55, leaving secondary
NEE, emissive hits, and non-NEE path radiance as the first GI candidate domain.
The debug path must also allow one secondary bounce and an RTXDI initial
path-length budget of at least 4, because RTXDI records secondary-surface NEE as
pathLength 4 while the default initial max is 3.
Modes 53-55 now run as staged initial-GI passes: a producer dispatch writes the
completed indirect initial reservoirs, then the visible mode consumes those
reservoirs instead of regenerating them in the same dispatch.
Keep them full-resolution and initial-only until source/path attribution is
stable. Do not add temporal GI reuse, spatial GI reuse, sparse updates, or mode
18 GI integration in the same slice.

Ray Sparsity Follow-Up
----------------------
`r_pathTracingRestirPTRaySparsity` is a default-off staged-direct-lighting
diagnostic. Values 2-8 update an interleaved subset of the direct reservoir
domain each frame, then reconstruct full-resolution consumers from active
representative reservoir pixels. It is wired only for modes 50-51 and mode 18
with `r_pathTracingRestirPTDirectLighting 1`.

`r_pathTracingRestirPTGiRaySparsity` is the matching default-off GI diagnostic
for modes 53-56. Values 2-8 sparsely update the full-resolution staged GI
initial producer, then visible GI modes load the nearest active GI initial reservoir
representative while keeping their current full-resolution surface context.

Combined DI + GI Follow-Up
--------------------------
Mode 56 is the debug-only combined direct + GI preview. It shades raw direct/DI
reservoir output with completed GI initial output in the normal path. The 01C GI
temporal candidate is debug-only via `r_pathTracingRestirPTGiDebugView 2` and
does not change normal mode 56 or mode 18.

Validate mode 56 by comparing it against:

- Mode 50 for direct spatial-reservoir lighting.
- Mode 54 for indirect/GI initial-reservoir lighting.
- Mode 55 for indirect source/path attribution.

Both sparsity controls can be tested in mode 56:

  r_pathTracingRestirPTRaySparsity
  r_pathTracingRestirPTGiRaySparsity

Validated Issues
----------------
Mode 56 is visibly working. Sparsity visibly changes the image, but it does not
meaningfully improve performance, so do not assume the first bottleneck is raw
producer ray count. User Nsight validation reports the bad cases are GPU-bound
below 60 FPS: `vkCmdTraceRays` dominates frame time, while RT core utilization
is only roughly 5-10%. The next performance step should profile shader work
inside `vkCmdTraceRays`, especially location-sensitive environment/sky,
material, light sampling, memory/cache, and branch-divergence paths.

`r_pathTracingRestirPTPrimarySurfacePrepass` is a primary-surface sharing
experiment. Do not interpret it as proof that DI/direct temporal/spatial
reservoir processing is finished. DI/direct remains raw reservoir output until a
future slice duplicates/adapts the accepted GI path.

Mode 56's final combined consumer also has a surface-based preview path when
the primary-surface prepass experiment is enabled and GI sparsity is 1. With GI
sparsity active, mode 56 falls back to tracing primary in the direct/final
passes so the sparse GI producer does not leave missing current primary-surface
records.

Runtime validation on 2026-05-15 reported that this path works but does not
improve performance. Do not spend more time on primary-ray reuse as the main
mode 56 optimization. The direct producer passes remain the target, but the
cost is likely after the primary hit is available: reservoir generation,
RTXDI/RAB sampling, direct/indirect NEE trial loops, visibility/continuation
rays, memory traffic, or shader divergence.

Mode 56 can still show legacy `vkCmdTraceRays` labels for direct temporal and
direct spatial producer events. Those labels are stale scaffolding names. Do not
prioritize DI temporal/spatial tuning until DI has been rebuilt from the
accepted GI reservoir path.

Some nominally dark base-game spaces appear to have open roofs. The raster path
apparently hides those spaces from skybox light through rules outside simple
closed geometry. The GI preview can therefore over-brighten them. Preserve
environment-lighting headroom, but add a future investigation for Doom
sky/environment visibility or light-mask rules before treating mode 56 exposure
as a pure GI-quality problem.
