ReSTIR PT Reference Input Map
Date: 2026-05-17

Purpose
-------
This is a contract map, not an implementation plan. It lists the inputs the
RTXDI ReSTIR PT reference path expects, where rbdoom currently connects each
input, and which RAB bridge file owns the shader-side callback or record.

The goal is to prevent future agents from inventing reservoir behavior when the
reference path is really failing because an input or RAB callback is incomplete.

Pass-Level Non-RAB Inputs
-------------------------
These are required by the reference ReSTIR PT passes, but are not themselves
RAB callbacks.

1. RTXDI_PTParameters / buffer indices

Reference need:
    ReSTIR PT parameters, reservoir buffer parameters, initial/temporal/spatial/
    final buffer indices, temporal thresholds, spatial sample count/radius,
    initial sampling settings, reconnection settings.

rbdoom connection:
    CPU: neo\renderer\NVRHI\PathTraceRestirPT.cpp
    CPU: neo\renderer\NVRHI\PathTraceRestirPasses.h
    shader binding: RestirPTParams at b28 in pathtrace_smoke.rt.hlsl and
    pathtrace_restir_combined_resolve.rt.hlsl

RAB file:
    None. This is RTXDI parameter plumbing.

Status:
    Connected. Use r_pathTracingRestirPTPassDump to verify active indices:
    initialOut, temporalIn, temporalOut, spatialIn, spatialOut,
    finalShadingIn.

2. Packed PT reservoir buffer

Reference need:
    RTXDI_LoadPTReservoir / RTXDI_StorePTReservoir access to the reservoir
    buffer pages selected by RTXDI_PTParameters.bufferIndices.

rbdoom connection:
    shader binding: RestirPTReservoirs at u29
    shader include: Rtxdi/PT/Reservoir.hlsli
    local debug readouts:
        pathtrace_restir_combined_resolve.rt.hlsl
        LoadRestirPTInitialDirectReservoir
        LoadRestirPTTemporalInputReservoir
        LoadRestirPTTemporalOutputReservoir
        LoadRestirPTSpatialOutputReservoir
        LoadRestirPTFinalShadingInputReservoir

RAB file:
    None. This is RTXDI reservoir storage.

Status:
    Connected for the reference RTXDI reservoir chain. Separate debug-only
    RestirPTDiReservoirs / RestirPTGiReservoirs are not the reference chain.

3. Pixel position, reservoir position, direct-domain dimensions

Reference need:
    Current pixel position, reservoir position, dispatch size, and any
    direct-domain resolution mapping.

rbdoom connection:
    pathtrace_smoke.rt.hlsl
    pathtrace_restir_combined_resolve.rt.hlsl
    helpers around PathTraceFullPixelToRestirDirectPixel,
    PathTraceRestirDirectSize, PathTraceRestirSparseRepresentativePixel.

RAB file:
    None. This is pass/domain mapping.

Status:
    Connected, but direct-domain/full-resolution conversion is a local
    adaptation and must be checked when diagnosing out-of-bounds or sparse
    representative mismatches.

4. Motion vector for temporal reuse

Reference need:
    Temporal resampling runtime uses a previous pixel position and a motion
    vector whose z component contributes expected previous linear depth.

rbdoom connection:
    shader textures:
        PathTraceMotionVectors u39
        PathTraceMotionVectorMask u40
        PathTraceRRGuideResetMask u52
    CPU constants:
        MotionVectorInfo
    producers:
        pathtrace_primary_surface_producer.rt.hlsl
        PathTraceSmokeDispatch.cpp

RAB file:
    None directly. Temporal validation then calls RAB surface/material helpers.

Status:
    Connected for the standalone primary-prepass route. rbdoom now exports xy
    pixel motion plus z depth delta in PathTraceMotionVectors, matching the
    reference temporal contract where expected previous depth is current
    surface linear depth plus motionVector.z. The reset/mask texture still
    carries rbdoom-local validity reasons.

data. Light index stability is a contract, not optional debug metadata.

rbdoom connection:
    Current emissive count + current Doom analytic count.

RAB file:
    neo\shaders\builtin\pathtracing\RtxdiBridge\RAB_LightInfoRuntime.hlsli

Status:
    Connected. Environment lights are not included.

Reference need:
    Needed for light-index domains and previous-frame remapping.

rbdoom connection:

    DoomAnalyticLightRemapInfo.w for previous emissive count.

RAB file:
    RAB_LightInfoRuntime.hlsli

Status:

Reference need:

rbdoom connection:

19. Load light info, current and previous frame

Reference need:
    RAB_LoadLightInfo(index, previousFrame).
    Temporal inverse shift and bias correction rely on previousFrame=true
    returning a coherent previous-frame light.

rbdoom connection:
    Current emissives:
        SmokeEmissiveTriangles t16
    Previous emissives:
        SmokePreviousEmissiveTriangles t57
    Emissive remap:
        SmokeEmissiveRemap t58
    Current Doom analytic:
        DoomAnalyticLights t27
    Previous Doom analytic:
        DoomAnalyticPreviousLights t45
    Analytic identities/remap:
        DoomAnalyticCurrentIdentities t42
        DoomAnalyticPreviousIdentities t43
        DoomAnalyticRemap t44

RAB file:
    RAB_LightInfoRuntime.hlsli
    RAB_LightInfoCore.hlsli

Status:
    Partial but now wired for emissive and analytic previous-frame loads.
    Bright green in debug view 12 means previous emissive light remap still
    failed. That is a RAB/light-identity contract failure, not reservoir math.
    Debug view 27 classifies the current RAB light-index domain as a tiled grid
    in the top half of the screen and the previous-frame translated load domain
    as a tiled grid in the bottom half. Visual testing showed current-frame
    loads are mostly valid while previous-frame suppliers are insufficient for
    full RTXDI PT temporal reuse: static analytic lights commonly reject as
    unproven continuity or non-sampleable, and motion briefly flashes missing
    previous/remap target or other remap/load failures before returning to
    valid as motion stops.

Visibility Inputs
-----------------
24. Conservative visibility to light sample

Reference need:
    RAB_GetConservativeVisibility(surface, lightSample).

RAB file:
    neo\shaders\builtin\pathtracing\RtxdiBridge\RAB_Visibility.hlsli

Status:
    Connected at the core callback. RAB_GetConservativeVisibility traces the
    local shadow-ray path and is the reference-shaped conservative visibility
    query. The upstream candidate/selected visibility supplier is not
    reference-compliant when policy 0 admits samples that strict conservative
    visibility rejects. Debug view 24 classifies the final input selected NEE
    sample against geometry, candidate, selected, and strict conservative
    visibility gates. Debug view 29 applies the same visibility gate
    classification to the current light-domain tile replay used by view 28, so
    visibility failures can be inspected without depending on a selected
    reservoir sample.

25. Conservative visibility to sample position

Reference need:
    RAB_GetConservativeVisibility(surface, samplePosition).
    Used for PT reservoir sample visibility in temporal/spatial reconnection.

RAB file:
    RAB_Visibility.hlsli

Status:
    Connected through local shadow rays. This is expensive and Doom-specific.
    The callback is reference-shaped, but the producer layer can provide
    geometry-only accepted samples that fail this strict query. Debug view 24
    exposes where the local policy defers strict visibility for the selected
    final input sample. Debug view 29 exposes the same split over current
    light-domain samples.

26. Candidate and selected NEE visibility

Reference need:
    The initial path tracer must decide whether NEE proposals and selected NEE
    samples are valid before recording them.

RAB file:
    RAB_Visibility.hlsli
    PathTracer\RAB_PathTracer.hlsli

Status:
    Connected with local visibility policy:
        policy 0 = geometry-only candidate/selected checks
        policy 1 = strict selected NEE visibility
        policy 2 = strict candidate and selected NEE visibility
    Debug view 24 classifies the active policy result for finalShadingInput.
    Debug view 29 classifies the active policy result over the current
    light-domain tile replay. Visual testing showed policy-accepted samples
    that strict visibility rejects; that is a supplier/producer mismatch, not a
    reservoir-math issue. The visibility supplier must be audited against the
    RTXDI reference before policy-0 output is treated as valid reuse input.

MIS / Light-Selection Inputs
----------------------------
27. MIS weight for NEE

Reference need:
    RAB_GetMISWeightForNEE and related helpers.

RAB file:
    neo\shaders\builtin\pathtracing\RtxdiBridge\RAB_MISCallbacks.hlsli

Status:
    Conservative adapter. Analytic samples return weight 1.0 as in the
    reference analytic branch. Emissive triangle NEE now uses rbdoom's existing
    emissive proposal PDF and emissive trial count to form the NEE MIS term
    instead of using solid-angle PDF alone. rbdoom still lacks the reference
    local-light PDF texture/sampling-mode inputs, so the adapter reads the
    local emissive `sampleWeightAndPdf.y` proposal field.

Missing inputs:
    local_light_pdf_texture
    local_light_sampling_mode
    nee_light_selection_pdf_in_callback
    exact_reference_num_local_light_samples

28. MIS for emissive surfaces and environment map

Reference need:
    GetMISWeightForEmissiveSurface, GetMISWeightForEnvironmentMap,
    RAB_EvaluateEnvironmentMapSamplingPdf, RAB_GetEnvironmentMapRandXYFromDir.

RAB file:
    RAB_MISCallbacks.hlsli

Status:
    Partial. BRDF-hit emissive-surface radiance now applies
    GetMISWeightForEmissiveSurface before recording the emissive sample. The
    callback conservatively remaps a ray hit to the current emissive light
    domain by matching instanceId, primitiveId, and materialId against
    SmokeEmissiveTriangles, then uses the same local emissive proposal PDF and
    emissive trial count as the NEE callback. RAB_RayPayload now carries the
    DXR GeometryIndex value, but rbdoom still lacks the reference direct
    hit-to-light lookup table. Debug view 26 classifies visible emissive
    primary surfaces against the current rbdoom linear hit-to-light scan:
    missing match, duplicate match, failed light-info load, mapped but zero
    loaded radiance, invalid loaded light record, or unique mapped current
    emissive. It no longer tries to replay the matched light from the visible
    emissive surface itself because the reference emissive-hit MIS path replays
    from the previous receiver surface with the ray hit barycentrics.
    Structurally loadable emissive records with zero CPU-estimated proposal
    weight now receive a tiny structural weight fallback so the map can be
    classified separately from the missing radiance estimate.

    Environment-map direction-to-UV is now a deterministic equirectangular
    mapping, but environment map pdf remains 0 and environment lights are not
    in the RAB light domain. The path tracer now marks
    RAB_PathTracerUserData.flags with
    RAB_PTUD_FLAG_ENVIRONMENT_MAP_MISS_UNBRIDGED when a continuation ray misses,
    documenting the exact place where the reference would optionally record an
    environment-map light sample. Debug view 18 classifies the environment MIS
    contract bands: left third direction-to-UV, middle third sampling pdf,
    right third environment light/radiance domain. This means the reference
    environment-light branch is still not implemented.
    rbdoom's current shader-visible outputs are insufficient for the NVIDIA
    reference environment-light branch. The systems that should supply
    environment sampling PDF, environment light-domain records, environment
    radiance, and miss-sample recording policy must be audited against the
    RTXDI reference before this callback is treated as connected.

Missing inputs:
    emissive_surface_hit_to_light_index_map
    geometry_instance_to_light_buffer
    emissive_triangle_barycentric_replay_vertices
    emissive_radiance_estimate_for_rab_light_records
    environment_sampling_pdf_source
    environment_light_domain_records
    environment_miss_sample_records_for_restir
    environment_map_radiance_source
    environment_map_recording_policy

    PathTracer\RAB_PathTracer.hlsli

RAB file:
    neo\shaders\builtin\pathtracing\RtxdiBridge\PathTracer\RAB_PathTracer.hlsli

Status:
    Connected. Local implementation is deliberately small and diffuse-focused.
    It is not a full production path tracer. Debug view 15 now classifies the
    RTXDI initial path-tracer output reservoir's path metadata: invalid
    reservoir, non-NEE sample, invalid sampledLightData, zero path length,

Known Contract Failures / Partial Inputs
----------------------------------------
Do not try to fix these by inventing reservoir math:

- Motion-vector depth delta:
  Reference temporal uses motionVector.z for expected previous depth. The
  primary-prepass exporter now stores this z delta for valid motion vectors;
  use debug view 12 to compare depth/normal rejection stability during motion.

- Emissive previous-frame identity:
  A conservative previous emissive buffer and remap now exists, but bright
  green in debug view 12 means remap failure remains.

- Material/surface similarity:
  Yellow object-local strobes mean the previous surface exists but the material
  or surface compatibility test rejected it.

- Duplication map:
  RAB_GetDuplicationMapCount returns bounds-aware 0. No sample-ID fill pass or
  duplication-map count UAV is bound yet.

- Environment lighting:
  Environment MIS/pdf callbacks are partial. Direction-to-UV is deterministic,
  but pdf is still 0 and environment lights are not part of
  RAB_GetCurrentLightCount.

- Denoiser callbacks:
  Reconnection and last-bounce denoiser callbacks are empty.

- BRDF model:
  RAB BRDF is diffuse-only. Specular/glossy reservoir behavior is not
  reference-complete.

- Custom DI/GI debug reservoirs:
  RestirPTDiReservoirs and RestirPTGiReservoirs are local debug buffers. They
  are not the RTXDI reference reservoir chain selected by
  RestirPTParams.bufferIndices.
