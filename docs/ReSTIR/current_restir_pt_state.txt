Recent Mode 56 / DLSS RR CVar Quick Reference
---------------------------------------------
Reference-first rule:

- Do not invent ReSTIR PT reservoir math or processing when local reference
  implementations already define the behavior. Temporal/spatial reuse,
  neighbor validation, target-function evaluation, normalization, bias
  correction, and final shading must be mapped from RTXDI / RTX Remix concepts
  into rbdoom-local passes and buffers.
- If a reference step does not fit this engine directly, document the exact
  rbdoom-specific mismatch before implementing a local adaptation. Keep such
  adaptations debug-scoped until visually accepted.

Core mode-56 route:

- `r_pathTracingRestirPTPrimarySurfacePrepass`
  Default 1. Uses the standalone primary-surface producer and standalone
  combined resolve path so mode 56 avoids the old combined monolithic shader.
  Set 0 only when deliberately testing the legacy fallback.
- `r_pathTracingRestirPTPassDump`
  Set 1 to print the current ReSTIR PT pass plan once. This is the easiest way
  to confirm `primaryPrepass`, standalone producer/resolve flags, sparse
  dispatch sizes, reflection producer state, RR guide debug state, and buffer
  indices.
- `r_pathTracingRestirPTGpuTimingDump`
  Set 1 to print one frame of per-dispatch ReSTIR PT GPU timestamp timings.
- `r_pathTracingNsightGpuMarkers`
  Set 1 before an Nsight capture to add GPU debug markers around RT/PT
  dispatches.
- `r_pathTracingRestirPTCombinedSpatialRadius`
  Legacy direct-spatial radius control. Do not use this CVar as evidence that
  DI/direct temporal or spatial reuse is complete.

Sparsity:

- `r_pathTracingRestirPTRaySparsity`
  Legacy sparse update control for raw DI/direct diagnostics. 1 disables
  sparsity; 2-8 were used for old direct-domain experiments and should not be
  treated as completed DI reservoir reuse.
- `r_pathTracingRestirPTGiRaySparsity`
  Interleaved sparse update rate for staged ReSTIR GI initial-reservoir producer
  rays in modes 53-56. 1 disables sparsity; 2-8 are roughly 1/N, not 1/N^2.

Lighting and visibility:

- `r_pathTracingRestirPTVisibilityPolicy`
  0 final/preview visibility only, 1 selected NEE sample visibility in the
  producer, 2 strict proposal-stream visibility. Policy 0 is the default
  validation path.
- `r_pathTracingRestirPTAnalyticLightTrials`
  Current-frame analytic Doom light proposal trials for ReSTIR PT NEE.
- `r_pathTracingToyLightScale`
  Now affects ReSTIR PT analytic-light preview through
  `r_pathTracingAnalyticLightIntensityScale * r_pathTracingToyLightScale`.
- `r_pathTracingToyEmissiveScale`
  Emissive source scale used by the RAB emissive light loader and primary
  material emissive.

Reflections:

- `r_pathTracingRestirPTReflectionMode`
  0 off, 1 sharp traced reflection preview for low-roughness/specular primary
  surfaces, 2 rough diagnostic preview. The current primary-prepass route uses
  the separate traced reflection producer and composites `RestirPTReflectionOutput`.
- `pathTraceToggleCrosshairZeroRoughness`
  Console command, not a CVar. Arms a one-shot crosshair pick on the next RT
  smoke frame and toggles a material override to roughness 0 / high F0 for
  reflection validation. Lowercase `pathtracetogglecrosshairzeroroughness` is
  also registered.

DLSS Ray Reconstruction bridge:

- `r_pathTracingDLSSRRGuideDebugView`
  0 off, 1 albedo, 2 encoded normal, 3 roughness, 4 depth, 5 hit distance,
  6 motion-vector mask/source category, 7 reset/disocclusion mask, 8
  specular albedo/F0, 9 RR input HDR preview.
- `r_pathTracingDLSSRRProbe`
  Default 1 and `CVAR_INIT`. Initializes the experimental Streamline DLSS/RR
  bridge at renderer startup and prints support. Change before startup or
  restart the renderer.
- `r_pathTracingDLSSRR`
  Experimental DLSS Ray Reconstruction evaluation gate. Use with mode 56,
  `r_pathTracingRestirPTPrimarySurfacePrepass 1`, and guide debug view 0. This
  now forces RR motion-vector export for the evaluated frame even when the
  standalone motion-vector debug CVar is 0.
- `r_pathTracingDLSSRRColorBuffersHDR`
  Default 1. Feeds Streamline `colorBuffersHDR`; the current Streamline/NGX RR
  path rejects the SDR-tagged option with `eErrorNGXFailed`.
- `r_pathTracingDLSSRRPreExposure`
  Default 1.0. Streamline `preExposure` diagnostic knob.
- `r_pathTracingDLSSRRExposureScale`
  Default 1.0. Streamline `exposureScale` diagnostic knob.
- `r_pathTracingDLSSRRSharpness`
  Default 0.0. Streamline RR sharpness option, clamped 0..1.
- `r_pathTracingDLSSRRVerbose`
  Extra Streamline/DLSS/RR requirement, tag, and evaluate logging.

The heavy ReSTIR path was separated from the core path tracing RT library.
Current routing keeps core mode 18 away from RAB/ReSTIR includes unless a
ReSTIR mode is selected.

  Legacy fallback for mode 56 combined direct spatial-reservoir plus

- pathtrace_restir_reflection_producer.rt.hlsl
  Mode-56 traced reflection producer. It consumes `PrimarySurfaceHistoryCurrent`,

  `RestirPTReflectionOutput` (`u47`) for the standalone combined resolve. Pass

- pathtrace_restir_indirect_initial_producer.rt.hlsl
  Standalone-ish mode 56 GI initial producer used when the primary prepass is
  active. It still includes `pathtrace_smoke.rt.hlsl` for hit/material plumbing,
  but compiles a producer-only raygen and avoids loading the much larger
  `pathtrace_smoke_restir_initial.rt.bin` library for the mode-56 GI prepass.
  Pass dumps report this route with `giInitialStandalone=1`.
- pathtrace_restir_direct_temporal_producer.rt.hlsl
  Legacy-named mode 56 direct producer wrapper. The name is scaffolding; do not
  treat it as completed DI temporal reuse.
- pathtrace_restir_direct_spatial_reservoir_producer.rt.hlsl
  Legacy-named mode 56 direct producer wrapper. The name is scaffolding; do not
  treat it as completed DI spatial reuse.

Modes 31, 32, 33, 50, 51, 56, and the mode 18 hybrid force-specialize the debug mode at compile time through
RB_PT_FORCE_DEBUG_MODE so the compiler can drop unrelated paths. User testing
reported roughly 13 second first-load stalls for each specialized ReSTIR mode,
down from roughly 197 seconds for the unspecialized ReSTIR library.

The primary-surface producer is deliberately not a macro wrapper over
`pathtrace_smoke.rt.hlsl`. It has its own raygen/hit/miss shader library so
future production passes can grow outside the mode-56 monolithic preview path.
When `r_pathTracingRestirPTPrimarySurfacePrepass 1` is enabled, the GI initial
producer, legacy-named direct producer wrappers, and mode 56 final combined
consumer can all consume `PrimarySurfaceHistoryCurrent`. Pass dumps report this
with `giConsumesPrimary`, `directConsumesPrimary`, and `finalConsumesPrimary`.
The same standalone primary-surface producer now exports the first DLSS Ray
Reconstruction guide scaffold: albedo `u48`, normal+roughness `u49`, depth
`u50`, hit distance `u51`, reset/debug mask `u52`, and specular albedo/F0
`u53`. The standalone combined resolve also writes `PathTraceRRInputColor`
`u54`, an `RGBA32_FLOAT` raw linear HDR input color for Streamline
`kBufferTypeScalingInputColor`; `SmokeOutput` stays the tonemapped mode-56
preview/debug image. Existing motion-vector outputs remain `u39/u40`.
The experimental Streamline DLSS RR bridge can now evaluate mode 56 when the
SDK reports support, but the guide data is still provisional. The bridge plan
lives in `docs/ReSTIR/dlss_rr_bridge_plan.txt`.
Use `r_pathTracingDLSSRRGuideDebugView` in mode 56 to inspect those guide
outputs through the standalone resolve: 1 albedo, 2 normal, 3 roughness, 4
depth, 5 hit distance, 6 motion-vector mask/source category, 7
reset/disocclusion mask, 8 specular albedo/F0, 9 RR input HDR preview.
When `r_pathTracingMotionVectorExport 1` is enabled, the standalone
primary-surface producer writes the bridge motion-vector/mask UAVs too, using
static previous snapshots, retained skinned previous positions, and continuous
rigid previous transforms. Mode 56 now also forces this export while primary
surface history is enabled because the DI/GI temporal debug reservoirs consume
`u39/u40` plus the reset mask for previous-pixel lookup. The reset mask is
written even when motion-vector export is disabled, so history validity can be
inspected without feeding `u39/u40` consumers. Mode 56 builds the skinned
previous-position scaffold for this guide export even when
`r_pathTracingGpuSkinning` is `0`; that CVar still only gates the diagnostic GPU
skinning compute path. The primary-prepass mode-56 path no longer depends on the
monolithic shader for these guide outputs.
The primary-prepass motion-vector export now follows the RTXDI PT temporal
contract shape: `u39` is `RGBA16F`, with xy pixel motion and z equal to expected
previous linear depth minus current surface linear depth. DI/GI temporal reuse
still uses the exported motion mask, reset mask, and previous-surface
similarity checks, but the reference-chain debug view can now classify
depth/normal rejection against `surface.linearDepth + motionVector.z` instead
of silently falling back to current depth for valid motion vectors. A local
debug-scoped fallback allows same-pixel temporal reuse for ordinary rigid
surfaces whose only reset reason is missing object-motion data, but only after
the current and previous same-pixel primary surfaces pass the existing
material/normal/roughness similarity check.
When `r_pathTracingDLSSRR 1` evaluates RR, motion-vector export is enabled for
that frame. The bitfield u52 texture remains the human-readable reset/debug
mask. A previous u53 float disocclusion-guide experiment was backed out because
the raw mode-56 primary-prepass path showed a geometry-disappearing regression
even with RR evaluation disabled; u53 is now reserved for the material-only
specular-albedo/F0 guide. u54 is reserved for the RR input color, not for
geometry/disocclusion metadata.
Mode 56 keeps the legacy dynamic rigid fallback even when a routed rigid TLAS
route exists. A short-lived change that pruned the fallback made a known
routed-geometry disappearance visible in mode 56; keep this conservative until
the routed-rigid missing-geometry case is fixed at the source.

Direct and GI sparsity now use real sparse producer dispatch widths instead of
only full-domain dispatch plus inactive-pixel early return. Pass dumps report
the reservoir domain and producer dispatch domain separately through
`directDomain`, `directDispatch`, and `giDispatch`. The active pattern is still
the existing diagonal `1/rate` pattern, not a `1/rate^2` block pattern.

The default CMake path-tracing shader target now builds the mode-56-required
set and skips legacy monolithic ReSTIR diagnostic libraries that are not needed
by the primary-prepass mode-56 path. Re-enable those old diagnostic rebuilds
with `PATH_TRACING_BUILD_RESTIR_LEGACY_DIAGNOSTICS=ON` when deliberately
working on modes such as 26-33 or 50-51 attribution/shading consumers.
The default mode-56 set now includes the traced reflection producer because it
is part of the current primary-prepass resolve graph when
`r_pathTracingRestirPTReflectionMode` is enabled.

ReSTIR PT NEE visibility is now policy controlled:

  r_pathTracingRestirPTVisibilityPolicy

Policy 0 keeps proposal and selected NEE reservoir recording unshadowed and
relies on final/preview visibility only. This is the default. The standalone
combined resolve now traces selected direct and GI preview reservoir visibility
when preview visibility is enabled, so obvious selected-sample leaks can be
blocked without rejecting energy during producer RIS. Policy 1 skips
per-proposal shadow rays but traces the selected NEE sample before recording it
into the reservoir; keep it diagnostic-only for now because rejecting one
occluded selected sample after RIS can darken the whole preview. Policy 2
restores the strict diagnostic behavior that traces visibility during proposal
streaming.

Mode 56 scale calibration:

- `r_pathTracingToyEmissiveScale` is the emissive source scale used by the
  RAB emissive light loader and primary-surface material emissive.
- ReSTIR PT analytic-light preview intensity now uses
  `r_pathTracingAnalyticLightIntensityScale * r_pathTracingToyLightScale`,
  clamped to 0..16, so the mode-18 toy light scale also affects ReSTIR PT
  analytic source brightness. The pass dump prints `toyLight`, `toyEmissive`,
  and the effective `analyticScale`.
- Changing either toy scale invalidates the ReSTIR reservoir dispatch signature,
  so stale direct/GI reservoirs should not mask the adjustment.
- RAB Doom analytic samples no longer hard-cut lighting at the authored Doom
  radius. The radius remains a source-size/proposal-domain hint, while sampled
  analytic sphere radiance is evaluated as a light sample without a game-radius
  emission boundary. This is necessary because compensating a hard cutoff with
  high intensity made shadows and falloff read as harsh stencils.

Skybox/wobblesky cube materials are forced to black fallback albedo and rejected
from the material emissive path. Doom 3 sometimes represents sky as large
geometry boxes, and earlier path-tracing material fallback made those boxes act
like unintended GI/light sources in dark areas.
The specific Doom material `textures/sfx/black` is also forced to an absorbing
black material: no diffuse/normal/specular/emissive texture binding, no alpha or
decal behavior, black fallback albedo, and forced fallback shading. It must not
inherit the material-id debug color or the `textures/sfx/black_2` stage.

Mode 56 with `r_pathTracingRestirPTPrimarySurfacePrepass 1` now uses the
standalone combined resolve library when available. Pass dumps report this with
`finalResolve=1`. If the standalone resolve or primary-surface producer cannot
load, mode 56 falls back to the older monolithic combined shader.
Set `r_pathTracingRestirPTPrimarySurfacePrepass 0` only when deliberately
testing that legacy fallback path.

Mode 56 reflection preview is controlled by:

  r_pathTracingRestirPTReflectionMode

Mode 0 disables it. Mode 1 only targets low-roughness/specular primary surfaces
and is the default validation shape for sharp reflections. Mode 2 is a rough
diagnostic mode that broadens the roughness threshold but should not be treated
as the final rough-reflection reconstruction path. With primary prepass and the
standalone resolve active, mode 56 now uses a separate traced reflection
producer and composites its `RestirPTReflectionOutput` texture. This can reflect
off-screen scene geometry because it traces the RT acceleration structure, but
it is still a single-bounce debug/preview pass and does not yet have
temporal/spatial reflection reservoirs.

For validation in Doom materials that are not naturally glossy, the console
command `pathTraceToggleCrosshairZeroRoughness` arms a one-shot crosshair pick
on the next RT smoke frame. The picked material id toggles a PT material-table
override that forces roughness to 0 and raises F0 to at least 0.85 in the
primary-surface producer and monolithic fallback. Running the command on the
same material again disables the override.

Primary-surface records now pack real material flags in
`materialAndSurface.z`. Earlier prepass records accidentally wrote
`surface.flags` there, which is triangle/class flag data. That can break
material-dependent behavior when mode 56 consumes primary history.
The triangle/class flags are preserved separately in `header.w` so standalone
resolve passes can still identify translucent subtypes such as GUI screens,
glass, smoke particles, and portal windows.
The standalone primary-surface producer now uses the same emissive sampling
contract as the monolithic shader, and the standalone mode 56 resolve adds the
packed visible emissive radiance to fallback/preview output so visible emissive
materials do not disappear when `primaryPrepass=1`.
The standalone producer also ports the monolithic any-hit transparency rejection
rules for GUI alpha, smoke-particle dithering, glass/portal fallback dithering,
additive decals, and filter decals. These stochastic rejects are required for
visible translucent surfaces when mode 56 consumes the standalone primary
history. This is still primary-hit parity, not full layered translucent
composition; if a scene depends on repeated GUI/glass layer compositing, that
needs a separate translucent resolve/fallback plan.

Known Fixes In This Slice
-------------------------
The RAB path tracer must account for RTXDI's PT convention that the primary
surface starts at `bounceDepth = 2`. A previous bridge cap reused the local
secondary-bounce count directly, often producing `maxBridgeBounces = 1`, so the
initial-sampling loop never executed and modes 26/27/32 could only show fallback
material context. The bridge now keeps the primary NEE-capable vertex alive and
only applies the local diffuse/path-depth limits to continuation bounces after
that primary vertex.

The temporal preview now feeds RTXDI the sub-pixel combined geometry motion
computed by the existing static/skinned/routed-rigid motion helpers. This
replaces the older integer previous-pixel delta in the temporal-resampling
runtime parameters and is intended to reduce routed-rigid edge flicker during
camera movement without changing candidate selection.

Mode 32 no longer rejects temporal reuse before RTXDI gets a chance to search
its previous-frame neighborhood. The earlier wrapper required the exact
reprojected previous pixel to have a valid matching surface and reservoir before
calling RTXDI_PTTemporalResampling. That defeated RTXDI's neighbor search and
made static, skinned, and routed-rigid edges flash during movement. The wrapper
now only computes motion/depth hints and lets RTXDI perform its own surface and
reservoir validation.

The RTXDI temporal/spatial neighbor thresholds are exposed for modes 31-33 and
50-51:

  r_pathTracingRestirPTTemporalDepthThreshold
  r_pathTracingRestirPTTemporalNormalThreshold

The normal threshold defaults to 0.35 instead of RTXDI's stock 0.6 because Doom
3 tangent-space normal maps and legacy bases can make shading-normal validation
too twitchy on particular object sides. Tighten it when looking for ghosting;
loosen it when stable motion sources still show small edge/object flashes.

Mode 50 adds the first spatial-reservoir shading preview. It uses RTXDI's
TemporalAndSpatial buffer-index contract, builds the current initial reservoir,
runs temporal reuse, then spatially resamples the temporal output before using
the same rough lighting preview path as mode 32. The branch does not have a
host-bound RTXDI neighbor-offset buffer yet, so modes 50-51 use a shader-local
32-entry offset table for these debug slices. Spatial samples and radius are
controlled by:

  r_pathTracingRestirPTSpatialSamples
  r_pathTracingRestirPTSpatialRadius

Mode 56 uses `r_pathTracingRestirPTCombinedSpatialRadius` instead of the shared
mode 50/51 radius. Its default is 1 because GPU timing isolated
`DirectSpatialReservoirProducer` as the dominant frame-time cost and runtime
testing showed radius 1 recovering roughly 20-40% FPS, while radius 4 and wider
had similar cost.

The spatial depth/normal validation thresholds currently reuse the temporal
threshold CVars above. The disocclusion boost sample count is clamped to the
same sample count as `r_pathTracingRestirPTSpatialSamples` to avoid a hidden
8x work spike during early Vulkan/device-removal testing.

Modes 50-51 still have legacy debug staging names in code and captures, but
DI/direct must be treated as raw reservoir output on this branch. Do not read
the temporal/spatial labels as evidence that DI has completed reservoir reuse.
Future DI reuse should duplicate or adapt the accepted GI reservoir path.

Modes 50-51 and the mode 18 ReSTIR direct-lighting hybrid have old
direct-resolution scaffolding through:

  r_pathTracingRestirPTDirectResolutionScale

The CVar and mapping language are historical scaffolding. They should not be
treated as completed DI temporal/spatial processing. Keep any future direct
domain explicit, but rebuild it from the accepted GI reservoir path instead of
assuming the current direct labels are already correct.

The old direct scaffolding also has default-off ray sparsity through:

  r_pathTracingRestirPTRaySparsity

Values 2-8 are legacy diagnostics over the raw direct/DI reservoir path. Do not
use this control as proof that DI temporal/spatial reuse is complete.

The staged GI initial path has a separate default-off sparse control:

  r_pathTracingRestirPTGiRaySparsity

Values 2-8 update roughly one out of N full-resolution GI initial reservoir
pixels per frame. Inactive GI producer pixels return before tracing. Visible
modes 53-56 keep their full-resolution primary material context, but load the
nearest active GI initial reservoir representative.

Mode 51 is the source-attribution diagnostic for the current raw direct/DI
reservoir output. It uses the mode 33 convention: cyan/blue for Doom analytic
lights, orange for emissive triangles, green for non-NEE/indirect reservoirs,
and red for invalid/unknown light data. It must not be read as proof that
direct temporal/spatial reuse is complete.

Modes 53-55 are the first indirect/GI reservoir diagnostics. They intentionally
reuse the initial RTXDI PT reservoir buffer and initial ReSTIR shader library
instead of adding a new resource family in this slice. They now run as a staged
initial-GI domain: an initial producer dispatch writes completed indirect
reservoirs, then the visible mode 53/54/55 pass consumes those reservoirs. In
these modes the RAB path tracer skips primary-surface NEE, so the first usable
candidates come from the secondary/indirect path domain: secondary NEE,
emissive hits, or non-NEE path radiance. Mode 53 shows reservoir metadata/heat,
mode 54 shades the rough indirect reservoir contribution, and mode 55 colors
source/path attribution. These modes are debug-only and have no temporal reuse,
spatial reuse, or mode 18 integration yet.

Modes 53-55 force the bridge path tracer to allow one secondary bounce and raise
the RTXDI initial path-length budget to 4 inside the debug path. RTXDI starts the
primary surface at bounceDepth 2 and records secondary-surface NEE as pathLength
4; leaving the default initial max at 3 accepts primary NEE but rejects all
secondary analytic NEE, making the GI views look like emissive-only/non-NEE
reservoirs. This override is debug-only and does not change mode 18.

Mode 56 is the combined direct + GI rough preview. Normal mode 56 shades raw
DI/direct reservoir output with GI initial output. The 01C GI temporal
candidate exists only behind `r_pathTracingRestirPTGiDebugView 2`; it does not
feed normal mode 56 yet. DI and GI are separate domains, and DI has no completed
temporal/spatial reuse.

An experimental default-off primary-surface reuse path is available through:

  r_pathTracingRestirPTPrimarySurfacePrepass

Default 0 preserves the current working behavior. Historical docs described this
as direct temporal/spatial producer reuse, but that overstated the DI state.
Treat it as primary-surface sharing scaffolding around raw DI/direct reservoir
output.
The mode 56 final/full-output consumer now has a surface-based combine path and
can load current primary-surface history instead of rebuilding a surface from a
fresh primary payload when this CVar is enabled and GI sparsity is 1. This path
is documented in `docs/ReSTIR/primary_surface_prepass_plan.txt` and should stay
opt-in until runtime testing confirms it does not produce the prior light-blue
clear-color failure.

Runtime validation on 2026-05-15 reported that the primary-surface prepass path
works visually but has no meaningful performance impact. Do not interpret this
as validation of completed DI temporal/spatial processing. The next direct-side
reuse work should wait until the GI reservoir path is accepted and can be
duplicated/adapted for DI.

User validation reported that mode 56 launches and the combined preview is
visibly working. Two issues are known:
- Performance is low in some locations, and increasing direct/GI sparsity did
  not visibly reduce the cost even though sparsity visibly changes the image.
  User Nsight validation reports the bad cases are GPU-bound below 60 FPS:
  `vkCmdTraceRays` dominates frame time, while RT core utilization is only
  roughly 5-10%. Treat the next performance pass as GPU shader/location
  profiling, not CPU setup or raw traversal ray-count profiling.
- Mode 56 can emit legacy-labeled `vkCmdTraceRays` events such as GI initial
  producer, direct temporal producer, direct spatial producer, and final
  combined consumer. Those direct labels are scaffolding and should not drive
  DI temporal/spatial tuning until DI has been rebuilt from the accepted GI
  reservoir path.
- Some dark base-game areas appear to have open roofs that are hidden from
  skybox lighting by non-geometry/light-projection rules in the raster path.
  With GI enabled they can become much too bright. Do not solve this by
  globally disabling environment/GI; investigate the Doom visibility/light-mask
  contract for sky/environment contribution.

Emissive proposal sampling now has an explicit selector boundary and CDF-backed
proposal table. The public shader entry point remains
`SelectSmokeWeightedEmissiveTriangle`; its legacy O(N) linear scan is preserved
as a fallback/A-B path, while the default path uses the CPU-built
`PathTraceEmissiveDistribution` SRV through
`r_pathTracingEmissiveDistribution 1`. This preserves the full authored
`SmokeEmissiveTriangles` universe for
shading, source attribution, visibility self-ignore, and RAB light-info
reconstruction. The CDF implementation built successfully in the
win64-pt-dev-release lane and was deployed to the prepared game folder on
2026-05-14. User runtime testing on 2026-05-14 reported the result looked fine
in the target ReSTIR/PT modes, so the first visual smoke check is accepted. The
same follow-up runtime testing reported roughly a 50% FPS boost in a previously
laggy spot. Treat the CDF selector as a validated first optimization, while
still using Nsight A/B timing with the distribution forced off versus on if a
per-pass attribution is needed.

Recommended Next Milestone
--------------------------
Stop treating mode 56 as the production performance path. Use it as the
validation preview while moving toward the real pass structure:

1. Full-resolution primary surface / G-buffer anchor.
2. Direct lighting producer in its own scaled/sparse domain.
3. GI/indirect producer in its own scaled/sparse domain.
4. Full-resolution resolve/composite that reconstructs from those domains.

Do this before adding reflections, temporal GI reuse, spatial GI reuse, DLSS RR,
or mode 18 GI integration. Continue using mode 50, 51, 54, 55, and 56 as
regression checks during the split. Keep
`r_pathTracingRestirPTTemporalAnalyticNeeReuse 0` while testing.

Mode 33 analytic-source note:

The Doom analytic light upload can contain deep portal candidates even when the
first printed candidate records are depth 0/1; the dump is sorted by portal
depth and capped by `r_pathTracingDoomLightDumpMax`. ReSTIR PT analytic NEE now
samples the full uploaded analytic domain uniformly. Do not bias proposals
toward the first compact-list entries unless a real spatial/local light proposal
structure replaces compact portal-depth ordering; that shortcut caused hard
source-attribution discontinuities across tiny camera-area movements.

Mode 51 spatial acceptance diagnostic:

`r_pathTracingRestirPTSpatialDiagnosticView 1` changes mode 51 from source
attribution to spatial acceptance/source-change status. Color key: gray temporal
input invalid, red spatial output invalid, blue no spatial neighbor contribution
detected/kept temporal, green spatial neighbor contribution detected with the
same packed source, cyan changed to Doom analytic, orange changed to emissive,
magenta changed to non-NEE, yellow changed to unknown/invalid source. In the
split producer/consumer path, contribution is inferred from the spatial
reservoir `M` value growing beyond the temporal reservoir `M`; the consumer no
longer sees RTXDI's internal `spatialResampled` bool. Use this when
`r_pathTracingRestirPTSpatialSamples` or `r_pathTracingRestirPTSpatialRadius`
appear visually inert in mode 50.

`r_pathTracingRestirPTSpatialDiagnosticView 2` keeps mode 51 on a
temporal-vs-spatial source comparison but adds a mode-50-style local replay
check for changed Doom analytic or emissive NEE sources. Yellow means the
spatial reservoir changed to a valid packed NEE source, but that source did not
reconstruct usable local rough-preview lighting at the current surface. Use this
view to separate expected spatial source changes from likely source leakage
across depth/object edges before treating TemporalAndSpatial as integration-ready.

Rough Preview Control
---------------------
Mode 32 is the rough temporal lighting preview. Its exposure is controlled by:

  r_pathTracingRestirPTPreviewExposure

Default is 1. The value is clamped to 0..16 and only affects the mode 32 and
mode 50 preview mapping. It does not change mode 18 and does not change
reservoir selection.

When `TargetFunction * WeightSum` is effectively black but the temporal
reservoir still contains a packed NEE light, mode 32 has a preview-only fallback
that reconstructs the selected Doom analytic or emissive light sample at the
current surface. This is intentionally not final shading; it exists so the rough
preview can show whether reservoir light selection is producing usable visible

- `r_pathTracingRestirPTTemporalReservoirReuse 0` is clean everywhere in the
  tested rooms, proving current-frame NEE sampling and analytic-light upload are
  sound enough for rough testing.
- With reservoir reuse enabled, a robot-train/strobe sequence in the same portal
  could keep nearby lights correct while the strobe animation was active, then
  make them break or flicker as soon as the source stopped in an offscreen hole.
  `com_pause 1` froze the symptom and `r_skipUpdates 1` did not, which points at
  shader/register time animation and light-domain state, not renderer update
  churn.
- The current safety path keeps NEE temporal reuse disabled by default through
  `r_pathTracingRestirPTTemporalAnalyticNeeReuse 0`. Despite the historical
  CVar name, the guard now rejects prior-frame NEE reservoirs from both Doom
  analytic lights and emissive/strobe sources before temporal combine. Non-NEE
  temporal reuse remains enabled.
- The proper future fix is stable light-domain identity, current/previous
  remapping, and per-light radiance/state validation so NEE temporal reuse can
  be restored without stale direct-light pollution.
