Current routing keeps core mode 18 away from RAB/ReSTIR includes unless a
ReSTIR mode is selected.

Modes 31, 32, 33, 50, 51, 56, and the mode 18 hybrid force-specialize the debug mode at compile time through
RB_PT_FORCE_DEBUG_MODE so the compiler can drop unrelated paths. User testing
reported roughly 13 second first-load stalls for each specialized ReSTIR mode,
down from roughly 197 seconds for the unspecialized ReSTIR library.

Known Fixes In This Slice
-------------------------
The RAB path tracer must account for RTXDI's PT convention that the primary
surface starts at `bounceDepth = 2`. A previous bridge cap reused the local
secondary-bounce count directly, often producing `maxBridgeBounces = 1`, so the
initial-sampling loop never executed and modes 26/27/32 could only show fallback
material context. The bridge now keeps the primary NEE-capable vertex alive and
only applies the local diffuse/path-depth limits to continuation bounces after
that primary vertex.

The temporal preview now feeds RTXDI the sub-pixel combined geometry motion
computed by the existing static/skinned/routed-rigid motion helpers. This
replaces the older integer previous-pixel delta in the temporal-resampling
runtime parameters and is intended to reduce routed-rigid edge flicker during
camera movement without changing candidate selection.

Mode 32 no longer rejects temporal reuse before RTXDI gets a chance to search
its previous-frame neighborhood. The earlier wrapper required the exact
reprojected previous pixel to have a valid matching surface and reservoir before
calling RTXDI_PTTemporalResampling. That defeated RTXDI's neighbor search and
made static, skinned, and routed-rigid edges flash during movement. The wrapper
now only computes motion/depth hints and lets RTXDI perform its own surface and
reservoir validation.

The RTXDI temporal/spatial neighbor thresholds are exposed for modes 31-33 and
50-51:

  r_pathTracingRestirPTTemporalDepthThreshold
  r_pathTracingRestirPTTemporalNormalThreshold

The normal threshold defaults to 0.35 instead of RTXDI's stock 0.6 because Doom
3 tangent-space normal maps and legacy bases can make shading-normal validation
too twitchy on particular object sides. Tighten it when looking for ghosting;
loosen it when stable motion sources still show small edge/object flashes.

Mode 50 adds the first spatial-reservoir shading preview. It uses RTXDI's
TemporalAndSpatial buffer-index contract, builds the current initial reservoir,
runs temporal reuse, then spatially resamples the temporal output before using
the same rough lighting preview path as mode 32. The branch does not have a
host-bound RTXDI neighbor-offset buffer yet, so modes 50-51 use a shader-local
32-entry offset table for these debug slices. Spatial samples and radius are
controlled by:

  r_pathTracingRestirPTSpatialSamples
  r_pathTracingRestirPTSpatialRadius

The spatial depth/normal validation thresholds currently reuse the temporal
threshold CVars above. The disocclusion boost sample count is clamped to the
same sample count as `r_pathTracingRestirPTSpatialSamples` to avoid a hidden
8x work spike during early Vulkan/device-removal testing.

Modes 50-51 now have explicit debug staging boundaries. Dispatch first runs the
temporal producer shader to write current primary surfaces and temporal
reservoirs, inserts UAV barriers, then runs a spatial-reservoir producer shader,
inserts another reservoir UAV barrier, and finally runs the mode 50/51 shading
or attribution consumer. Spatial consumers read the completed spatial output
reservoir instead of regenerating it. RTXDI's spatial pass expects completed
current-frame neighbor surfaces and temporal reservoirs; doing all stages in one
raygen dispatch left the diagnostic 100% blue because neighbor inputs were not
globally available yet. This is still debug-only staging, not the final
production pass graph.

Modes 50-51 and the mode 18 ReSTIR direct-lighting hybrid now have a first
explicit direct-lighting reservoir resolution domain through:

  r_pathTracingRestirPTDirectResolutionScale

The CVar defaults to 1.0 and clamps to 0.25..1.0. Temporal/spatial producer
prepasses dispatch at the scaled direct size and map each direct pixel to a
representative full-resolution primary pixel. RAB surface callbacks map
direct-domain neighbor pixels back to the full-resolution primary-surface
history. Mode 18, mode 50, and mode 51 final consumers still dispatch at full PT
output size and map each full pixel to the nearest completed direct reservoir
pixel. Visible consumers must never treat the scaled direct-domain flag as their
output dispatch domain; they stay full-resolution and only use explicit
full-to-direct reservoir lookup. Mode 32 remains a full-resolution
single-dispatch temporal preview for now. The packed reservoir UAV binding is
still full-output capacity in this slice; RTXDI reservoir parameters and
dispatch dimensions restrict active ReSTIR direct reads/writes to the scaled
domain.

The staged direct path also has default-off ray sparsity through:

  r_pathTracingRestirPTRaySparsity

Values 2-8 update roughly one out of N direct-domain reservoir pixels per frame.
Inactive sparse producer pixels return before tracing. Spatial neighbor lookup
and full-resolution consumers snap to active representative reservoir pixels,
and temporal previous-frame lookup snaps to the previous sparse phase so it does
not deliberately read inactive previous slots.

The staged GI initial path has a separate default-off sparse control:

  r_pathTracingRestirPTGiRaySparsity

Values 2-8 update roughly one out of N full-resolution GI initial reservoir
pixels per frame. Inactive GI producer pixels return before tracing. Visible
modes 53-56 keep their full-resolution primary material context, but load the
nearest active GI initial reservoir representative.

Mode 51 is the spatial source-attribution diagnostic. It runs the same
initial->temporal->spatial reservoir path as mode 50, then colors the selected
spatial reservoir source using the mode 33 convention: cyan/blue for Doom
analytic lights, orange for emissive triangles, green for non-NEE/indirect
reservoirs, and red for invalid/unknown light data. It exists to catch source
bleeding or unexpected source loss after spatial reuse before final integration.

Modes 53-55 are the first indirect/GI reservoir diagnostics. They intentionally
reuse the initial RTXDI PT reservoir buffer and initial ReSTIR shader library
instead of adding a new resource family in this slice. They now run as a staged
initial-GI domain: an initial producer dispatch writes completed indirect
reservoirs, then the visible mode 53/54/55 pass consumes those reservoirs. In
these modes the RAB path tracer skips primary-surface NEE, so the first usable
candidates come from the secondary/indirect path domain: secondary NEE,
emissive hits, or non-NEE path radiance. Mode 53 shows reservoir metadata/heat,
mode 54 shades the rough indirect reservoir contribution, and mode 55 colors
source/path attribution. These modes are debug-only and have no temporal reuse,
spatial reuse, or mode 18 integration yet.

Modes 53-55 force the bridge path tracer to allow one secondary bounce and raise
the RTXDI initial path-length budget to 4 inside the debug path. RTXDI starts the
primary surface at bounceDepth 2 and records secondary-surface NEE as pathLength
4; leaving the default initial max at 3 accepts primary NEE but rejects all
secondary analytic NEE, making the GI views look like emissive-only/non-NEE
reservoirs. This override is debug-only and does not change mode 18.

Mode 56 is the first combined direct + GI rough preview. It keeps the staged
TemporalAndSpatial direct path from mode 50, also runs the staged GI initial
producer used by modes 53-55, then a specialized consumer reads the completed
direct spatial reservoir and the completed indirect/GI initial reservoir
together. It remains debug-only: direct and GI still have separate reservoir
domains, GI has no temporal/spatial reuse, and mode 18 is unchanged.

User validation reported that mode 56 launches and the combined preview is
visibly working. Two issues are known:
- Performance is low, and increasing direct/GI sparsity did not visibly reduce
  the cost even though sparsity visibly changes the image. Treat the next
  performance pass as a CPU-first profiling task; CPU-side ReSTIR supply/setup,
  dispatch submission, pass orchestration, or barrier overhead is now the
  leading suspect rather than raw producer ray count.
- Some dark base-game areas appear to have open roofs that are hidden from
  skybox lighting by non-geometry/light-projection rules in the raster path.
  With GI enabled they can become much too bright. Do not solve this by
  globally disabling environment/GI; investigate the Doom visibility/light-mask
  contract for sky/environment contribution.

Main Gaps
---------
- Spatial reuse is debug-only in modes 50-51; it has temporal, spatial, and
  shading/diagnostic dispatches, but not a production pass graph or
  denoiser-facing output yet.
- Indirect/GI is diagnostic-only in modes 53-56; it is initial-reservoir only
  and deliberately avoids temporal/spatial reuse until source/path identity is
  easier to inspect. Mode 56 only combines the completed GI initial reservoir
  with the completed direct spatial reservoir for a rough preview.
- No production final ReSTIR PT pass yet; mode 18 has only an opt-in primary
  direct-lighting hybrid hook.
- Light identity is still index-fragile across changing emissive/light order.
- RAB visibility remains conservative/preview-oriented.
- Environment lighting is not bridged.
- Reflection/glass/transmission are not integrated into ReSTIR final history.
- Motion vectors are good enough for rough experiments, but ReSTIR consumers
  still need stricter rejection and disocclusion handling.
- High resolution remains risky; temporal previews can device-remove above
  roughly 720p-class output.
- Mode 56 performance is not yet a trustworthy ray-count signal because
  sparsity visibly changes reconstruction but has not shown the expected cost
  reduction.
- Environment/sky contribution does not yet honor all base-game dark-area
  occlusion or light-hiding rules.

Mode 33 analytic-source note:

The Doom analytic light upload can contain deep portal candidates even when the
first printed candidate records are depth 0/1; the dump is sorted by portal
depth and capped by `r_pathTracingDoomLightDumpMax`. ReSTIR PT analytic NEE now
samples the full uploaded analytic domain uniformly. Do not bias proposals
toward the first compact-list entries unless a real spatial/local light proposal
structure replaces compact portal-depth ordering; that shortcut caused hard
source-attribution discontinuities across tiny camera-area movements.

Mode 51 spatial acceptance diagnostic:

`r_pathTracingRestirPTSpatialDiagnosticView 1` changes mode 51 from source
attribution to spatial acceptance/source-change status. Color key: gray temporal
input invalid, red spatial output invalid, blue no spatial neighbor contribution
detected/kept temporal, green spatial neighbor contribution detected with the
same packed source, cyan changed to Doom analytic, orange changed to emissive,
magenta changed to non-NEE, yellow changed to unknown/invalid source. In the
split producer/consumer path, contribution is inferred from the spatial
reservoir `M` value growing beyond the temporal reservoir `M`; the consumer no
longer sees RTXDI's internal `spatialResampled` bool. Use this when
`r_pathTracingRestirPTSpatialSamples` or `r_pathTracingRestirPTSpatialRadius`
appear visually inert in mode 50.

`r_pathTracingRestirPTSpatialDiagnosticView 2` keeps mode 51 on a
temporal-vs-spatial source comparison but adds a mode-50-style local replay
check for changed Doom analytic or emissive NEE sources. Yellow means the
spatial reservoir changed to a valid packed NEE source, but that source did not
reconstruct usable local rough-preview lighting at the current surface. Use this
view to separate expected spatial source changes from likely source leakage
across depth/object edges before treating TemporalAndSpatial as integration-ready.

Rough Preview Control
---------------------
Mode 32 is the rough temporal lighting preview. Its exposure is controlled by:

  r_pathTracingRestirPTPreviewExposure

Default is 1. The value is clamped to 0..16 and only affects the mode 32 and
mode 50 preview mapping. It does not change mode 18 and does not change
reservoir selection.

When `TargetFunction * WeightSum` is effectively black but the temporal
reservoir still contains a packed NEE light, mode 32 has a preview-only fallback
that reconstructs the selected Doom analytic or emissive light sample at the
current surface. This is intentionally not final shading; it exists so the rough
preview can show whether reservoir light selection is producing usable visible
lighting before the final ReSTIR PT shading pass is split out.

Confirmed temporal NEE issue:

- `r_pathTracingRestirPTTemporalReservoirReuse 0` is clean everywhere in the
  tested rooms, proving current-frame NEE sampling and analytic-light upload are
  sound enough for rough testing.
- With reservoir reuse enabled, a robot-train/strobe sequence in the same portal
  could keep nearby lights correct while the strobe animation was active, then
  make them break or flicker as soon as the source stopped in an offscreen hole.
  `com_pause 1` froze the symptom and `r_skipUpdates 1` did not, which points at
  shader/register time animation and light-domain state, not renderer update
  churn.
- The current safety path keeps NEE temporal reuse disabled by default through
  `r_pathTracingRestirPTTemporalAnalyticNeeReuse 0`. Despite the historical
  CVar name, the guard now rejects prior-frame NEE reservoirs from both Doom
  analytic lights and emissive/strobe sources before temporal combine. Non-NEE
  temporal reuse remains enabled.
- The proper future fix is stable light-domain identity, current/previous
  remapping, and per-light radiance/state validation so NEE temporal reuse can
  be restored without stale direct-light pollution.
