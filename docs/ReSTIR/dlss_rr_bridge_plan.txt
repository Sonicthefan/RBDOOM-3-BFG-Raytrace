DLSS Ray Reconstruction Bridge Plan
Date: 2026-05-16

Purpose
-------
ReSTIR PT mode 56 is now useful enough that raw noisy previews are the main
visual blocker. The next renderer milestone is not to tune raw reservoir output
forever, but to make the mode-56 pass graph DLSS Ray Reconstruction-ready.

This document tracks the bridge between the current ReSTIR/PT renderer and a
future DLSS RR SDK integration.

Local References
----------------
Use both local references, but for different reasons:

- `E:\prog\references\dxvk-remix-git`
  - Treat this as the modern DLSS RR shape reference.
  - Relevant concepts: explicit primary depth, primary world/shading normal,
    screen-space motion vector, roughness, hit distance, disocclusion/mask, and
    ReSTIR GI compatibility handling before the RR dispatch.

- `E:\prog\references\Q2RTX-rayreconstruction-master`
  - Treat this as the smaller Vulkan/NGX guide-buffer reference.
  - Relevant concepts: noisy color plus depth, motion, normal, roughness,
    albedo/specular/emissive/material ID, and ray-length/hit-distance inputs.

Do not infer that rbdoom should copy either renderer directly. The current Doom
branch already has a full-resolution primary surface record and a mode-56 pass
graph; the bridge should expose those facts as stable guide buffers first.

Current rbdoom Anchor
--------------------
The primary-surface prepass is the full-resolution reconstruction anchor. Its
`PathTracePrimarySurfaceRecord` already carries:

- world position and view depth,
- geometric normal and roughness,
- shading normal and opacity,
- albedo,
- specular F0,
- emissive radiance,
- previous-position or motion placeholder,
- material/surface identifiers,
- instance/primitive identifiers.

Mode 56 should remain a renderer-debug preview until the guide buffers and
history policy are reliable. The production-like path is:

1. Full-resolution primary surface / guide export.
2. Sparse or lower-rate lighting producers.
3. Full-resolution resolve/composite.
4. DLSS RR or another temporal reconstruction stage.

Task 01: Guide Buffer Scaffold
------------------------------
Status: implemented as the first bridge step.

Frame resources now allocate full-resolution DLSS RR guide UAVs:

- `PathTraceRRGuideAlbedo` at `u48`, `RGBA16_FLOAT`.
- `PathTraceRRGuideNormalRoughness` at `u49`, `RGBA16_FLOAT`.
- `PathTraceRRGuideDepth` at `u50`, `R32_FLOAT`.
- `PathTraceRRGuideHitDistance` at `u51`, `R32_FLOAT`.
- `PathTraceRRGuideResetMask` at `u52`, `R32_UINT`.
- `PathTraceRRGuideSpecularAlbedo` at `u53`, `RGBA16_FLOAT`.
- `PathTraceRRInputColor` at `u54`, `RGBA32_FLOAT`.

The standalone primary-surface producer writes these guide buffers from the
same surface that it stores into `PrimarySurfaceHistoryCurrent`:

- albedo rgb plus opacity in alpha,
- specular albedo/F0 rgb from the resolved material, currently Doom fake-PBR
  `specularF0` and later real PBR F0/specular albedo,
- encoded shading normal rgb plus roughness in alpha,
- current primary linear depth,
- current primary hit distance, currently the same value as linear depth.
- conservative reset/disocclusion reasons for invalid history.

The standalone combined resolve writes `PathTraceRRInputColor` from raw
scene-referred linear HDR direct/GI/emissive radiance before preview tonemapping
or 0..1 saturation. `SmokeOutput` remains the tonemapped mode-56 preview/debug
image. Streamline `kBufferTypeScalingInputColor` must consume `u54`, not the
preview output, or RR raises blacks and flattens contrast.

Existing motion-vector outputs remain:

- `PathTraceMotionVectors` at `u39`, `RG16_FLOAT`.
- `PathTraceMotionVectorMask` at `u40`, `R32_UINT`.

When `r_pathTracingMotionVectorExport 1` is enabled, the standalone
primary-surface producer writes bridge-grade current-to-previous pixel motion
vectors and masks from static previous snapshots, retained skinned previous
positions, and continuous rigid previous transforms. The reset mask is written
even when the motion-vector export CVar is disabled, so mode 56 can inspect
history validity without feeding `u39/u40` consumers.
When `r_pathTracingDLSSRR 1` is enabled in mode 56, motion-vector export is
forced for that frame even if the debug CVar is 0, because RR evaluation needs
the motion-vector texture as an actual input.

Mode 56 primary-prepass/RR-guide export builds the skinned previous-position
scaffold when needed, even if `r_pathTracingGpuSkinning` is `0`. That CVar still
controls the diagnostic GPU skinning compute path; it is no longer required
just to validate mode-56 skinned RR motion guides.

This is intentionally an export scaffold, not an SDK call.

Known Placeholders
------------------
- Hit distance currently equals primary hit depth. This is enough to reserve the
  RR input and debug the resource path, but diffuse/specular lobe hit distances
  must be split later if the RR integration needs per-lobe data.
- Material/surface identity exists in primary-surface history, but there is not
  yet a dedicated material-id texture.
- Motion vectors now cover the primary static/skinned/rigid history bridge, but
  DLSS RR still needs a stricter validation gate before it is trusted as a
  player-facing reconstruction input.
- Diffuse/specular radiance separation is not final. Mode 56 still composites a
  preview color and writes one raw HDR RR input color; future RR integration
  should keep noisy diffuse/specular channels separable where possible.
- Transparent/translucent handling is not final. Stochastic primary hits are
  flagged in the reset mask, but layered glass/transmission needs its own
  policy.

Task 02: Guide Debug Views
--------------------------
Status: implemented for the first bridge views.

Runtime CVars:

- `r_pathTracingDLSSRRGuideDebugView`
  Selects the mode-56 guide/debug output:

Values:

- 0: off.
- 1: albedo.
- 2: encoded normal.
- 3: roughness.
- 4: depth.
- 5: hit distance.
- 6: motion-vector mask and source category.
- 7: reset/disocclusion mask.
- 8: specular albedo/F0.
- 9: RR input HDR preview, tonemapped for inspection only.

- `r_pathTracingDLSSRRProbe`
  Default 1 and `CVAR_INIT`. Initializes the experimental Streamline DLSS/RR
  bridge at renderer startup and prints DLSS/RR support. Set 0 before startup
  to disable the probe.
- `r_pathTracingDLSSRR`
  Experimental DLSS Ray Reconstruction evaluation gate. Use with mode 56,
  `r_pathTracingRestirPTPrimarySurfacePrepass 1`, and
  `r_pathTracingDLSSRRGuideDebugView 0`.
- `r_pathTracingDLSSRRColorBuffersHDR`
  Default 1. Sets Streamline `colorBuffersHDR`; the current Streamline/NGX RR
  path rejects the SDR-tagged option with `eErrorNGXFailed`.
- `r_pathTracingDLSSRRPreExposure`
  Default 1.0. Sets Streamline `preExposure`.
- `r_pathTracingDLSSRRExposureScale`
  Default 1.0. Sets Streamline `exposureScale`.
- `r_pathTracingDLSSRRSharpness`
  Default 0.0. Sets Streamline RR sharpness, clamped 0..1.
- `r_pathTracingDLSSRRVerbose`
  Extra Streamline/DLSS/RR requirement, resource-tag, and evaluate logging.

These views use the standalone mode-56 combined resolve, not the old monolithic
ReSTIR shader. They require the primary-surface prepass path because that pass
is the current guide producer.

The reset mask debug view is deliberately conservative:

- green: reusable history.
- red or orange: rejected previous history or material mismatch.
- yellow: object/skinned/rigid motion unavailable.
- purple: stochastic translucent primary hit.
- black: invalid primary surface.

Further debug views may still be useful for:

- future material-id texture,
- future diffuse/specular separated noisy radiance.

Task 03: History And Reset Contract
-----------------------------------
Status: partially implemented.

The current frame-level reset maps into the Streamline constants reset flag.
Reset sources include:

- output resize,
- scene-resource replacement,
- reservoir dispatch signature changes,
- primary-history invalidation,
- camera cuts,
- object/skinned motion unavailable,
- material/object identity mismatch,
- translucent/stochastic-primary mismatch.

The mask/disocclusion policy should be conservative. Bad history is worse than
underusing history while the renderer is still changing.

Do not add new guide UAV writes to the primary-surface producer casually. A
previous u53 float disocclusion-guide experiment coincided with a raw mode-56
geometry-disappearing regression even with RR evaluation disabled. The current
u53 use is deliberately limited to the material-only specular-albedo/F0 guide.
The current u54 use is not geometry metadata; it is the mode-56 combined
resolve's raw linear HDR RR input color. New geometry, disocclusion, or routing
guide data needs separate validation.

Task 04: SDK Integration
------------------------
The experimental Streamline/NGX bridge is wired into the rbdoom repo, but keep
the SDK layer replaceable. Do not bury the guide/input contract inside the
mode-56 shader work.

The first experimental Streamline evaluation layer now consumes:

- noisy mode-56 raw linear HDR color from `PathTraceRRInputColor` `u54`,
- RR guide albedo,
- RR guide normal/roughness,
- RR guide depth,
- RR guide hit distance,
- motion vectors and mask,
- frame reset data.

The bridge now also sends the mode-56 world-to-camera and camera-to-world
matrices in DLSS RR options so the specular-hit-distance input is not evaluated
against identity transforms.

The SDK integration should be replaceable. Keep the guide-export contract useful
for a non-DLSS denoiser or an internal debug reconstruction path.

Validation
----------
For the current scaffold:

1. Build the PT release/dev lane.
2. Copy `RBDoom3BFG.exe` to the prepared game folder.
3. Copy `base\renderprogs2` after shader changes.
4. Run mode 56 with `r_pathTracingRestirPTPrimarySurfacePrepass 1`.
5. Confirm the frame-resource log reports:
   `RT DLSS RR guide scaffold ... u48 ... u54`.
6. Confirm mode 56 visuals remain unchanged apart from expected compile/load
   churn from the touched primary-surface producer shader.
7. Use debug view 6 to inspect valid motion source categories, debug view 7
   to inspect conservative reset reasons, and debug view 9 to inspect the
   tonemapped RR input HDR preview.
