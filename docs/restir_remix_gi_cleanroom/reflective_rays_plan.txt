Clean ReSTIR GI Reflective Rays Plan
====================================

Purpose
-------

The current GI lane evaluates diffuse + specular receiver lobes for the chosen
GI reservoir sample, but the producer still traces the original diffuse-style
secondary ray. That gives specular response to cached GI samples; it does not
produce actual reflective rays or mirror/glossy hit chains.

This plan records the missing work so the reflection path is not lost across
context compaction.


RGI-RR-00: NEE-Cache Seed Contract Blocker
------------------------------------------

Status: first corrective slice implemented. The old GI NEE-cache seed path was
not Remix-shaped enough to build more reflection work on top of it: it could
visibly stamp one selected light across an entire cache cell, producing
cell-sized strobing blocks. The primary seed now uses a per-pixel proposal
draw; secondary-hit cache consume is still opt-in and now uses the same
per-pixel no-provider-slot shape, but remains default-off until real
bounce-hit cache population exists. The current secondary implementation is a
cheap single-slot cache probe because the full weighted scan cost about half
the frame rate.

Observed failure:

    - r_pathTracingCleanRestirGiNeeCacheSeed 1 can make GI sample one light
      source across a whole cell.
    - r_pathTracingCleanRestirGiNeeCacheSecondary 0 does not fix this because
      the primary seed path still consumes the cell provider.
    - The symptom is cell-correlated light identity, not just generic temporal
      noise.

Old wrong shape:

    CleanGiSeedInitPageFromNeeCache maps the primary surface to a cache cell,
    picks a provider/candidate for that cell, samples that light once, divides
    by providerResult.sourcePdf and lightSample.solidAnglePdf, then combines
    the result directly into the GI reservoir. If the cache cell has one strong
    candidate, every pixel mapped to that cell can get the same visible light.

    CleanGiBuildNeeCacheFallbackProvider is also cell-hash/frame-index driven,
    so fallback can change whole-cell identity under motion.

Why this is incorrect:

    - The NEE cache provider must be a proposal distribution, not a
      cell-owned final light or finished GI sample.
    - Each pixel must draw its own proposal from the mapped cell/fallback
      mixture.
    - The chosen light must replay through normal RAB light sampling,
      visibility, target-PDF evaluation, and reservoir update.
    - The selected source PDF must include the full cache/fallback identity
      mixture, not just the branch that happened to select the sample.
    - Cached radiance or a cell-owned selected light must never be painted
      directly as final GI.

Reference shape from the working clean-DI provider consume:

    1. Map the receiving surface to a cache cell.
    2. Use the pixel RNG to choose cache vs fallback.
    3. If cache is chosen, draw a candidate from the cell candidate
       distribution with RNG.
    4. If cache misses, draw from the fallback light domain with RNG.
    5. Compute:

           sourceSelectionPdf =
               cacheProbability * cacheIdentityPdf
             + fallbackProbability * fallbackIdentityPdf

       where both terms are evaluated for the selected dense RLU light.
    6. Sample that exact light with a per-pixel UV.
    7. Use the same light sample for solidAnglePdf, visibility, target PDF,
       and the GI reservoir update.

Required GI replacement shape:

    - Replace direct PathTraceNeeCacheProviderResults[cell] consumption in
      CleanGiSeedInitPageFromNeeCache with a per-pixel proposal helper shaped
      like PathTraceCleanRoomNeeCacheStreamProviderIntoReservoir.
    - Keep the GI reservoir as the owner of the selected path sample. The NEE
      cache may bias proposal selection only.
    - Reuse the DI helper logic where possible, but evaluate GI target PDF
      against the primary GI surface and build a GI reservoir sample, not a DI
      reservoir.
    - Keep secondary-hit NEE-cache consume opt-in until it has real bounce-hit
      population; even while opt-in, it must never read the provider slot as
      final lighting. The current fast path probes one random candidate slot
      and falls back to the existing secondary proposal on miss. The full
      weighted secondary scan worked visually but cost too much, so it remains
      deferred behind performance work.
    - Match the Remix after-first-bounce control shape instead of treating
      secondary enable as an all-rays query. The local dxvk-remix reference
      exposes enableModeAfterFirstBounce as None / Specular Only / All, with
      the specular-only path rejecting diffuse and rough specular rays. The
      clean GI route mirrors that policy with:
          r_pathTracingCleanRestirGiNeeCacheSecondaryMode
          r_pathTracingCleanRestirGiNeeCacheSecondaryRoughness
          r_pathTracingCleanRestirGiNeeCacheSecondaryProbability
      Mode 1 is the default so rough diffuse producer rays do not pay the
      secondary cache probe cost.

Cell-frame fix:

    PathTraceNeeCacheMapWorldPositionToCell used snapped camera origin to
    choose its LOD level. That made the effective cell identity
    camera-dependent. The non-onion NEE-cache provider now uses fixed
    world-anchored cells so a stationary world point keeps the same cache cell
    identity under camera movement.

Acceptance before continuing NEE-cache-assisted reflection work:

    - No one deterministic light square per cache cell in normal GI output.
    - Moving the camera does not make stationary surfaces change cache cell
      identity because of camera-relative LOD shells.
    - Seed on/off changes variance or proposal distribution, not mean energy
      in blocky cell regions.
    - Empty/stale cache cells fall back through the explicit proposal mixture
      and do not black out, brighten, or strobe whole cells.
    - With secondary cache enabled, mode 1 should be materially cheaper than
      mode 2 in rough/diffuse views while still allowing glossy/specular
      secondary cache reuse. Probability below 1 should trade cache influence
      for cost without introducing rare over-bright hits.
    - Russian roulette is valid only after the first producer segment exists.
      The current implementation applies it to the secondary-to-tertiary
      continuation when MaxBounces is 2. Applying it to the first producer ray
      behaves as a sparsity setting, not Remix-style path-continuation
      roulette.
    - Multi-bounce first slice is controlled by
      r_pathTracingCleanRestirGiMaxBounces. Value 2 adds one
      secondary-to-tertiary continuation ray and terminal tertiary lighting.
      The tertiary direct-light proposal is budgeted separately by
      r_pathTracingCleanRestirGiContinuationDirectProbability so the second
      bounce can be tested without always paying full terminal NEE cost.
      Deeper recursion and path-depth reservoir metadata are future work.


RGI-RR-01: Specular Initial Candidate
-------------------------------------

Status: first slice implemented with r_pathTracingCleanRestirGiSpecularProducer.
It traces a specular/GGX producer candidate and preloads it into the INIT page
so the existing raw diffuse initial sample can merge with it. The existing
view 1/2 producer textures still show the diffuse producer only; dedicated
specular candidate debug views remain part of RGI-RR-05.
The producer is limited to meaningful reflective receivers: F0 luminance at
least 0.035 and either roughness <= 0.35 or high-F0/metal-like F0 >= 0.12.
Ordinary rough dielectric walls keep the diffuse-only producer path to avoid
tracing an extra specular candidate ray across most of the screen.

Add a cvar-gated specular/glossy GI producer candidate from the primary
surface:

    r_pathTracingCleanRestirGiSpecularProducer 0/1

For each valid primary surface:

    1. Read primary normal, geometric normal, view dir, roughness, and F0 from
       the existing primary surface record.
    2. Sample a specular/GGX direction using the same local material
       interpretation as the GI/NEE specular lobe.
    3. Gate the ray by geometric normal and roughness/F0 sanity checks.
    4. Trace the ray into the existing TLAS.
    5. Shade the secondary hit with the same secondary-hit lighting path used
       by the diffuse GI producer.
    6. Store as a second initial candidate for the GI reservoir.

Do not replace the diffuse producer. The first implementation should merge one
diffuse candidate and one specular candidate into the initial reservoir so
existing diffuse GI remains testable.


RGI-RR-02: Mixture PDF / Weight Contract
----------------------------------------

Status: first slice implemented. When r_pathTracingCleanRestirGiSpecularProducer
is off, diffuse samples keep the old cosine PDF. When it is on, both diffuse
and specular producer rays divide incoming radiance by a deterministic material
mixture PDF:

    mixturePdf = diffuseProbability * diffusePdf
               + specularProbability * specularPdf

The current probabilities are derived from diffuse luminance, F0 luminance, and
roughness, with a conservative clamp to keep both lobes sampleable on mixed
materials. PDF health is visible in view 15.

Mixed diffuse/specular candidates need a consistent PDF contract before they
are allowed through temporal/spatial reuse.

Required contract:

    producerRadiance = incomingRadiance / mixturePdf

where:

    mixturePdf = diffuseProbability * diffusePdf
               + specularProbability * specularPdf

The diffuse probability and specular probability must be deterministic from
the primary material, for example based on diffuse luminance, F0 luminance, and
roughness. The target PDF must evaluate the same diffuse + specular reflected
luminance model used by final GI shading.

Acceptance checks:

    - diffuse-only materials produce the old diffuse behavior
    - high-F0/glossy materials can select specular directions
    - zero/invalid PDFs reject the candidate instead of creating fireflies
    - view 3/4/5 radiance*W does not explode when specular producer is enabled


RGI-RR-03: Temporal / Spatial Reuse Rules
-----------------------------------------

Status: first conservative slice implemented. With
r_pathTracingCleanRestirGiSpecularProducer enabled, receiver surfaces with
roughness < 0.28 and F0 luminance > 0.04 bypass temporal reuse and spatial
reuse. This avoids blindly reusing near-mirror reflective samples before the
reservoir carries lobe/source metadata. Rougher specular materials still use
the existing temporal/spatial policy and material similarity gates.

Reflective samples are strongly view-dependent. Do not let the first specular
producer blindly inherit the diffuse temporal/spatial policy.

Initial safe policy:

    - allow temporal/spatial reuse for rough specular only
    - disable or heavily clamp reuse for near-mirror roughness
    - require stricter normal, roughness, F0, and view-direction similarity
    - keep the existing maxReservoirAge fallback until lighting validation is
      implemented

Useful gates:

    roughness < mirrorThreshold:
        either no temporal reuse or maxHistoryLength = 1

    abs(dot(currentViewDir, previousViewDir)) below threshold:
        reject temporal reuse

    roughness/F0 material mismatch:
        reject temporal and spatial reuse


RGI-RR-04: Output Shape / Denoising
-----------------------------------

Status: first output-shape slice implemented. The raygen now writes the
existing combined shaded GI output plus separate final shaded diffuse and
specular lobe UAVs. Debug views:

    r_pathTracingCleanRestirGiView 11   final diffuse lobe
    r_pathTracingCleanRestirGiView 12   final specular lobe

The boiling filter and resolve still consume the existing combined output.
Denoiser/RR-aware consumption of the split outputs remains deferred.
The specular split output alpha carries final sample hit distance; debug view
16 visualizes it as distance bands.

RR hit-distance route: opt-in slice implemented with
r_pathTracingCleanRestirGiRrHitDistance. When both it and the specular producer
are enabled, eligible reflective GI specular lobe distances are written to
PathTraceRRGuideHitDistance in linear world/engine units. The write uses the
same reflective-receiver gate as the specular producer and remains off by
default for A/B validation.

RR specular input route: opt-in slice implemented with
r_pathTracingCleanRestirGiRrSpecularInput. When both it and the specular
producer are enabled, eligible stored specular GI is added into
PathTraceRRInputColor for RR/DLSS experiments. It is independent from the
hit-distance CVar but uses the same reflective receiver/lobe eligibility.
For clean DI DLSS-RR views 12/16, view-0 RR export dispatches before the
DLSS-RR bridge evaluates so the added specular GI and hit distance are
available to RR instead of being overlaid after RR output copy. GI debug
views remain late overlays and should still be tested with DLSS-RR off.
The boiling-filter beauty resolve must not also add shaded indirect GI into
PathTraceRRInputColor; doing both makes r_pathTracingCleanRestirGiRrSpecularInput
look ineffective at 0 and double-counts the specular lobe at 1.

The current rbdoom output is one shaded GI RGB buffer. That is enough for a
proof of reflective rays, but not a complete Remix/NRD-shaped output contract.

Deferred output work:

    - validate the cvar-gated pre-DLSSRR specular color and hit-distance
      export with RR consumers
    - keep the current combined buffer as a debug/proof fallback

Until this exists, reflective rays can be accumulated into the shared GI output
only for proof. Expect temporal instability on glossy/mirror surfaces.


RGI-RR-05: Debug Views
----------------------

Status: first debug slice implemented. Debug views:

    r_pathTracingCleanRestirGiView 13   specular producer radiance
    r_pathTracingCleanRestirGiView 14   specular producer hit geometry
    r_pathTracingCleanRestirGiView 15   specular producer PDF health
    r_pathTracingCleanRestirGiView 17   specular producer eligibility
    r_pathTracingCleanRestirGiView 18   specular reuse state
    r_pathTracingCleanRestirGiView 19   stored specular output

View 15 packs red = specular sampling probability, green = log-scaled mixture
PDF, blue = log-scaled specular PDF. It is a health/debug view, not energy.
View 17 is a no-extra-ray material gate view: red marks receivers eligible for
the specular producer, green is smoothness (1 - roughness), and blue is scaled
F0 luminance.
View 18 is a no-extra-ray reuse-state view for reflective receivers: red marks
mirror/glossy reuse quarantine, orange marks temporal disabled, yellow marks no
valid temporal history, green marks accumulated history, blue marks specular
producer off, and dark green/gray mark non-eligible or invalid surfaces.
View 19 reads the stored split specular output UAV after final shading writes
it. It should match view 12 in RGB; a mismatch means the split-output path is
not what later denoiser/RR consumers would receive.

View 6 should remain the final shaded diffuse+specular GI view. It is not
enough by itself to diagnose reflective ray producer failures.


Implementation Order
--------------------

    0. Fix the NEE-cache GI seed contract in RGI-RR-00 before using it as a
       reflective-ray proposal source.
    1. Add the cvar-gated specular initial candidate with temporal/spatial off
       for that candidate if needed.
    2. Prove candidate hit/radiance/debug views without changing diffuse GI.
    3. Implement mixture PDF and target PDF parity.
    4. Enable limited temporal/spatial reuse for rough specular only.
    5. Add split outputs and denoiser/RR-aware resolve when proof is stable.


Known Non-Goals For First Slice
-------------------------------

    - no external RTXDI/Remix header copying
    - no portal-sensitive validation substitute
    - no mirror-quality reflection denoising
    - no broad rewrite of shared DI RAB_Brdf behavior
