GI Field Notes
==============

These notes record runtime findings that must shape the next GI work. Treat
them as constraints, not as completed validation.


2026-06-11 Crash Triage
-----------------------

Symptom:
    Repeatable access-violation crashes in action/enemy-spawn areas. The crash
    reproduced even with the GI lane disabled and also reproduced when testing
    the pre-GI decal checkpoint `41c5a115`, once the test build was cleaned up.

Test contamination found:
    Early "old commit" tests were not clean because the prebuilt
    `base/renderprogs2` directory had been overlay-copied. It retained hundreds
    of stale shader binaries from later commits. For any future commit-to-commit
    runtime comparison, clean-mirror the renderprogs2 contents instead of
    overlay-copying them.

Resolved crash cause:
    The stable crash path was CPU-side drawSurf capture, not GI shader code.
    Windows crash offsets mapped to virtual calls through
    `renderEntity->hModel` reached from view drawSurf metadata:

        `ClassifySmokeSurface` -> `hModel->IsStaticWorldModel()`
        `CapturePathTraceDrawSurfMirror` -> `hModel->Name()`

    A drawSurf can carry an entity/model pointer that is stale during action or
    spawn churn. Calling virtual methods through that pointer is unsafe in the
    drawSurf mirror/classification path.

Fix checkpoint:
    `b09a8f8a Avoid stale hModel calls during drawSurf capture`

Runtime result:
    User looped the previous crash area four times without a crash, then tested
    with GI enabled and still saw no crash.

Follow-up rule:
    Do not reintroduce `hModel` virtual calls in drawSurf-derived capture,
    classification, or diagnostic paths unless the model pointer is proven live
    by a stronger ownership/lifetime contract. Entity/material/tri data are
    acceptable for this path; model name/dynamic/static-world model diagnostics
    are not worth an unsafe dereference.


2026-06-11 GI Material Correctness
----------------------------------

Current GI producer limitation:
    The RGI-02 producer traces a secondary ray and interprets the hit through a
    small GI-local material path. That path is not equivalent to the primary
    material/composite path. It does not fully model rbdoom's detail decals,
    alpha/translucent cards, dynamic material records, spectrum-light decal
    behavior, material-classifier routing, or emissive-source ownership.

Observed issues:
    - Some decal/card surfaces appear black or invalid in GI views.
    - Characters or dynamic objects can appear to seed indirect light when they
      should not be treated as emissive GI sources.
    - Broadly rejecting decal/alpha card material flags at secondary-hit time
      caused severe decal/alpha texture destruction and repeatable crashes in
      `0e586ea7 Filter unsupported GI secondary surfaces`; that approach was
      backed out and must not be reapplied.

Implication:
    The producer must eventually consume shared/main material semantics for
    secondary hits, or an explicitly validated subset of those semantics. Adding
    ad-hoc material guesses to the GI shader is likely to regress decals,
    alpha cards, spectrum decals, or emissive ownership again.

Near-term guardrails:
    - Keep crash fixes separate from material filtering experiments.
    - Validate emissive inventory/source ownership before suppressing dynamic
      character/object emission.
    - Treat decal/alpha behavior as receiver-aware material composition, not as
      a coarse secondary-hit reject list.
    - Test RGI-02 views in scenes containing animated decals, alpha cards,
      spectrum decals, characters, and ordinary static emissives before moving
      to temporal/spatial reuse.

2026-06-11 RGI-08 Status + Deferred Features
--------------------------------------------

Boiling filter: ported as a GI compute pass
(pathtrace_clean_restir_gi_boiling_filter.cs.hlsl) that clamps shaded
indirect-GI outliers to the 8x8-group average luminance. The rbdoom lane now
feeds diffuse + specular into the shared GI output, while the separate
NRD-shaped split outputs remain deferred. Threshold cvar
r_pathTracingCleanRestirGiBoilingFilter, default 15 (HL2 RTX value);
0 disables. The same pass owns the resolve add, so the combined outputs
receive the filtered contribution.

Remaining deferred features and their trigger conditions:

    virtual samples                 OFF in shipped HL2 RTX; revisit only if
                                    glossy GI is added (needs roughness-aware
                                    sample repositioning)
    reflection reprojection         needs specular GI lane; see
                                    reflective_rays_plan.txt for the staged
                                    producer/PDF/reuse/output plan
    portal-space sample transforms  needs active ray portals (none in rbdoom)
    sample stealing                 ON in HL2 RTX; port after the resolve
                                    path is proven (feeds PT integrator)
    lighting validation /           needs the gradients pipeline; HL2 RTX
      validateVisibilityChange      ships validateVisibilityChange=True, so
                                    this is the highest-value deferred item
                                    (kills stale GI when lights/doors change;
                                    baseline relies on maxReservoirAge)
    historyDiscardStrength          part of the validation feature set
    pairwise / ray-traced bias      upstream supports it; BASIC is proven
      correction                    baseline first
    DLSS-RR compatibility mode      ON in HL2 RTX; needs RR-aware temporal
                                    randomization; revisit with denoiser work
    pre-denoise RR export           view-0 RR export now runs before the clean
                                    DI DLSS-RR evaluate when the specular
                                    producer and RR export CVars are enabled;
                                    GI debug views remain late overlays
    RR input ownership              the beauty resolve used to add full shaded
                                    GI to PathTraceRRInputColor, making
                                    r_pathTracingCleanRestirGiRrSpecularInput
                                    look always-on and double-counting the
                                    specular lobe when enabled; RR input is
                                    now owned by explicit RR export only
    analytic portal domain          clean DI views 12/16 intentionally clamp
                                    analyticLightCount to the portal-region
                                    prefix, but GI secondary bounces need the
                                    full uploaded current Doom analytic
                                    domain. The GI dispatcher overrides only
                                    its private copy of analyticLightCount and
                                    leaves DI's portal proof path unchanged.
    secondary direct sampling       fallback secondary-vertex NEE used to
                                    sample one analytic and one emissive light,
                                    so enabling analytic lights added an extra
                                    shadow ray per GI producer hit. It now
                                    uses one mixed analytic/emissive proposal;
                                    emissive count changes the PDF/domain, not
                                    the number of fallback shadow rays.

Known debug-view caveats:
    GI debug views share SmokeOutput with the DI route; with DI views 12/16 +
    DLSS-RR active the RR output copy can overwrite GI views. Use GI views
    with DLSS-RR off. View 6 shows the pre-filter indirect diffuse; the
    beauty resolve uses the filtered value.
    View 20 is a NEE-cache provider-state diagnostic for the same primary
    surface query used by r_pathTracingCleanRestirGiNeeCacheSeed, not a shaded
    output: dark blue means the provider is not ready, green/orange-red mean
    cache emissive/analytic providers, yellow means fallback was selected even
    though usable cache candidates exist, orange-brown means candidate records
    exist but do not replay as usable, magenta means candidates exist but the
    provider result slot was empty, bright cyan/blue/violet mean shared
    provider fallback sources with no usable candidates, dark cyan/blue/violet
    mean GI synthesized the provider fallback with no usable candidates, and
    purple/cyan indicate a provider mapping or replay-contract fault.
    GI primary seed consume: r_pathTracingCleanRestirGiNeeCacheSeed must not
    consume PathTraceNeeCacheProviderResults[cell] as a cell-owned selected
    light. The seed path now draws a per-pixel cache/fallback proposal from
    the mapped cell candidate distribution, evaluates the full
    cacheProbability * cacheIdentityPdf + fallbackProbability *
    fallbackIdentityPdf mixture for the selected dense RLU light, then samples
    that light with per-pixel UV before merging a GI reservoir sample. View 20
    remains a provider-state diagnostic, not proof that the shaded GI seed path
    should use the provider slot directly.
    The clean-DI/GI constants contract stores NEE-cache candidateSlots in
    CleanRtxdiDiNeeCacheInfo0.w. CleanRtxdiDiNeeCacheInfo1.w is the provider
    result count on this route and must not be used as the candidate stride.
    The NEE-cache primary-surface compute prepass must also carry the live
    analytic intensity scale and toy emissive scale. Its candidate admission
    replays RLU records through the RAB loader; leaving those scales at the
    primary-surface defaults makes analytic candidates look invalid/zero.
    Field result: the NEE cache can reduce noise heavily in the clean-DI view,
    so the shared provider is not globally broken. View 20 must not force a
    live provider rebuild, because that compares GI against the cache's
    noisiest mid-build state while DI consumes the delayed/frozen provider
    snapshot. The useful GI acceptance test is normal GI output or reservoir
    views with r_pathTracingCleanRestirGiNeeCacheSeed toggled, after the
    provider has had time to reach snapshot hold.
    The current cache producer is still primary-surface populated. That is
    enough for DI and the GI primary-seed path, but secondary-bounce NEE cache
    queries can legitimately miss off-screen/behind-corner cells that the
    primary prepass never populated. Treat secondary-bounce cache population
    as follow-up design work, not proof that the primary seed path is broken.
    r_pathTracingCleanRestirGiNeeCacheSecondary gates that secondary-hit query
    and defaults off so r_pathTracingCleanRestirGiNeeCacheSeed can be validated
    independently against normal GI reservoir/output views. When opted in, the
    secondary-hit query must not read the provider slot as a cell-owned
    selected light. Field result: the full weighted candidate scan worked but
    cost about half the frame rate, so the opt-in secondary path now uses one
    random cache-candidate slot probe and falls through to the existing
    secondary-light proposal on cache miss. This is a performance compromise,
    not the final full-PDF secondary cache consumer.
    The local dxvk-remix reference gates NEE-cache use after the first bounce
    with an explicit mode: none, specular-only, or all. Its specular-only path
    rejects diffuse and rough specular rays because ReSTIR GI handles diffuse
    rays better than highly specular ones and the extra cache work is not worth
    paying everywhere. The clean GI route now mirrors that control principle:
    r_pathTracingCleanRestirGiNeeCacheSecondary remains the master switch,
    r_pathTracingCleanRestirGiNeeCacheSecondaryMode defaults to specular-only,
    r_pathTracingCleanRestirGiNeeCacheSecondaryRoughness defaults to 0.1, and
    r_pathTracingCleanRestirGiNeeCacheSecondaryProbability throttles cache
    attempts after policy gating. Mode 2 restores the prior broad secondary
    behavior for A/B testing.
    RTX Remix field/reference note: Remix commonly pairs Russian roulette with
    two-or-more indirect bounces; in that shape it can terminate whole
    continuation paths and gave a reported 10-15 percent FPS gain in-game.
    Applying roulette to the first clean-GI producer ray is just stochastic
    sparsity, not path-continuation roulette, and the experiment was removed.
    Reintroduce roulette only with real second-bounce/path-depth controls.
    Multi-bounce bring-up starts with r_pathTracingCleanRestirGiMaxBounces.
    The default remains 1. Value 2 traces one secondary-to-tertiary
    continuation ray, shades the tertiary hit with terminal emissive/direct
    lighting, and reflects that contribution through the secondary material
    with the same diffuse/specular mixture PDF used by the producer. This is a
    bounded two-indirect-segment proof, not a recursive path tracer.
    Russian roulette is now valid only for that continuation segment:
    r_pathTracingCleanRestirGiContinuationRoulette defaults on and compensates
    accepted secondary-to-tertiary paths by the continuation probability. It
    does not affect MaxBounces 1 and must not be moved back onto the first
    producer ray.
    The tertiary hit's direct-light proposal is separately budgeted by
    r_pathTracingCleanRestirGiContinuationDirectProbability. This keeps the
    expensive terminal light/shadow work from running on every accepted
    continuation path while preserving a full-rate reference at value 1.
    The first secondary hit's direct-light proposal is separately budgeted by
    r_pathTracingCleanRestirGiSecondaryDirectProbability. This is the first
    one-bounce cost reducer because it avoids thinning the primary-to-secondary
    material ray while measuring and reducing the secondary NEE/shadow cost.
    The optional secondary-to-tertiary ray defaults to opaque traversal via
    r_pathTracingCleanRestirGiContinuationOpaqueTrace. This mirrors the
    reference's use of specialized opaque/translucent traversal variants more
    closely than forcing the alpha any-hit path on every continuation ray. The
    first producer ray remains alpha-capable, preserving the alpha-clip support
    validated earlier.
    The shared PathTraceNeeCacheMapWorldPositionToCell helper now uses fixed
    world-anchored cells for the non-onion provider. The old snapped-camera
    LOD frame could change a stationary point's cell identity under camera
    movement and produced movement-correlated cell strobe.
