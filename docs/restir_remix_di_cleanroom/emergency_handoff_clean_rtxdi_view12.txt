Current High-Level State
========================

The clean RTXDI DI path can prove accumulation with synthetic lights, but real
Doom analytic lights in view 12 remain unstable.

Important current observations:

1. Synthetic view 11 can accumulate.
   This proves the RTXDI temporal machinery can work under a clean synthetic
   light/surface contract.

2. Real Doom analytic view 12 is still not reliably accumulating.
   Single isolated Doom lights were once observed working in older test states,
   but in the current state the user reports even single Doom lights can break.
   Do not assume overlap-only failure.

3. Full-map / full analytic brute force is worse.
   A full-map brute force test roughly doubled the noise. The view-12 full
   analytic domain toggle also made things worse. This rules out "just include
   all lights" as a solution.

4. The mirror glitch is real but not a solution.
   The mirror can produce strong apparent accumulation, sometimes around 90%
   except for very bright areas near lights. A weaker "half glitch" from a
   different portal can produce around 50%. But this is an accidental state
   perturbation, not a compliant RTXDI solution.

5. Mirror dump evidence says the mirror changes the analytic domain.
   Example dump while looking at mirror:
       subview=1 mirror=1 area=23 drawSurfs=129
       cleanCurrentAnalytic=100
       cleanPortalAnalytic=31
       cleanAnalyticRemap=545

   Same spot with mirror offscreen:
       subview=0 mirror=0 area=28 drawSurfs=27
       cleanCurrentAnalytic=101
       cleanPortalAnalytic=5
       cleanAnalyticRemap=546

   This is the key fact: the mirror path changes the portal/area analytic
   domain. It is not merely denoising or reusing the same data better.

6. Subview/mirror toggle findings:
       r_pathTracingCleanRtxdiDiSubviewDispatch 1
           Primary driver of the mirror glitch.

       r_pathTracingCleanRtxdiDiSubviewSurfacePromote 1
           Adds another layer of denoising, but cannot help without subview
           dispatch.

       r_pathTracingCleanRtxdiDiSubviewReservoirPromote
           Not identified as the main driver.

   Away from mirrors, these toggles have no impact, as expected.

7. Target floor made things noisier and erased the mirror glitch.
   r_pathTracingCleanRtxdiDiDoomTargetFloor 1 promotes weak/bad Doom analytic
   samples and removes useful target filtering. Keep it off for baseline tests.

8. BRDF gate relaxation fixed some binary diagnostics but did not fix view 12.
   r_pathTracingCleanRtxdiDiRelaxBrdfGates 1 can make view 15 greener, proving
   some RAB receiver gates are real. It did not restore stable view-12 temporal
   accumulation by itself.

    r_pathTracingCleanRtxdiDiView 15
        Binary gate diagnostic. Green/red bands for:
            1. surface and light domain valid
            2. opaque diffuse BRDF supported
            3. replayable light sample
            4. radiance > 0
            5. light hemisphere passes
            6. view hemisphere passes
            7. BRDF luminance > epsilon
            8. left targetPdf > epsilon, right reflected luminance > epsilon

    Cases with red 5,7,8 indicate light hemisphere / BRDF / target failure.
    Cases with red 2,7,8 indicate BRDF support rejection, likely surface class
    or receiver classification. RelaxBrdfGates can clear some of this, but that
    does not fix view 12.


Interpretation of "Analytic Domain"
===================================

The analytic domain is the set of Doom analytic lights RTXDI is allowed to draw
candidate lights from for a pixel/view.

Current relevant domains:

    cleanCurrentAnalytic
        All uploaded Doom analytic lights currently known to the renderer.

    cleanPortalAnalytic
        Smaller portal/area subset considered relevant to the active view.

    mirror/subview domain
        Different portal/area subset produced by the mirror render view.

Domain failure modes:

    Too small:
        Missing the light that actually illuminates the surface.

    Too large:
        Many irrelevant lights are sampled, noise increases, current reservoirs
        become poor.

    Unstable:
        History resets, light identity remap changes, or previous reservoirs do
        not map cleanly.

The mirror appears useful because it accidentally produces a more favorable
portal/subview domain, not because it reveals a correct RTXDI path.
