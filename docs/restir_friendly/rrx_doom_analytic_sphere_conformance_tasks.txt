RRX Doom Analytic Sphere Conformance Tasks
=========================================
Date: 2026-05-22

Purpose
-------

This document defines the next narrow implementation plan for Doom analytic
lights in the active RRX / ReSTIR-friendly light path.

The current Doom analytic light path is prototype scaffolding. It represents
Doom lights as crude analytic spheres and adapts them into RAB_LightInfo, but
it is not yet equivalent to RTX Remix's imported/game light representation.
The closest Remix equivalent is a sphere light with shaping disabled.

This task does not rebuild all Remix light types. It makes Doom point lights
behave like a disciplined sphere-light contract, while preserving Doom-specific
radius/falloff semantics where rbdoom already depends on them.

Reference Behavior To Match
---------------------------

RTX Remix imported/game local lights have these relevant properties:

1. CPU lights are typed.

   Remix uses RtLight variants: Sphere, Rect, Disk, Cylinder, and Distant.
   A sphere light carries position, radiance, radius, optional shaping, and
   validation/sanitization before GPU upload.

2. GPU lights are polymorphic.

   Remix writes a 64-byte MemoryPolymorphicLight payload. Shader-side code
   decodes the type and routes sampling/weighting through the corresponding
   light helper. For this task, only the sphere-light behavior is relevant.

3. Sampling and target weighting are paired.

   Remix sphere sampling samples the visible sphere solid angle, reports a
   solid-angle PDF, computes a radiance sample, and target weighting uses the
   same sphere-light model.

4. Identity is separate from payload changes.

   The light manager preserves current/previous mapping through buffer indices
   and remaps. Payload changes are not supposed to imply a broad reservoir
   reset.


rbdoom Current Mismatch
----------------------

The current rbdoom path is not a Remix-style polymorphic light representation.

It currently has:

    PathTraceDoomAnalyticLightCandidate
    PathTraceUnifiedLightRecord
    RAB_LightInfo
    RAB_SampleDoomAnalyticSphereLight

This is enough to sample a crude sphere-like light, but it is missing a clear
contract for:

    sphere-light parameter validation
    physical sphere radius versus Doom influence radius
    Doom radius/falloff behavior
    stable payload semantics for animated lights
    target-PDF parity with the sampled light model
    diagnostics proving analytic samples survive RAB sampling and target PDF

The native rbdoom Doom NEE path in pathtrace_nee.hlsli already contains Doom
influence behavior. The RAB analytic path must not silently drop those
semantics while claiming to represent Doom lights.


Required End State
------------------

The active RRX path should see Doom analytic lights as a stable sphere-light
contract:

    Doom light candidate
        -> validated sphere-compatible payload
        -> current/previous mapped light-manager record
        -> RAB_LoadLightInfo
        -> RAB_SamplePolymorphicLight
        -> RAB_GetLightSampleTargetPdfForSurface

Only Doom point/sphere lights are in scope.

Out of scope:

    Rect lights
    Disk lights
    Cylinder lights
    Distant lights
    light shaping
    NEE cache
    best-light sampling
    temporal reuse policy
    spatial reuse policy
    DLSS RR input changes
    reservoir finalization changes

Task RRX-AL-02: Harden CPU Sphere-Compatible Payload
----------------------------------------------------

Goal:

    Make the CPU payload for Doom analytic lights explicitly sphere-compatible
    and validated before RAB consumes it.

Owned files:

    neo\renderer\NVRHI\PathTraceDoomLights.cpp
    neo\renderer\NVRHI\PathTraceUnifiedLight.cpp
    neo\renderer\NVRHI\PathTraceUnifiedLight.h

Required behavior:

    Preserve current Doom analytic candidate generation and identity/remap
    scaffolding.
    Preserve full authored Doom analytic light universe; do not prune lights
    for quality.
    Ensure the unified Doom analytic record has:
        position
        physical sphere radius
        area derived from physical sphere radius
        radiance or color/intensity after current rbdoom convention
        Doom influence radius
        source identity fields
        stable previousIndex when remappable
        positive sourceWeight only when sampleable
    Reject or mark unsampleable records with invalid/zero radius, invalid
    influence radius, non-finite position, or non-positive radiance.

Layout rule:

    Prefer using the existing PathTraceUnifiedLightRecord fields if they are
    sufficient. Do not change its size unless absolutely required. If layout
    changes are required, update the matching HLSL layout and static_assert in
    the same slice.

Forbidden behavior:

    Do not add random analytic-light priority.
    Do not cap or remove dynamic lights to hide the problem.
    Do not classify payload changes as structural reset triggers.
    Do not delete legacy split Doom analytic outputs.


Task RRX-AL-03: Make Shader Sampling Sphere-Contract Clean
----------------------------------------------------------

Goal:

    Make RAB_SampleDoomAnalyticSphereLight match a disciplined sphere-light
    sampling contract, using Remix sphere-light behavior as the structural
    guide and rbdoom native Doom NEE as the Doom-specific influence guide.

Owned files:

    neo\shaders\builtin\pathtracing\RtxdiBridge\RAB_LightSamplingCore.hlsli
    neo\shaders\builtin\pathtracing\RtxdiBridge\RAB_LightTarget.hlsli
    neo\shaders\builtin\pathtracing\RtxdiBridge\RAB_LightLoadRuntime.hlsli

Required behavior:

    Keep RAB_SamplePolymorphicLight as the public entry point.
    Keep emissive triangle sampling unchanged.
    For Doom analytic sphere lights:
        validate surface and light info
        reject samples outside Doom influence radius if native Doom semantics
        require that behavior
        sample the sphere solid angle consistently
        report a valid sample position, normal, radiance, distance, and
        solidAnglePdf
        apply Doom influence/falloff consistently with native Doom analytic
        NEE, not as an undocumented extra multiplier
        make target PDF use the same radiance/PDF model as the sample

Reference constraint:

    Use RTX Remix sphere_light.slangh as the structural reference for sphere
    sampling. Do not copy the source. Translate the behavior into rbdoom's
    existing RAB_LightSample and RAB_Surface types.

Forbidden behavior:

    Do not change RTXDI reservoir math.
    Do not alter temporal reuse thresholds.
    Do not change RRX page ownership or clear policy.
    Do not change DLSS RR inputs.
    Do not add full Remix rect/disk/cylinder/distant support in this slice.


Task RRX-AL-04: Add Analytic Survival Diagnostics
-------------------------------------------------

    Make it visible whether analytic lights survive the RAB sampling and target
    PDF path.

    Extend an existing mode-56 RRX debug subview if there is already an
    analytic-light attribution view.

Acceptable diagnostic states:

    selected emissive
    selected Doom analytic
    invalid light info
    invalid light sample
    zero target PDF
    outside Doom influence radius
    non-finite payload

Forbidden diagnostic work:

    Do not add per-pixel terminal spam.
    Do not make a broad new debug system.
    Do not reuse a color meaning from an existing view with a different meaning.

Success condition:

    In a scene with active Doom analytic lights, the user can determine whether
    analytic lights are:
        entering initial sampling
        surviving RAB_SamplePolymorphicLight
        producing positive target PDF
        being selected into the reservoir
