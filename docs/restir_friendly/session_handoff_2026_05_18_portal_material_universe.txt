Current User Observations
=========================

The user has narrowed the active failure to a ReSTIR-wide temporal/domain
problem, not merely a new DI/GI implementation bug.

Observed behavior:

- Debug mode 12 can become cyan while stationary.
- Movement produces orange or magenta noise/strobing depending on which light
  domains are enabled.
- Portal regions strongly correlate with failure.
- Some portal crossings are clean, so "crossing any portal" is not inherently
  broken.
- Bad portal regions seem to corrupt adjacent regions.
- Looking at a surface through a portal, then moving closer into a different
  portal zone, changes the green/cyan or orange/cyan ratio.
- With analytic lights disabled, stationary often becomes 100% cyan, but
  portal/motion failures remain and show as emissive failures.
- Debug mode 18's old accumulator is broken by the same locations.
- That means the problem is upstream of the newest RTXDI temporal code.
- Turning normal maps off produces lots of magenta in ReSTIR mode 12.
- Turning normal maps back on does not recover, even after loading another
  level.

Important interpretation:

The portal issue can plausibly be caused by frame-local compact material,
texture, emissive, or unified-light indices changing when portal/area selection
changes. Safe portals may simply preserve enough of the same compact domain,
while bad portals cause the list to reorder or lose entries when deeper areas
drop out.


Mode 12 Diagnostic Color Meanings
=================================

These meanings are based on the current diagnostic split in
pathtrace_restir_reference_temporal_debug.hlsli:

- cyan:
    Accepted temporal candidate, no remap failure, no history growth.

- green:
    No remap failure and temporal history grew.

- orange:
    Selected previous light is an emissive unified light and previous-to-current
    unified light translation failed.

- magenta:
    Selected previous light is an analytic unified light and previous-to-current
    unified light translation failed.

- blue:
    Previous light index is outside previous unified light count.

- gray:
    Analytic light state incompatible.

- white:
    Invalid sampled light data.

Primary diagnostic mode:
    r_pathTracingRestirPTTemporalNeighborDebugMode 2

Useful runtime toggles the user has used:
    r_pathTracingRestirPTUnifiedLightLoad 1
    r_pathTracingRestirPTUnifiedLightSample 1
    r_pathTracingRestirPTDiDebugView 12
    r_pathTracingLightUniverseDump 1
    r_pathTracingAnalyticLightScale 0


Current Strongest Finding
=========================

The strongest concrete bug candidate is an emissive remap ordering mismatch in
PathTraceSmokeSceneBuild.cpp.

Current suspicious graph:

1. Build raw current emissive triangle records.
2. Read previous emissive triangles from m_smokePreviousEmissiveTriangles.
3. BuildSmokeEmissiveLightRemap(rawCurrent, previousMerged).
4. Mutate/rebuild current emissives via m_smokeLightUniverse.MergeFrameCandidates.
5. BuildPathTraceUnifiedLights(postMergeCurrent, previousMerged, rawRemap, ...).
6. Store postMergeCurrent as next frame's previous emissive list.

This means the remap can describe a different current emissive array than the
array actually uploaded and sampled through the unified light buffer.

That directly matches orange mode-12 failures: previous emissive light cannot
translate to current unified emissive light.

The dedicated audit is:
    docs/restir_friendly/geometry_universe_rtxdi_remix_audit.txt

Material Universe Finding
=========================

The material universe can plausibly cause or amplify the portal bug.

Key points from:
    docs/restir_friendly/material_universe_rtxdi_remix_audit.txt

rbdoom currently lets frame-local or table-local material state participate in
temporal identity:

- PathTracePrimarySurfacesAreSimilar rejects if materialId, materialIndex, or
  surfaceClass differ.
- Emissive identity includes materialIndex.
- Emissive remap compatibility requires materialId, universeMaterialIndex, and
  emissiveTextureIndex to match.

RTXDI does not use material ids or compact table indices for ReSTIR material
similarity. It compares material properties: roughness, specular luminance, and
diffuse luminance.

RTX Remix is a better model for an old/uncooperative game:

- persistent material/texture identity is separate from per-frame buffer index;
- textures and materials are kept alive explicitly;
- scene reset and cache lifetime are explicit;
- original game material/texture hashes are used for identity/categories.

The user's normal-map toggle poisoning behavior strongly suggests a missing
reset/invalidation boundary. Changing normal-map policy changes the current
RAB surface normal/material semantics but does not obviously clear all dependent
previous history domains.

Possible Portal Mechanism
=========================

The user's theory is plausible:

Safe portals may happen to preserve the same compact material/light domain.
Bad portals may cull a deeper area or 4th portal back, causing the frame-local
material/emissive/unified-light list to reorder or drop entries.

If previous reservoirs or remap compatibility use:

- materialIndex
- emissiveTextureIndex
- unified light index
- a remap built against a different current list

then ReSTIR will reject or fail previous-to-current translation. The visual
result is orange/magenta/blue strobing depending on which domain fails.

This is not "textures" in the simple visual sense. It is texture/material/light
identity being represented by unstable compact indices.
