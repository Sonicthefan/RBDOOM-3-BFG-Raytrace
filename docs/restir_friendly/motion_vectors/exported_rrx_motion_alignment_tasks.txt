Purpose
-------

This file is the worker-facing plan for the current motion-vector failure:
the older rbdoom motion-vector debug modes can look solved, while the active
RRX/RTXDI path still sees holes or bad data in the exported
PathTraceMotionVectors / PathTraceMotionVectorMask inputs.

This is not a ReSTIR math task. Do not alter reservoir finalization, sampling,

Observed Problem
----------------

The user verified that mode 56 / DI view 70 exposes routed-rigid and large
object regions with bad temporal input coverage:

    column 0: motion source / validity
    column 1: raw exported motion xy
    column 2: reset-mask safety
    column 3: previous-surface validity at RTXDI reprojected pixel

Some large routed-rigid objects show:

    column 0 valid routed-rigid motion
    column 1 unstable or gapped exported vector visualization
    column 2 unsafe reset-mask regions
    column 3 missing/not-applicable previous surface

Disabling rigid motion-vector export with r_pathTracingMotionVectorDisableRigid
does not solve accumulation quality. That CVar is a proof/quarantine tool only:
it converts bad rigid motion into missing rigid motion. Missing temporal input
still leaves holes.

--------------

The old debug modes and the active RRX input path do not prove the same
contract.

Old diagnostics:

Relevant modes:

    40 = routed-rigid primary object-motion vector diagnostic
    41 = combined skinned/routed-rigid object-motion vector diagnostic
    47 = combined static/skinned/routed-rigid motion-vector diagnostic
    48 = combined geometry reprojection-match diagnostic
    49 = combined geometry motion-source diagnostic

Active RRX input:

    producer:
        neo/shaders/builtin/pathtracing/pathtrace_primary_surface_producer.rt.hlsl

    exported buffers:
        PathTraceMotionVectors
        PathTraceMotionVectorMask
        PathTraceRRGuideResetMask

    consumer/probe:
        neo/shaders/builtin/pathtracing/pathtrace_restir_local_debug_reservoirs.hlsli
        mode 56 / r_pathTracingRestirPTDiDebugView 70

Concrete divergence found by audit:

1. Old mode 47 draws motion from helper results. It is diagnostic-only and does
   not prove the exported PathTraceMotionVectors UAV is usable by RTXDI.

2. pathtrace_primary_surface_producer.rt.hlsl has a local projection helper:

       ProjectPrimarySurfaceToPreviousPixelFloatAndDepth

   That helper rejects previous NDC outside the frame as a producer failure.

3. The shared/status-aware helper in PathTracePrimarySurface.hlsli:

       ProjectPathTracePrimarySurfaceToPreviousPixelFloatAndDepthWithStatus

   distinguishes previous xy outside the frame as a diagnostic status. The
   reference contract expects finite offscreen previous xy to remain a produced
   motion vector; consumer bounds checks decide whether previous history can be
   read.

4. Active RRX DI temporal currently builds:

       desc.screenSpaceMotion = float3(motionPixels, 0.0)

   RTXDI expects motion.z to be:

       expectedPreviousLinearDepth - currentLinearDepth

   Passing 0.0 makes RTXDI compare previous depth as though expected previous
   depth equals current depth. This is especially suspicious for forward/back
   movement and "short reach" motion-vector behavior.

View 70 diagnostic correction:

    The final two view-70 stripes were previously still named and implemented
    as packed-object export checks. That made them reject level/static motion
    even though debug mode 47's full combined helper legitimately accepts
    level/static through the previous-static snapshot path.

    The combined resolve shader does not bind the static, skinned, and routed
    rigid geometry buffers required to recompute debug mode 47's full combined
    helper result. The honest diagnostic contract for those final stripes is
    therefore the exported data itself:

        stripe 10 = exported PathTraceMotionVectorMask source/valid color
        stripe 11 = exported PathTraceMotionVectors.xy color

    These correspond to the mode-49 source-mask contract and mode-47
    full-combined motion-vector contract as exported by the primary-surface
    producer.

    by pathtrace_smoke.rt.hlsl primary raygen/export paths. The producer keeps
    the smoke-compatible primary ray TMin/TMax and skips GUI-screen payloads
    instead of writing them as usable primary surfaces.

    View 70 now separates expected producer-record object motion from exported
    motion. Stripe 0/1 are the packed object motion represented by the current
    primary-surface record. Stripe 0 only reports rigid/cyan when that record
    has both instanceId >= 2 and surfaceClass == RIGID_ENTITY. Stripe 10/11
    remain the exported
    PathTraceMotionVectorMask and PathTraceMotionVectors.xy. Stripe 12 reports
    current producer-record source classification; for a known routed-rigid
    door it must be rigid/cyan, which proves instanceId >= 2 and
    surfaceClass == RIGID_ENTITY reached the producer record. Stripe 13 reports
    the current producer-record packed-motion status. For rigid doors, stripe
    0/1 and stripe 10/11 should match. If stripe 0/1 are good while 10/11 are
    broken, the export write is wrong. If both are broken in the same places,
    the producer primary-surface record/packed previous position is already

Follow-up: Mode-56 Rigid Duplicate Root Cause
--------------------------------------------

Coordinator finding:

    The old motion-vector debug modes and mode 56 both enabled routed rigid
    TLAS instances, but only the old debug modes were included in the dynamic
    fallback removal predicate in PathTraceDrawSurfCapture.cpp. That meant a
    route-ready rigid surface could appear twice in mode 56: once as routed
    rigid with instanceId >= 2, and once as the legacy dynamic fallback with
    instanceId 1. View 70 then split the same object between cyan routed-rigid
    stripes and non-cyan/broken fallback stripes, matching the observed
    z-fighting-style instability.

Implemented correction:

    The routed-rigid dynamic fallback removal predicate now includes ReSTIR PT
    debug modes, including mode 56, and the integrator debug modes that also
    enable routed rigid. This preserves the old mode-47/49 behavior: once a
    rigid mesh is route-ready, the dynamic fallback copy is skipped for every
    routed-rigid path, not just the old diagnostics.
