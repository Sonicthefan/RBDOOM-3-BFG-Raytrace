Motion Vector Worker Launch Protocol
====================================
Date: 2026-05-19
Updated: 2026-05-21

Purpose
-------

This file explains how the coordinating agent should launch implementation
workers for the motion-vector work. The goal is to reproduce the effective
workflow:

    coordinator researches and chooses the slice
    worker implements one bounded task
    coordinator reviews, builds/deploys, and asks the user to test

Workers must not be asked to design strategy while coding.


Current Active Worker: MV-RRX-01 Exported Producer Parity
---------------------------------------------------------

Use this before any older MV-00A/MV-01A task on the current branch.

The current failure is not simply "motion vectors are bad." The specific
problem is that the older debug modes can look solved while the active RRX
input path still consumes holes or bad data from the exported
PathTraceMotionVectors / PathTraceMotionVectorMask buffers.

The worker-facing task and paste-ready prompt are in:

    docs/restir_friendly/motion_vectors/exported_rrx_motion_alignment_tasks.txt

Coordinator instruction:

    Launch MV-RRX-01 exactly as written in that file. Do not ask the worker to
    design a broader motion-vector system. Do not combine it with MV-RRX-02.

Reason:

    MV-RRX-01 fixes the producer contract first. MV-RRX-02 then feeds the full
    motion.xyz contract into RRX temporal reuse. Doing both at once would make
    user visual evidence ambiguous.


Coordinator Rules
-----------------

The coordinator owns:

- choosing the next slice
- reading the reference files
- deciding the intended shape
- keeping the worktree safe
- reviewing worker changes
- building/deploying
- explaining exactly what the user should test

The worker owns only:

- the concrete files named in its prompt
- the exact implementation checklist in its prompt
- reporting changed paths and any blockers

Do not launch a worker with a broad request like:

    "Fix motion vectors against RTXDI."

Do launch a worker with a bounded request like:

    "In pathtrace_smoke_rab_motion_supplier.hlsli only, add diagnostic status
    codes that distinguish offscreen previous xy from behind-camera projection.
    Do not change behavior."


Worker Prompt Required Fields
-----------------------------

Every worker prompt must include these fields:

1. Repository:

       E:/prog/rbdoom-3-bfg-rt

2. Slice id:

       Example: MV-00A

3. Mission:

       One sentence. No strategy language.

4. Read first:

       Exact docs and exact source files.

5. Reference files:

       Exact RTXDI/RTX Remix files relevant to the slice.

6. Owned write set:

       Exact rbdoom files the worker may edit.

7. Forbidden changes:

       Reservoir math, unrelated modules, broad refactors, fallback hacks,
       code copied from NVIDIA, and worktree cleanup.

8. Implementation checklist:

       Concrete steps the worker should do.

9. Verification:

       Local compile/build command if applicable, or "no build required" for
       docs-only/diagnostic-code-not-built assignments.

10. Final response requirements:

       Changed paths, whether build passed, and blockers. No broad proposal.

11. Shared-worktree warning:

       "You are not alone in the codebase. Do not revert edits made by others.
       Work with existing changes and list any conflict."


When To Use Workers
-------------------

Use one worker when:

- the slice has a clear owned write set;
- the worker can make progress without further design decisions;
- the coordinator can review the diff afterward.

Use two workers only when write sets are disjoint. Example:

- Worker A: shader diagnostics only.
- Worker B: documentation/test instructions only.

Do not run two workers against the same shader file at the same time.

Do not delegate the immediate blocker if the coordinator needs that answer
before doing anything else. In that case, the coordinator should inspect it
directly.

Coordinator Review Checklist
----------------------------

After a worker returns:

1. Inspect the diff.
2. Confirm it only touched the owned write set.
3. Confirm forbidden areas were not changed.
4. Confirm no NVIDIA code was pasted.
5. Confirm behavior matches the slice.
6. Build/deploy if not already done.
7. Tell the user exactly what to test.
8. Stop for user visual feedback.
