RRX Light Sampling Range Conformance Tasks
=========================================
Date: 2026-05-22

Purpose
-------

This document defines the next narrow implementation slice for the active RRX
DI reservoir path.

The current rbdoom RRX DI reservoir path can store and reuse reservoir records,
but its initial light sampling still does not match the RTX Remix / RTXDI light
contract. The active shader samples the current light manager as one flat list
with:

    sampleCount = min(currentCount, 32)

That is not how RTX Remix feeds RTXDI. Remix groups lights into typed ranges,
assigns bounded per-range sample counts, guarantees at least one sample for
each non-empty type, and exposes those ranges through RAB_GetLightRange for the
RTXDI helpers.

This task is not a reservoir redesign. It is a conformance correction for how
the existing light-manager output is consumed by initial sampling.

Reference Behavior To Match
---------------------------

RTX Remix light handling has three relevant contracts:

1. Active lights are grouped into typed ranges.

   Remix counts active lights by type, assigns each type a sequential range,
   writes the typed light records into those ranges, and builds current-to-
   previous plus previous-to-current light mapping.

2. Initial RTXDI sample counts are distributed across ranges.

   Remix derives a requested sample density from the requested initial sample
   count and the total active light count. Each non-empty range receives at
   least one RTXDI sample. Empty ranges receive zero samples.

3. Shaders consume ranges through RAB.

   Remix RAB_GetLightRange returns first index, count, and sample count for a
   light type. RTXDI initial sampling uses the RAB contract instead of a custom
   flat-list walk over every light.

Do not copy RTX Remix code. Reproduce the contract in rbdoom's style.


Current rbdoom Mismatch
----------------------

The active RRX DI initial reservoir builder currently lives in:

    neo\shaders\builtin\pathtracing\pathtrace_restir_local_debug_reservoirs.hlsli

Function:

    RestirPTRrxDiBuildInitialReservoir

The current code:

    uses RAB_GetCurrentRestirLightManagerCount()
    sets sampleCount = min(currentCount, 32)
    chooses samples from one combined flat light domain
    streams each candidate with invSourcePdf/currentCount

This makes analytic lights compete against emissives in a single capped list.
In the current observed scene, the manager dump showed:

    emissive range count: 666
    Doom analytic range count: 450
    total active count: 1116

A flat 32-sample cap can underrepresent or miss one type in ways Remix avoids
with per-type sample distribution.


Required End State
------------------

The RRX DI initial reservoir path must consume the light-manager range/sample
contract instead of sampling the flat combined domain directly.

The shader must sample at least these two current rbdoom ranges:

    emissive triangle range
    Doom analytic range

Each non-empty range must receive at least one sample when the total initial
sample budget is non-zero. Source PDF / inverse source PDF must account for
the range size and sample count in the same way the existing flat path accounts
for the full current count.

The current/previous light index mapping behavior must remain unchanged.

The previous-best-light path must remain separate. Do not make this task about
best-light sampling or Portal RTX's gradient/confidence pass.

Task RRX-LS-02: Make CPU Sample Contract Bounded
------------------------------------------------

Goal:

    Make the rbdoom light-manager sample contract Remix-shaped: bounded total
    initial samples distributed across non-empty ranges.

Owned files:

    neo\renderer\NVRHI\PathTraceRemixLightManager.cpp
    neo\renderer\NVRHI\PathTraceSmokeDispatch.cpp

Expected behavior:

    Use the existing RRX DI initial sample budget if one already exists.
    If no explicit RRX initial sample count exists, keep the current effective
    cap of 32 as the temporary requested total.
    Distribute that total across active ranges proportional to range count.
    Guarantee at least one sample for each non-empty range when requested total
    is non-zero.
    Clamp so no range asks for more samples than it has lights.

Forbidden behavior:

    Do not use raw range count as per-pixel sample count.
    Do not add a new light weighting policy.
    Do not bias analytic lights by hand.
    Do not change reservoir temporal reuse.
    Do not change final resolve.

Debug output:

    Existing light-manager dumps must show the bounded sample contract. The
    dump should make it obvious when the active scene has emissive and Doom
    analytic ranges and both receive non-zero sample counts.


Task RRX-LS-03: Consume Ranges In RRX DI Initial Sampling
---------------------------------------------------------

Goal:

    Replace the flat combined-list loop in RestirPTRrxDiBuildInitialReservoir
    with a per-range loop over the light-manager sample contract.

Owned file:

    neo\shaders\builtin\pathtracing\pathtrace_restir_local_debug_reservoirs.hlsli

Required behavior:

    Read current emissive range offset/count/sample count.
    Read current Doom analytic range offset/count/sample count.
    For each non-empty range with non-zero sample count:
        choose a random start inside that range
        stride through that range
        load the current-frame light by unified light index
        sample the light through existing RAB light sampling helpers
        stream the candidate into the reservoir with source PDF corrected for
        that range's count and sample count

The total result should still be one RTXDI_DIReservoir for the pixel.

Previous-best behavior:

    Keep the existing previous-best candidate handling after random/ranged
    sampling. Continue using existing light-index translation. Do not rewrite
    previous-best as part of this slice.

Forbidden behavior:

    Do not add new target-PDF math.
    Do not change RAB_GetLightSampleTargetPdfForSurface.
    Do not change RAB_SamplePolymorphicLight behavior.
    Do not change temporal reuse thresholds.
    Do not change debug color meanings for existing views.
