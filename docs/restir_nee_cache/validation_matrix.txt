Remix-Style NEE Cache Validation Matrix
=======================================

Rung V-00: Read-Only Contract Audit
-----------------------------------

Purpose:

    Compare Remix NEE cache, RTXDI ReGIR, current rbdoom ReGIR scaffolding,
    current RLU, current PDFNEE producer, clean RTXDI DI initial sampling,
    and path/indirect integration before implementation.

Pass:

    worker names exact reference files, rbdoom read set, proposed write set,
    whitelist ids, CVars, source-provider boundary, the split between
    GI/secondary NEE, clean DI proposals, and experimental PDFNEE visual mode,
    and first missing contract

Fail:

    worker starts coding before proving the boundary


Rung V-01: CPU Resource Shell
-----------------------------

Purpose:

    Prove the new lane can own fixed-budget cache buffers and dump their
    state without affecting active lighting.

Expected:

    CVars exist
    disabled state writes no active route
    enable creates only this lane's resources
    dump reports buffer bytes and first missing contract
    provider ABI is concrete before candidate algorithm work

Pass:

    enabling/disabling changes only this lane's resources and dumps
    worker names shader struct, binding slots, buffer owner, provider
    function boundary, and future consumer call/read boundaries for:
        GI / secondary NEE
        clean RTXDI DI proposal
        experimental PDFNEE visual mode
    structurally valid resources with current dense RLU and an active selected
        source domain report firstMissingContract=none

Fail:

    clean RTXDI, PDFNEE, temporal, spatial, or mode 56 behavior changes
    candidate algorithm work starts while provider ABI is ambiguous
    resource descriptor reports old PDFNEE consumer or Doom-analytic-only
        missing contracts when emissive-only or clean-DI proposal roles are
        otherwise valid

Rung V-05: Analytic Candidate Map
---------------------------------

Purpose:

    Prove cached Doom analytic candidates are visible and replayable.

Status:

    implemented by NEECACHE-05 as current-RLU Doom analytic candidate records

Expected:

    selected analytic identities are stable while stationary
    light animation does not turn the identity map into a rejection mask or
        leave stale per-cell candidate relevance alive
    portal-dependent replay state does not rank or invalidate the identity map
    RLU payload replay validates selected dense RLU index
    analytic candidates are written only when the dense RLU record is marked
        stableCacheable
    currentSampleable but unstableDynamic analytics can still be direct-light
        fallback inputs, but cannot be stored in persistent candidate cells
    RLU structural, mapping, payload, and payloadOnly flags invalidate provider,
        cell, task, and candidate buffers
    RLU payloadOnlyChange clears persistent candidate identities so animated
        light payload changes cannot leave stale cell relevance alive
    provider selection rejects any buffered candidate that cannot replay through
        RAB_LoadActiveRrxLightInfo in the current frame
    provider selection recomputes candidate weight from the current RLU payload
        before calculating selected sourcePdf
    candidate PDF is explicit
    debugView 6 shows selected analytic identity
    debugView 7 shows analytic sourcePdf / invSourcePdf with identity tint
        rather than a flat uniform field
    debugView 10 shows analytic RLU payload replay validity
    debugView 11 shows a flat consumed-candidate render from the same candidate
        records and current dense RLU payload
    debugView 12 shows a flat full current-RLU render from dense RLU identities
        replayed through the active RLU/RAB helpers, without the candidate slot
        filter or any direct raw DoomAnalyticLights sourceIndex dereference

Pass:

    user can correlate candidate identity with clean RTXDI view-12 lighting
    candidate records use dense current RLU identity and explicit uniform
        current-Doom-analytic-range source PDF
    view 12 shows whether portal-dependent payload strobing exists before the
        candidate filter
    view 11 shows whether the fixed-budget candidate filter is dropping or
        strobing lights relative to view 12
    the dump reports rluSignatures, rluSignatureChanged, cacheInvalidation,
        candidateInvalidationPolicy, and current replay/reweight selection
        policy

Fail:

    analytic candidate brightness/shape diverges from RLU/RAB replay
    animated zero-energy lights poison unrelated cell identities
    payload position/radiance/sourceWeight changes reshuffle source identities
    payloadOnlyChange is treated as safe cache reuse or only current-payload
        reweighting without clearing persistent candidate contents
    cached candidateWeight is used as provider or PDFNEE consume proof without
        current RAB replay and current-payload reweighting
    broad lights are suppressed to make the view look correct

    for visual testing. This lane is preserved but is not the default Remix-like
    NEE cache consumer.

Expected:

    temporal off
    spatial off
    clean RTXDI temporal untouched
    selected dense RLU identity
    r_pathTracingRestirPdfNeeVerifierSourcePolicy 2 selects the provider lane
    PDFNEE binding layout exposes PathTraceNeeCacheProviderResults at t74
    PDFNEE binding layout exposes PathTraceNeeCacheCells at t75
    PDFNEE binding layout exposes PathTraceNeeCacheCandidates at t77
    PDFNEE maps the surface world position through the same NEE cache log/hash

    cache candidate reads reject hash-cell collisions where the stored cell

        the fallback domain instead of reusing one selected provider result for

Rung V-08: Experimental PDFNEE Importance / RIS Proposal Selection
------------------------------------------------------------------

Purpose:

    Prove sourcePolicy 2 is a real RIS proposal source over the current
    cache/fallback distribution, not a single selected light from the mapped
    cache cell.

    This rung validates only the rbdoom research/visual lane. It must not be
    used to claim RTX Remix NEE-cache parity, ReSTIR PT parity, or production
    primary-DI readiness.

Runtime result:

    A valid NEECACHE-08 dump proves the RIS wrapper and sourcePolicy 2 plumbing
    are active, but similar visual behavior to the previous direct NEE-cache
    lane is not a deployment failure. If conservative RLU mapping/payload
    invalidation is active, the lane may not retain enough useful state to
    stabilize as primary DI.

Expected:

    sourcePolicy 2 streams multiple proposals per pixel
    proposal count follows r_pathTracingRestirPdfNeeVerifierSamples
    cache candidates are populated by persistent per-hit bounded RIS slot
        updates over the source domain
    candidate admission refreshes duplicates, uses stable identity-owned target
        slots, fills empty slots, and replaces an occupied target slot only with
        a materially stronger current candidate
    PDFNEE samples populated cache candidates by current replay/importance
        weight
    fallback proposals remain mixed with fallbackProbability
    every streamed proposal has exact sourcePdf/invSourcePdf
    RTXDI_StreamSample and RTXDI_FinalizeResampling perform RIS
    dense current RLU identity output
    same RAB replay boundary
    temporal and spatial remain off

Pass:

    changing PDFNEE sample count changes sourcePolicy 2 quality/noise
    dump reports the RIS helper sequence
    dump reports persistent-slot bounded-RIS candidate build policy
    retained candidates survive new samples unless a stronger candidate is
        admitted
    no cell-sized one-light projection remains as the dominant behavior
    report says whether the remaining instability makes the lane research-only

Fail:

    sourcePolicy 2 streams only one cache/fallback proposal per pixel
    sourcePdf omits the cache/fallback mixture probability
    RIS is faked by final direct-light contribution inside the cache
    worker claims RTX Remix or ReSTIR PT use this NEE cache as primary DI
    worker treats sourcePolicy 2 as the production proof path for primary DI
    worker weakens RLU signature invalidation to make this visual lane retain
        candidates without a separate proof against stale light poisoning


Rung V-09: Clean RTXDI DI Spatial Proposal Provider
---------------------------------------------------

Purpose:

    Prove clean RTXDI DI can use ReGIR-style spatial proposals from the
    provider without making the provider a reservoir owner or visible cell-grid
    lighting path.

Expected:

    clean RTXDI DI current reservoir construction remains the owner
    provider returns dense current RLU identity and sourcePdf/invSourcePdf only
    r_pathTracingCleanRtxdiDiNeeCacheProvider enables the provider path
    clean sentinel binds provider/cell/candidate buffers as SRV t74/t75/t77
    each pixel draws its own candidate from the mapped cell / fallback mixture
    full-RLU or typed fallback remains explicit
    selected candidate replays through normal RAB helpers
    RTXDI_StreamSample and RTXDI_FinalizeResampling perform selection
    clean DI temporal accumulation is unchanged

Pass:

    clean RTXDI DI dump reports cleanNeeCacheProvider requested/ready=1/1
        when the provider is enabled and ready
    provider pipeline initialization is allowed before the first buildable map
        location/source domain, but providerReady remains false until the
        provider result/cell/candidate buffers are valid for consumption
    sourceDomain 0 can reach provider-ready once dense current RLU lightCount is
        nonzero, even if typed ranges are not active yet; sourceDomain 1/2/3
        still require their selected typed source range
    on first provider request or fresh NEE-cache resource allocation, clean DI
        dump reports startupDelay counting down from 20 and requested/ready=1/0
        before the first provider snapshot is built
    before the delayed refresh completes, camera movement/rotation reports
        stableView=0, resets stableFrames, restarts startupDelay, and keeps
        providerSnapshotHold=0
    after providerSnapshotHold=1, camera movement may report stableView=0 but
        must not disable the clean NEE-cache provider solely because the camera
        moved
    after startupDelay reaches zero, clean RTXDI DI dump reports startupRefresh
        counting down from 8 with buildPrepass=1, then providerSnapshotHold=1
        while the delayed refreshed snapshot is reused
    entering clean-DI view 8 band 10 releases providerSnapshotHold so the
        diagnostic visual refresh owns the shared cache buffers
    leaving clean-DI view 8 band 10 reports band10ExitedClear=1 once and
        clears the diagnostic-owned cache buffers before view 12 can hold a
        clean provider snapshot again
    frames that begin with taskClearPending report deferredByClear=1 and do not
        consume a new clean provider snapshot until the next clean frame, but
        may run buildPrepass=1 after clearing so moving/animated scenes do not
        wait for a quiet RLU-signature frame
    resource-allocation and diagnostic-ownership clear frames may warm the
        pipeline/cache buffers but must not set providerSnapshotHold=1 until a
        later normal clean frame rebuilds the snapshot
    clean RTXDI DI dump reports providerResultSrv=t74 cellSrv=t75
        candidateSrv=t77
    fallback-only provider matches current clean RTXDI DI full-RLU baseline
    enabling cache proposals changes proposal distribution/noise, not the
        light universe or reservoir page contract
    selected-dense-index debug view is per-pixel stochastic/dithered and does
        not show one deterministic light square per cell
    clean RTXDI DI view 12 remains the proof target

Fail:

    provider returns a finished DI reservoir
    provider computes final direct-light color for clean DI
    all pixels in a cell reuse one selected light
    cell boundaries are visible as synchronized light-selection squares
    clean RTXDI temporal accumulation is edited to make the provider work


Rung V-10: GI / Secondary NEE Cache Consumer
--------------------------------------------

Purpose:

    Prove the Remix-like NEE cache is consumed by path/secondary NEE instead
    of primary DI.

Expected:

    task feedback comes from real path/secondary events
    emissive triangle/surface candidate consume is the first production target
    selected candidate carries exact selection PDF, solidAnglePdf, emitted
        radiance, visibility, and MIS inputs
    bounce policy is explicit
    disabling this consume path leaves clean RTXDI DI unchanged

Pass:

    GI/secondary NEE cache can be A/B tested independently from clean DI and
        from PDFNEE sourcePolicy 2
    stale/invalid candidates reject or fall back with explicit reason
    no cached radiance is painted directly as final lighting

First-slice runtime recipe:

    r_pathTracingRemixLightUniverseEnable 1
    r_pathTracingNeeCacheEnable 1
    r_pathTracingNeeCacheSecondaryEnable 1
    r_pathTracingNeeCacheSecondaryDump 1

Isolation checks:

    clean RTXDI DI may be disabled or left unchanged
    r_pathTracingNeeCacheSecondaryEnable 0 returns to the existing ReSTIR PT
        path-tracer NEE path
    empty or stale cells must not black out lighting; they reject and fall back
    the dump must report task=NEECACHE-10 and firstMissingContract=none before
        visual A/B is meaningful
    validation is normal-render A/B with r_pathTracingNeeCacheSecondaryEnable
        0/1 plus clean-DI view 8 band 10 candidate-field diagnostics
    firstMissingContract=no-emissive-secondary-consume-domain means the first
        slice is correctly disabled because it only consumes emissive cache
        candidates and the current RLU has no emissive range
    legacy mode 18 is not a valid NEECACHE-10 validation route
    ReSTIR PT debug modes 53-55 are not NEE-cache/RLU/clean-RTXDI proof views
    clean-DI view 8 band 10 is only a candidate-field visualizer: it must show
        the visible primary surface mapped into cache cells and current-RLU
        emissive candidate identities/PDF scale, not final secondary lighting
    r_pathTracingNeeCacheSecondaryVisualRefresh 1 is a one-shot diagnostic
        refresh with a held band-10 snapshot until the view is changed;
        continuous refresh is not a performance baseline

Fail:

    GI cache replaces primary DI
    cached candidate radiance is used without MIS/replay
    stale map or animated-light candidates survive reset as valid lighting


Hard Stop
---------

Stop at the first failing rung.

Do not continue to PDFNEE, clean DI, GI/secondary consume, temporal, spatial,
best lights, or full ReGIR to hide a failing cell/candidate/PDF contract.
