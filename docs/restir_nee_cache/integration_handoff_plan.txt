Remix-Style NEE Cache Integration Handoff Plan
==============================================

Purpose
-------

This file defines handoffs from standalone NEE cache / spatial proposal proof
to the three allowed consumers:

    default GI / secondary NEE cache consumer
    clean RTXDI DI spatial proposal provider
    preserved experimental PDFNEE visual consumer

The old assumption that PDFNEE is the default destination is stale. PDFNEE
sourcePolicy 2 remains useful and must stay available as an alternate visual
testing lane, but it is not the default Remix-like architecture.

Reference caveat:

    RTX Remix consumes its NEE cache from indirect/path integration, after the
    RTXDI DI pass. ReSTIR PT / RTXDI do not use a Remix-style NEE cache as the
    primary DI path. The rbdoom PDFNEE sourcePolicy 2 lane is therefore a
    research/visual current-frame proposal mode, not a reference-derived primary
    DI design.


Producer Output
---------------

The shared proposal/cache layer may expose:

    cache buffers
    cache parameters
    selected dense current RLU identity for DI/PDFNEE proposal lanes
    selected emissive triangle/surface candidate for GI/secondary NEE
    sourcePdf / invSourcePdf
    source label
    fallback mixture probability
    cell id
    empty/fallback reason

It must not expose:

    final direct-light color for primary DI
    temporal reservoir
    spatial reservoir
    best-light candidate
    old NEE output


Provider ABI Requirement
------------------------

NEECACHE-01 must define the concrete rbdoom-owned provider ABI before any
candidate algorithm work begins.

The ABI must state:

    HLSL struct name and fields
    C++ mirror struct name if CPU readback or upload uses one
    binding/register slots for provider result, cell, task, and candidate
        buffers
    buffer owner
    buffer reset/lifetime policy
    provider function name
    future consumer call/read sites:
        GI / secondary NEE
        clean RTXDI DI proposal
        experimental PDFNEE visual
    sourceLabel enum values
    fallbackReason enum values

The minimum provider result is:

    selectedDenseRluIndex
    sourcePdf
    invSourcePdf
    sourceLabel
    mixtureProbability
    fallbackReason
    cellIndex
    candidateSlot
    flags

No consumer task may start while its ABI is ambiguous.


Default GI / Secondary NEE Consumer Boundary
--------------------------------------------

The Remix-aligned NEE cache consumer is indirect/path integration.

It may consume the cache as:

    emissive triangle/surface proposal
    exact selection PDF
    solidAnglePdf
    emitted radiance
    visibility result
    MIS input
    empty-cell fallback decision

It must not:

    replace primary DI
    write clean RTXDI DI reservoir pages
    paint cached radiance directly without MIS/replay
    persist animated analytic-light candidates unless RLU marks them
        stableCacheable and the consumer replays current payloads


Clean RTXDI DI Proposal Boundary
--------------------------------

Clean RTXDI DI may consume a ReGIR-style spatial provider only as an initial
proposal source.

The provider returns:

    selectedDenseRluIndex
    sourcePdf
    invSourcePdf
    sourceLabel
    mixtureProbability
    fallbackReason

Clean DI still owns:

    current RAB surface
    RAB_LoadLightInfo
    RAB_SamplePolymorphicLight
    RAB_GetLightSampleTargetPdfForSurface
    RTXDI_StreamSample
    RTXDI_FinalizeResampling
    visibility
    temporal/spatial reuse
    final shading

The provider must not return a finished reservoir or a cell-owned final light.
Each pixel must draw its own proposal from the mapped cell / fallback mixture
so the image does not reveal square cell boundaries.

NEECACHE-09 implementation hook:

    r_pathTracingCleanRtxdiDiNeeCacheProvider
    clean sentinel SRV bindings t74/t75/t77 for provider result, cell, and
        candidate buffers
    dump proof:
        cleanNeeCacheProvider requested/ready=1/1
        providerResultSrv=t74 cellSrv=t75 candidateSrv=t77


Experimental PDFNEE Visual Consumer Boundary
--------------------------------------------

PDFNEE may consume the NEE cache only as one source distribution for selecting
a current-frame light candidate. This is an opt-in visual experiment, not the
default architecture.

PDFNEE still owns:

    current RAB surface
    sample UV
    RAB_LoadActiveRrxLightInfo
    RAB_SampleActiveRrxPolymorphicLight
    RAB_GetLightSampleTargetPdfForSurface
    RAB_GetReflectedBsdfRadianceForSurface
    visibility policy
    RTXDI current reservoir construction


Experimental PDFNEE Current Consume Shape
-----------------------------------------

NEECACHE-07 adds the first current-frame consume bridge:

    r_pathTracingRestirPdfNeeVerifierSourcePolicy 2 selects the NEE cache
        provider
    PDFNEE reads PathTraceNeeCacheProviderResults at t74
    PDFNEE reads PathTraceNeeCacheCandidates at t77
    PDFNEE maps the primary-hit world position to the same log/hash cell
    PDFNEE draws a per-pixel cache/fallback proposal from the mapped cell and
        streams selectedDenseRluIndex into the existing RTXDI current reservoir
        path using the selected proposal's invSourcePdf
    keep temporal off
    keep spatial off
    dump effective cache parameters and mixture probabilities
    report neeCacheProvider requested/ready and candidateSrv=t77 in the PDFNEE
        dump

It must not:

    edit clean RTXDI temporal accumulator
    edit RRX
    enable spatial reuse
    enable best-light feedback
    route through mode 56
    revive old PDFNEE lightMode 8/9
    silently force different cache parameters than the NEE cache dump reports


Negative Paths
--------------

PDFNEE must handle:

    NEE cache disabled
    cache resource invalid
    no current RLU lights
    empty cell
    invalid selected dense index
    sourcePdf <= 0
    RAB load failure
    targetPdf <= 0
    solidAnglePdf <= 0
    visibility rejection
    scene/map/light-domain reset
    sourcePolicy 2 with an unavailable provider result buffer


First Runtime Proof
-------------------

Standalone NEE cache proof first:

    r_pathTracingDebugMode 0
    r_pathTracingNeeCacheEnable 1
    r_pathTracingNeeCacheDebugView 2 through 10
    r_pathTracingNeeCacheDump 1

Current consume validation:

    r_pathTracingRestirPdfNeeVerifierSourcePolicy 2
    PDFNEE dump reports sourcePolicy=2(nee-cache-provider)
    PDFNEE dump reports neeCacheProvider requested/ready=1/1
    PDFNEE dump reports candidateSrv=t77
    temporal off
    raw current-frame output
    noisy but plausible lighting

Clean RTXDI temporal consume is not proven by this lane. If PDFNEE visual
quality is good, keep it as a research/visual test mode and require a later
task before routing it into temporal reuse.


Stop Conditions
---------------

Stop before any consumer integration if:

    cells are screen-space anchored
    cells or candidates strobe while standing still
    selected identity is not dense RLU replayable
    sourcePdf / invSourcePdf is missing
    fallback mixture is missing
    valid candidates cannot replay through RAB
    emissives require an old debug route to populate first
    old map candidates survive reset
    output falls back to beauty/albedo


Next Provider Rule
------------------

If the first sourcePolicy 2 consume pass is too sparse, do not treat that as a
failure of the default Remix-like NEE cache. It is only the experimental PDFNEE
visual lane.

For clean DI, the next rung is a lightweight ReGIR-style spatial proposal
provider that feeds clean RTXDI DI initial sampling:

    dense RLU identity
    sourcePdf / invSourcePdf
    source label
    fallback mixture
    per-pixel stochastic candidate draw

For GI / secondary NEE, the next rung is real path/secondary task feedback and
emissive triangle/surface candidate consume with MIS:

    triangle/surface identity
    selection PDF
    solidAnglePdf
    emitted radiance
    visibility
    MIS data

NEECACHE-10 first slice consumes only current-RLU emissive cache candidates in
the ReSTIR PT path-tracer NEE boundary, RAB_RecordSmokeNeeSample during initial
path sampling. The selected candidate is replayed through RAB light
loading/sampling, visibility, reflected BSDF radiance, and MIS. Empty, stale,
non-emissive, or non-replayable records fall back to the existing path-tracer
NEE path. This slice must not replace primary DI or write clean DI reservoir
pages. Legacy mode 18 is not a valid validation route for this task.
ReSTIR PT debug modes 53-55 are not NEE-cache/RLU/clean-RTXDI proof views in
the current branch. The first dedicated visualizer is clean-DI view 8 forced
band 10:

    r_pathTracingCleanRtxdiDiView 8
    r_pathTracingCleanRtxdiDiView8Band 10
    r_pathTracingNeeCacheSecondaryVisualRefresh 1

This view maps the visible primary surface into the same cache cells and shows
current-RLU emissive candidate occupancy, deterministic dense-RLU identity, and
sourcePdf/weight scale. It is a candidate-field diagnostic only; normal-render
A/B remains the consume test until a true secondary-bounce visualizer is added.
The visual refresh CVar is intentionally default-off because continuous refresh
dispatches the cache candidate builder from the clean route and can be expensive
when the view crosses portal-heavy surfaces.

For the preserved PDFNEE visual lane, proper importance/RIS selection remains
behind the same producer output:

    selected dense RLU identity
    sourcePdf / invSourcePdf
    source label
    fallback mixture

PDFNEE should stream multiple cache/fallback proposals per pixel using
r_pathTracingRestirPdfNeeVerifierSamples, then let RTXDI_StreamSample and
RTXDI_FinalizeResampling choose the current-frame reservoir sample.

If this remains unstable under camera motion, dense light churn, or animated
lights, preserve the lane as a diagnostic/research mode and switch production
work to either the GI/secondary NEE consumer or the separate clean DI proposal
provider.

A full ReGIR/onion provider remains a later escape hatch only after the
lightweight DI proposal provider fails, not merely because the experimental
PDFNEE visual lane is noisy.
