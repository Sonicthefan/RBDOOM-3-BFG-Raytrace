NEE Cache Provider ABI
======================

Status
------

Implemented for NEECACHE-01 as a CPU/resource-shell ABI.

This file resolves the previous first missing contract:

    provider-abi

The next missing contract after NEECACHE-01 is:

    cell-mapping-not-implemented-neecache-02


Owner
-----

CPU/resource owner:

    neo/renderer/NVRHI/PathTraceNeeCache.cpp
    neo/renderer/NVRHI/PathTraceNeeCache.h
    PathTracePrimaryPass::m_smokeNeeCacheState

No old ReGIR/PDFNEE verifier route owns this ABI.

Consumer roles:

    GI / secondary NEE:
        default Remix-style NEE cache destination; consumes emissive
        triangle/surface candidates with explicit PDFs and MIS.

    Clean RTXDI DI proposal:
        may consume dense current RLU identities and sourcePdf/invSourcePdf as
        initial proposals only. Clean DI owns reservoir construction,
        temporal/spatial reuse, visibility, and final shading.

    PDFNEE visual:
        preserved alternate sourcePolicy 2 testing lane. Keep it available for
        image-quality and research comparisons, but do not treat it as the
        default Remix-like consume path.


Provider Function Boundary
--------------------------

Future shader function name:

    PathTraceNeeCacheSelectProposal

Input side:

    current RLU dense light count
    current RLU typed ranges
    current dense RLU payloads
    current RAB surface/world position
    provider RNG/sample state

Output side:

    PathTraceNeeCacheProviderResult

The provider returns identity and source-distribution metadata only. It does
not compute final direct-light contribution, visibility, temporal reservoirs,
spatial reservoirs, best-light state, or denoiser/confidence/gradient data.
It must not return a finished DI reservoir or one deterministic cell-owned
light for all pixels in a cell.


Clean RTXDI DI Proposal Boundary
--------------------------------

Clean RTXDI DI may consume this provider only as an initial proposal source.

Allowed output:

    selectedDenseRluIndex
    sourcePdf
    invSourcePdf
    sourceLabel
    mixtureProbability
    fallbackReason

Required consume shape:

    opt in with r_pathTracingCleanRtxdiDiNeeCacheProvider
    bind provider result, cell, and candidate buffers as SRV t74/t75/t77 in
        the clean RTXDI DI sentinel
    map current surface world position to cache/proposal cell
    draw a per-pixel stochastic candidate from the cell or full-RLU/typed
        fallback
    replay the selected dense RLU identity through the normal RAB helpers
    stream the proposal through RTXDI_StreamSample / RTXDI_FinalizeResampling

Forbidden consume shape:

    read one preselected provider result for every pixel in a cell
    consume a provider-owned finished reservoir
    consume provider-computed final direct-light color
    edit clean RTXDI temporal accumulation to hide proposal artifacts


GI / Secondary NEE Boundary
---------------------------

The default Remix-style NEE cache consumer is the path/indirect integrator.
In rbdoom that boundary is RAB_RecordSmokeNeeSample during ReSTIR PT initial
path sampling. Legacy mode 18 is not part of the validation contract.

The GI/secondary consumer may use:

    emissive triangle/surface identity
    selected candidate PDF
    solidAnglePdf
    emitted radiance
    visibility result
    MIS input
    fallback reason

This consumer must be testable independently from clean RTXDI DI and from the
experimental PDFNEE visual lane.


PDFNEE Visual Boundary
----------------------

This boundary is preserved as an alternate visual testing mode. Do not delete
it unless a later task explicitly asks for that. Do not claim default Remix
NEE-cache parity from this lane alone.

Call/read site:

    pathtrace_restir_pdf_nee_rlu_current.rt.hlsl:current-frame-source-policy

The wrapper shader includes pathtrace_smoke.rt.hlsl, so the concrete
implementation lives in the RB_PT_RESTIR_PDF_NEE_RLU_CURRENT_PRODUCER_ONLY
section of:

    neo/shaders/builtin/pathtracing/pathtrace_smoke.rt.hlsl

PDFNEE selects the provider lane with:

    r_pathTracingRestirPdfNeeVerifierSourcePolicy 2

The PDFNEE binding layout exposes:

    t74 PathTraceNeeCacheProviderResults
    t75 PathTraceNeeCacheCells
    t77 PathTraceNeeCacheCandidates

The shader maps the current primary-hit surface world position through the
same PathTraceNeeCacheMapWorldPositionToCell helper, then draws a proposal per
pixel from that cell's fixed candidate list or from the current-frame fallback
domain. This is deliberately not a read of one preselected provider result per
cell; that older shape made cell-sized sample correlation visible in PDFNEE.
The selected dense RLU index is streamed into the existing RTXDI
current-reservoir construction using an invSourcePdf computed from:

    cache:    weightedCandidateIdentityPdf * cacheProbability
    fallback: fallbackDomainPdf * fallbackProbability

If no usable cache candidate exists, fallback is forced with mixture
probability 1.

PDFNEE keeps RAB ownership of:

    RAB_LoadActiveRrxLightInfo
    RAB_SampleActiveRrxPolymorphicLight
    RAB_GetLightSampleTargetPdfForSurface
    RAB_GetReflectedBsdfRadianceForSurface
    visibility policy
    RTXDI current reservoir construction

Provider unavailability is explicit. If sourcePolicy is 2 but the NEE cache
result buffer is not ready for the frame, the dump reports:

    firstMissingContract=nee-cache-provider-not-ready-neecache-07

No fallback to the typed policy is hidden under sourcePolicy 2.

Task Accumulation Diagnostic
----------------------------

NEECACHE-03 uses the existing task buffer at u76.

Insert policy:

    debug-primary-visible-hit-slot0-atomic-count

For each primary hit in debugView 4, the shader maps the world position to a
cache cell, writes task slot 0 for that cell, and atomically increments a
diagnostic hit-count accumulator. The task class is diagnostic only:

    taskClass = 1
    denseRluIndex = 0xffffffff

Decay policy:

    full-cell-prepass-decay-7of8

Before debugView 4 draws the screen, the same RT debug shader runs a decay
prepass over the physical cell range. One dispatch lane owns one physical cell
and decays stale diagnostic counts by 7/8 per elapsed frame. The visible draw
path then performs only atomic hit-count increments and color reads, so the
screen cannot observe a transient per-pixel decay lock.

Reset policy:

    clear-task-and-cell-buffers-on-allocation-or-scene-reset

The CPU clears task and cell buffers after cache resource allocation. Scene
resource reset destroys the cache state and forces a fresh clear on the next
allocation.

This diagnostic does not prove candidate quality, source PDFs, RAB replay, or
final direct lighting.


Candidate Diagnostics
---------------------

NEECACHE-04 and NEECACHE-05 use the candidate buffer at u77 and the current RLU
payload SRV at t66.

Build policy:

    primary-visible-hit-persistent-slot-update-bounded-ris-over-source-domain

When the cache is enabled and either current dense RLU typed range is nonempty,
the NEE cache requests the current dense RLU producer during scene build and the
debug RT shader runs a build-only prepass before the final dispatch. The prepass
traces primary-visible hits, hashes each mapped cell, and updates one persistent
candidate slot for that hit. The replacement candidate is selected by bounded
RIS over the current source domain using current cell-importance weights. Other
slots in the cell remain cached until reset, signature invalidation, hash-cell
collision, or later slot admission. Admission refreshes duplicate dense
identities in place, prefers a stable identity-owned target slot, fills empty
slots, and replaces an occupied target slot only when the new candidate's
current replay weight is at least 2x stronger. This prevents primary-visible
pixels from churning all slots every frame. This build is independent of debug
views; debug views only visualize the populated records. Views 5 through 10 are
read-only in the visible debug pass; only the build prepass writes provider,
cell, and candidate records. A separate light-universe CVar must not be
required for this lane.

For source domains that can consume emissives, the NEE cache scene-build request
also enables the emissive material/texture and rigid-route inventory producer
used by the accepted RLU consumers. Without that producer handoff, the RLU shell
can exist while the emissive typed range contains only inert zero-weight records.

Candidate identity:

    dense current RLU index

Source-domain policy:

    r_pathTracingNeeCacheSourceDomain is consumed by the shader. Domain 0 uses
    the full dense current RLU as a uniform source domain. Domain 1 restricts
    candidates and typed fallback to the current emissive range. Domain 2
    restricts candidates and typed fallback to the current Doom analytic range.
    Domain 3 builds typed candidate lists and multiplies the per-type source
    PDF by the selected typed class probability.

Source PDF policy:

    candidate sourcePdf = selectedCandidateWeight / risWeightSum
    candidate invSourcePdf = risWeightSum / selectedCandidateWeight
    domain 0 proposals cover currentEmissiveRluRangeCount +
        stableCurrentDoomAnalyticRluRangeCount
    domain 1 proposals cover the current emissive range
    domain 2 proposals cover the stable Doom analytic prefix
    domain 3 proposals multiply typed proposal probability by the selected
        typed class probability
    invSourcePdf = 1 / sourcePdf

If a current emissive RLU record carries a zero sourcePdf, the diagnostic uses
uniform emissive-range fallback for that candidate and the dump reports the
fallback policy. This preserves a nonzero replay field for validation without
claiming final PDFNEE consumption.

Analytic records are selected from the current dense RLU Doom analytic typed
range and store dense current RLU identity. Candidate slots also carry a
spatial weight. Emissive weights use a cell-distance, sourceWeight, area, and
normal-facing proxy. Analytic weights use a cell-distance, sourceWeight,
luminance, area, radius, and influence-radius proxy. These are proposal weights
only; target PDF and final contribution remain outside the cache.

The current dense RLU Doom analytic typed range is not automatically a cacheable
range. RLU records expose explicit stability flags:

    currentSampleable
    stableCacheable
    unstableDynamic

and rejection reason flags:

    noRemap
    payloadChanged
    unprovenContinuity
    unknownIdentity
    duplicateIdentity
    portalDisconnected
    outOfSelectedArea

The provider may use currentSampleable lights as fallback/current-frame direct
lighting only through RAB. Persistent candidate records for Doom analytics must
require stableCacheable. If an analytic is unstableDynamic, the provider must
not write it into candidate cells and must reject any stale buffered candidate
that no longer carries stableCacheable in the current RLU payload.
Portal depth is diagnostic metadata only. It must not be interpreted as
outOfSelectedArea, and it must not reject otherwise stable analytic lights from
the stableCacheable prefix.

For dense Remix RLU mode the manager sorts stableCacheable Doom analytics to
the front of the Doom analytic typed range and reports the stable count. The
NEE cache and ReGIR-style persistent consumers sample only that stable prefix
when writing candidate cells. Current-frame full-RLU diagnostics may still
iterate the full current typed range to inspect direct-light replay.
Fallback is current-frame only and may sample the full current RLU or full
current typed range; unstable analytic fallback selections must not be copied
back into persistent candidate cells.

Candidate validity policy:

    reject-zero-area-or-zero-energy-emissive
    reject-analytic-not-rlu-stableCacheable
    analytic-zero-cell-weight
    provider-rejects-rab-replay-failed-or-zero-current-weight

The emissive typed range can contain structurally emissive payloads with zero
sourceWeight and zero luminance. NEECACHE-04 does not write those records as
valid cache candidates because they cannot be useful NEE proposals. DebugView 5
uses a separate dark-red failure color for this state.

Doom analytic identity selection is typed-range based and uses only current RLU
payload fields, never raw DoomAnalyticLights sourceIndex dereferences. DebugView
6 is therefore a proposal identity map, while debugView 10 reports whether the
selected current RLU payload can replay as a nonzero RAB light this frame.

Debug views:

    5 emissive candidate map
    6 analytic candidate identity map
    7 analytic sourcePdf / invSourcePdf map, identity-tinted because the
        current analytic PDF is uniform over the typed range
    8 cache/fallback source label
    9 fallback reason
    10 analytic RLU payload replay validity
    11 flat consumed-candidate render
    12 flat full current-RLU render

Provider result policy:

    PathTraceNeeCacheSelectProposal writes PathTraceNeeCacheProviderResults
    at u74 with one result per physical cell. The selected identity is always a
    dense current RLU index. Cache selection scans the fixed candidate slot
    list, but a buffered record is selectable only after
    RAB_LoadActiveRrxLightInfo(candidate.denseRluIndex, false) succeeds for
    the current frame and the current RLU payload type still matches the
    candidate class. The provider then recomputes the candidate weight from the
    current RLU payload before contributing it to selection or sourcePdf.

    If the selected dense identity appears in more than one slot, the provider
    sums duplicate-slot current replay weights and writes
    sourcePdf = currentReplayWeightedCandidateIdentityPdf * cacheProbability.
    A fallback selection writes sourcePdf = domainPdf * fallbackProbability. If
    no valid cache candidate exists, fallback is forced with
    mixtureProbability = 1. Disabled/no-RLU/zero-PDF cases write an invalid
    result with the documented fallbackReason. Buffered candidateWeight is a
    build-time diagnostic hint, not provider consume proof.

This diagnostic does not consume old ReGIR/PDFNEE routes and does not perform
final direct-light contribution inside the cache. The NEECACHE-07 PDFNEE bridge
is a separate alternate visual consumer of the same provider result buffer.

DebugView 11 is a visual inspection route for the light data consumed by this
diagnostic lane. It shades flat surfaces from the current candidate records and
the dense current RLU payload at t66 by replaying candidate dense indexes
through RAB_LoadActiveRrxLightInfo, RAB_SampleActiveRrxPolymorphicLight,
RAB_GetLightSampleTargetPdfForSurface, and RAB_GetReflectedBsdfRadianceForSurface.
Geometry SRVs are used only to recover a flat diagnostic surface. It
intentionally skips textures, denoising, temporal/spatial reuse, best-light
feedback, visibility proof, and PDFNEE consumption.

DebugView 12 uses the same flat shading and the same current dense RLU identity
payload but bypasses the candidate slot filter and loops over the full current
typed RLU ranges. It must not dereference DoomAnalyticLights[record.sourceIndex]
for shading; raw split-domain SRVs may be bound only because the shared RAB
bridge helpers declare them. The diagnostic replay boundary is the dense
current RLU index. It is the control view for separating bad source payload or
range data from bad cache proposal quality.


Buffer Lifetime
---------------

Buffer owner:

    PathTraceNeeCacheState

Lifetime:

    Created when r_pathTracingNeeCacheEnable 1 and mode is nonzero.
    Destroyed when disabled, invalid, or scene resources are reset.
    Recreated when the fixed budget or stride changes.
    Cleared when resources are allocated, scene resources reset, or the current
    RLU structural, mapping, payload, or payload-only signature state changes.
    Candidate identities remain dense current RLU indexes for DI/PDFNEE
    proposal lanes, and every provider/PDFNEE consume path must replay the
    current payload and recompute current cell weight before selection.
    GI/secondary NEE triangle candidates must likewise replay current geometry
    and material identity before use.

Buffers:

    PathTraceNeeCacheProviderResult
    PathTraceNeeCacheCells:
        one record per hash-cell index
        stores the source cell hash as the collision tag
        provider and PDFNEE cache reads reject candidate lists whose stored
            hash does not match the currently mapped world cell
    PathTraceNeeCacheTasks
    PathTraceNeeCacheCandidates
    PathTraceNeeCachePlaceholderSRV

Default fixed budget:

    cellCount = 65536
    taskSlots = 8 per cell
    candidateSlots = 8 per cell
    providerResultCount = one result record per cell

Invalidation:

    PathTraceNeeCacheState observes the current Remix light-manager structural,
    mapping, and payload signatures after resource creation.

    The provider result, cell, task, and candidate buffers are cleared on:

        resource allocation
        scene/resource reset
        RLU structural signature change
        RLU mapping signature change
        RLU payload signature change
        RLU payloadOnlyChange

    Payload-only animation changes can preserve dense identity while changing
    the light state that made a candidate relevant to a cell. Persistent
    candidate records are therefore cleared on payloadOnlyChange. Selected
    dense RLU indexes are still replayed through the active RLU/RAB helpers,
    and both the provider and PDFNEE consumer recompute current cell weights
    before choosing or using a cached candidate. Stored candidateWeight is a
    build-time hint, not consume proof.
