Remix-Style NEE Cache Worker Tasks
==================================

Run tasks in order within the selected lane. Do not skip ahead.

Architecture note:

    The default Remix-aligned NEE cache lane feeds GI / secondary NEE.
    Clean RTXDI DI gets a separate lightweight ReGIR-style spatial proposal
    provider over dense RLU identities.
    The PDFNEE sourcePolicy 2 consume path is preserved as an alternate visual
    testing lane and must not be deleted, but it is not the default Remix-like
    consumer.
    RTX Remix and ReSTIR PT do not use the Remix-style NEE cache as their
    primary DI path. Do not treat sourcePolicy 2 as a reference primary-DI
    design.


================================================================================
BINDING CONTRACTS  (authoritative; override any conflicting wording anywhere
in this folder, including older task bodies below and the other *.txt files)
================================================================================

These three contracts were added after a clean-DI consume implementation
(NEECACHE-09) shipped that BUILT correctly but produced emissive noise and
movement strobing. Root cause was under-specified PDF/measure wording plus an
ambiguous cell frame. Every consume task (06, 07, 08, 09, 10) and every
validation step must satisfy these.


CONTRACT A — Single PDF / measure convention (end to end)
---------------------------------------------------------

There is ONE measure: solid angle at the shading surface. Every proposal that is
streamed into a reservoir, and the final resolve, must use it consistently.

    targetPdf (p_hat):
        = RAB_GetLightSampleTargetPdfForSurface(lightSample, surface)
        unnormalized importance of the actual contribution
        (luminance of brdf * radiance * NdotL), in solid-angle measure.
        RAB owns it. Do not reconstruct it in the cache.

    sourcePdf (the pdf the proposal was actually drawn from):
        = P_select(light) * lightSample.solidAnglePdf
        where:
          - P_select(light) is the probability the proposal step chose this
            light, INCLUDING the cache/fallback mixture (see Contract A.2), and
          - lightSample.solidAnglePdf is the per-sample area/solid-angle
            sampling pdf returned by RAB_SamplePolymorphicLight.
        invSourcePdf = 1.0 / max(sourcePdf, eps).

        REQUIRED: the per-sample solidAnglePdf MUST be folded into sourcePdf.
        A selection-only sourcePdf (P_select alone, no solidAnglePdf) is a BUG.
        It is the direct cause of emissive-area-light noise, because large/far/
        grazing emitters have tiny solidAnglePdf and explode the RIS weight.

    resolve:
        final = (brdf * radiance * NdotL) * UCW * visibility
        UCW = RTXDI_GetDIReservoirInvPdf(reservoir).
        Do NOT divide again by solidAnglePdf in the resolve. solidAnglePdf
        already lives in sourcePdf, and UCW already carries 1/targetPdf.
        => r_pathTracingCleanRtxdiDiResolveSolidAnglePdf must be 0 under this
           convention. The dividing path is retired; keep the cvar only as a
           diagnostic, default 0.

    Equivalently: a streamed proposal's RIS weight is
        w = targetPdf / sourcePdf
          = p_hat / (P_select * solidAnglePdf)
    and the integrand multiplied by UCW at resolve is the raw contribution with
    no extra pdf division. Source, target, and resolve must never disagree on
    where solidAnglePdf lives.

CONTRACT A.2 — Cache/fallback as a real mixture
-----------------------------------------------

    A proposal may come from the cache cell OR the fallback domain. P_select for
    the drawn sample is the MIXTURE probability, not the probability of the
    branch that happened to be taken:

        P_select(light) =
            cacheProbability   * p_cache(light)
          + fallbackProbability* p_fallback(light)

        cacheProbability + fallbackProbability = 1.

    Both p_cache and p_fallback are evaluated for the SAME drawn light, even
    when only one branch generated it (defensive / one-sample MIS). Using only
    the taken branch's pdf is a BUG: it biases cache-served vs fallback-served
    pixels and produces brightness seams between cells.

    fallback-only (cacheProbability = 0) MUST reduce exactly to the existing
    uniform/full-RLU baseline.


CONTRACT B — Cell coordinate frame is WORLD-ANCHORED
----------------------------------------------------

    Cells are anchored in world space, NOT relative to the camera origin.
    A pixel maps world position -> a fixed world cell whose id does not move
    when the camera moves.

    Camera-relative ("onion") grids shift their cell boundaries under camera
    motion, which pops the per-cell candidate set frame to frame and strobes
    under movement. That mode is demoted to the later escape hatch ONLY.

    Required: a stationary world point keeps the same cell id regardless of
    camera motion or rotation. "world-space stable while standing still" is not
    sufficient wording; the test is stability under camera movement.

    A world-anchored grid may still use variable cell size by distance for
    budget (finer near the view, coarser far) as long as a given world point's
    membership only changes when it crosses a fixed world boundary, not when the
    camera moves.


CONTRACT C — Validation proves CONVERGENCE, not just "distribution changed"
--------------------------------------------------------------------------

    Acceptance for every consume task must include a ground-truth / convergence
    check, because a measure bug (Contract A) affects the cache and the baseline
    the same way and therefore passes a baseline-only A/B.

    Required validation, in addition to the existing baseline A/B:

      1. Energy / convergence: with temporal accumulation on and the camera
         held still, cache+fallback must converge to the SAME image as
         fallback-only (within noise), not merely "look different." Compare a
         held-still accumulated frame, or a long-accumulation reference.
      2. Emissive stress: a bright/large emissive panel must not show
         brightness that scales with how rarely it is sampled. Doubling emitter
         intensity must not increase variance disproportionately once Contract A
         holds.
      3. Movement: walking past world cells must not strobe (Contract B).

    "fallback-only matches baseline" and "cache changes proposal distribution"
    are necessary but NOT sufficient and must never be the only acceptance.

NEECACHE-09R: Bring clean RTXDI DI consume into compliance (START HERE)
================================================================================

Context:

    NEECACHE-09 already streams per-cell candidates into the per-pixel clean DI
    reservoir (RunNeeCacheProviderProducer -> StreamProviderIntoReservoir ->
    RTXDI_StreamSample), which is the correct architecture: cells are a
    candidate SOURCE, shading stays per-pixel. Do NOT redesign that. It is
    non-compliant only on Contract A (source PDF omits solidAnglePdf and uses
    branch-only mixture) and Contract B (camera-relative cell mapping).

Goal:

    Make the existing clean DI consume satisfy Contract A, A.2, and B, and
    validate it with Contract C. No new architecture, no temporal/spatial
    redesign.

Required:

    Contract A:
        - In PathTraceCleanRoomNeeCacheStreamLightIntoReservoir, the streamed
          invSourcePdf must be 1 / (P_select * lightSample.solidAnglePdf).
          P_select is the value from selection (cache) or fallback, made into a
          mixture per Contract A.2.
        - The light sample used to derive solidAnglePdf must be the SAME sample
          (same UV) that is streamed. Do not select with one UV and stream with
          a fresh unrelated UV; draw the UV, sample the light, read its
          solidAnglePdf, then stream that exact sample.
        - Set r_pathTracingCleanRtxdiDiResolveSolidAnglePdf default 0 and make
          the resolve NOT divide by solidAnglePdf (Contract A resolve rule).
    Contract A.2:
        - selectedInvSourcePdf reflects the cache+fallback mixture probability
          for the drawn light, evaluated in both branches.
    Contract B:
        - PathTraceNeeCacheMapWorldPositionToCell maps to a WORLD-ANCHORED cell
          (remove the camera-origin dependence from cell identity). Variable
          cell size by view distance is allowed; cell membership of a fixed
          world point must not change under camera motion.
    Keep:
        - clean RTXDI DI remains reservoir and accumulation owner
        - per-pixel proposal draw, no cell-owned reservoir, no cell-owned light
        - selected candidate replays through RAB_LoadLightInfo,
          RAB_SamplePolymorphicLight, RAB_GetLightSampleTargetPdfForSurface
        - RTXDI_StreamSample / RTXDI_FinalizeResampling stay the selection path
        - no clean RTXDI temporal/spatial edits

Forbidden:

    no full ReGIR/onion provider
    no cell-owned final light or finished reservoir
    no resolve that divides by solidAnglePdf while sourcePdf also includes it
    no temporal/spatial reuse changes

Acceptance (must satisfy Contract C):

    build/deploy succeeds
    clean RTXDI DI dump reports:
        cleanNeeCacheProvider requested/ready=1/1
        providerResultSrv=t74 cellSrv=t75 candidateSrv=t77
        cellFrame=world-anchored
        resolveSolidAnglePdf=0
    fallback-only (cacheProbability=0) matches the existing full-RLU clean DI
        baseline
    CONVERGENCE: with the camera held still, cache+fallback converges to the
        same accumulated image as fallback-only (not just "different")
    EMISSIVE: a bright/large emissive panel converges without variance that
        scales with emitter intensity; doubling intensity does not blow up noise
    MOVEMENT: walking past world cells does not strobe; debug view shows the
        selected dense index is stochastic/dithered, not square cell blocks
    disabling the provider returns to the known clean DI baseline
