Remix-Style NEE Cache Worker Tasks
==================================

Run tasks in order within the selected lane. Do not skip ahead.

Architecture note:

    The default Remix-aligned NEE cache lane feeds GI / secondary NEE.
    Clean RTXDI DI gets a separate lightweight ReGIR-style spatial proposal
    provider over dense RLU identities.
    The current PDFNEE sourcePolicy 2 consume path is preserved as an
    alternate visual testing lane and must not be deleted, but it is not the
    default Remix-like consumer.
    RTX Remix and ReSTIR PT do not use the Remix-style NEE cache as their
    primary DI path. Do not treat sourcePolicy 2 as a reference primary-DI
    design.

NEECACHE-08: Experimental PDFNEE Importance / RIS Proposal Selection
--------------------------------------------------------------------

Goal:

    Turn the sourcePolicy 2 PDFNEE consume bridge into a real current-frame
    RIS proposal source over the cache/fallback distribution instead of a
    single light draw from the mapped cache cell.

    This improves the preserved PDFNEE visual lane. It is not the clean DI
    proposal-provider integration.

    Current outcome: sourcePolicy 2 can be visually interesting and sometimes
    stabilizes in simple scenes, but it is not stable or performant enough to
    promote to production primary DI. Further work in this lane is optional
    research only; the next production-oriented work should be NEECACHE-09 or
    NEECACHE-10, depending on the selected consumer.

    Validation outcome:

        The deployed NEECACHE-08 path reports the expected sourcePolicy 2 RIS
        sequence and buffer bindings, but visual output remains the same class
        of slightly unstable direct NEE-cache current-frame lighting. Runtime
        dumps can show mapping/payload RLU signature changes and cache
        invalidation, which is required to avoid stale light poisoning but also
        prevents this lane from retaining enough history to behave as a stable
        primary-DI replacement.

Later Escape Hatch: Full ReGIR / Onion Provider
-----------------------------------------------

Only if the lightweight clean DI spatial proposal provider is still
insufficient, add a full ReGIR/onion provider behind the same source-provider
interface. That fallback is not the next rung after NEECACHE-07; PDFNEE visual
noise alone does not justify it.
