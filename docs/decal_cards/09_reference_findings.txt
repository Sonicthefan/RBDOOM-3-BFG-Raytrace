1. RAY TRACING GEMS II, CH.27 "RAY TRACING DECALS" (Bahnassi, NVIDIA, 2021)
   E:/prog/references/Ray_Tracing_Decals.pdf
================================================================================
CORE MODEL (what the chapter is mostly about): a decal = a TEXTURE + a directed
PROJECTION BOX. The receiving surface point is transformed by world2DecalUnit; if
inside [-0.5,0.5]^3 it receives the decal, UV = transformed.xy+0.5. Decals live in
a SEPARATE decal TLAS (or instance-mask subset) and are queried from the base hit.

  => This is the CLUSTERED / box-projection architecture. We REJECTED it (02 sec.3):
     D3 detail decals are authored as coplanar CARDS, not boxes; synthesizing boxes
     loses the authored UV mapping; the per-hit decal query wants a second trace
     (recursion / inline RayQuery), and we are pinned at maxRecursionDepth=1. Its
     unique strength (projecting onto arbitrary/curved receivers) is irrelevant for
     flat static detail decals. So the MODEL is not our path.

WHAT WE TOOK FROM IT:
  (a) PERFORMANCE TABLE (27.8, RTX 3080, exaggerated 36,582 decals on Sponza):
        Single/Mesh/CHS (TraceRay loop)  ~12.40 ms   <- catastrophic, AVOID
        Single/Mesh/Any-hit              ~1.31  ms
        Single/AABB (intersection)       ~0.81  ms   <- near-free
        Multi(max 3)/AABB                ~0.84  ms   <- near-free
      Conclusions we use: TraceRay-in-a-loop is ~10x worse AND impossible here
      (maxRecursionDepth=1); ANY-HIT ACCUMULATION is both viable and cheap; a SMALL
      MAX decal count (2-3) keeps multi-decal essentially free. This validates our
      N<=3-4 bin + any-hit design.
  (b) "LIMITED SORTED DECALS" (27.3.2): any-hit collects up to n decal IDs into the
      payload, then SORT + blend in correct order (n tiny -> O(n^2) sort is free).
      This is exactly our payload-bin + sort-at-surface-build (02 M4 / DECAL-04/05).
  (c) EVALUATION ORDER (27.5.4): evaluate decal first; if it fully overrides a
      channel, skip base eval for it. Optional micro-opt (03 sec.5).
  (d) TIGHTER BLAS / ray-length / inclusion-lists (27.5/27.6): box-model concerns,
      not applicable to the card model. Noted, not used.

2. RTX REMIX — PRODUCTION BLEND-THROUGH DECALS
   E:/prog/references/dxvk-remix-git/src/dxvk
================================================================================
Remix ships EXACTLY the architecture we chose: decals stay as geometry, accumulate
via ANY-HIT, composite onto the base. It is the concrete reference implementation.
Mechanisms + anchors:

  M-A  SEPARATE "UNORDERED" TLAS for decals/blended/emissive geometry.
       rtx_instance_manager.cpp:1132-1147. Non-opaque, NO_DUPLICATE_ANY_HIT,
       FORCE_NO_OPAQUE; mask OBJECT_MASK_UNORDERED_ALL_BLENDED. "Unordered resolve
       only accumulates via any-hits and ignores opaque hits." No recursion.
       (We keep decals in the existing world BLAS as non-opaque rather than a second
       TLAS — same any-hit-accumulation effect, less infra; the separate-TLAS is a
       perf option Remix offers, not a requirement.)

  M-B  DRAW-ORDER SORT KEY (not depth). rtx_instance_manager.cpp:1118-1126 assigns
       a monotonic decalSortOrder counter (0..255) in DRAW order. Shader sort key
       resolve.slangh:1389:
         (decalSortOrder<<24) | (geometryIndex<<16) | primitiveIndex
       This is our DECAL-03 / 02 M2. Coplanar depth is unreliable; draw order is
       what raster blended in. >255 logs a corruption warning.
       RBDOOM PACKING NOTE (from a prior attempt): the per-triangle flags word read
       by LoadSmokeTriangleClassAndFlags already spends bits ~24-27 on the
       translucent subtype field, so the 8-bit draw-order key cannot sit in a single
       contiguous byte there. A prior attempt straddled it across bits 20-23 and
       28-31 (leaving 24-27 untouched). Verify the live bitfield layout before
       reusing those exact bits — do not assume it; confirm against the current
       RT_SMOKE_* flag/subtype definitions in PathTraceDynamicMaterialState.h.

  M-C  FIXED-BIN COLLECT + SORT + COMPOSITE. resolve.slangh:1202 (4 bins),
       1374-1401 (isDecal -> conditionallyStoreDecal by sort key), 1616-1643 (find
       lowest sortOrder, draw first). This is our DECAL-04/05.

  M-D  RESERVOIR-SAMPLED SINGLE DECAL FOR INDIRECT RAYS. resolve.slangh:1402-1409
       (A-Res weighted reservoir, weight by opacity). Primary rays get the full
       sorted bin; bounces get one decal to bound cost. Our DECAL-11 (later).

  M-E  PER-DECAL NORMAL OFFSET (the coplanar fix the user read about).
       rtx_instance_manager.cpp:1942 ("offset each decal separately"); options
       rtx_options.h:231-259: decalTextures / dynamicDecalTextures /
       singleOffsetDecalTextures (one offset per batch) / nonOffsetDecalTextures
       (already offset, skip). Driven by a GLOBAL INCREMENTING offset index capped
       by decals.maxOffsetIndex -> separates STACKED decals, not just decal-from-wall.
       This is our 04 / DECAL-02. NOTE: Remix tags decals by hand-maintained TEXTURE
       HASH lists; WE replace that with the authoritative classifier (sort/polyoffset)
       — strictly better, less brittle.

  M-F  MATERIAL BLENDING + flags. resolve.h:58-61 (resolveModeDecalMaterialBlending:
       decals stored to a DecalMaterial channel, continueResolving set, next
       non-decal surface gets them composited). Options rtx_options.h:852-861:
       enableDecalMaterialBlending, translucentDecalAlbedoFactor (~10x albedo for
       decals on glass). decalSortOrder packed rtx_materials.h:256-257,480.

  M-G  NORMAL BLENDING IS PUNTED EVEN IN REMIX. resolve.slangh:1398-1400 stores the
       decal tangent frame but: "TODO[REMIX-3080] Unused for now, will be needed for
       normal blending ... any decal's tangent space should work." This is why our
       plan defers EXACT normal projection (03 sec.3 / DECAL-10) and only attempts
       the base-TBN approximation later.
