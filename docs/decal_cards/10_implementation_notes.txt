  - RT_SMOKE_MATERIAL_DETAIL_DECAL (0x2000) / _DETAIL_DECAL_DYNAMIC (0x4000)
    flags (PathTraceDynamicMaterialState.h), mapped + hashed + compared in

DECAL-02  Face-normal offset at static capture
  - ApplySmokeDetailDecalNormalOffset (PathTraceSceneCapture.cpp), called in the
    STATIC pass right after a successful AppendSmokeSurfaceGeometry. Face normal
    per triangle (cross of edges, oriented by the authored vertex normal), all
    verts of a triangle pushed by the same vector; each vertex lifted once.
  - DEVIATION from 04: the offset index is NOT a global monotonic counter. It is
    hashed from the persistent static-surface key into [1, maxOffsetIndex]. The
    static cache appends incrementally across frames and evicts/re-appends, so a
    counter is capture-order-dependent (view-dependent) and would churn the BVH;
    the key hash is stable per surface forever. Stacked same-spot decals get
    different keys -> different indices with collision chance 1/maxOffsetIndex.
    If a real stacked pair collides, raise r_pathTracingDecalMaxOffsetIndex.
  - The offset is per-SURFACE (per drawSurf batch), i.e. Remix's per-batch mode;
    D3 batches static decals per material per area so same-material stacking
    within one surface is the rare case.
  - CAVEAT: toggling the cvar does not re-capture already-cached static
    surfaces; reload the map (or vid_restart) after changing stage 0 <-> 1+.

DECAL-03  Sort key (conservative v1)
  - sortKey = (instanceId << 30) | primitiveIndex, computed in the any-hit.
    Static capture appends in submission order so BLAS primitive order is the
    draw-order proxy. NOTE: the plan's 16-bit primitiveIndex packing was NOT
    used - static world primitive indexes exceed 16 bits. No CPU-side metadata
    buffer yet; triangleClassAndFlags untouched.

DECAL-04  Payload bin + any-hit accumulation
  - PathTraceSmokePayload grows: decalCount + 3x {decalMaterialIndex,
    decalPackedTexCoord (2x f16), decalSortKey, decalHitT} (13 dwords).
  - AnyHit: in collect stages (1/3/4) a primary-mode ray (payload.value==0)
    hitting an instance<2 triangle whose material has DETAIL_DECAL stores via
    ConditionallyStoreDetailDecal (keeps the N HIGHEST sort keys = topmost
    layers; dedupes by sort key) then IgnoreHit(). The decal never commits.
  - In collect stages RayGen launches with mode 0 (not 3) and the filter-decal
    pass-through experiment (ResolvePrimaryFilterDecalReceiver) is bypassed.

DECAL-05  Composite
  - ApplyDetailDecalComposite (producer raygen path, after
    BuildSurfaceFromPayload). Sorts ascending by sort key (selection sort, n<=3),
    applies per blend kind:
      FILTER  : albedo = lerp(albedo, albedo*decalRgb, coverage), coverage from
                white/black key (the headline modulate fix; no 0.12 floor - the
                raster contract is a true multiply).
      ADDITIVE: emissiveRadiance += decalRgb * additiveOpacity. DEVIATION: adds
                radiance, not albedo, mirroring the unlit-color fallback the
                stochastic path used for additive cards; albedo addition would
                double-count under DI.
      OVER    : albedo = lerp(albedo, decalRgb, SmokeAlphaCoverage).
    Decal-material EMISSIVE flag additionally folds SampleSmokeEmissive * coverage.
  - RECEIVER GUARD: |baseHitT - decalHitT| * |dot(rayDir, baseGeometricNormal)|
    <= step*maxOffsetIndex*1.5 + 0.05 (normal-space, never raw ray-T).
  - v1.5 roughness/metallic/F0 projection (DECAL-06) NOT implemented yet.

POST-V1 FIX: ZERO-ALBEDO VOID RECURRED (2026-06-10, same day)
-------------------------------------------------------------
First in-game test reproduced the 08 sec.3 failure THROUGH THE NEW PATH: the
dark sections of textures/decals/stainwallfade cut apparent holes. Mechanism
(confirmed in code, not speculation): the v1 modulate was the raster-pure
product base*decalRgb with no floor, so authored-black texels drive the
composited diffuse albedo to EXACTLY zero. The clean DI/RR pipeline treats
near-zero albedo as an invalid surface: CleanTexturedSurfaceAlbedo
(pathtrace_clean_rtxdi_di_spatial.rt.hlsl:618-626) rejects guide albedo with
luminance <= 1e-5, and a zero albedo also zeroes the DI target pdf (NaN risk in
reservoir weights). The result renders as holes/garbage, not as a dark stain.

LESSON: the 08 ZERO-ALBEDO VOID warning applies to ANY composite that can
produce exact-zero albedo, not just to the pass-through experiment. The
experiment's max(decalRgb, 0.12) floor was load-bearing, not cosmetic.

Fix shipped:
  - Modulate multiply factor floored per channel: max(decalRgb, DecalInfo.w),
    cvar r_pathTracingDecalModulateFloor (default 0.12, the value the prior
    experiment validated).
  - After any composite, final diffuseAlbedo floored at 0.01 so every blend
    kind (incl. OVER with black stickers) stays above the DI validity
    threshold.
