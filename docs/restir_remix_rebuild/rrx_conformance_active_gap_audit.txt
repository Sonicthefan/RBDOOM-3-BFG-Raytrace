RRX-CF-01 Active Producer-Consumer Gap Audit
============================================

Scope
-----

This is a read-only audit of the current active producer-consumer gap between the Remix-shaped manager/resources and the active rbdoom RAB/reservoir paths. It does not propose new ReSTIR math and does not treat diagnostic-only scaffolding as active parity.

Reference contract
------------------

RTX Remix binds shader-visible light data from the scene light manager: current lights, previous lights, and light mapping are fetched from `getSceneManager().getLightManager()` and bound as `BINDING_LIGHT_DATA_BUFFER`, `BINDING_PREVIOUS_LIGHT_DATA_BUFFER`, and `BINDING_LIGHT_MAPPING` (`E:/prog/references/dxvk-remix-git/src/dxvk/rtx_render/rtx_context.cpp:1377-1379`, `E:/prog/references/dxvk-remix-git/src/dxvk/rtx_render/rtx_context.cpp:1396-1398`).

RTXDI expects current/previous light domains plus translation through `RAB_TranslateLightIndex`: integration docs call out current/previous light data and mapping (`E:/prog/references/RTXDI-main/Doc/Integration.md:76-78`), and DI temporal resampling locally invalidates a previous reservoir sample when translation fails by zeroing that sample, not by requesting a global reservoir clear (`E:/prog/references/RTXDI-main/Libraries/Rtxdi/Include/Rtxdi/DI/TemporalResampling.hlsli:135-146`).

Direct Answers
--------------

Which light manager feeds shader-visible RAB records?
    The active shader-visible RAB records are fed by `m_restirLightManager`, not `m_remixLightManager`. The producer is `m_restirLightManager.BeginFrame/Update/EndFrame` and `GetActive*` vectors (`neo/renderer/NVRHI/PathTraceSmokeSceneBuild.cpp:2210-2226`), uploaded into smoke buffers (`neo/renderer/NVRHI/PathTraceSmokeSceneBuild.cpp:3014-3019`), bound as t62-t67 (`neo/shaders/builtin/pathtracing/pathtrace_restir_combined_resolve.rt.hlsl:142-147`), and consumed by `RAB_LightLoadRuntime.hlsli` when the manager RAB gate is enabled (`neo/shaders/builtin/pathtracing/RtxdiBridge/RAB_LightLoadRuntime.hlsli:146-152`, `neo/shaders/builtin/pathtracing/RtxdiBridge/RAB_LightLoadRuntime.hlsli:203-237`). `m_remixLightManager` currently reports `behavior=cpu-diagnostics-only` (`neo/renderer/NVRHI/PathTraceSmokeSceneBuild.cpp:2127-2154`).

Which reservoir domain do mode-56 probe views read?
    The RRX probe views 60 and 63-66 read/write the RRX DI domain through `RemixRtxdiDiReservoirs` at u68, with parameters from `RestirPTRemixDiReservoirInfo` and pages from `RestirPTRemixDiReservoirPageInfo` (`neo/renderer/NVRHI/PathTraceSmokeDispatch.cpp:1240-1269`, `neo/shaders/builtin/pathtracing/pathtrace_restir_local_debug_reservoirs.hlsli:493-535`, `neo/shaders/builtin/pathtracing/pathtrace_restir_local_debug_reservoirs.hlsli:1074-1092`). Normal mode-56 GI/debug/resolve paths still read the legacy PT/GI domains (`RestirPTReservoirs`, `RestirPTGiReservoirs`, and `RestirPTDiReservoirs`) declared at `neo/shaders/builtin/pathtracing/pathtrace_restir_combined_resolve.rt.hlsl:149-153`.

Which paths can still broad-clear old smoke/ReSTIR reservoirs?
    `PathTraceSmokeResources` scene-signature changes set `smokeReservoirNeedsClear`, `restirPTReservoirNeedsClear`, `restirPTDiReservoirNeedsClear`, and `restirPTGiReservoirNeedsClear` (`neo/renderer/NVRHI/PathTraceSmokeResources.cpp:1935-1942`). `PathTraceSmokeDispatch` dispatch-signature and reset paths can also set these flags (`neo/renderer/NVRHI/PathTraceSmokeDispatch.cpp:891-917`, `neo/renderer/NVRHI/PathTraceSmokeDispatch.cpp:952-954`, `neo/renderer/NVRHI/PathTraceSmokeDispatch.cpp:1593-1596`). The actual broad clears are `ClearSmokeReservoirBuffers` and `ClearRestirPTReservoirBuffers` in `neo/renderer/NVRHI/PathTraceSmokeDispatch.cpp:1606-1644`.

Which paths can clear RRX reservoirs?
    `PathTraceRemixRtxdiResources::ClearPendingDomain` clears a pending RRX domain with `clearBufferUInt` (`neo/renderer/NVRHI/PathTraceRemixRtxdiResources.cpp:287-312`). In the current dispatch path, only the RRX DI domain is cleared, and only when writes are enabled, mode 56 combined is active, debug view is 60 or 63-66, and the DI domain has `clearPending` (`neo/renderer/NVRHI/PathTraceSmokeDispatch.cpp:1645-1658`). I found no active dispatch call clearing the RRX GI domain.

Where does failed light translation become local rejection?
    In the active rbdoom RAB translation helper, failed mappings return `-1` (`neo/shaders/builtin/pathtracing/RtxdiBridge/RAB_LightLoadRuntime.hlsli:203-231`, `neo/shaders/builtin/pathtracing/RtxdiBridge/RAB_LightLoadRuntime.hlsli:233-342`). The reference RTXDI DI temporal path turns a failed previous-light mapping into local reservoir rejection by clearing only that previous sample (`E:/prog/references/RTXDI-main/Libraries/Rtxdi/Include/Rtxdi/DI/TemporalResampling.hlsli:135-146`). The current RRX DI probe path only visualizes/probes translation and heartbeat state (`neo/shaders/builtin/pathtracing/pathtrace_restir_local_debug_reservoirs.hlsli:538-579`, `neo/shaders/builtin/pathtracing/pathtrace_restir_local_debug_reservoirs.hlsli:718-837`); the active rbdoom DI temporal helper combines previous direct reservoirs without calling `RAB_TranslateLightIndex` on the selected previous light (`neo/shaders/builtin/pathtracing/pathtrace_restir_local_debug_reservoirs.hlsli:269-347`). Therefore the RRX-local rejection behavior is not active yet; it belongs to RRX-CF-04.

First Follow-up Code Slice
--------------------------

Do not change reservoir math or broad-clear policy in that slice.
