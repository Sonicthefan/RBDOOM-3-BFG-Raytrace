Worker Protocol
===============

Workers have repeatedly failed when asked to improvise ReSTIR behavior. Use
this protocol for every worker in this folder.


Mandatory First Step
--------------------

Before editing code, the worker must write a short local plan in its response:

    Remix file-equivalent being matched
    rbdoom file being edited
    exact input contract
    exact output contract
    forbidden areas it will not touch

If the worker cannot name those five items, it must stop.


Clean-Room Direct Port Rule
---------------------------

RTX Remix is the authoritative file responsibility graph. RTXDI is the
authoritative reservoir math contract. Do not copy source text. Do not redesign
algorithms.

Use "reference" only to identify source files that were inspected. It is not
permission to invent alternate behavior.

When translating a Remix file-equivalent:

    keep the same responsibility
    keep the same dataflow shape
    keep the same invalid/missing mapping behavior
    keep the same reservoir ownership idea
    adapt only the engine-specific data source

If the worker cannot identify the exact Remix file-equivalent, it must stop
before editing. Do not replace a Remix responsibility with an rbdoom smoke
reservoir equivalent.


Compliance Review Gate
----------------------

Structural resemblance is not compliance.

Do not accept a worker result because it creates similarly named objects,
similar buffers, plausible maps, successful dumps, or a non-crashing build.
Those are only scaffolding checks.

Every implementation or review that claims RTX Remix / RTXDI conformance must
prove all three items below for the assigned slice:

    active consumer proof
        exact file/line evidence that the new producer output is consumed by
        the next active rbdoom module, shader binding, RAB callback, or
        dispatch path. A diagnostic dump, inactive shader, or unbound buffer is
        not active consumption.

    negative-path proof
        exact behavior when the reference path encounters the important failure
        case. For light/reservoir work this means failed light translation
        rejects only the selected candidate/sample, not the whole reservoir
        domain. If the negative path is deferred, the slice is not full
        conformance.

    runtime diagnostic proof
        a dump or debug view that distinguishes storage survival, mapping
        failure, payload change, global clear, and inactive consumer state. A
        green/raw storage view alone is not proof of temporal correctness.

If any of these proofs are missing, mark the review P1 for incomplete
conformance unless the task was explicitly documented as diagnostic-only.


Forbidden Patterns
------------------

Workers must not:

    tune around black/noisy output
    tune temporal thresholds to hide overlap-light reservoir fighting
    feed previous temporal reservoirs back as Remix previous-best unless a task
    explicitly marks it as a debug-only non-Remix approximation
    invent new ReSTIR math
    add broad dynamic-light suppression
    route new ReSTIR reservoirs through old smoke reservoir clear policy
    keep old light/reservoir fallback active in the default RRX path
    use old mode-56 accumulation to prove RRX reservoir stability
    use Doom analytic count changes as temporal validity
    remove legacy outputs unless explicitly commissioned
    claim DLSS RR validation before DI/GI temporal debug views are sane
    claim feature parity for diagnostic-only scaffolding
    claim reservoir correctness from storage-only probes


Required Review Output
----------------------

Every worker final response must include:

    files changed
    Remix file-equivalent matched
    exact contract added
    active consumer proof, with file/line evidence
    negative-path proof, or explicit statement that it is deferred
    runtime diagnostic proof, or explicit statement that no runtime path exists
    any remaining old-smoke dependency
    build result or "not built"
    runtime CVars to test, if applicable


Build And Deploy
----------------

Build from:

    E:/prog/rbdoom-3-bfg-rt/neo

Build command:

    cmake --build --preset win64-pt-dev-release

Deploy exe:

    copy E:/prog/rbdoom-3-bfg-rt/build-win64-pt-dev/Release/RBDoom3BFG.exe
    to E:/prog/rbdoom-3-BFG-prebuilt/RBDoom3BFG.exe

Deploy shaders after shader changes:

    copy the contents of E:/prog/rbdoom-3-bfg-rt/base/renderprogs2
    to E:/prog/rbdoom-3-BFG-prebuilt/base/renderprogs2

Do not create:

    E:/prog/rbdoom-3-BFG-prebuilt/base/renderprogs2/renderprogs2
