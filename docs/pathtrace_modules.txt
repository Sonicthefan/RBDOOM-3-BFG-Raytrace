RBDoom3 BFG RT/PT module map
============================

This document maps the experimental Vulkan/NVRHI RT smoke/path tracing files and
the main feature flows that pass through them. It is meant as a navigation guide
for humans working on the branch, not as a replacement for the implementation or
the broader path tracing handoff notes. The `pathtrace_scene_producer_*` files
are historical intervention/build logs; this document is the durable module map.

For persistent geometry guardrails around lookup cost, validation cost,
lifecycle, thread ownership, material links, allocation, and transient geometry
exclusions, see docs/pathtrace_geometry_universe.md.

For persistent material guardrails around stable facts, descriptor ownership,
reset boundaries, lookup identity, validation cost, thread ownership, allocation,
and churn stats, see docs/pathtrace_material_universe.md.

For game-driven material/register behavior such as switchable fluorescent lights
and red/green door panels, see docs/pathtrace_interactive_surfaces.md.

Current scope
-------------

The current RT/PT path is still an experimental "smoke" path:

- r_pathTracing mode 18: toy path tracer / textured smoke path with selected
  lights, emissives, accumulation, ambient/lighting CVars, and debug modes.
  `r_pathTracingToyFakePBRSpecular` defaults on and can be set to 0 to opt out
  of the shader-local rbdoom3 legacy specmap-to-PBR F0/roughness approximation
  for direct specular testing. The same fake-PBR toggle is shared by the
  path-tracer visualizer modes 34-37 so their reflection/path-state views
  validate the same material policy. The mode now also has an explicit early
  integrator contract: samples per pixel, max path depth, diffuse/specular/
  transmission bounce limits, reflection mode, Russian-roulette depth
  reservation, NEE toggle, secondary selected-light NEE mode, and a one-shot
  ray-budget dump. Defaults preserve the previous one-sample,
  one-diffuse-bounce behavior while replacing secondary selected-light fan-out
  with one uniformly sampled selected point light; reflection is opt-in.
- r_pathTracing mode 19: emissive inventory/debug pass for future ReSTIR/PT
  preparation.
- r_pathTracing mode 20: first single-frame reservoir direct-lighting debug view
  from emissive candidates. It writes current reservoirs only; spatial and
  temporal reuse are still off.
- r_pathTracing modes 34-37: path tracer core visualizers for path depth,
  reflection hit/miss, reflection roughness/F0 gating, and max-depth /
  Russian-roulette state. They are diagnostics over the mode 18 integrator
  contract, not alternate renderers.
- r_pathTracing mode 21: PT-native solid drawSurf mirror bounds boxes for
  spatial/BVH/source diagnostics.
- r_pathTracing mode 22: PT-native wireframe drawSurf mirror bounds boxes. This
  is the preferred human spatial inspection mode because all-category solid
  boxes can occlude too much.
- r_pathTracing mode 23: experimental routed rigid TLAS instance view. It traces
  only routed rigid BLAS instances with a rigid-only ray mask and reads the
  packed rigid route vertex/index/material buffers, so it can validate
  local-space BLAS/TLAS transforms without changing mode 18/20 shading.
- r_pathTracing mode 24: routed rigid overlap validation. It keeps the old
  fallback scene intact, traces fallback-only and routed-rigid-only masks
  separately, color-codes hit-distance agreement, material/class mismatches,
  and front-most ownership, and can dump per-color pixel proportions with
  `r_pathTracingRigidRouteOverlapDump 1`. The dump includes full-frame counts,
  a 33x33 center-region summary, and the center-pixel bucket so crosshair-aimed
  problem objects can be separated from room-wide background coverage.
- r_pathTracing mode 25: routed rigid lighting validation. It traces the
  combined fallback plus routed rigid TLAS scene with simple material/normal
  lighting, after optional routed-rigid removal from the dynamic fallback.
- r_pathTracingRigidRouteMode18: default-on mode 18 integration gate for
  routed rigid TLAS instances. It preserves mode 18's toy PT shader path while
  routing eligible rigid meshes through reusable local-space BLAS/TLAS records.
- r_pathTracingMode18TestPreset: one-shot helper for applying the current
  source3 routed-rigid/static-preload/rigid-residency mode 18 stack. The stack
  is already default-on; the preset remains a convenience/debug shortcut.
- r_pathTracingRigidRouteRemoveDynamic: default-on subtraction gate used by
  routed-rigid validation/integration modes, including mode 18, mode 20,
  ReSTIR PT modes 26-33/50/51/53-56, and integrator debug modes 34-37. It
  skips routed-ready rigid candidates from the source3 dynamic fallback after
  the reusable rigid route is enabled. Mode 56 depends on this gate for clean
  primary-surface producer motion export; otherwise the same rigid object can
  be present as both a routed rigid TLAS instance and an `instanceId == 1`
  dynamic fallback copy. This gate is a temporary workaround: the current
  route-ready classification can also remove chunks of level airlocks and some
  doors from the fallback path. Replace it with a proper object and geometry
  classification system tied to Doom 3's render-world/game geometry before
  treating routed-rigid removal as final production policy.
- r_pathTracingMode20TestPreset: one-shot helper for the current mode 20 source3
  routed-rigid/static-preload/rigid-residency stack. The stack is already
  default-on; the preset remains a convenience/debug shortcut for mode 20 and
  optional light-area diagnostics.
- r_pathTracingAnalyticLightCandidates: default-on bridge from authored Doom
  renderer/game lights into a separate analytic sphere-light buffer. The first
  shader path keeps old selected point lights available for comparison, while
  r_pathTracingAnalyticLightReplaceSelected can suppress that legacy constant
  array so the analytic lights can be inspected visually on their own.
- r_pathTracingRestirPTAnalyticLightCandidates: default-on ReSTIR PT debug
  convenience gate. Modes 26-33 and 50-51 can build/use Doom analytic light
  candidates even when the global analytic-light CVar is off, but setting this
  to 0 restores emissive-only ReSTIR validation and profiling isolation.
- r_pathTracingAnalyticLightMaxGpu: cap for analytic Doom light candidates
  uploaded/sampled by the smoke shader. This is still a direct per-hit loop,
  so keep it conservative until ReSTIR candidate selection exists.
- r_pathTracingAnalyticLightIntensityScale: radiance calibration multiplier for
  analytic Doom sphere lights. The current default is 4.0 after visual testing
  showed the early low diagnostic scale under-lit authored Doom helper lights.
- r_pathTracingAnalyticSphereLightRadiusScale and min/max: convert Doom point
  light radius into the first diagnostic sphere-emitter size. This is not final
  physical classification of round/square/projected/area light materials. These
  CVars are consumed by PathTraceDoomLights.cpp when it builds
  PathTraceDoomAnalyticLightCandidate records; the default radius scale is 0.08
  and the default min/max clamp is 4.0/64.0 world units.
- Source3 RT residency uses Doom renderer area/portal selection rather than
  current raster visibility alone. The final drawSurf stream is the freshness
  source for transforms/material overrides/runtime registers; portal-selected
  static preload and rigid residency are the persistence/residency layer for
  occluded or off-camera RT geometry. Depth 4 is the current source3 default
  because visually continuous rooms can span several Doom portal areas.
- Source3/static/rigid/light portal walks seed from all valid camera-area
  probes at depth 0: `viewDef->areaNum`, initial view-area origin, exact view
  origin, and short +/- view-axis offsets. This avoids one-cell portal-border
  flips adding artificial hops to every resident object and light. The concrete
  regression was a valid-but-bad camera area on a Mars City portal border: PT
  residency started in a tiny connector/sliver cell, consumed several walks
  before reaching the visually occupied room, and then culled rigid residents
  plus Doom analytic lights that appeared to be in the next visible portal.
  Multi-seeding does not flatten real portal topology; it only prevents the
  current-area probe from being the sole traversal root when the camera is on a
  border.
- Routed rigid closest-hit shading now transforms local-space geometric and
  interpolated normals into world space through the TLAS instance transform and
  constructs a world-space tangent basis for later lighting validation.
- r_pathTracingSceneSource 3: default scene producer replacement scaffold. It
  keeps the existing drawSurf-seeded static cache and mirrors dynamic-frame
  drawSurfs as mesh identities plus per-frame instances while preserving the
  old shader-compatible static/dynamic render buffers. The original
  drawSurf-only source0 path remains available for emergency comparison, but is
  deprecated as the normal BVH source.
- In mode 18, r_pathTracingSkipRaster3D defaults to 1 so the RT output owns the
  3D scene layer and the normal Doom 3D raster view is skipped. GUI/HUD/menu
  draw commands still rasterize afterward. Leaving full 3D raster enabled below
  the path traced output roughly doubled frame cost in testing and allowed
  raster-side hitches to be misread as RT/PT hitches.
- Many dynamic Doom 3 material behaviors are not fully represented yet. Some are
  deliberately hidden, rejected, or forced to fallback/debug handling rather than
  guessed incorrectly.

Most files live under:

    neo/renderer/NVRHI

File map
--------

PathTracePrimaryPass.h
    Declares PathTracePrimaryPass, the frame-level shell for the experimental
    RT smoke state. The class still owns scene/build m_smoke* handles,
    counters, and private pass methods, but output-sized frame resources now
    live behind PathTraceFrameResources and static-world geometry cache storage
    lives in PathTraceGeometryUniverse. Most behavior is implemented in smaller
    PathTrace* modules.

PathTracePrimaryPass.cpp
    Thin pass shell. Performs RT feature detection, per-frame entry flow, output
    sizing, and debug-present blit. The PT output size is controlled by
    r_pathTracingDebugWidth/r_pathTracingDebugHeight, not the application window
    size. Internal PT output-size changes are delegated to
    PathTraceFrameResources, which waits for the GPU to idle before old output,
    accumulation, readback, smoke reservoir, ReSTIR PT reservoir, or primary
    history resources can be replaced. When PT is active, a known native-size
    resize also waits for the GPU to idle before ResizeImages() so
    previous-frame RT/AS work cannot overlap swapchain teardown, then
    invalidates PT output, scene, BLAS, smoke reservoirs, ReSTIR PT
    reservoirs/context, and the cached LDR blit framebuffer. The resize frame
    skips PT execution and debug
    presentation; if r_pathTracingSkipRaster3D owns the 3D view, it skips raster
    3D too instead of falling back into the normal 3D path. Avoid putting scene
    build, dispatch, resource lifetime, or diagnostics back here unless the pass
    shell itself truly owns the work.

PathTraceFrameResources.h / PathTraceFrameResources.cpp
    Output-sized PT frame resource owner. It owns the smoke output texture,
    accumulation texture, diagnostic readback texture, smoke reservoir buffers,
    RTXDI/ReSTIR PT packed reservoirs, primary-surface history buffers, ReSTIR
    PT context state, frame settings, reset reason flags, accumulation counters,
    primary-history view state, and readback queue bookkeeping. It is the single
    owner for destructive output-sized replacement policy: replacing existing PT
    output-sized resources waits for GPU idle, runs NVRHI garbage collection,
    recreates resources, and marks affected histories invalid. Its diagnostics
    are printed by `r_pathTracingReservoirDump` alongside the reservoir dump.

PathTracePrimarySurface.h
    CPU-side mirror for the shader primary-surface record. The current record is
    version 2 and 176 bytes, with fields for valid/status flags, world position
    plus depth, geometric and shading normals, view direction, diffuse albedo,
    alpha cutoff, roughness, F0/specular, emissive radiance, material flags,
    surface class, instance/primitive IDs, a deliberate object/entity ID
    placeholder, height/parallax/displacement placeholder, and previous-position
    or object-motion placeholder. Do not choose the final object/entity ID here
    without confirming the durable Doom renderer/game identity source. Treat the
    height, previous-position/motion, object/entity, roughness, F0, opacity, and
    alpha fields as denoise/upscale/reflection reservations; they are not a
    license to wire DLSS/RR before real motion and reset policy exist. The
    internal PT motion convention is current pixel to previous pixel, in
    internal PT render-resolution pixels, with unjittered current/previous
    camera state. Conversions for consumers with different sign or resolution
    conventions must be explicit.

PathTraceSceneInputs.h
    CPU-side PT scene input contract. It packages the committed TLAS/BLAS
    handles, static/dynamic/rigid/skinned geometry counts, static dirty-range
    summary, conservative previous static buffer/count/history facts, material
    table buffer and signature, texture descriptor table and active texture
    count, emissive and Doom analytic light buffers/counts, source3
    view/current-area portal policy facts, history-availability placeholders,
    skinned previous CPU-bridge match/invalid diagnostics, off-by-default
    skinned GPU-skinning scaffold resource handles/counts, and CPU upload/timing
    diagnostics. This is not a shader ABI. The static dirty-range summary
    mirrors the geometry-universe CPU bookkeeping and reports when reused static
    geometry buffers consumed it for partial dirty-range uploads. New or resized
    static buffers still use full uploads. The previous static bridge is
    conservative scene-input metadata only: it records previous committed static
    buffer handles/counts, including the material-index buffer, and exposes
    whether current static records have complete previous ranges with matching
    counts. It also reports when previous static handles alias the current
    persistent static buffers; after the dedicated previous static snapshot path
    is populated, aliasing should normally be false. The geometry universe
    separately retains and reports a one-frame CPU previous static snapshot;
    PathTracePrimaryPass retains the previous static material-index array next
    to it. Scene build uploads those previous static vertex/index/triangle-
    class/triangle-material/material-index arrays into separate GPU buffers and
    binds them at explicit reserved shader slots for future static/rigid
    temporal consumers. No current shader path reads them for shading or motion.
    The material-index count is tracked independently because the static-side
    material-index table can include full-level emissive and routed-rigid
    material IDs beyond the static triangle count.
    `staticPreviousGpuSnapshotUploadUsed` reports whether the retained previous
    static GPU snapshot actually needed a fresh upload in the current scene
    build. The same geometry section reports a dynamic
    fallback split for rigid, CPU-current skinned, likely-base-pose skinned, RT
    CPU-skinned, particle, unknown, and retained occluder surface/triangle
    counts. The classified total deliberately excludes retained occluders
    because retained occluders are also counted by class; its match flag reports
    whether the class split accounts for the aggregate dynamic triangle count.
    These counts are diagnostics only and do not split the dynamic BLAS. Upload
    diagnostics also expose static, previous-static, dynamic, and rigid-route
    geometry byte buckets next to the aggregate geometry/material/light upload
    totals. The static previous bridge dump reports `prevGpuSkipBytes` when a
    retained previous-static GPU snapshot avoids a repeated upload.
    The skinned previous bridge currently reports CPU-retained data in the
    default path. Routed rigid instances report previous-transform availability
    when the instance mirror observed the same stable instance in the
    consecutive previous frame. `r_pathTracingGpuSkinning 1` additionally
    creates and binds diagnostic source/output/previous-position/dispatch
    buffers so primary-surface packing can consume validated skinned previous
    positions without making the BLAS path consume the scaffold output. The
    opt-in `r_pathTracingGpuSkinning 2` path retargets the same compute dispatch
    to validated skinned spans in the dynamic vertex buffer before BLAS build.

RenderBackend.cpp
    Calls PathTracePrimaryPass from DrawViewInternal for 3D views. When
    r_pathTracingSkipRaster3D is 1, the backend returns after the RT output is
    produced, so the normal 3D raster pipeline does not run underneath mode 18.
    Later GUI/HUD/menu commands still pass through the raster backend. Keep this
    PT-owns-3D / raster-owns-GUI split unless a specific missing side effect is
    identified and restored narrowly.

PathTraceSmokeSceneBuild.cpp
    Per-frame scene build orchestration. It calls scene capture, material table
    construction, emissive inventory generation, buffer creation/upload,
    BLAS/TLAS submission, binding creation, resource commit, and scene build
    diagnostics. It also asks PathTraceDoomLights for analytic Doom light
    candidates before scene buffer creation when that CVar path is enabled. At
    the end of a successful build it also packages RtPathTraceSceneInputs with
    split geometry/material/light/output/camera/debug/upload signatures and
    source3 portal-policy facts. This is the right place to adjust high-level
    build order, but it should delegate specialized work to narrower modules.
    Reused static geometry buffers now use geometry-universe dirty ranges for
    partial vertex/index/triangle-class/triangle-material uploads when the
    ranges are valid; static material-index uploads remain full because they
    are derived from the current material table. The previous static CPU
    snapshot plus the previous static material-index array are uploaded into
    separate non-shader-bound GPU buffers for future motion/history consumers.
    Previous material-index validity is checked against the previous material-
    index count, not against static triangle count. When the previous static
    snapshot signature matches the retained previous buffers, scene build skips
    those five previous-static buffer uploads and keeps the existing GPU data.

PathTraceSmokeResources.h
    Resource-facing structs for smoke geometry buffers, binding build inputs,
    binding build results, scene resource commit descriptors, and resource
    commit packaging. The Doom analytic light buffer is part of this resource
    package, separate from emissive triangle and emissive light-candidate
    buffers.

PathTraceSmokeResources.cpp
    Persistent NVRHI smoke resource lifetime for shader/pipeline, scene geometry
    buffers, binding resources, commit packaging, smoke state reset, and
    successful scene-build commit back into PathTracePrimaryPass state.
    `ResizeRayTracingSmokeOutput` remains the pass-facing entry for PT internal
    render-resolution changes from r_pathTracingDebugWidth/
    r_pathTracingDebugHeight, but it delegates output-sized resource ownership
    and destructive resize safety to PathTraceFrameResources. The smoke raygen
    dispatch path now reserves private PT motion-vector export textures: RG16F
    vectors at shader UAV `u39` and an R32_UINT validity/source mask at UAV
    `u40`; they are allocated, bound, and reported by frame-resource
    diagnostics. `r_pathTracingMotionVectorExport` defaults to 1 and lets raygen populate them
    with combined static/skinned/routed-rigid current-to-previous pixel motion
    plus an explicit mask. Mask bit 0 is valid, bits 1-4 are source kind, and
    bits 5-12 carry a compact invalid reason from the primary-surface debug
    status when valid is zero. Invalid pixels therefore do not rely on zero
    motion as a sentinel, and valid stationary pixels can write zero motion with
    a non-zero mask. Exported vectors preserve sub-pixel previous projection;
    integer previous pixels are only for history-buffer lookups. When export is
    disabled, dispatch setup clears the mask texture to zero. No consumer reads
    them and normal PT output is unchanged.
    Scene
    commits now retain the latest RtPathTraceSceneInputs package and can print
    it with `r_pathTracingSceneInputsDump 1`. `r_pathTracingPortalTransitionDump`
    logs scene/portal/resource-handle transitions, including descriptor table
    create/write flags and BLAS/binding/resource replacement facts. Its normal
    test value is `1`, which captures the next portal-area transition and then
    disables itself; value `3` is the intentionally noisy broad resource
    transition dump.
    `r_pathTracingWaitForIdleOnPortalChange` is a diagnostic-only device idle
    inserted immediately before replacing committed PT scene resources when the
    scene transition signature changes; use it to prove or disprove an in-flight
    lifetime hazard. Normal replacement now uses frame-lagged retirement:
    `r_pathTracingSceneRetireFrames` defaults to 6 and keeps replaced scene
    buffers, BLAS/TLAS handles, binding sets, descriptor tables, and active
    texture handles alive until their retire frame passes. With retirement
    enabled, scene binding creation uses a fresh descriptor table instead of
    rewriting the previous committed table in place. Use
    `r_pathTracingSceneRetireDump 1` or `2` to inspect retire/release events.

PathTraceSmokeDispatch.h
    Exposes the smoke constants buffer size without exposing the full constants
    layout.

PathTraceSmokeDispatch.cpp
    Ray tracing dispatch path. Builds raygen constants, selects debug lights,
    sets resource states, manages accumulation, dispatches the NVRHI RT pipeline,
    and queues optional readback. It uploads Doom analytic light counts/scales
    through dispatch constants and can suppress the legacy selected point-light
    constant array when r_pathTracingAnalyticLightReplaceSelected is enabled.
    It also owns the current `PathTraceIntegratorSettings` CPU packing for mode
    18 and the ReSTIR PT bridge: SPP, max path depth, diffuse/specular/
    transmission limits, reflection mode, Russian-roulette depth reservation,
    NEE enable, secondary selected-light NEE mode, accumulation-signature
    participation, and the one-shot `r_pathTracingIntegratorDump` ray-budget
    estimate. Slow dispatch logs and
    `r_pathTracingPassTimingDump 1` now report pseudo-pass CPU submit timing for
    setup, ReSTIR context update, constants upload, barriers, clears, RT
    dispatch submission, primary-history copy, and optional readback. These are
    command-submission timings around the monolithic raygen, not GPU duration
    measurements for separated shader passes. `r_pathTracingNsightGpuMarkers 1`
    adds NVRHI command-list markers around each PT/ReSTIR `DispatchRays` call so
    Nsight can attribute mode-56 cost to primary surface, GI initial, direct
    temporal, direct spatial, reflection, or combined resolve stages. If Nsight
    does not show those labels, `r_pathTracingRestirPTGpuTimingDump 1` captures
    one frame of NVRHI GPU timestamp timings for the same dispatch stages and
    prints the result once the query data is ready. Mode 56 uses the separate
    `r_pathTracingRestirPTCombinedSpatialRadius` CVar for its direct spatial
    producer so the combined preview can default to radius 1 without changing
    the wider mode 50/51 diagnostic radius. The dispatch constants also carry
    shader-side safety metadata: a diagnostic kill-mask plus static/dynamic/
    rigid-route geometry capacities and primary-history/smoke-reservoir
    capacities. `r_pathTracingSafetyDump 1`
    prints the pre-dispatch safety state once, and the `r_pathTracingDisable*`
    kill switches can independently disable any-hit alpha rejection, selected
    light loops, analytic light loops, emissive triangle sampling, diffuse
    secondary rays, reflection rays, primary-surface history writes/copies,
    reservoir writes, and ReSTIR preview visibility rays.
    `r_pathTracingDispatchTileEnable` can split the same
    full-screen raygen work into sequential screen-space `DispatchRays` tiles
    for device-removal diagnosis; clears remain outside the tile loop and the
    shader uses absolute full-frame pixels for output/history/reservoir indexing.

PathTraceSmokeReadback.cpp
    Optional diagnostic readback of the smoke output texture. Handles delayed
    map/unmap and throttled terminal logging.

PathTraceAcceleration.h
    Acceleration helper declarations and small structs for geometry ranges,
    BLAS creation inputs/results, buffer upload batches, TLAS/BLAS submit inputs,
    and timing results.

PathTraceAcceleration.cpp
    Pure-ish acceleration helpers. Initializes NVRHI triangle geometry, creates
    BLAS objects, computes the static BLAS signature, uploads acceleration input
    buffers, and submits BLAS/TLAS builds. Long-lived cache state is not owned
    here.

PathTraceSceneCapture.h
    Scene capture constants, stats structs, bucket range structs, reason sample
    structs, and the capture entry points. This is the main interface for
    converting Doom draw surfaces into RT smoke triangle buckets.

PathTraceSceneCapture.cpp
    Doom draw-surface capture implementation. Walks view draw surfaces, chooses
    static/rigid/skinned/particle buckets, transforms vertices, records skip and
    classification stats, and appends captured geometry for the scene builder.
    Also stamps per-triangle runtime emissive state from evaluated draw-surface
    shader registers; stable material metadata only describes emissive
    capability, not whether a surface is currently emitting. Skinned/deforming
    captures also emit CPU-side key/range records after bucket offsets are
    finalized so previous matching does not rely on same dynamic-buffer offsets.

PathTraceSceneUniverse.h
    Diagnostics/static-preload scene-universe declarations. Defines direct
    renderer-world static surface records, area/material stats, selected-area
    portal-walk stats, and static geometry build stats. This module is useful
    for map inventory, area membership, and source3 static preload support; it
    is not the primary live scene representation.

PathTraceSceneUniverse.cpp
    Direct render-world scanner for static map/world data. It walks renderer
    entities and model surfaces without depending on the final raster drawSurf
    list, computes static surface keys, emissive-capable material summaries,
    bounds/area membership, selected-area portal diagnostics, and optional
    selected/full static geometry appends. It should remain diagnostics/preload
    infrastructure: do not feed its records as the universal RT scene source or
    synthesize live rigid/skinned/dynamic drawSurfs from it. Its portal-area
    selected static append path is used to seed static-world cache records that
    Doom raster visibility has not submitted this frame. The selected-area walk
    now uses the shared multi-seed camera-area policy so static preload does
    not disappear when `viewDef->areaNum` lands in a portal-border connector
    while the view is effectively in an adjacent room.

PathTraceDrawSurfCapture.h
    DrawSurf mirror declarations for the source3 scene producer scaffold.
    Exposes mesh observations, instance observations, bounds-overlay records,
    and the capture entry point that mirrors the final visible raster drawSurf
    stream without synthesizing fake drawSurfs or fake viewEntities.

PathTraceDrawSurfCapture.cpp
    Live drawSurf mirror implementation. For each usable visible drawSurf it
    builds a mesh key that excludes transform, records per-frame instance
    transform/material/entity metadata, compares static-world matches against
    PathTraceSceneUniverse diagnostics when requested, feeds geometry-universe
    rigid candidate diagnostics, and optionally emits bounds records consumed by
    debug modes 21/22. It also contains the conservative source3 promotion gate
    for stable entity-attached translucent signage/glow cards, so such cards can
    travel with routed rigid residency instead of remaining raster-visible-only
    particle/alpha drawSurfs. The same file owns routed-ready dynamic fallback
    removal. Its predicate must stay in sync with the routed rigid enablement
    predicate in `PathTraceSmokeSceneBuild.cpp`: old motion-vector diagnostics,
    mode 18/20 integration, ReSTIR PT modes including mode 56, and integrator
    diagnostics all remove route-ready rigid surfaces from the fallback when
    `r_pathTracingRigidRouteRemoveDynamic 1` is set. The predicate is known to
    be too coarse for final shipping behavior: when chunks of level airlocks or
    some doors disappear, this fallback-removal gate is the first suspect. The
    durable replacement is a real Doom 3 object/geometry classifier, not a
    broader material-name or visibility heuristic.

PathTraceInstanceUniverse.h
    Per-frame and lifetime instance-universe diagnostic declarations. This is
    the current home for observed mesh records, instance records, transform
    history, moved-rigid counters, and compact dump stats. It is not a TLAS
    source yet.

PathTraceInstanceUniverse.cpp
    Instance-universe implementation. Tracks mesh/instance observations across
    frames, separates same-transform and changed-transform history, records max
    observed matrix/origin deltas for rigid movers, reports missing
    material/skin override data, and prints one-shot dumps for
    r_pathTracingInstanceUniverseDump.

PathTraceGeometry.h
    Shared GPU-facing vertex layout and common geometry helper declarations.
    PathTraceSmokeVertex is consumed by NVRHI buffers and shader code, so layout
    changes here must be treated as shader/resource contract changes.

PathTraceGeometry.cpp
    Pure geometry/math helpers such as finite checks, normal packing support,
    and triangle/UV calculations shared by capture, acceleration, and emissive
    candidate code.

PathTraceGeometryUniverse.h
    Persistent static-geometry and rigid-diagnostic universe declarations. This
    is the first ownership boundary for static-world geometry cache data and
    currently exposes the static surface keys, vertices, indexes, triangle
    classes, triangle material IDs, static surface records, per-frame lifecycle
    fields, previous range fields, and stats used by scene build. It also owns
    rigid mesh candidate/local-source records, validation stats, CPU rigid BLAS
    plan/input descriptor dumps, and default-on rigid BLAS GPU scaffold records
    for per-mesh local-space vertex/index buffers plus optional standalone BLAS
    handles. It also owns source3 rigid residency records, live revalidation
    state, selected-area emissive rigid area-walk observations, and bounds
    samples. It builds rigid TLAS instance plans by matching current-frame
    revalidated instance observations to rigid mesh records when the route
    gates are enabled.
    The universe must evolve with previous-frame geometry history in mind so
    future motion vector output can be generated from geometry, not guessed
    later.

PathTraceGeometryUniverse.cpp
    Static geometry universe and rigid-diagnostic implementation. At this stage
    it owns the former PathTracePrimaryPass static-cache vectors and
    generation/stat tracking while preserving the existing shader-facing buffer
    behavior. Static surface lookup, per-frame BeginFrame/Touch/EndFrame
    lifecycle updates, append completion, and static record validation now go
    through the universe. It snapshots the static CPU arrays at BeginFrame so
    the scene-input layer can distinguish previous static CPU data ownership
    from aliasing current persistent GPU buffers. The snapshot covers vertex,
    index, triangle-class, and triangle-material ID arrays; PathTracePrimaryPass
    retains the previous static material-index array generated by the material
    table because that array includes material-table-side contributors outside
    the geometry-universe triangle arrays. `r_pathTracingGeometryUniverseRangeDump 1` prints a
    one-shot current/previous static range-history dump for validating normal
    range mapping, not only failure cases. The dump also reports union dirty
    vertex/index/triangle extents as CPU bookkeeping for future partial uploads;
    current static buffer uploads are still behavior-preserving and
    all-or-nothing. It also records eligible rigid local-space mesh sources,
    validates them against the baked dynamic rigid fallback payload, builds
    CPU-only reusable rigid BLAS/TLAS plans and NVRHI-shaped input descriptor
    dumps, and now owns a default-on GPU scaffold that can
    create/upload local-space rigid mesh input buffers and submit per-mesh BLAS
    builds. With the explicit rigid route gates, it can feed routed rigid TLAS
    instances into modes 18/20/23/24/25. Rigid residency no longer emits a
    blind cache of ordinary rigid objects: visible drawSurf observations are
    the authoritative freshness source, records not revalidated in the current
    frame are pruned, and the selected-area portal walk only supplements
    conservative emissive/signage/glow rigid surfaces after the entity has been
    previously validated by visible drawSurfs without a later render-entity
    modification. The area-walk supplement suppresses exact visible rigid
    surface instances, not whole entities, so a visible non-emissive sibling
    surface cannot disable an offscreen emissive sibling on the same model.
    Rigid residency was one visible symptom of the portal seed
    bug: a valid-but-bad current area could make cached rigid objects in the
    next visually adjacent portal fall outside the selected set despite
    sufficient nominal portal depth. The residency walk now multi-seeds from
    the plausible camera areas at depth 0 before applying the configured portal
    depth. Future work should split emissive/light residency from geometry
    residency more cleanly and expand persistent records, dirty ranges,
    current/previous transform history, and per-frame material-state handling
    needed for moving/game-driven surfaces and temporal reconstruction.

pathtrace_nee.hlsli
    Native mode 18 NEE/direct-lighting include. It owns the selected-light NEE
    source category constants, small surface/light-sample/result records, the
    legacy selected-light full-loop evaluator, the one-sample bounded selected
    point-light evaluator, and the legacy full Doom analytic sphere-light loop.
    Primary selected-light and analytic direct lighting still route through the
    legacy full loops.
    Secondary diffuse/reflection hits use `r_pathTracingSecondaryNeeMode`: 0 =
    no secondary selected-light NEE, 1 = one uniformly selected point light with
    inverse-PDF compensation, 2 = legacy full selected-light loop. Secondary
    Doom analytic-light NEE uses `r_pathTracingSecondaryAnalyticNeeMode`: 0 =
    off, 1 = uniformly sampled bounded proposals from the current uploaded
    analytic candidate set, 2 = legacy full analytic-light loop. The sampled
    mode is the default and uses `r_pathTracingSecondaryAnalyticNeeSamples` as the per-vertex
    proposal count and compensates the discrete light-selection PDF; it does
    not shrink the authored/uploaded analytic-light universe to the sample
    count. Doom analytic lights remain under the existing analytic-light
    candidate/upload CVars, but their loop now builds and evaluates native
    SmokeNee light-sample/result records instead of bypassing the NEE contract.
    The sample/result records now explicitly carry
    native extension fields for source type/index, future material/instance/
    primitive identity, light-selection/sample proposal PDFs, target weight,
    target-PDF reservation, visibility, path event classification, and an MIS
    placeholder. These records are RTXDI-shaped extension points, not
    RTXDI-equivalent sampling state yet: selected point lights have a discrete
    light-selection PDF and no finite solid-angle PDF, while `targetPdf` remains
    zero until a real reflected-radiance target evaluator is added. Doom
    analytic samples expose replay helpers for future reservoir final shading:
    `ValidateSmokeDoomAnalyticLightSampleForReplay` rejects stale source
    indexes, changed render-light identities, changed Doom entity identities,
    invalid PDFs, and invalid distances; `EvaluateSmokeDoomAnalyticLightSampleForSurface`
    evaluates one already-selected analytic sample without choosing a new light
    or looping the light buffer; and `SmokeNeeAnalyticTargetWeight` computes the
    local target score. Analytic NEE radiance is currently marked as
    solid-angle-scaled local radiance, so reservoir code must not divide it by
    `solidAnglePdf` again.
    `r_pathTracingSecondaryNeeVisibility` can skip the sampled selected-light
    shadow ray as a diagnostic isolation gate, but the default traces
    visibility.

RtxdiBridge/*.hlsli
    Local RBDoom3 shader bridge declarations for future RTXDI RAB callbacks.
    These files live next to the path tracing shader instead of copying NVIDIA
    sample bridge files into a distant third-party area. The current Step 4
    content is type-only: RAB_Material, RAB_Surface, RAB_RayPayload,
    RAB_LightInfo, RAB_LightSample, and RAB_RandomSamplerState plus small empty
    constructors/accessors.

    Step 5 surface callback scaffolding currently lives in pathtrace_smoke.rt.hlsl
    because it depends on the smoke shader's local payload and material helpers.
    `RAB_BuildSurfaceFromSmokePayload` and `RAB_BuildMaterialFromSmokePayload`
    translate a traced smoke hit into bridge records without changing the
    existing mode 18/20 lighting paths. `RAB_GetGBufferSurface` and
    `RAB_GetGBufferMaterial` now call the named PathTracePrimarySurface helper
    layer instead of reading an ad hoc local history struct. Raygen writes the
    current primary hit, and the backend copies current to previous after
    dispatch for next-frame validation. When `r_pathTracingToyFakePBRSpecular 1`
    is enabled, modes 26/27/28/29/30/31/32/33/50/51 sample Doom specular textures and
    store the same converted fake-PBR roughness/F0 that mode 18 uses. Modes
    34/35/36/37 use the same fake-PBR material toggle for their path-tracer core
    visualizers.

    Step 6 RNG scaffolding lives in `RAB_RandomSamplerState.hlsli`. It keeps a
    local `RAB_RandomSamplerState` wrapper but initializes and advances it
    through RTXDI's own random sampler helpers. This preserves the seed/index
    contract that ReSTIR PT reservoirs need for path replay, while keeping the
    application bridge name local and upgradeable.

    Step 7 surface/BRDF scaffolding is now split by include cost.
    `RAB_SurfaceCore.hlsli` owns the RAB surface record, accessors, safe
    normalization helpers, and opaque-diffuse support predicate.
    `RAB_Brdf.hlsli` owns the first conservative opaque-diffuse callbacks:
    `RAB_GetSurfaceBrdfSample`, `RAB_GetSurfaceBrdfPdf`,
    `RAB_EvaluateSurfaceBrdf`, `RAB_EvaluateSurfaceBrdfOverPdf`, plus
    `RAB_SurfaceImportanceSampleBrdf` and `RAB_SurfaceEvaluateBrdfPdf` aliases.
    `RAB_Surface.hlsli` remains a compatibility wrapper that includes the BRDF
    file for older bridge code. New pass-local shaders should include
    `RAB_SurfaceCore.hlsli` when they only need surface records or history
    callbacks.
    The current model is cosine-hemisphere Lambert sampling over the RAB shading
    normal with geometry-normal hemisphere rejection. Unsupported translucent or
    invalid surfaces return no sample/zero contribution so they can keep using
    the regular path until real BSDF support exists. The diffuse-only callback
    is intentionally quieter than the material contract: RAB material/history
    records still carry roughness, specular F0, opacity, alpha cutoff, material
    flags, and glass/window fallback flags. Do not delete or collapse those
    fields just because the current validation path does not sample glossy,
    mirror, refraction, or transmission lobes yet.

    Reflection, glass, denoise, and upscale reservations are documented without
    allocating fake final resources. Future reflection/specular work should keep
    separate output/history names, hit distance, confidence, roughness/F0,
    normals, and motion inputs instead of folding into diffuse/ReSTIR history.
    Future glass/transmission work needs explicit opacity, alpha cutoff, IOR,
    tint, thickness, nested-surface, and Doom window/liquid policy; the current
    glass handling remains a temporary visibility approximation until the user
    confirms that policy. Future denoise/upscale/RR work needs color, albedo,
    normal, roughness, depth, motion vectors, hit distance, diffuse/specular
    separation, exposure, and history reset reasons, with geometry-derived
    motion rather than camera-only reprojection.

    Step 8 light scaffolding is now split across capability files:
    `RAB_LightInfoCore.hlsli` for light records/constants,
    `RAB_LightInfoRuntime.hlsli` for current-frame smoke/Doom light loading,
    `RAB_LightSamplingCore.hlsli` for sampling emissive and Doom analytic
    proposals, `RAB_LightTarget.hlsli` for target/replayed radiance evaluation,
    `RAB_Visibility.hlsli` for the conservative visibility callbacks,
    `RAB_DuplicationMap.hlsli` for the placeholder duplication-map callback,
    and `RAB_MISCallbacks.hlsli` for MIS/environment placeholders.
    `RAB_LightInfo.hlsli` and `RAB_LightSampling.hlsli` remain compatibility
    wrappers for the current smoke/ReSTIR library. The current bridge
    can load and sample current-frame smoke emissive triangles and Doom analytic
    sphere lights from the existing shader buffers, compute a simple reflected
    target weight for supported opaque-diffuse RAB surfaces, return direction
    and distance, and provide a conservative visibility callback that only
    rejects invalid surfaces/samples. RAB Doom analytic samples now mirror the
    native mode 18 analytic convention: they reject surfaces outside the Doom
    influence radius (`doomRadiusAndArea.x`) and apply the same quadratic
    influence falloff before RIS/target evaluation. Doom analytic previous-frame
    NEE light indices now translate through the uploaded universe remap and load
    previous analytic light records for inverse-shift target evaluation. The
    remap also rejects matched Doom analytic lights whose current/previous
    color, intensity, position, or radius diverges beyond
    `r_pathTracingRestirPTTemporalAnalyticLightChangeTolerance`, because stable
    identity alone is not enough for strobe/animated light phases. Emissive
    previous-frame light reuse remains conservative until an emissive identity
    map exists. Environment sampling is still zero/unsupported. MIS callbacks
    are present but intentionally neutral until `RAB_PathTrace` and consistent
    NEE/BRDF path accounting exist.

    Step 9 path-trace scaffolding lives under `RtxdiBridge/PathTracer/`.
    `RAB_PathTracerUserData.hlsli` provides the minimal user-data object and
    path-type setter required by RTXDI PT entry points. `RAB_PathTracer.hlsli`
    compiles RTXDI's initial-sampling path tracer context against the local RAB
    callbacks and defines `RAB_PathTrace`. The skeleton samples the current
    conservative RAB BRDF, fills `RTXDI_BrdfRaySample`, sets a continuation ray,
    lets RTXDI analyze reconnectibility, traces `SmokeScene`, converts the smoke
    hit payload back into `RAB_RayPayload`/`RAB_Surface`, records the
    intersection, and records emissive surface hits. It is capped to a small
    bridge bounce depth and does not yet implement NEE, environment radiance,
    Russian roulette, PSR/denoiser state, traced RAB visibility, temporal replay,
    or spatial reuse.

PathTraceDebugDumps.h
    Shared diagnostic structs and dump/trigger APIs for scene build, material,
    texture, timing, debug-mode classification, and one-shot diagnostic CVars.

PathTraceDebugDumps.cpp
    Console diagnostics implementation and diagnostic trigger/reset orchestration.
    Should format and gate terminal output, not own render resources. It also
    owns the current debug-mode classification table that labels modes as
    pass-output visualizers, scene/geometry diagnostics, material/texture
    diagnostics, light diagnostics, or temporary behavior-changing experiments.

PathTraceSurfaceDebugDumps.h
    Surface-targeted diagnostic dump declarations.

PathTraceSurfaceDebugDumps.cpp
    Crosshair material and GUI surface diagnostic dump implementation. Uses the
    already-built material table and current view.

PathTraceCVars.h
    Central extern declarations for experimental RT/PT CVars used by the
    extracted modules.

PathTraceCVars.cpp
    CVar definitions for the smoke/path tracing prototype, including the mode
    18 toy PT controls such as `r_pathTracingToyFakePBRSpecular`.

Material processing / classification
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Primary files involved:

- PathTraceDoomMaterialClassifier.*
- PathTraceSurfaceClassification.*
- PathTraceMaterialTextureDiscovery.*
- PathTraceTextureRegistry.*
- PathTraceMaterialUniverse.*
- PathTraceDynamicMaterialState.*
- PathTraceSceneCapture.*
- PathTraceDebugDumps.*
- PathTraceSurfaceDebugDumps.*
- PathTraceCVars.*

Typical flow:

1. PathTraceSceneCapture walks Doom draw surfaces.
2. PathTraceSurfaceClassification decides the surface bucket.
3. PathTraceDoomMaterialClassifier extracts coarse material traits.
4. PathTraceMaterialTextureDiscovery registers visible material texture metadata.
5. PathTraceTextureRegistry stores safe texture handles and fallback reasons.
6. PathTraceMaterialUniverse builds or reuses persistent material records and
   exposes stable material facts.
7. PathTraceDynamicMaterialState builds the compact GPU-facing active material
   table and triangle material indexes from the universe records.
8. PathTraceDebugDumps and PathTraceSurfaceDebugDumps report what happened.

Mode 18 fake-PBR specular test:

1. PathTraceCVars defines `r_pathTracingToyFakePBRSpecular`, default 1.
2. PathTraceSmokeDispatch folds the CVar into the accumulation reset signature,
   enables specular-map sampling for mode 18, ReSTIR PT bridge modes, and
   path-tracer visualizer modes when the toggle is active, and passes a texture
   flag bit to the raygen constants.
3. pathtrace_smoke.rt.hlsl tests that flag, samples the existing
   `specularTextureIndex`, derives approximate F0/roughness from the Doom legacy
   spec map, and uses that for selected/analytic direct specular and related
   validation records.

This is deliberately a visual smoke test. It does not add CPU material fields,
does not detect packed `_rmao`/`rmaomap` usage, and does not replace the material
classification/facts work needed for alpha, blend, decal, GUI, portal, glass,
or special-stage behavior.

Safer places to touch:

- Add a new material trait name or diagnostic string in
  PathTraceDoomMaterialClassifier or PathTraceDebugDumps.
- Add a fallback reason or registry metadata field if it does not change shader
  layout.
- Add cached material facts in PathTraceMaterialUniverse when they avoid
  repeated per-frame classification and can be validated against existing
  metadata.
- Add a diagnostic dump for a suspected material class.
- Extend material texture discovery to record more metadata while preserving
  current fallback behavior.

Riskier places to touch:

- PathTraceSmokeMaterial layout in PathTraceEmissiveCandidates.h. This is
  shader-facing data.
- Material flag bit values in PathTraceDynamicMaterialState.h. These are read by
  shaders and debug code.
- Per-triangle material index generation in PathTraceDynamicMaterialState.cpp.
  Bad indexes can break shader reads.
- Active material table row order. It must stay compatible with cached static
  triangle material-index buffers unless the static cache/resource contract is
  deliberately changed.
- Capture filtering in PathTraceSceneCapture.cpp. It can make geometry appear or
  disappear from the RT scene.

Rule of thumb:

If the change only improves classification metadata or diagnostics, it is usually
safe. If it changes what surfaces/materials are admitted, which texture slot is
bound, the active material row order, or the GPU material layout, test mode 18,
mode 19, crosshair dump, texture fallback dump, universe table validation, and
scene rebuild behavior.

BVH / BLAS / TLAS
~~~~~~~~~~~~~~~~~

Primary files involved:

- PathTraceSmokeSceneBuild.cpp
- PathTraceDrawSurfCapture.*
- PathTraceInstanceUniverse.*
- PathTraceAcceleration.*
- PathTraceSmokeResources.*
- PathTraceGeometry.*
- PathTraceGeometryUniverse.*
- PathTraceSceneUniverse.*
- PathTraceSceneCapture.*
- PathTraceSmokeDispatch.cpp

Typical flow:

1. PathTraceSmokeSceneBuild borrows static cache storage from
   PathTraceGeometryUniverse.
2. PathTraceSceneCapture currently produces static/dynamic triangle buckets and
   asks the universe to look up or complete static surface appends.
3. PathTraceSceneUniverse can provide static map diagnostics and, when source3
   static area preload is enabled, selected-area static-world preload records.
   It does not own the live dynamic/rigid/skinned scene. Its selected-area walk
   uses Doom renderer portal areas, and currently counts but does not block on
   `PS_BLOCK_VIEW`.
4. For source3, PathTraceDrawSurfCapture mirrors the final visible drawSurf
   stream after old capture has done the stable static-cache work. It records
   mesh identity without transform and per-frame instance identity with
   transform/material/entity metadata. Treat this as freshness input, not the
   complete RT residency rule.
5. PathTraceInstanceUniverse tracks those observations for diagnostics and
   moved-rigid history. The geometry universe consumes the current/cached rigid
   observations when rigid route/residency gates are enabled.
6. PathTraceGeometryUniverse records eligible rigid local-source mesh candidates,
   CPU rigid BLAS plans/input descriptors, default-on GPU scaffold records for
   local-space rigid mesh buffers/standalone BLAS builds, rigid residency
   records, and rigid TLAS instance plans. In gated validation/integration
   modes, routed-ready rigid triangles are removed from the dynamic fallback
   and traced through the reusable rigid route. This includes mode 56. Keeping
   both copies in the TLAS makes the primary producer nondeterministically hit
   either the routed rigid (`instanceId >= 2`, `RIGID_ENTITY`) or the dynamic
   fallback (`instanceId == 1`), which poisons the exported motion-vector
   source/mask/vector classification. The removal path is still a temporary
   compromise and can remove level airlock chunks or some doors when the
   current route-ready classifier overreaches; this is why a future object and
   geometry classification rebuild is required. Rigid residency now keeps only
   current-frame revalidated resident instances. Its selected-area portal walk
   supplements previously visible emissive/signage/glow rigids, but does not
   discover or preserve arbitrary opaque rigid props from raw area refs because
   destructible gameplay objects can remain render-side while no longer being
   valid PT geometry. It suppresses exact visible rigid instances rather than
   entire entities so multi-surface models keep offscreen emissive siblings
   resident when a non-emissive sibling is on camera.
7. PathTraceSmokeSceneBuild calculates geometry ranges and static BLAS cache
   eligibility.
8. PathTraceSmokeResources creates geometry/material/emissive buffers.
9. PathTraceAcceleration creates BLAS objects, uploads build-input buffers, and
   submits BLAS/TLAS builds. Its TLAS submit helper appends routed rigid
   instances when the explicit rigid route gates and compatible debug or
   integration mode are enabled.
10. PathTraceSmokeResources commits the successful handles to pass state.
11. PathTraceSmokeDispatch transitions committed buffers and dispatches rays.
12. RenderBackend.cpp blits the RT output as the 3D scene. With
   r_pathTracingSkipRaster3D enabled, the backend skips the remaining normal 3D
   raster passes for that view; subsequent GUI/HUD/menu views still rasterize.

Safer places to touch:

- Add timing/stat fields around acceleration helpers.
- Improve debug names for buffers/accel structures.
- Add validation/logging around build inputs.
- Change helper packaging structs when the same fields are passed through.
- Move static-cache bookkeeping into PathTraceGeometryUniverse when the capture
  behavior remains equivalent and can be tested in isolation.
- Add drawSurf mirror diagnostics, candidate counts, and validation dumps.
- Add PT-native debug views that consume diagnostic buffers, such as modes 21
  and 22, while keeping mode 18/20 resource contracts stable.

Riskier places to touch:

- Static BLAS cache eligibility in PathTraceSmokeSceneBuild.cpp.
- Static surface identity or append semantics in PathTraceSceneCapture.cpp or
  PathTraceGeometryUniverse.cpp.
- Treating PathTraceSceneUniverse as the live universal scene source. It lacks
  the final raster-submitted dynamic, skinned, material-override, and
  game-logic state needed for the primary scene producer.
- Treating current raster visibility as the complete BVH residency rule. That
  repeats the earlier failure mode where occluded/off-camera emissives and
  shadow-casting rigid geometry disappear together.
- Treating current+1 portal depth as a proven global rule. It was too shallow
  in Mars City; depth 4 is now the default source3 residency/preload standard.
- Mesh key semantics in PathTraceDrawSurfCapture.cpp. Mesh identity must not
  include model matrices or current transforms.
- Upload skip flags in the RtSmokeBufferUploadItem array.
- Resource state transitions in PathTraceSmokeDispatch.cpp.
- PathTraceSmokeVertex layout or index/vertex counts.
- Any TLAS instance setup in PathTraceAcceleration.cpp.
- Re-enabling the full 3D raster tail under mode 18. This should be treated as
  a diagnostic comparison path, not the default PT path, because it doubles work
  and can reintroduce raster-only hitches.

Rule of thumb:

Treat BLAS/TLAS state as behavior-sensitive. A tiny cache or upload mistake can
look like a rendering bug, stale geometry, or a performance regression. Prefer
diagnostic additions before ownership changes. The geometry universe is now the
intended home for persistent static-world records, but it is still a scaffold:
move one responsibility at a time and keep mode 18 visual behavior stable.

For source3, keep the representation split explicit:

- mesh = geometry plus material metadata, no transform
- instance = mesh plus current transform, entity/material override, and
  visibility this frame
- reusable rigid BLAS = future local-space mesh cache, not the current render
  path

Only route rendering through reusable rigid records or remove rigid triangles
from the dynamic fallback behind the existing explicit route gates or test
presets. The current source3 validation path expects mesh identity to remain
transform-free, instance identity to carry transform/material/entity metadata,
and residency to be driven by Doom area/portal selection rather than pure
raster visibility.

For mode 18 performance validation, keep r_pathTracingSkipRaster3D at 1 unless
the test explicitly compares against old raster-under-RT behavior. If something
is missing with raster skipped, prefer restoring the narrow dependency rather
than falling back to the full 3D raster frame.

Motion-vector / Ray Reconstruction requirement:

DLSS Ray Reconstruction needs high-quality motion-vector inputs. A path-traced
renderer cannot get there with only current-frame packed geometry or camera-only
motion: moving rigid instances, animated/skinned meshes, reflected/refracted
hits, and disocclusions all need stable current-vs-previous geometry data. The
Q2RTX Ray Reconstruction reference fork in
E:\prog\references\Q2RTX-rayreconstruction-master stores previous triangle
positions and previous instance transforms, then projects current and previous
hit positions in shader to write motion.

ReSTIR PT mode 56 now has the first DLSS RR guide-buffer bridge. The
full-resolution primary-surface producer writes:

- `u48` `PathTraceRRGuideAlbedo`, `RGBA16_FLOAT`.
- `u49` `PathTraceRRGuideNormalRoughness`, `RGBA16_FLOAT`.
- `u50` `PathTraceRRGuideDepth`, `R32_FLOAT`.
- `u51` `PathTraceRRGuideHitDistance`, `R32_FLOAT`.
- `u52` `PathTraceRRGuideResetMask`, `R32_UINT`.
- `u53` `PathTraceRRGuideSpecularAlbedo`, `RGBA16_FLOAT`.
- `u54` `PathTraceRRInputColor`, `RGBA32_FLOAT`.

The existing motion-vector scaffold remains `u39/u40`. `u54` is written by the
standalone combined resolve, not by the primary-surface producer, and must stay
raw scene-referred linear HDR input color rather than the tonemapped
`SmokeOutput` preview. This is not a DLSS RR SDK integration; it is the explicit
guide/input export needed before one. The task plan is
`docs/ReSTIR/dlss_rr_bridge_plan.txt`. Use
`r_pathTracingDLSSRRGuideDebugView` in mode 56 to inspect albedo, normal,
roughness, depth, hit distance, motion-mask/source-category output,
reset/disocclusion output, specular albedo/F0, and a tonemapped RR-input preview
without loading the old monolithic ReSTIR shader.
With `r_pathTracingMotionVectorExport` defaulting to 1, the standalone primary-surface
producer writes `u39/u40` directly for this bridge path from static previous
snapshots, retained skinned previous positions, and continuous rigid previous
transforms. Mode 56 also forces this export while primary surface history is
enabled because the DI/GI temporal debug reservoir paths use `u39/u40` plus
`u52` reset/disocclusion bits for previous-pixel lookup. Same-pixel DI history
fallback is only allowed for valid near-zero exported motion or for an
ordinary-rigid `OBJECT_MOTION_UNAVAILABLE` reset when the previous same-pixel
surface still matches. The reset mask is written even when motion-vector export
is disabled so history validity can be inspected independently. Mode 56 builds
the skinned previous-position scaffold for this guide export even when
`r_pathTracingGpuSkinning` is `0`; that CVar still controls the diagnostic GPU
skinning compute path. Mode 56 removes routed-ready rigid surfaces from the
legacy dynamic fallback when the routed rigid TLAS route exists. This matches
the mode-47/49 diagnostic contract and prevents duplicate rigid geometry from
feeding mixed producer classifications into `u39/u40`.
The local bridge still differs from the RTXDI PT reference motion-vector
contract: `u39` is xy pixel motion only, while the reference carries xy plus a
depth delta for temporal validation.

Implications for PathTraceGeometryUniverse:

- Persistent surface/triangle records need stable identities across frames.
- Records need current and previous range mapping, or explicit invalidation when
  a match is not trustworthy.
- Static-world records already carry current/previous ranges and lifecycle
  validity as CPU-side scaffolding, but no previous-position GPU buffers or
  motion-vector output exist yet.
- Static world can often reuse identical previous positions, but rigid entities
  need previous transforms and skinned/deformed geometry eventually needs
  previous skinned positions or a GPU skinning history source.
- Future GPU buffers should plan for current/previous position data or compact
  position deltas, not only the current PathTraceSmokeVertex stream.
- Motion-vector generation should be treated as part of the geometry-universe
  contract before DLSS/RR integration begins. Retrofitting it after the universe
  only stores current-frame data would risk another major rebuild.

Current state:

Interactive GUI surfaces can be admitted into the RT smoke scene through the
GUI-surface path, but this is still conservative and diagnostic-oriented. Doom
GUIs are not "just a texture"; they are generated by Doom's GUI system and may
change due to game/menu interaction.

Important postmortem:

- Do not modify GuiModel.cpp to synthesize frontEndGeo for generated GUI draw
  surfaces in normal play. That experiment made the UI painter-order quads look
  like physical world geometry. It broke in-world monitors/video panels in the
  shared raster path, produced black or overlapping screen layers, and caused
  severe cinematic/GUI hitches. GuiModel.cpp is intentionally restored to the
  original behavior where generated GUI drawSurfs keep frontEndGeo = NULL.
- The viable long-term route is a narrow GUI bridge: keep Doom's raster GUI/SWF
  system responsible for panel contents, expose or render a composited panel
  texture, and project that texture onto the actual physical panel surface in
  the PT material path. The generated GUI draw commands themselves should stay
  out of the TLAS.

How to pipe in a better GUI idea:

1. Start in PathTraceGuiSurfaces.cpp.
   - Keep the CVar gate intact.
   - Make the admission test explicit and easy to dump.

2. Ensure PathTraceSceneCapture.cpp gets the surface into the intended bucket.
   - Do not silently merge GUI behavior into unrelated translucent/particle
     handling.
   - Record reason samples so r_pathTracingGuiDump and related diagnostics can
     explain the decision.

3. Make texture discovery explicit.
   - If the GUI can provide a stable image/texture handle, record it through
     PathTraceMaterialTextureDiscovery and PathTraceTextureRegistry.
   - If it cannot, preserve a fallback reason rather than binding a guessed or
     unsafe texture.

4. Keep material table changes isolated.
   - PathTraceDynamicMaterialState should receive registry metadata and produce
     material records. It should not know Doom GUI internals.

5. Add diagnostics before broadening behavior.
   - Use PathTraceSurfaceDebugDumps for view/crosshair/GUI details.
   - Use PathTraceDebugDumps for one-shot CVar trigger/reset behavior.

6. Test with:
   - r_pathTracingAllowGuiSurfaces
   - r_pathTracingGuiDump
   - r_pathTracingCrosshairMaterialDump
   - mode 18 at normal and high resolution
   - a live interactive GUI surface, not just a static menu texture

What to avoid:

- Do not reintroduce the abandoned generated-GUI frontEndGeo/layer-offset
  experiment as a normal play path. It made painter-order panel layers stick out
  or clip, and was the source of the broken monitor/screen behavior and
  repeatable GUI/cinematic hitch route.
- Do not make scene capture reach directly into unrelated Doom GUI systems if a
  narrow bridge can expose the needed texture/state.
- Do not treat all GUI-like translucent surfaces as safe GUI surfaces.
- Do not overwrite fallback/rejection diagnostics just to force a texture to
  appear. It makes later debugging much harder.

Skinning
~~~~~~~~

Primary files involved:

- PathTraceSkinning.*
- PathTraceSceneCapture.*
- PathTraceSurfaceClassification.*
- PathTraceGeometry.*
- PathTraceAcceleration.*

Current state:

Skinned/deformed surfaces are captured through a CPU skinning bridge. This works
as a temporary path but is expected to move GPU-side later. Current captures now
record explicit skinned surface identity/range records, match them to previous
records by key, retain a skinned-only CPU copy of prior RT CPU-skinned vertices,
and expose conservative validity/invalid-reason diagnostics through
`r_pathTracingSceneInputsDump`. The identity key is bridge-grade, not final:
it uses currently available entity/model/surface pointer fields plus
material/class and depends on scene reset/invalidation boundaries for safety.
Future long-lived temporal consumers still need explicit StableInstanceID and
StableGeometryID ownership. The bridge deliberately does not make previous
skinned positions shader-visible in the default path.

`r_pathTracingGpuSkinning` is the off-by-default scaffold gate:

- `0`: default CPU-skinned bridge only.
- `1`: build diagnostic skinned source/current-output/previous-position/
  dispatch/triangle-dispatch-map/current-joint/previous-joint buffers from the
  CPU bridge and report counts through scene inputs. The compute pass writes
  the diagnostic current output and overwrites validated previous-position
  ranges from previous joint rows when those rows exist.
- `2`: retarget the current-output writes to validated skinned spans in the
  dynamic vertex buffer before dynamic BLAS build while still updating the
  previous-position buffer.

The scaffold does not change BLAS input or modify raster GPU skinning in the
default `0` mode. In `r_pathTracingGpuSkinning 1`, the scaffold dispatches the
PT-owned
`pathtrace_skinning.cs.hlsl` compute shader to write the diagnostic current
output buffer, then leaves BLAS input and hit shading on the existing dynamic
CPU-skinned output. The compute dispatch path does not upload the diagnostic
output buffer from the CPU before overwriting it. It does upload the
previous-position buffer from the CPU bridge first, then overwrites validated
ranges from previous joint rows when they are available. The previous-position
buffer, skinned dispatch records, and dynamic-triangle-to-dispatch-index map are
bound explicitly at the smoke shader level so primary-surface records can
reconstruct a previous world-space hit position for validated skinned dynamic
triangles without a per-hit linear scan over dispatch records.
The shader-visible constants carry the map count and closest-hit code bounds the
lookup against that count, so future map compaction does not turn into an
out-of-bounds read.
Current/previous joint matrix buffers are filled with copied CPU-side joint rows when
`r_pathTracingGpuSkinning 1` is enabled.
In `r_pathTracingGpuSkinning 2`, those same dispatch records write the dynamic
vertex range for each validated skinned surface, and the dynamic vertex buffer
is transitioned back to acceleration-structure input before the dynamic BLAS
build.

Routed rigid primary-surface motion is transform-based rather than vertex-copy
based. `PathTraceRigidRouteInstance` carries current and previous object rows
plus continuity flags; the shader reconstructs previous hit position from the
current rigid triangle barycentrics and the previous transform. Do not add a
duplicated previous rigid vertex buffer for this path.

Safer places to touch:

- Improve base-pose detection or diagnostics in PathTraceSkinning.cpp.
- Add stats in PathTraceSceneCapture for skinned surface counts or previous
  match reasons.
- Extend GPU skinning diagnostics around joint availability, dispatch coverage,
  and mode 2 visual validation.
- Add validation around joint availability.
- Add a new narrow data-source abstraction if it preserves current CPU behavior.

Riskier places to touch:

- Vertex transform math in PathTraceSkinning.cpp.
- Bucket assignment for skinned surfaces in PathTraceSurfaceClassification.cpp.
- BLAS range/index accounting for dynamic buckets.

Future GPU-side direction:

Keep PathTraceSkinning as the bridge boundary. A future GPU path should provide
already-skinned vertex data or a GPU-readable skinning source to scene build /
acceleration code without making material classification or surface bucketing
depend on GPU skinning details.

Higher-level path tracing / emissives
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Primary files involved:

- PathTraceEmissiveCandidates.*
- PathTraceDoomLights.*
- PathTraceMaterialUniverse.*
- PathTraceDynamicMaterialState.*
- PathTraceSmokeDispatch.cpp
- PathTraceLightSelection.*
- PathTraceSmokeSceneBuild.cpp
- PathTraceDebugDumps.*
- PathTraceCVars.*

Current state:

Mode 18 uses a toy path tracing/debug lighting path. Mode 19 builds and dumps an
emissive inventory that can become a future sampling input. This is not yet the
NVIDIA RTXPT/ReSTIR implementation.

Emissive proposal selection now has a shader include boundary:
`pathtrace_emissive_sampling.hlsli`. `SelectSmokeWeightedEmissiveTriangle`
keeps its public name for existing mode 20/ReSTIR/RAB call sites, uses the
CPU-built CDF distribution when `r_pathTracingEmissiveDistribution 1` and the
table is valid, and falls back to the legacy linear scan when disabled or
invalid. Do not optimize by capping or shrinking the authored emissive triangle
inventory; optimize proposal sampling or add a later alias table behind the same
selector.

Doom renderer lights now have a separate authored-light path. PathTraceDoomLights
builds stable identity/probe diagnostics, a CPU-only Doom analytic light
universe scaffold, and an opt-in analytic sphere-light candidate buffer from
active Doom point lights in the selected current-area plus portal-neighbor set.
Compact candidate index is not treated as identity, unknown entity numbers
remain invalid sentinels in the universe state, duplicate keys reject remap,
and dynamic/temporary lights without proven continuity stay remap-invalid.
Task 01 keeps this domain CPU-only. The shader samples the separate candidate buffer for visual
inspection when r_pathTracingAnalyticLightCandidates is enabled. ReSTIR PT debug
modes 26-33 and 50-51 also have the separate
`r_pathTracingRestirPTAnalyticLightCandidates` convenience gate, default 1, so
normal ReSTIR validation sees the authored-light list; set it to 0 with the
global analytic-light CVar off for emissive-only ReSTIR isolation. The bridge is
verified at a basic level: lights appear, preserve RGB color, follow game
animation/on-off state, and survive level-load transitions. The current light
model is intentionally crude and treats every eligible point light as a sphere
emitter, but it now samples the sphere's solid angle instead of tracing a hard
center-point shadow ray. Future work should classify light materials such as
square, round, triangle, projected, and helper textures into better physical
light types before feeding ReSTIR.

Projected Doom lights have enough authored geometry for a future analytic
record: origin/axis plus local target/right/up/start/end define a rectangular
projection footprint. At the target plane, width is approximately
`2 * right.Length()`, height is `2 * up.Length()`, and projected area is
`4 * cross(right, up).Length()`, scaling quadratically with distance along the
target direction. Treat this as a projected/spotlight footprint first, not
automatically as the physical emitter size.

Potential helper-light cleanup should be an opt-in light-to-emissive
association layer. A nearby emissive surface can inherit an authored light's
evaluated RGB, on/off animation state, and calibrated intensity when area,
distance, facing, and material checks produce a strong match. Keep unmatched
helper/fill lights as analytic candidates and dump associations before
suppressing linked helper lights.

Emissive summaries and uploaded light-candidate records now carry stable
material universe indexes in addition to active material-table indexes. This is
intended to let future ReSTIR/PT code refer to persistent material identity even
while the shader-facing active table remains compact and frame-local.

Per-triangle emissive records also carry explicit sampling contract fields:
area-weighted luminance, normalized inventory PDF, area PDF, material universe
index, material id, emissive texture slot/dimensions, UV bounds, and a 64-bit
identity hash derived from material/surface-side triangle identity. These are
prep fields for ReSTIR/PT and are not a final light sampler.

Reservoir resource ownership exists and mode 20 consumes the first slice of it.
Current, previous, and spatial scratch buffers are screen-sized and reset when
the output, camera/FOV, material-table signature, emissive triangle records,
light-candidate records, or static-BLAS signature change. Use
r_pathTracingReservoirDump 1 to inspect the current signature/reset state. Mode
20 writes current reservoirs for one-frame direct emissive lighting; previous
and spatial scratch reservoirs are not consumed yet.

RtPathTraceSceneInputs now records the committed scene/material/light package
for the smoke path. The package names TLAS/BLAS and buffer handles, geometry
counts, material table/texture descriptor inputs, emissive and Doom analytic
light inputs, split signatures, source3 portal policy facts, current/previous
history placeholders, and CPU upload/timing estimates. It intentionally keeps
the existing shader buffer layouts and dispatch behavior unchanged while giving
future ReSTIR/PT passes one contract to consume. Use
`r_pathTracingSceneInputsDump 1` for a one-shot console dump after the next
successful scene commit.

RTXDI-compatible ReSTIR PT host plumbing now exists separately from the smoke
reservoirs. PathTraceRestirPTReservoirs allocates a packed
RTXDI_PackedPTReservoir buffer sized by RTXDI's tiled reservoir parameters and
two PT reservoir slices. PathTraceRestirPT owns an RTXDI ReSTIRPTContext and
uploads an RTXDI_PTParameters constant buffer each smoke dispatch, with
resampling mode still forced to None. The smoke ray tracing binding set and
shader now expose the Step 3 RTXDI PT reservoir ABI at b28/u29 and compile
`Rtxdi/PT/Reservoir.hlsli` through `RTXDI_PT_RESERVOIR_BUFFER`, but no RTXDI
shader pass, temporal reuse, or spatial reuse consumes it yet. Step 4 local RAB
type declarations exist under `RtxdiBridge/`, Step 5 smoke-payload-to-RAB
surface helpers plus intentionally invalid screen G-buffer callbacks are present
in `pathtrace_smoke.rt.hlsl`, and Step 6 RAB RNG callbacks wrap RTXDI's own
sampler helpers. Step 7 RAB BRDF callbacks provide only conservative
opaque-diffuse sampling/evaluation; glossy, transmission, decal, GUI, particle,
and volumetric BSDFs remain future work. Step 8 RAB light callbacks expose
current smoke emissive triangles and Doom analytic sphere lights to the bridge,
but do not add traced visibility, environment sampling, previous-frame light
history, or active RTXDI passes. Step 9 compiles the RTXDI initial-sampling
path-tracer context against a local `RAB_PathTrace` skeleton. Step 10 adds the
first gated dispatch path in `r_pathTracingDebugMode 26`: raygen builds the
current primary RAB surface, calls RTXDI `GenerateInitialSamples`, and stores
the result through `RTXDI_StorePTReservoir` into
`bufferIndices.initialPathTracerOutputBufferIndex`. Misses, GUI surfaces, and
unsupported translucent/special surfaces store an empty reservoir. Mode 26 is a
debug reservoir-writer/heat view only. The local path tracer now records one
conservative NEE sample from the current RAB light list per visited path vertex
before BRDF continuation so initial reservoirs are visible near valid light
candidates; the first BRDF-only probe and the first single uniform light pick
were too sparse to validate visually. The current NEE picker uses the existing
weighted emissive-triangle distribution plus uniform full-domain analytic-light
proposals, then streams valid candidates through a small local RIS selector.
This replaced the earlier best-target heuristic, whose recorded raw light PDF
was not valid after max-score selection. A later compact-list local-window
proposal experiment was also rejected because the compact analytic list is
sorted by portal depth/distance, not per-surface relevance; biasing toward its
first 32 entries made mode 33 change abruptly across tiny portal-area movements.
Doom analytic RAB samples also honor the same Doom influence radius and falloff
used by mode 18 native analytic NEE, so ReSTIR-side proposals no longer reuse
out-of-volume Doom lights. This is still a debug visibility aid, not final light sampling.
It does not shade from the reservoir, does not enable temporal/spatial reuse,
does not use previous-frame G-buffer history, does not trace final RAB
visibility, and does not change modes 18/20. The RAB path-tracer skeleton now
reads the same integrator max-depth, diffuse-bounce, and NEE settings as mode
18 where those settings apply, instead of keeping an unrelated hard-coded
continuation policy.

Modes 26 through 33 now also build a CPU-side `RtPathTraceRestirPassPlan` before
the smoke raygen dispatch. The pass plan records whether the current mode is an
initial-reservoir writer, temporal-reservoir writer, reservoir-shading preview,
primary-surface debug view, source-attribution view, or visibility-tracing
preview. PathTraceSmokeDispatch uses this plan to choose the RTXDI
ReSTIRPTContext resampling mode and to dump the RTXDI initial/temporal/final
buffer indices with `r_pathTracingRestirPTPassDump 1`. The shader still executes
the existing validation paths inside one dispatch, so the pass plan is a data
contract for the next split rather than a behavior change.

Modes 26 through 33 use the same source3 scene-membership baseline as the
playable modes for validation: rigid TLAS routing, rigid residency, routed
rigid emissive-card inventory, and Doom analytic light candidate upload are
enabled for the whole ReSTIR PT debug family. Without this, ReSTIR PT can test
against a culled/fallback scene or an emissive-only light list that does not
match modes 18/20.

Mode 27 is the paired initial-reservoir shading preview. It uses the same
initial reservoir write path and source3 scene-membership baseline as mode 26,
but visualizes `TargetFunction * WeightSum` instead of the temporary diagnostic
fallback color key. It is intentionally initial-only because temporal
resampling still needs previous primary-surface data and motion vectors.
User validation reported that mode 27 works fairly well, runs well
performance-wise, and has no serious stability or gameplay issues. Remaining
black objects/rough normals should be handled as local RAB material and normal
contract cleanup. `r_pathTracingNormalMapFlipGreen` is available as a targeted
diagnostic for Doom 3's legacy green/Y normal-map convention, but the larger
normal fidelity fix is still to pass the original idTech4 vertex tangent/sign
basis into the PT geometry path instead of reconstructing TBN per triangle in
the hit shader. The RAB surface builder now constrains decoded shading normals
to the oriented geometric-normal hemisphere before BRDF checks, which avoids
fully black reservoir-lit surfaces when a normal map tilts past the valid side.

Mode 28 is the paired initial-reservoir visibility preview. It keeps the same
mode 27 reservoir generation and preview shading, but multiplies the preview by
one conservative visibility ray toward the selected reservoir target. NEE
reservoirs decode the packed RTXDI sampled-light reference and resample the
current RAB light; non-NEE reservoirs use the translated reservoir world
position. This is still initial-only validation, not temporal/spatial ReSTIR PT
or final path replay.
User validation reported no issues. Hard-edged shadows are expected because the
mode traces a single binary visibility ray to the selected reservoir sample;
soft shadows require later multi-sample/temporal/spatial accumulation and final
shading.

Mode 29 validates the primary-surface history buffer contract needed before
temporal/spatial ReSTIR PT. It compares current and previous same-pixel
PathTracePrimarySurface records and highlights missing current/previous records,
material mismatches, normal mismatches, and roughness mismatches. Rejected
previous and no-object-motion statuses are reserved for later reprojection and
object-motion validation; they are not part of the normal mode 29 success path.
It intentionally does not reproject or use motion vectors yet. Primary-surface
history clearing is deliberately separate from
`m_smokeReservoirNeedsClear`: the reservoir reset signature is camera-sensitive,
so using it to clear primary history made every moving-camera frame look like
"current valid, no previous" full-screen blue even though the previous frame
copy was otherwise valid.
After splitting those lifetimes, user validation reports the moving-camera
full-screen blue case is gone. Purple normal mismatches and orange roughness
mismatches while moving are expected same-pixel-history artifacts until motion
vectors and reprojection are wired.

Mode 30 validates a temporary screen-space reprojection path before RTXDI
temporal resampling is enabled. PathTraceSmokeDispatch stores the current camera
basis/tangents after copying current primary history to previous; the next
frame's shader projects each current primary surface into that previous camera
and samples the previous history pixel. Static hits use current-world-position
camera reprojection. The shared reprojection helper now consumes the packed
primary-surface `previousPositionOrMotion` plus object-motion/previous-position
flags, so validated skinned hits and routed rigid hits use their packed
previous world position for the previous-pixel lookup. If an eligible dynamic
hit needs object motion but lacks valid previous state, the mode reports the
skinned/rigid invalid color instead of silently falling back to camera-only
reprojection. It uses the same color key as mode 29 plus the object-motion
invalid colors.
User validation reports mode 30 is mostly green, with purple/orange outlines on
moving-object and silhouette edges. Treat those outlines as the expected limit
of camera-only screen-space reprojection, not a failure of the primary history
copy or previous-camera state.

Mode 31 is the first temporal-reservoir validation view. PathTraceSmokeDispatch
sets the ReSTIR PT context to temporal mode for modes 31 through 33, which gives
RTXDI the alternating initial/temporal input/output reservoir indices. The
shader writes the current initial reservoir, projects the current primary
surface to the previous pixel, rejects missing or mismatched previous
surface/reservoir data, then calls RTXDI_PTTemporalResampling and stores the
temporal output reservoir. The previous-pixel lookup is object-aware for
validated skinned/routed rigid hits and remains camera-only for static hits.
This remains a diagnostic color view; it is not final shading and it does not
enable spatial reuse.
`RtPathTraceFrameResources` reports primary-history `objectMotion=1` only when
scene inputs contain a concrete object-motion source: skinned previous-position
dispatch ranges or routed rigid previous transforms.
User validation reports mode 31 is broadly similar to mode 30; violent camera
motion can leave blue edge bands until history catches up. Treat those bands as
offscreen/disocclusion misses from the temporary reprojection path rather than
reservoir corruption.

Mode 32 shades the temporal reservoir output from the same debug raygen path.
It uses the mode 31 temporal context/indices and conservative previous-surface
gates, stores the current initial reservoir before calling
RTXDI_PTTemporalResampling, then previews `TargetFunction * WeightSum` from the
temporal output reservoir. The visible preview uses
`r_pathTracingRestirPTPreviewExposure` and suppresses the strong rejection
colors used by mode 31; invalid reservoirs fall back to near-black material
context so ordinary movement does not look like an edge-detection overlay. The
preview also has a selected-NEE reconstruction fallback when
`TargetFunction * WeightSum` is effectively black but the packed reservoir
still identifies a Doom analytic or emissive light. That fallback is for rough
visualization only and is not the final ReSTIR PT estimator. The
mode 28-style conservative visibility ray is controlled by
`r_pathTracingRestirPTPreviewVisibility` and defaults off because it adds
another ray per pixel on top of temporal PT replay. It remains a rough preview;
final shading and spatial reuse are still separate steps.

The RAB path tracer accounts for RTXDI's primary-surface convention:
`bounceDepth = 2` is the first NEE-capable vertex. Do not cap the bridge loop
directly to the local secondary-bounce count, or initial reservoirs will be
empty and modes 26/27/32 will fall back to flat material context.

Temporal resampling now uses the combined geometry motion helper's sub-pixel
motion vector rather than an integer previous-pixel delta. That keeps routed
rigid objects on the same motion contract as the exported motion-vector path
and should reduce rigid edge instability while moving.

The mode 32 wrapper does not pre-reject temporal reuse on exact previous-pixel
surface/reservoir validity anymore. RTXDI_PTTemporalResampling owns the
previous-neighborhood search and validation; keeping an exact-pixel gate in
front of it caused visible edge flashes during movement even when mode 49 showed
stable static/skinned/routed-rigid motion-source coverage.

Modes 31-33 and 50-51 expose RTXDI's temporal/spatial relative-depth and normal-dot
thresholds as
`r_pathTracingRestirPTTemporalDepthThreshold` and
`r_pathTracingRestirPTTemporalNormalThreshold`. The normal threshold defaults to
0.35 for Doom asset stability; RTXDI's stock 0.6 was too strict for some
normal-mapped object sides during motion.
`r_pathTracingRestirPTTemporalReservoirReuse` defaults to 1 and exists only as
a diagnostic split: set it to 0 when mode 32/50 flickers in exact alternating
frames while mode 52 stays stable. If the flash disappears, the current-frame
geometry and initial reservoir are probably sound and the bug is previous
reservoir reuse/history validation.
`r_pathTracingRestirPTTemporalFallbackSampling` defaults to 1 now that the
motion-vector path and pre-combine bad-NEE rejection guard the rough temporal
preview. Set it to 0 only to isolate zero-motion fallback behavior from the
motion-projected path.
`r_pathTracingRestirPTTemporalAnalyticNeeReuse` defaults to 1 and allows
previous-frame NEE light reservoirs during temporal replay. The name is
historical; set it to 0 only to isolate current-frame NEE sampling from previous
light history. The robot/strobe endpoint case is validated by translating
previous NEE reservoirs through the Doom analytic remap and rejecting stale light
state before temporal combine. Non-NEE temporal reuse remains enabled.
Mode 32/50 temporal validation also passes RTXDI an expected previous depth
computed from the projected previous world position's ray distance. Do not
derive that value from the sampled previous surface at the projected pixel; if
the pixel lands on a wrong same-material triangle, that would let the wrong
triangle validate itself. Do not use previous-camera forward-Z here either,
because `RAB_Surface.linearDepth` is the smoke ray `hitT`, and mixing the two
depth metrics causes a centered circular temporal-validation region.
The packed ReSTIR PT reservoir history is also cleared when the scene reservoir
signature changes. That signature includes Doom analytic light topology and
identity, but not analytic light color/intensity. Mode 32/50 debug preview
therefore re-evaluates NEE reservoirs against current light data before display
instead of trusting stale target energy packed in the reservoir. This avoids
global history wipes from unrelated flickering lights while preserving the proof
that stale bright-phase light reservoirs were the source of the earlier
polygon-brightness bug. A production path still needs stable light remapping and
per-light radiance-version rejection.
`RAB_LoadLightInfo(..., previousFrame=true)` is used by RTXDI PT inverse
shift/bias correction. The bridge treats the incoming light index as the current
reservoir light index, remaps it through the Doom analytic universe to a
previous packed analytic index, and only then reads the previous analytic light
table. Failed remaps return an empty light instead of reusing a compact slot.
Before temporal combine, the mode 32/50 wrapper also scans the reprojected 3x3
previous-reservoir neighborhood and sets temporal max age to zero if a previous
NEE reservoir cannot translate through the current Doom analytic remap or fails
the analytic light-state compatibility guard. This rejects bad NEE history
before RTXDI's combine/bias-correction bookkeeping can keep it alive.
This was validated on 2026-05-14 against the robot/strobe endpoint case. Stable
identity, previous light records, inverse-shift loading, and light-state checks
were necessary infrastructure but did not change the visible flicker by
themselves. The fix landed only when bad previous NEE reservoirs were rejected
before RTXDI temporal combine. The confirmed symptom was stationary lights
turning off/flickering until movement, then being temporarily repaired while a
nearby robot/strobe light animation was active.
The current temporary safety rule is stronger than preview-side reconstruction:
when NEE temporal reuse is disabled and the current initial reservoir selects a
Doom analytic light, mode 32/50 stores that current reservoir directly as the
temporal output and skips RTXDI temporal combine/bias correction for that pixel.
This prevents the inverse-shift path from treating the current analytic light
index as a previous-frame light-table entry before real light remapping exists.
Separately, before temporal combine, the mode 32 wrapper scans the reprojected
previous 3x3 reservoir neighborhood. If NEE temporal reuse is disabled and any
candidate previous reservoir connects to a sampled light, that pixel's temporal
resampling is forced to current-frame behavior by zeroing the local temporal
max reservoir age. This avoids stale direct-light reservoirs affecting RTXDI's
combine/bias-correction normalization even when the stale NEE reservoir is not
the final selected sample.
The bridge also clamps `RAB_LightSampleSolidAnglePdf` for rejected/non-replayable
light samples to a finite neutral value. RTXDI's hybrid-shift helper divides by
that callback even after conservative visibility rejects the light sample, so a
stale light whose current radiance or influence volume disappeared must produce
zero contribution, not NaN/Inf state that can make unrelated lights flicker or
turn off.

Mode 32 has a validated high-resolution workload limit. It is stable enough for
basic testing around 1280x720, but user validation reports that pushing much
beyond 720p can device-remove the GPU. This is expected for the current
prototype shape: the monolithic raygen performs full-screen current initial PT
sampling, RTXDI temporal resampling / hybrid-shift work, and optional visibility
inside one heavy RT dispatch. `r_pathTracingRestirPTPreviewMaxPixels` is an
experimental safety cap, but the reliable testing rule is to keep mode 32/33/50/51
at or below 720p until the pass is restructured.

Modes 53-55 are the first indirect/GI reservoir diagnostics. They reuse the

The GI debug path explicitly allows one secondary bridge bounce and raises the

bounceDepth 2; secondary-surface NEE is recorded as pathLength 4, so the default
initial max of 3 accepts primary NEE but silently rejects the first indirect
analytic NEE candidate. This override is scoped to modes 53-56 and does not
change mode 18.

Mode 56 is the combined direct + GI debug preview. DI/direct is still raw
initial-reservoir output; do not describe it as completed temporal/spatial

toward shader work inside raygen: environment/sky paths, material/light

Mode 38 is the skinned primary object-motion vector diagnostic. It visualizes
skinned hits with retained previous positions and intentionally treats routed
rigid hits as no-object-motion until rigid coverage has a separate validation
pass. Valid skinned motion encodes current-to-previous pixel displacement as
signed red/green direction around 0.5 plus blue magnitude, capped at 32 internal
PT pixels. Static surfaces and missing/invalid skinned history use the existing
no-object-motion or skinned invalid colors. It does not write a production
motion-vector buffer. `r_pathTracingGpuSkinning 1` is required for the
diagnostic skinned source/output/previous-position scaffold; the current mode 1
scaffold does not switch final rendering to PT-owned GPU skinning.

Mode 39 is the routed-rigid object-motion eligibility diagnostic. It does not
visualize vectors; it reports whether rigid-class primary hits have a routed
rigid instance with a continuous previous transform. Green means the current
hit has valid routed rigid object-motion data, cyan means the hit is rigid-class
but still on the non-routed fallback path, and the existing rigid invalid
colors cover missing previous transform or range/index mismatch. The mode also
uses routed-rigid dynamic fallback removal so overlap between the routed TLAS
instance and the dynamic fallback copy does not look like a coverage failure.
Non-rigid surfaces use the no-object-motion color. This mode is diagnostic-only
and does not change final PT rendering.

Mode 40 is the routed-rigid primary object-motion vector diagnostic. It mirrors
mode 38's current-to-previous pixel encoding, but only for routed rigid hits
whose previous transform is continuous. Missing previous transforms, range/index
mismatches, and non-routed rigid fallback hits use the same invalid/no-object
colors as mode 39. The mode uses routed-rigid dynamic fallback removal so the
vector field comes from the routed instance rather than an overlapping fallback
copy. It is diagnostic-only and does not export a production motion-vector
buffer.

Mode 41 is the combined skinned/routed-rigid object-motion vector diagnostic.
It uses the shared packed-object-motion pixel helper to visualize the packed
`previousPositionOrMotion` field when the record has object-motion and
previous-position flags. Static surfaces stay in the no-object-motion color,
and skinned or routed-rigid invalid states keep their existing
missing/range/out-of-range colors. Routed-ready dynamic fallback removal is
active so the combined vector field is not polluted by overlapping fallback
geometry. Its skinned branch needs `r_pathTracingGpuSkinning 1` for the
diagnostic previous-position scaffold; with that CVar off, skinned surfaces
correctly report missing skinned previous data. It is diagnostic-only and does
not export a production motion-vector buffer.

Mode 42 is the packed primary object-motion flags diagnostic. It repacks the
current primary surface and visualizes whether the stored
`PathTracePrimarySurfaceRecord` has object-motion and previous-position flags
plus a valid `previousPositionOrMotion.w` selector. Valid packed skinned records
are green, valid packed routed-rigid records are teal, static/no-object records
use the no-object-motion color, and invalid states keep the existing
primary-surface debug colors. The skinned branch still requires
`r_pathTracingGpuSkinning 1` for the diagnostic previous-position scaffold.

Mode 43 is the packed object-motion reprojection match diagnostic. It uses the
same packed object-motion flags as mode 42, uses the shared packed-object-motion
pixel helper to project the stored `previousPositionOrMotion.xyz` into the
previous camera, reads the previous primary-surface history record at that
pixel, and reports the normal
material/normal/roughness match colors. Green means a packed skinned or
routed-rigid object-motion source lands on a conservative previous-hit match;
static/no-object and invalid dynamic states keep the existing primary-surface
debug colors. It is diagnostic-only and does not export a production
motion-vector buffer.

Mode 44 is the previous static snapshot shader-binding diagnostic. It verifies
that static-world primary hits can read the reserved previous-static
vertex/index/triangle-class/triangle-material/material-index buffers by current
persistent static primitive index. Green means the previous snapshot is
shader-visible, range-safe, materially consistent, and reconstructs a previous
position matching the current static hit. Non-static hits use the
no-object-motion color and failures reuse the existing primary-surface
missing/range/material/position mismatch colors. It is diagnostic-only and does
not consume previous static data for production motion or shading.

Mode 45 is the previous static snapshot reprojection match diagnostic. It uses
the same previous-static lookup as mode 44, projects the reconstructed previous
static position through the previous camera, reads the previous primary-surface
history record at that pixel, and applies the normal material/normal/roughness
match colors. Green means the shader-visible previous static snapshot is usable
as a conservative temporal correspondence source for static-world hits.
Non-static hits and static lookup failures reuse the mode 44 colors. It is
diagnostic-only and does not consume previous static data for production motion
or shading.

Mode 46 is the previous static snapshot motion-vector diagnostic. It uses the
same previous-static lookup as mode 44, projects the reconstructed previous
static position through the previous camera, and visualizes current-to-previous
pixel displacement with the same signed red/green plus blue magnitude convention
as the skinned and routed-rigid object-motion vector modes. Non-static hits and
static lookup failures reuse the mode 44 colors. It is diagnostic-only and does
not consume previous static data for production motion or shading.

Mode 47 is the combined geometry motion-vector diagnostic. Static-world hits
use the previous-static snapshot path from mode 46, while skinned and
routed-rigid hits use the packed object-motion path from mode 41. Other dynamic
or invalid hits keep the existing object-motion rejection colors. Like mode 41,
it removes routed-ready rigid candidates from the dynamic fallback path so the
rigid portion visualizes the routed source rather than overlapping fallback
geometry. It is diagnostic-only and does not consume previous geometry data for
production motion or shading.

Mode 48 is the combined geometry reprojection-match diagnostic. Static-world
hits use the previous-static reprojection path from mode 45, while skinned and
routed-rigid hits use the packed object-motion reprojection path from mode 43.
Other dynamic or invalid hits keep the existing object-motion rejection colors.
Like mode 47, it removes routed-ready rigid candidates from the dynamic fallback
path so the rigid portion validates the routed source rather than overlapping
fallback geometry. It is diagnostic-only and does not consume previous geometry
data for production motion or shading. The shared current/previous
surface-similarity helper compares geometric normals for temporal identity
while leaving normal-mapped shading normals intact for lighting.

Mode 49 is the combined geometry motion-source diagnostic. It uses the shared
combined geometry motion helper that future production export code should call,
then colors valid static, skinned, and routed-rigid motion sources distinctly.
Invalid or unsupported hits keep the same rejection colors used by modes 47 and
48. Like modes 47 and 48, it removes routed-ready rigid candidates from the
dynamic fallback path. It is diagnostic-only and does not consume previous
geometry data for production motion or shading.

Mode 52 is the routed-rigid transform parity diagnostic. It traces routed rigid
TLAS instances, removes their fallback dynamic copies, and compares the actual
DXR hit position against both TLAS `ObjectToWorld3x4()` reconstruction and the
CPU-uploaded `PathTraceRigidRouteInstance.currentObjectToWorld` reconstruction.
Green means both reconstructions match; yellow/orange/red show increasing route
error; purple means route metadata disagrees with TLAS transform; blue is
non-routed/unsupported coverage. It exists to diagnose rotated rigid props whose
flicker respects polygon boundaries.

Optimization path: split initial sampling, temporal resampling, visibility, and
preview/final shading into separate profiled passes; keep the visibility ray
off by default; add lower internal resolution / checkerboard execution before
presentation; reduce initial sample and replay parameters after correctness is
locked; and wire real motion vectors so temporal history rejection does not
thrash around moving objects and disocclusions.

Rejected temporal-resolve shortcut: an uncommitted mode 33 prototype reused
`SmokeAccumulation` as previous resolved color and copied `SmokeOutput` back
into it after dispatch. User validation reported little regular lighting,
mostly emissive contribution, orange outlines, and device removal after roughly
30 seconds. That prototype was removed; the active mode 33 slot is now source
attribution. Future temporal resolve work should not reuse this shortcut; it
needs explicit history ownership, separated passes, lower internal
resolution/checkerboard execution, and per-stage GPU timing.

Mode 20 has a known device-remove stress risk under heavier/high-FPS testing.
When it fails, the scene/surface buffers can visibly degrade before the driver
removes the device. This resembles the older mode 18 `r_pathTracingLightCount
32` at 120 FPS failure, where brute-force primary plus one-bounce selected-light
shadowing could produce up to 64 shadow rays per pixel, saturate queued GPU
work while the camera was moving, and trigger transient surface corruption
followed by device removal. Keep reservoir experiments conservative until
buffer lifetime, barriers, UAV writes, and per-pixel ray workload have dedicated
validation.

Mode 20 now samples a merged light list: persistent static emissives plus
current-frame dynamic emissives. Semi-static promoted dynamic records are
tracked for identity/history diagnostics, but missing dynamic records are not
injected by default because only retaining the light without the dynamic
occluder geometry produces ghost lighting. Use r_pathTracingLightUniverseDump 1
to inspect persistent/merged counts.

Q2RTX's Ray Reconstruction reference fork handles off-camera emissives by
promoting static emissive BSP surfaces into light polygon records, assigning
them to BSP clusters, and sampling per-cluster light lists rather than the
current screen-visible emissive set. Dynamic/model emissive lights are injected
into those cluster lists each frame and temporal ReSTIR history uses explicit
previous-to-current light identity remapping. If a previous dynamic light cannot
be matched in the current frame, the history is invalidated. For this branch,
that points toward a future light-universe/area-membership contract for static
and stable entity emissives, not stale dynamic candidate retention.

Modes 18, 19, and 20 all honor per-surface evaluated emissive state.
Switchable or dimming lights can have a material that is capable of emissive
output while the current draw surface has its emissive stage disabled or dimmed
by shader registers. Treat material-universe emissive facts as stable
capability only; runtime active/dimmed emissive state belongs to
capture/emissive-candidate records.

Texture binding and fallback behavior
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Primary files involved:

- PathTraceMaterialTextureDiscovery.*
- PathTraceTextureRegistry.*
- PathTraceMaterialUniverse.*
- PathTraceDynamicMaterialState.*
- PathTraceSmokeResources.*
- PathTraceSmokeDispatch.cpp
- PathTraceDebugDumps.*

Current state:

Textures are discovered from visible materials, validated into safe handles, then
stored in the texture registry and reflected into persistent material-universe
records. PathTraceDynamicMaterialState then packs the current active bindless
texture table and patches descriptor indexes into the compact shader-facing
material records. Unsupported or unsafe cases keep fallback reasons for
diagnostics.

Safer places to touch:

- Better fallback names/reasons.
- Additional texture metadata that does not change shader layout.
- More diagnostic dumps around active texture windows.

Riskier places to touch:

- Descriptor table capacity and active texture cap.
- Force-fallback behavior.
- Texture index assignment in material table records.
- Binding set/layout creation in PathTraceSmokeResources.cpp.

Scene build ownership
~~~~~~~~~~~~~~~~~~~~~

Primary files involved:

- PathTracePrimaryPass.*
- PathTraceSmokeSceneBuild.cpp
- PathTraceSmokeResources.*
- PathTraceSmokeDispatch.cpp
- PathTraceSmokeReadback.cpp

Current state:

PathTracePrimaryPass still owns persistent scene/build m_smoke* state. Several
modules define private PathTracePrimaryPass methods so they can work with that
state without a large public state object. Output-sized frame resources are the
first broad orchestration exception: they now live behind
RtPathTraceFrameResources so resize/reset policy and frame diagnostics can grow
without adding more ad hoc fields to PathTracePrimaryPass. Static-world
geometry cache storage also lives behind RtSmokeGeometryUniverse so persistent
geometry can grow without making PathTracePrimaryPass larger.

Safe next architectural step:

For geometry, continue growing PathTraceGeometryUniverse deliberately:

1. Expand the current static surface records into explicit persistent
   surface/range records.
2. Add one-frame history requirements to those records before finalizing the
   range/buffer model: previous transform, previous range, previous vertex data
   or delta data, and invalidation flags.
3. Move static GPU buffer and dirty-range ownership only after the CPU cache
   record model is stable.

For broader smoke pass state, design a PathTraceSmokeState or PathTraceSmokeScene
object first, then move ownership in one dedicated architecture change. Do not
gradually leak pieces of m_smoke* state into unrelated helper modules without a
plan.
