RBDoom3 BFG RT/PT module map
============================

This document maps the experimental Vulkan/NVRHI RT smoke/path tracing files and
the main feature flows that pass through them. It is meant as a navigation guide
for humans working on the branch, not as a replacement for the implementation or
the broader path tracing handoff notes. The `pathtrace_scene_producer_*` files
are historical intervention/build logs; this document is the durable module map.

For persistent geometry guardrails around lookup cost, validation cost,
lifecycle, thread ownership, material links, allocation, and transient geometry
exclusions, see docs/pathtrace_geometry_universe.md.

For persistent material guardrails around stable facts, descriptor ownership,
reset boundaries, lookup identity, validation cost, thread ownership, allocation,
and churn stats, see docs/pathtrace_material_universe.md.

For game-driven material/register behavior such as switchable fluorescent lights
and red/green door panels, see docs/pathtrace_interactive_surfaces.md.

Current scope
-------------

The current RT/PT path is still an experimental "smoke" path:

- r_pathTracing mode 18: toy path tracer / textured smoke path with selected
  lights, emissives, accumulation, ambient/lighting CVars, and debug modes.
  `r_pathTracingToyFakePBRSpecular` can opt this mode into a shader-local
  rbdoom3 legacy specmap-to-PBR F0/roughness approximation for direct specular
  testing.
- r_pathTracing mode 19: emissive inventory/debug pass for future ReSTIR/PT
  preparation.
- r_pathTracing mode 20: first single-frame reservoir direct-lighting debug view
  from emissive candidates. It writes current reservoirs only; spatial and
  temporal reuse are still off.
- r_pathTracing mode 21: PT-native solid drawSurf mirror bounds boxes for
  spatial/BVH/source diagnostics.
- r_pathTracing mode 22: PT-native wireframe drawSurf mirror bounds boxes. This
  is the preferred human spatial inspection mode because all-category solid
  boxes can occlude too much.
- r_pathTracing mode 23: experimental routed rigid TLAS instance view. It traces
  only routed rigid BLAS instances with a rigid-only ray mask and reads the
  packed rigid route vertex/index/material buffers, so it can validate
  local-space BLAS/TLAS transforms without changing mode 18/20 shading.
- r_pathTracing mode 24: routed rigid overlap validation. It keeps the old
  fallback scene intact, traces fallback-only and routed-rigid-only masks
  separately, color-codes hit-distance agreement, and can dump per-color pixel
  proportions with `r_pathTracingRigidRouteOverlapDump 1` before any dynamic
  fallback rigid removal is attempted.
- r_pathTracing mode 25: routed rigid lighting validation. It traces the
  combined fallback plus routed rigid TLAS scene with simple material/normal
  lighting, after optional routed-rigid removal from the dynamic fallback.
- r_pathTracingRigidRouteMode18: default-on mode 18 integration gate for
  routed rigid TLAS instances. It preserves mode 18's toy PT shader path while
  routing eligible rigid meshes through reusable local-space BLAS/TLAS records.
- r_pathTracingMode18TestPreset: one-shot helper for applying the current
  source3 routed-rigid/static-preload/rigid-residency mode 18 stack. The stack
  is already default-on; the preset remains a convenience/debug shortcut.
- r_pathTracingRigidRouteRemoveDynamic: default-on subtraction gate used by
  routed-rigid validation/integration modes, including mode 18 and mode 20. It
  skips routed-ready rigid candidates from the source3 dynamic fallback after
  the reusable rigid route is enabled.
- r_pathTracingMode20TestPreset: one-shot helper for the current mode 20 source3
  routed-rigid/static-preload/rigid-residency stack. The stack is already
  default-on; the preset remains a convenience/debug shortcut for mode 20 and
  optional light-area diagnostics.
- r_pathTracingAnalyticLightCandidates: opt-in bridge from authored Doom
  renderer/game lights into a separate analytic sphere-light buffer. The first
  shader path keeps old selected point lights available for comparison, while
  r_pathTracingAnalyticLightReplaceSelected can suppress that legacy constant
  array so the analytic lights can be inspected visually on their own.
- r_pathTracingAnalyticLightMaxGpu: cap for analytic Doom light candidates
  uploaded/sampled by the smoke shader. This is still a direct per-hit loop,
  so keep it conservative until ReSTIR candidate selection exists.
- r_pathTracingAnalyticLightIntensityScale: radiance calibration multiplier for
  analytic Doom sphere lights. The current default is 4.0 after visual testing
  showed the early low diagnostic scale under-lit authored Doom helper lights.
- r_pathTracingAnalyticSphereLightRadiusScale and min/max: convert Doom point
  light radius into the first diagnostic sphere-emitter size. This is not final
  physical classification of round/square/projected/area light materials. These
  CVars are consumed by PathTraceDoomLights.cpp when it builds
  PathTraceDoomAnalyticLightCandidate records; the default radius scale is 0.08
  and the default min/max clamp is 4.0/64.0 world units.
- Source3 RT residency uses Doom renderer area/portal selection rather than
  current raster visibility alone. The final drawSurf stream is the freshness
  source for transforms/material overrides/runtime registers; portal-selected
  static preload and rigid residency are the persistence/residency layer for
  occluded or off-camera RT geometry. Depth 4 is the current source3 default
  because visually continuous rooms can span several Doom portal areas.
- Source3/static/rigid/light portal walks seed from all valid camera-area
  probes at depth 0: `viewDef->areaNum`, initial view-area origin, exact view
  origin, and short +/- view-axis offsets. This avoids one-cell portal-border
  flips adding artificial hops to every resident object and light. The concrete
  regression was a valid-but-bad camera area on a Mars City portal border: PT
  residency started in a tiny connector/sliver cell, consumed several walks
  before reaching the visually occupied room, and then culled rigid residents
  plus Doom analytic lights that appeared to be in the next visible portal.
  Multi-seeding does not flatten real portal topology; it only prevents the
  current-area probe from being the sole traversal root when the camera is on a
  border.
- Routed rigid closest-hit shading now transforms local-space geometric and
  interpolated normals into world space through the TLAS instance transform and
  constructs a world-space tangent basis for later lighting validation.
- r_pathTracingSceneSource 3: default scene producer replacement scaffold. It
  keeps the existing drawSurf-seeded static cache and mirrors dynamic-frame
  drawSurfs as mesh identities plus per-frame instances while preserving the
  old shader-compatible static/dynamic render buffers. The original
  drawSurf-only source0 path remains available for emergency comparison, but is
  deprecated as the normal BVH source.
- In mode 18, r_pathTracingSkipRaster3D defaults to 1 so the RT output owns the
  3D scene layer and the normal Doom 3D raster view is skipped. GUI/HUD/menu
  draw commands still rasterize afterward. Leaving full 3D raster enabled below
  the path traced output roughly doubled frame cost in testing and allowed
  raster-side hitches to be misread as RT/PT hitches.
- Many dynamic Doom 3 material behaviors are not fully represented yet. Some are
  deliberately hidden, rejected, or forced to fallback/debug handling rather than
  guessed incorrectly.

File map
--------

PathTracePrimaryPass.h
    Declares PathTracePrimaryPass, the frame-level owner for the experimental
    RT smoke state. The class still owns most persistent m_smoke* handles,
    counters, and private pass methods, but static-world geometry cache storage
    now lives in PathTraceGeometryUniverse. Most behavior is implemented in
    smaller PathTrace* modules.

PathTracePrimaryPass.cpp
    Thin pass shell. Performs RT feature detection, per-frame entry flow, output
    sizing, and debug-present blit. The PT output size is controlled by
    r_pathTracingDebugWidth/r_pathTracingDebugHeight, not the application window
    size. Internal PT output-size changes wait for the GPU to idle before old
    output, accumulation, readback, smoke reservoir, or ReSTIR PT reservoir
    resources can be replaced. When PT is active, a known native-size resize
    also waits for the GPU to idle before ResizeImages() so previous-frame RT/AS
    work cannot overlap swapchain teardown, then invalidates PT output, scene,
    BLAS, smoke reservoirs, ReSTIR PT reservoirs/context, and the cached LDR
    blit framebuffer. The resize frame skips PT execution and debug
    presentation; if r_pathTracingSkipRaster3D owns the 3D view, it skips raster
    3D too instead of falling back into the normal 3D path. Avoid putting scene
    build, dispatch, resource lifetime, or diagnostics back here unless the pass
    shell itself truly owns the work.

RenderBackend.cpp
    Calls PathTracePrimaryPass from DrawViewInternal for 3D views. When
    r_pathTracingSkipRaster3D is 1, the backend returns after the RT output is
    produced, so the normal 3D raster pipeline does not run underneath mode 18.
    Later GUI/HUD/menu commands still pass through the raster backend. Keep this
    PT-owns-3D / raster-owns-GUI split unless a specific missing side effect is
    identified and restored narrowly.

PathTraceSmokeSceneBuild.cpp
    Per-frame scene build orchestration. It calls scene capture, material table
    construction, emissive inventory generation, buffer creation/upload,
    BLAS/TLAS submission, binding creation, resource commit, and scene build
    diagnostics. It also asks PathTraceDoomLights for analytic Doom light
    candidates before scene buffer creation when that CVar path is enabled. This
    is the right place to adjust high-level build order, but it should delegate
    specialized work to narrower modules.

PathTraceSmokeResources.h
    Resource-facing structs for smoke geometry buffers, binding build inputs,
    binding build results, scene resource commit descriptors, and resource
    commit packaging. The Doom analytic light buffer is part of this resource
    package, separate from emissive triangle and emissive light-candidate
    buffers.

PathTraceSmokeResources.cpp
    Persistent NVRHI resource lifetime. Creates pipeline/output/readback
    resources, creates scene geometry buffers and binding resources, packages
    committed scene resources, resets smoke state, and commits successful scene
    builds back into PathTracePrimaryPass state. `ResizeRayTracingSmokeOutput`
    is the owner for PT internal render-resolution changes from
    r_pathTracingDebugWidth/r_pathTracingDebugHeight. When replacing existing
    output-sized resources it must wait for GPU idle and run NVRHI garbage
    collection before old output, accumulation, readback, smoke reservoir, or
    ReSTIR PT reservoir handles are released. This fixed a hard device-removal
    crash that initially looked like an app-resize or aspect-ratio issue but was
    actually old PT resources being replaced while previous-frame GPU work could
    still reference them.

PathTraceSmokeDispatch.h
    Exposes the smoke constants buffer size without exposing the full constants
    layout.

PathTraceSmokeDispatch.cpp
    Ray tracing dispatch path. Builds raygen constants, selects debug lights,
    sets resource states, manages accumulation, dispatches the NVRHI RT pipeline,
    and queues optional readback. It uploads Doom analytic light counts/scales
    through dispatch constants and can suppress the legacy selected point-light
    constant array when r_pathTracingAnalyticLightReplaceSelected is enabled.

PathTraceSmokeReadback.cpp
    Optional diagnostic readback of the smoke output texture. Handles delayed
    map/unmap and throttled terminal logging.

PathTraceAcceleration.h
    Acceleration helper declarations and small structs for geometry ranges,
    BLAS creation inputs/results, buffer upload batches, TLAS/BLAS submit inputs,
    and timing results.

PathTraceAcceleration.cpp
    Pure-ish acceleration helpers. Initializes NVRHI triangle geometry, creates
    BLAS objects, computes the static BLAS signature, uploads acceleration input
    buffers, and submits BLAS/TLAS builds. Long-lived cache state is not owned
    here.

PathTraceSceneCapture.h
    Scene capture constants, stats structs, bucket range structs, reason sample
    structs, and the capture entry points. This is the main interface for
    converting Doom draw surfaces into RT smoke triangle buckets.

PathTraceSceneCapture.cpp
    Doom draw-surface capture implementation. Walks view draw surfaces, chooses
    static/rigid/skinned/particle buckets, transforms vertices, records skip and
    classification stats, and appends captured geometry for the scene builder.
    Also stamps per-triangle runtime emissive state from evaluated draw-surface
    shader registers; stable material metadata only describes emissive
    capability, not whether a surface is currently emitting.

PathTraceSceneUniverse.h
    Diagnostics/static-preload scene-universe declarations. Defines direct
    renderer-world static surface records, area/material stats, selected-area
    portal-walk stats, and static geometry build stats. This module is useful
    for map inventory, area membership, and source3 static preload support; it
    is not the primary live scene representation.

PathTraceSceneUniverse.cpp
    Direct render-world scanner for static map/world data. It walks renderer
    entities and model surfaces without depending on the final raster drawSurf
    list, computes static surface keys, emissive-capable material summaries,
    bounds/area membership, selected-area portal diagnostics, and optional
    selected/full static geometry appends. It should remain diagnostics/preload
    infrastructure: do not feed its records as the universal RT scene source or
    synthesize live rigid/skinned/dynamic drawSurfs from it. Its portal-area
    selected static append path is used to seed static-world cache records that
    Doom raster visibility has not submitted this frame. The selected-area walk
    now uses the shared multi-seed camera-area policy so static preload does
    not disappear when `viewDef->areaNum` lands in a portal-border connector
    while the view is effectively in an adjacent room.

PathTraceDrawSurfCapture.h
    DrawSurf mirror declarations for the source3 scene producer scaffold.
    Exposes mesh observations, instance observations, bounds-overlay records,
    and the capture entry point that mirrors the final visible raster drawSurf
    stream without synthesizing fake drawSurfs or fake viewEntities.

PathTraceDrawSurfCapture.cpp
    Live drawSurf mirror implementation. For each usable visible drawSurf it
    builds a mesh key that excludes transform, records per-frame instance
    transform/material/entity metadata, compares static-world matches against
    PathTraceSceneUniverse diagnostics when requested, feeds geometry-universe
    rigid candidate diagnostics, and optionally emits bounds records consumed by
    debug modes 21/22. It also contains the conservative source3 promotion gate
    for stable entity-attached translucent signage/glow cards, so such cards can
    travel with routed rigid residency instead of remaining raster-visible-only
    particle/alpha drawSurfs.

PathTraceInstanceUniverse.h
    Per-frame and lifetime instance-universe diagnostic declarations. This is
    the current home for observed mesh records, instance records, transform
    history, moved-rigid counters, and compact dump stats. It is not a TLAS
    source yet.

PathTraceInstanceUniverse.cpp
    Instance-universe implementation. Tracks mesh/instance observations across
    frames, separates same-transform and changed-transform history, records max
    observed matrix/origin deltas for rigid movers, reports missing
    material/skin override data, and prints one-shot dumps for
    r_pathTracingInstanceUniverseDump.

PathTraceGeometry.h
    Shared GPU-facing vertex layout and common geometry helper declarations.
    PathTraceSmokeVertex is consumed by NVRHI buffers and shader code, so layout
    changes here must be treated as shader/resource contract changes.

PathTraceGeometry.cpp
    Pure geometry/math helpers such as finite checks, normal packing support,
    and triangle/UV calculations shared by capture, acceleration, and emissive
    candidate code.

PathTraceGeometryUniverse.h
    Persistent static-geometry and rigid-diagnostic universe declarations. This
    is the first ownership boundary for static-world geometry cache data and
    currently exposes the static surface keys, vertices, indexes, triangle
    classes, triangle material IDs, static surface records, per-frame lifecycle
    fields, previous range fields, and stats used by scene build. It also owns
    rigid mesh candidate/local-source records, validation stats, CPU rigid BLAS
    plan/input descriptor dumps, and default-on rigid BLAS GPU scaffold records
    for per-mesh local-space vertex/index buffers plus optional standalone BLAS
    handles. It also owns source3 rigid residency records, live revalidation
    state, selected-area emissive rigid area-walk observations, and bounds
    samples. It builds rigid TLAS instance plans by matching current-frame
    revalidated instance observations to rigid mesh records when the route
    gates are enabled.
    The universe must evolve with previous-frame geometry history in mind so
    future motion vector output can be generated from geometry, not guessed
    later.

PathTraceGeometryUniverse.cpp
    Static geometry universe and rigid-diagnostic implementation. At this stage
    it owns the former PathTracePrimaryPass static-cache vectors and
    generation/stat tracking while preserving the existing shader-facing buffer
    behavior. Static surface lookup, per-frame BeginFrame/Touch/EndFrame
    lifecycle updates, append completion, and static record validation now go
    through the universe. It also records eligible rigid local-space mesh
    sources, validates them against the baked dynamic rigid fallback payload,
    builds CPU-only reusable rigid BLAS/TLAS plans and NVRHI-shaped input
    descriptor dumps, and now owns a default-on GPU scaffold that can
    create/upload local-space rigid mesh input buffers and submit per-mesh BLAS
    builds. With the explicit rigid route gates, it can feed routed rigid TLAS
    instances into modes 18/20/23/24/25. Rigid residency no longer emits a
    blind cache of ordinary rigid objects: visible drawSurf observations are
    the authoritative freshness source, records not revalidated in the current
    frame are pruned, and the selected-area portal walk only supplements
    conservative emissive/signage/glow rigid surfaces after the entity has been
    previously validated by visible drawSurfs without a later render-entity
    modification. Rigid residency was one visible symptom of the portal seed
    bug: a valid-but-bad current area could make cached rigid objects in the
    next visually adjacent portal fall outside the selected set despite
    sufficient nominal portal depth. The residency walk now multi-seeds from
    the plausible camera areas at depth 0 before applying the configured portal
    depth. Future work should split emissive/light residency from geometry
    residency more cleanly and expand persistent records, dirty ranges,
    current/previous transform history, and per-frame material-state handling
    needed for moving/game-driven surfaces and temporal reconstruction.

PathTraceSkinning.h
    Narrow CPU skinning bridge declarations for captured skinned/deformed Doom
    surfaces. Kept small so the data source can later move GPU-side.

PathTraceSkinning.cpp
    Current CPU skinning implementation used by scene capture. Handles joint
    lookup, skinned vertex position/normal transforms, and base-pose detection.

PathTraceSurfaceClassification.h
    Surface bucket classification enums, subtype constants, and classification
    helper declarations.

PathTraceSurfaceClassification.cpp
    Maps Doom draw surfaces into RT smoke buckets: static world, rigid entity,
    skinned/deformed, particle/alpha, or unknown. This is surface-level
    classification, not full material declaration analysis.

PathTraceDoomMaterialClassifier.h
    Doom material classification structs and helper declarations. This is where
    material declarations and stage properties become coarse traits for the RT
    smoke path.

PathTraceDoomMaterialClassifier.cpp
    Material classifier implementation. Interprets Doom material coverage,
    lighting, blend modes, texgen, deform/dynamic model hints, and translucent
    subtypes for capture, material table construction, and diagnostics.

PathTraceMaterialTextureDiscovery.h
    Per-frame visible-material texture discovery interface and timing/stat
    structs.

PathTraceMaterialTextureDiscovery.cpp
    Walks visible Doom materials, discovers usable diffuse/alpha/normal/specular
    /emissive textures, records fallback/rejection reasons, and updates the
    texture registry. This is separate from material classification.

PathTraceTextureRegistry.h
    Texture metadata registry interface. Stores safe texture handles, image
    names, dimensions, usage flags, and fallback reasons keyed by material id.

PathTraceTextureRegistry.cpp
    Texture registry implementation. Owns frame-persistent material texture
    metadata used by dynamic material table construction and debug dumps.

PathTraceMaterialUniverse.h
    Persistent material record and fact declarations for the RT smoke path.
    Records carry stable universe indexes, shader-facing base material data, and
    cached facts used by emissive inventory and future PT/ReSTIR preparation.
    Persistent records store stable material facts only; descriptor indexes,
    visible texture bindings, and compact active table row order stay in
    PathTraceDynamicMaterialState.

PathTraceMaterialUniverse.cpp
    Owns the persistent material universe cache. Builds/rebuilds per-material
    records from texture registry metadata, preserves stable universe indexes,
    validates cached records on request, exposes material facts without requiring
    each consumer to reclassify Doom material metadata, and reports cumulative
    plus frame-local churn counters. Lookup uses materialId plus a 64-bit
    material-name hash so the shader-facing materialId contract can remain
    unchanged while reducing persistent-record collision risk.

PathTraceDynamicMaterialState.h
    GPU-facing material table structs, material flags, texture descriptor limits,
    cache/compare stats, and material table build APIs.

PathTraceDynamicMaterialState.cpp
    Builds the active smoke material table from captured material ids and
    persistent material universe records. Produces the compact shader-facing
    material records, per-triangle material indexes, and texture descriptor
    slots consumed by dispatch/shaders. The legacy active-table builder remains
    available for validation and fallback.

PathTraceGuiSurfaces.h
    Small GUI surface admission helper declaration.

PathTraceGuiSurfaces.cpp
    Tests whether a Doom draw surface should be admitted as a GUI surface in the
    RT smoke scene. Currently CVar-gated and conservative.

PathTraceEmissiveCandidates.h
    GPU-facing emissive triangle record layout, inventory stats, and emissive
    inventory APIs. The triangle record is the current ReSTIR/PT light-candidate
    contract boundary: center/area, normal/luminance, UV bounds, centroid UV,
    emissive texture slot, material table row, persistent material universe
    index, stable-ish identity hash, and placeholder sampling weight/PDF fields.

PathTraceEmissiveCandidates.cpp
    Builds compact emissive triangle metadata from static/dynamic triangle
    buckets after material table construction. Used by mode 19 diagnostics and
    as a boundary for future path tracing light sampling. It normalizes
    per-triangle inventory PDFs from area-weighted luminance and total emissive
    area, but deliberately does not build a full sampler yet.

PathTraceDoomLights.h
    Doom renderer/game light diagnostic and analytic candidate declarations.
    Defines the GPU-facing PathTraceDoomAnalyticLightCandidate record used by
    the separate analytic light buffer.

PathTraceDoomLights.cpp
    Doom authored-light bridge. Scans renderer lightDefs, links renderer light
    indexes back to game idLight entities through GetLightDefHandle, evaluates
    current light color/state, computes current-area plus portal-depth
    selection over the light's renderer bounds/area membership, prints
    identity/probe/candidate dumps, and builds active point-light analytic
    sphere candidates. It does not add light geometry to the BVH, skips
    game-entity scans while gameLocal is not GAMESTATE_ACTIVE, and does not
    classify Doom light textures into physical light types yet. The diagnostic
    GPU cap defaults high enough to upload the current Mars City selected
    Doom-light set; the candidate dump reports any cap drops explicitly. Doom
    analytic light selection uses the same multi-seed portal policy as rigid
    residency, then resolves each light through bounds/area membership. This
    specifically avoids losing authored lights near a visible portal when the
    camera area's single seed falls into a tiny connector area. Candidate sphere
    size is controlled here by `r_pathTracingAnalyticSphereLightRadiusScale`
    with `r_pathTracingAnalyticSphereLightRadiusMin` /
    `r_pathTracingAnalyticSphereLightRadiusMax` clamps.

PathTraceLightSelection.h
    Debug light selection structs and constants used by smoke dispatch.

PathTraceLightSelection.cpp
    Chooses a small set of Doom lights for debug/direct-lighting constants in
    the smoke raygen path.

PathTraceLightUniverse.h
    Persistent emissive light universe declarations. Static emissive triangles
    are retained by stable-ish identity. Stable rigid-entity emissives can be
    promoted into semi-static persistence; other dynamic emissives remain
    frame-local. Area-selection diagnostics and optional connected-overflow
    filtering are declared here for mode 20 light-candidate validation.

PathTraceLightUniverse.cpp
    Merges persistent static records with current-frame dynamic emissive
    candidates, tracks semi-static rigid/object-like dynamic emissives for
    identity diagnostics, renormalizes inventory PDFs, and reports compact
    universe stats. Missing dynamic records are not injected by default because
    their matching occluder geometry is not persistent yet. The merged list is
    still capped by r_pathTracingEmissiveInventoryMaxTriangles. The area filter
    path can report selected, connected-overflow, disconnected, and unknown
    light candidates using the same broad portal traversal policy as geometry
    residency. It also uses the multi-seed camera-area walk so future
    ReSTIR/emissive filtering does not inherit the portal-border current-area
    bug. Apply mode is separate and remains an explicit render-affecting
    experiment.

PathTraceReservoirs.h
    ReSTIR/PT reservoir record layout and screen-sized buffer ownership
    declarations. Owns current/previous reservoir buffers plus a spatial
    scratch buffer, while keeping shading integration out of the smoke shader
    until the sampler contract is ready.

PathTraceReservoirs.cpp
    Creates/reuses reservoir buffers, clears reservoir history, and builds the
    scene-side reservoir reset signature from material-table, static-BLAS,
    emissive-triangle, and light-candidate data. Dispatch adds output and
    camera/FOV to form the full reset signature.

PathTraceRestirPT.h
    Host-side RTXDI ReSTIR PT context scaffold. Owns the pass-local
    rtxdi::ReSTIRPTContext state, cached RTXDI_PTParameters block, dimensions,
    checkerboard mode, resampling mode, and frame index used for RTXDI temporal
    random state. Initial mode is ReSTIRPT_ResamplingMode::None.

PathTraceRestirPT.cpp
    Creates/recreates the RTXDI PT context on output-size/checkerboard changes,
    calls SetFrameIndex and SetResamplingMode each dispatch, and fills
    RTXDI_PTParameters from the context's reservoir parameters, buffer indices,
    initial sampling, reconnection, temporal, hybrid-shift, boiling-filter, and
    spatial parameters. It does not dispatch shader passes.

PathTraceRestirPTReservoirs.h
    NVIDIA ReSTIR PT packed reservoir declarations. This is the new RTXDI ABI
    bridge boundary for a single RWStructuredBuffer<RTXDI_PackedPTReservoir>
    using RTXDI reservoir addressing and the stock two PT reservoir slices. It
    is separate from the mode 20 smoke reservoirs.

PathTraceRestirPTReservoirs.cpp
    Creates/reuses and clears the packed ReSTIR PT reservoir buffer. Sizing uses
    rtxdi::CalculateReservoirBufferParameters plus
    rtxdi::c_NumReSTIRPTReservoirBuffers instead of width*height linear
    addressing. The buffer is allocated with the RT smoke output size and bound
    into the smoke ray tracing shader ABI at u29, but no ReSTIR pass writes or
    reads it yet.

pathtrace_smoke.rt.hlsl
    Existing smoke ray tracing shader. It now includes the RTXDI ReSTIR PT
    parameter and reservoir headers for ABI validation only. The ABI bridge binds
    `ConstantBuffer<RTXDI_PTParameters>` at b28 and
    `RWStructuredBuffer<RTXDI_PackedPTReservoir>` at u29, defines
    `RTXDI_PT_RESERVOIR_BUFFER`, and compiles NVIDIA's PT reservoir load/store
    helpers. The shader does not call those helpers yet, and the existing mode
    18/20 smoke behavior should remain unchanged.

RtxdiBridge/*.hlsli
    Local RBDoom3 shader bridge declarations for future RTXDI RAB callbacks.
    These files live next to the path tracing shader instead of copying NVIDIA
    sample bridge files into a distant third-party area. The current Step 4
    content is type-only: RAB_Material, RAB_Surface, RAB_RayPayload,
    RAB_LightInfo, RAB_LightSample, and RAB_RandomSamplerState plus small empty
    constructors/accessors.

    Step 5 surface callback scaffolding currently lives in pathtrace_smoke.rt.hlsl
    because it depends on the smoke shader's local payload and material helpers.
    `RAB_BuildSurfaceFromSmokePayload` and `RAB_BuildMaterialFromSmokePayload`
    translate a traced smoke hit into bridge records without changing the
    existing mode 18/20 lighting paths. `RAB_GetGBufferSurface` and
    `RAB_GetGBufferMaterial` now read output-sized current/previous
    primary-surface history buffers. Raygen writes the current primary hit, and
    the backend copies current to previous after dispatch for next-frame
    validation. The history record includes material identity plus the RAB
    material roughness/F0. When `r_pathTracingToyFakePBRSpecular 1` is enabled,
    modes 26/27/28/29/30/31 sample Doom specular textures and store the same converted
    fake-PBR roughness/F0 that mode 18 uses.

    Step 6 RNG scaffolding lives in `RAB_RandomSamplerState.hlsli`. It keeps a
    local `RAB_RandomSamplerState` wrapper but initializes and advances it
    through RTXDI's own random sampler helpers. This preserves the seed/index
    contract that ReSTIR PT reservoirs need for path replay, while keeping the
    application bridge name local and upgradeable.

    Step 7 BRDF scaffolding lives in `RAB_Surface.hlsli`. It implements the
    first conservative opaque-diffuse callbacks:
    `RAB_GetSurfaceBrdfSample`, `RAB_GetSurfaceBrdfPdf`,
    `RAB_EvaluateSurfaceBrdf`, `RAB_EvaluateSurfaceBrdfOverPdf`, plus
    `RAB_SurfaceImportanceSampleBrdf` and `RAB_SurfaceEvaluateBrdfPdf` aliases.
    The current model is cosine-hemisphere Lambert sampling over the RAB shading
    normal with geometry-normal hemisphere rejection. Unsupported translucent or
    invalid surfaces return no sample/zero contribution so they can keep using
    the regular path until real BSDF support exists. The diffuse-only callback
    is intentionally quieter than the material contract: RAB material/history
    records still carry roughness, specular F0, opacity, alpha cutoff, material
    flags, and glass/window fallback flags. Do not delete or collapse those
    fields just because the current validation path does not sample glossy,
    mirror, refraction, or transmission lobes yet.

    Step 8 light scaffolding is split across `RAB_LightInfo.hlsli`,
    `RAB_LightSampling.hlsli`, and `RAB_MISCallbacks.hlsli`. The current bridge
    can load and sample current-frame smoke emissive triangles and Doom analytic
    sphere lights from the existing shader buffers, compute a simple reflected
    target weight for supported opaque-diffuse RAB surfaces, return direction
    and distance, and provide a conservative visibility callback that only
    rejects invalid surfaces/samples. Previous-frame light loading returns
    empty data. Environment sampling is still zero/unsupported. MIS callbacks
    are present but intentionally neutral until `RAB_PathTrace` and consistent
    NEE/BRDF path accounting exist.

    Step 9 path-trace scaffolding lives under `RtxdiBridge/PathTracer/`.
    `RAB_PathTracerUserData.hlsli` provides the minimal user-data object and
    path-type setter required by RTXDI PT entry points. `RAB_PathTracer.hlsli`
    compiles RTXDI's initial-sampling path tracer context against the local RAB
    callbacks and defines `RAB_PathTrace`. The skeleton samples the current
    conservative RAB BRDF, fills `RTXDI_BrdfRaySample`, sets a continuation ray,
    lets RTXDI analyze reconnectibility, traces `SmokeScene`, converts the smoke
    hit payload back into `RAB_RayPayload`/`RAB_Surface`, records the
    intersection, and records emissive surface hits. It is capped to a small
    bridge bounce depth and does not yet implement NEE, environment radiance,
    Russian roulette, PSR/denoiser state, traced RAB visibility, temporal replay,
    or spatial reuse.

PathTraceDebugDumps.h
    Shared diagnostic structs and dump/trigger APIs for scene build, material,
    texture, timing, and one-shot diagnostic CVars.

PathTraceDebugDumps.cpp
    Console diagnostics implementation and diagnostic trigger/reset orchestration.
    Should format and gate terminal output, not own render resources.

PathTraceSurfaceDebugDumps.h
    Surface-targeted diagnostic dump declarations.

PathTraceSurfaceDebugDumps.cpp
    Crosshair material and GUI surface diagnostic dump implementation. Uses the
    already-built material table and current view.

PathTraceCVars.h
    Central extern declarations for experimental RT/PT CVars used by the
    extracted modules.

PathTraceCVars.cpp
    CVar definitions for the smoke/path tracing prototype, including the mode
    18 toy PT controls such as `r_pathTracingToyFakePBRSpecular`.

Feature-flow guide
------------------

Material processing / classification
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Primary files involved:

- PathTraceDoomMaterialClassifier.*
- PathTraceSurfaceClassification.*
- PathTraceMaterialTextureDiscovery.*
- PathTraceTextureRegistry.*
- PathTraceMaterialUniverse.*
- PathTraceDynamicMaterialState.*
- PathTraceSceneCapture.*
- PathTraceDebugDumps.*
- PathTraceSurfaceDebugDumps.*
- PathTraceCVars.*

Typical flow:

1. PathTraceSceneCapture walks Doom draw surfaces.
2. PathTraceSurfaceClassification decides the surface bucket.
3. PathTraceDoomMaterialClassifier extracts coarse material traits.
4. PathTraceMaterialTextureDiscovery registers visible material texture metadata.
5. PathTraceTextureRegistry stores safe texture handles and fallback reasons.
6. PathTraceMaterialUniverse builds or reuses persistent material records and
   exposes stable material facts.
7. PathTraceDynamicMaterialState builds the compact GPU-facing active material
   table and triangle material indexes from the universe records.
8. PathTraceDebugDumps and PathTraceSurfaceDebugDumps report what happened.

Mode 18 fake-PBR specular test:

1. PathTraceCVars defines `r_pathTracingToyFakePBRSpecular`, default 0.
2. PathTraceSmokeDispatch folds the CVar into the accumulation reset signature,
   enables specular-map sampling for mode 18 when the toggle is active, and
   passes a texture flag bit to the raygen constants.
3. pathtrace_smoke.rt.hlsl tests that flag in mode 18, samples the existing
   `specularTextureIndex`, derives approximate F0/roughness from the Doom
   legacy spec map, and uses that for selected/analytic direct specular.

This is deliberately a visual smoke test. It does not add CPU material fields,
does not detect packed `_rmao`/`rmaomap` usage, and does not replace the material
classification/facts work needed for alpha, blend, decal, GUI, portal, glass,
or special-stage behavior.

Safer places to touch:

- Add a new material trait name or diagnostic string in
  PathTraceDoomMaterialClassifier or PathTraceDebugDumps.
- Add a fallback reason or registry metadata field if it does not change shader
  layout.
- Add cached material facts in PathTraceMaterialUniverse when they avoid
  repeated per-frame classification and can be validated against existing
  metadata.
- Add a diagnostic dump for a suspected material class.
- Extend material texture discovery to record more metadata while preserving
  current fallback behavior.

Riskier places to touch:

- PathTraceSmokeMaterial layout in PathTraceEmissiveCandidates.h. This is
  shader-facing data.
- Material flag bit values in PathTraceDynamicMaterialState.h. These are read by
  shaders and debug code.
- Per-triangle material index generation in PathTraceDynamicMaterialState.cpp.
  Bad indexes can break shader reads.
- Active material table row order. It must stay compatible with cached static
  triangle material-index buffers unless the static cache/resource contract is
  deliberately changed.
- Capture filtering in PathTraceSceneCapture.cpp. It can make geometry appear or
  disappear from the RT scene.

Rule of thumb:

If the change only improves classification metadata or diagnostics, it is usually
safe. If it changes what surfaces/materials are admitted, which texture slot is
bound, the active material row order, or the GPU material layout, test mode 18,
mode 19, crosshair dump, texture fallback dump, universe table validation, and
scene rebuild behavior.

BVH / BLAS / TLAS
~~~~~~~~~~~~~~~~~

Primary files involved:

- PathTraceSmokeSceneBuild.cpp
- PathTraceDrawSurfCapture.*
- PathTraceInstanceUniverse.*
- PathTraceAcceleration.*
- PathTraceSmokeResources.*
- PathTraceGeometry.*
- PathTraceGeometryUniverse.*
- PathTraceSceneUniverse.*
- PathTraceSceneCapture.*
- PathTraceSmokeDispatch.cpp

Typical flow:

1. PathTraceSmokeSceneBuild borrows static cache storage from
   PathTraceGeometryUniverse.
2. PathTraceSceneCapture currently produces static/dynamic triangle buckets and
   asks the universe to look up or complete static surface appends.
3. PathTraceSceneUniverse can provide static map diagnostics and, when source3
   static area preload is enabled, selected-area static-world preload records.
   It does not own the live dynamic/rigid/skinned scene. Its selected-area walk
   uses Doom renderer portal areas, and currently counts but does not block on
   `PS_BLOCK_VIEW`.
4. For source3, PathTraceDrawSurfCapture mirrors the final visible drawSurf
   stream after old capture has done the stable static-cache work. It records
   mesh identity without transform and per-frame instance identity with
   transform/material/entity metadata. Treat this as freshness input, not the
   complete RT residency rule.
5. PathTraceInstanceUniverse tracks those observations for diagnostics and
   moved-rigid history. The geometry universe consumes the current/cached rigid
   observations when rigid route/residency gates are enabled.
6. PathTraceGeometryUniverse records eligible rigid local-source mesh candidates,
   CPU rigid BLAS plans/input descriptors, default-on GPU scaffold records for
   local-space rigid mesh buffers/standalone BLAS builds, rigid residency
   records, and rigid TLAS instance plans. In gated validation/integration
   modes, routed-ready rigid triangles can be removed from the dynamic fallback
   and traced through the reusable rigid route. Rigid residency now keeps only
   current-frame revalidated resident instances. Its selected-area portal walk
   supplements previously visible emissive/signage/glow rigids, but does not
   discover or preserve arbitrary opaque rigid props from raw area refs because
   destructible gameplay objects can remain render-side while no longer being
   valid PT geometry.
7. PathTraceSmokeSceneBuild calculates geometry ranges and static BLAS cache
   eligibility.
8. PathTraceSmokeResources creates geometry/material/emissive buffers.
9. PathTraceAcceleration creates BLAS objects, uploads build-input buffers, and
   submits BLAS/TLAS builds. Its TLAS submit helper appends routed rigid
   instances when the explicit rigid route gates and compatible debug or
   integration mode are enabled.
10. PathTraceSmokeResources commits the successful handles to pass state.
11. PathTraceSmokeDispatch transitions committed buffers and dispatches rays.
12. RenderBackend.cpp blits the RT output as the 3D scene. With
   r_pathTracingSkipRaster3D enabled, the backend skips the remaining normal 3D
   raster passes for that view; subsequent GUI/HUD/menu views still rasterize.

Safer places to touch:

- Add timing/stat fields around acceleration helpers.
- Improve debug names for buffers/accel structures.
- Add validation/logging around build inputs.
- Change helper packaging structs when the same fields are passed through.
- Move static-cache bookkeeping into PathTraceGeometryUniverse when the capture
  behavior remains equivalent and can be tested in isolation.
- Add drawSurf mirror diagnostics, candidate counts, and validation dumps.
- Add PT-native debug views that consume diagnostic buffers, such as modes 21
  and 22, while keeping mode 18/20 resource contracts stable.

Riskier places to touch:

- Static BLAS cache eligibility in PathTraceSmokeSceneBuild.cpp.
- Static surface identity or append semantics in PathTraceSceneCapture.cpp or
  PathTraceGeometryUniverse.cpp.
- Treating PathTraceSceneUniverse as the live universal scene source. It lacks
  the final raster-submitted dynamic, skinned, material-override, and
  game-logic state needed for the primary scene producer.
- Treating current raster visibility as the complete BVH residency rule. That
  repeats the earlier failure mode where occluded/off-camera emissives and
  shadow-casting rigid geometry disappear together.
- Treating current+1 portal depth as a proven global rule. It was too shallow
  in Mars City; depth 4 is now the default source3 residency/preload standard.
- Mesh key semantics in PathTraceDrawSurfCapture.cpp. Mesh identity must not
  include model matrices or current transforms.
- Upload skip flags in the RtSmokeBufferUploadItem array.
- Resource state transitions in PathTraceSmokeDispatch.cpp.
- PathTraceSmokeVertex layout or index/vertex counts.
- Any TLAS instance setup in PathTraceAcceleration.cpp.
- Re-enabling the full 3D raster tail under mode 18. This should be treated as
  a diagnostic comparison path, not the default PT path, because it doubles work
  and can reintroduce raster-only hitches.

Rule of thumb:

Treat BLAS/TLAS state as behavior-sensitive. A tiny cache or upload mistake can
look like a rendering bug, stale geometry, or a performance regression. Prefer
diagnostic additions before ownership changes. The geometry universe is now the
intended home for persistent static-world records, but it is still a scaffold:
move one responsibility at a time and keep mode 18 visual behavior stable.

For source3, keep the representation split explicit:

- mesh = geometry plus material metadata, no transform
- instance = mesh plus current transform, entity/material override, and
  visibility this frame
- reusable rigid BLAS = future local-space mesh cache, not the current render
  path

Only route rendering through reusable rigid records or remove rigid triangles
from the dynamic fallback behind the existing explicit route gates or test
presets. The current source3 validation path expects mesh identity to remain
transform-free, instance identity to carry transform/material/entity metadata,
and residency to be driven by Doom area/portal selection rather than pure
raster visibility.

For mode 18 performance validation, keep r_pathTracingSkipRaster3D at 1 unless
the test explicitly compares against old raster-under-RT behavior. If something
is missing with raster skipped, prefer restoring the narrow dependency rather
than falling back to the full 3D raster frame.

Motion-vector / Ray Reconstruction requirement:

DLSS Ray Reconstruction needs high-quality motion-vector inputs. A path-traced
renderer cannot get there with only current-frame packed geometry or camera-only
motion: moving rigid instances, animated/skinned meshes, reflected/refracted
hits, and disocclusions all need stable current-vs-previous geometry data. The
Q2RTX Ray Reconstruction reference fork in
E:\prog\references\Q2RTX-rayreconstruction-master stores previous triangle
positions and previous instance transforms, then projects current and previous
hit positions in shader to write motion.

Implications for PathTraceGeometryUniverse:

- Persistent surface/triangle records need stable identities across frames.
- Records need current and previous range mapping, or explicit invalidation when
  a match is not trustworthy.
- Static-world records already carry current/previous ranges and lifecycle
  validity as CPU-side scaffolding, but no previous-position GPU buffers or
  motion-vector output exist yet.
- Static world can often reuse identical previous positions, but rigid entities
  need previous transforms and skinned/deformed geometry eventually needs
  previous skinned positions or a GPU skinning history source.
- Future GPU buffers should plan for current/previous position data or compact
  position deltas, not only the current PathTraceSmokeVertex stream.
- Motion-vector generation should be treated as part of the geometry-universe
  contract before DLSS/RR integration begins. Retrofitting it after the universe
  only stores current-frame data would risk another major rebuild.

~~~~~~~~~~~~~~~~~~~~~~~

Current state:

Interactive GUI surfaces can be admitted into the RT smoke scene through the
GUI-surface path, but this is still conservative and diagnostic-oriented. Doom
GUIs are not "just a texture"; they are generated by Doom's GUI system and may
change due to game/menu interaction.

Important postmortem:

- Do not modify GuiModel.cpp to synthesize frontEndGeo for generated GUI draw
  surfaces in normal play. That experiment made the UI painter-order quads look
  like physical world geometry. It broke in-world monitors/video panels in the
  shared raster path, produced black or overlapping screen layers, and caused
  severe cinematic/GUI hitches. GuiModel.cpp is intentionally restored to the
  original behavior where generated GUI drawSurfs keep frontEndGeo = NULL.
- The viable long-term route is a narrow GUI bridge: keep Doom's raster GUI/SWF
  system responsible for panel contents, expose or render a composited panel
  texture, and project that texture onto the actual physical panel surface in
  the PT material path. The generated GUI draw commands themselves should stay
  out of the TLAS.

How to pipe in a better GUI idea:

1. Start in PathTraceGuiSurfaces.cpp.
   - Keep the CVar gate intact.
   - Make the admission test explicit and easy to dump.

2. Ensure PathTraceSceneCapture.cpp gets the surface into the intended bucket.
   - Do not silently merge GUI behavior into unrelated translucent/particle
     handling.
   - Record reason samples so r_pathTracingGuiDump and related diagnostics can
     explain the decision.

3. Make texture discovery explicit.
   - If the GUI can provide a stable image/texture handle, record it through
     PathTraceMaterialTextureDiscovery and PathTraceTextureRegistry.
   - If it cannot, preserve a fallback reason rather than binding a guessed or
     unsafe texture.

4. Keep material table changes isolated.
   - PathTraceDynamicMaterialState should receive registry metadata and produce
     material records. It should not know Doom GUI internals.

5. Add diagnostics before broadening behavior.
   - Use PathTraceSurfaceDebugDumps for view/crosshair/GUI details.
   - Use PathTraceDebugDumps for one-shot CVar trigger/reset behavior.

6. Test with:
   - r_pathTracingAllowGuiSurfaces
   - r_pathTracingGuiDump
   - r_pathTracingCrosshairMaterialDump
   - mode 18 at normal and high resolution
   - a live interactive GUI surface, not just a static menu texture

What to avoid:

- Do not reintroduce the abandoned generated-GUI frontEndGeo/layer-offset
  experiment as a normal play path. It made painter-order panel layers stick out
  or clip, and was the source of the broken monitor/screen behavior and
  repeatable GUI/cinematic hitch route.
- Do not make scene capture reach directly into unrelated Doom GUI systems if a
  narrow bridge can expose the needed texture/state.
- Do not treat all GUI-like translucent surfaces as safe GUI surfaces.
- Do not overwrite fallback/rejection diagnostics just to force a texture to
  appear. It makes later debugging much harder.

Dynamic game logic driven materials

Current state:

This area is intentionally incomplete. Many Doom materials driven by game logic,
dynamic expressions, animated stages, special texgens, overlays, post-process
behavior, or callbacks are hidden, rejected, forced to fallback, or represented
only through coarse diagnostics. That is preferable to rendering them as if the

Safer places to touch:

- Add explicit classifier fields for a dynamic behavior.
- Add fallback/rejection reasons that explain why a material is hidden.
- Add diagnostics that report counts, examples, or material names.

- Broadly enabling callback/dynamic materials in scene capture.
- Treating animated/generated textures as stable bindless textures.
- Collapsing several dynamic cases into one "good enough" material flag.
- Changing fallback behavior without adding diagnostics.

Rule of thumb:

Make unsupported dynamic material behavior more visible before making it more
rendered. First improve detection, names, and dumps; then add one narrow dynamic
bridge at a time.

Higher-level path tracing / emissives
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Primary files involved:

- PathTraceEmissiveCandidates.*
- PathTraceDoomLights.*
- PathTraceMaterialUniverse.*
- PathTraceDynamicMaterialState.*
- PathTraceSmokeDispatch.cpp
- PathTraceLightSelection.*
- PathTraceSmokeSceneBuild.cpp
- PathTraceDebugDumps.*
- PathTraceCVars.*

Current state:

Mode 18 uses a toy path tracing/debug lighting path. Mode 19 builds and dumps an
emissive inventory that can become a future sampling input. This is not yet the
NVIDIA RTXPT/ReSTIR implementation.

Doom renderer lights now have a separate authored-light path. PathTraceDoomLights
builds stable identity/probe diagnostics and an opt-in analytic sphere-light
candidate buffer from active Doom point lights in the selected current-area plus
portal-neighbor set. The shader samples that separate buffer for visual
inspection when r_pathTracingAnalyticLightCandidates is enabled. The bridge is
verified at a basic level: lights appear, preserve RGB color, follow game
animation/on-off state, and survive level-load transitions. The current light
model is intentionally crude and treats every eligible point light as a sphere
emitter, but it now samples the sphere's solid angle instead of tracing a hard
center-point shadow ray. Future work should classify light materials such as
square, round, triangle, projected, and helper textures into better physical
light types before feeding ReSTIR.

Projected Doom lights have enough authored geometry for a future analytic
record: origin/axis plus local target/right/up/start/end define a rectangular
projection footprint. At the target plane, width is approximately
`2 * right.Length()`, height is `2 * up.Length()`, and projected area is
`4 * cross(right, up).Length()`, scaling quadratically with distance along the
target direction. Treat this as a projected/spotlight footprint first, not
automatically as the physical emitter size.

Potential helper-light cleanup should be an opt-in light-to-emissive
association layer. A nearby emissive surface can inherit an authored light's
evaluated RGB, on/off animation state, and calibrated intensity when area,
distance, facing, and material checks produce a strong match. Keep unmatched
helper/fill lights as analytic candidates and dump associations before
suppressing linked helper lights.

Emissive summaries and uploaded light-candidate records now carry stable
material universe indexes in addition to active material-table indexes. This is
intended to let future ReSTIR/PT code refer to persistent material identity even
while the shader-facing active table remains compact and frame-local.

Per-triangle emissive records also carry explicit sampling contract fields:
area-weighted luminance, normalized inventory PDF, area PDF, material universe
index, material id, emissive texture slot/dimensions, UV bounds, and a 64-bit
identity hash derived from material/surface-side triangle identity. These are
prep fields for ReSTIR/PT and are not a final light sampler.

Reservoir resource ownership exists and mode 20 consumes the first slice of it.
Current, previous, and spatial scratch buffers are screen-sized and reset when
the output, camera/FOV, material-table signature, emissive triangle records,
light-candidate records, or static-BLAS signature change. Use
r_pathTracingReservoirDump 1 to inspect the current signature/reset state. Mode
20 writes current reservoirs for one-frame direct emissive lighting; previous
and spatial scratch reservoirs are not consumed yet.

RTXDI-compatible ReSTIR PT host plumbing now exists separately from the smoke
reservoirs. PathTraceRestirPTReservoirs allocates a packed
RTXDI_PackedPTReservoir buffer sized by RTXDI's tiled reservoir parameters and
two PT reservoir slices. PathTraceRestirPT owns an RTXDI ReSTIRPTContext and
uploads an RTXDI_PTParameters constant buffer each smoke dispatch, with
resampling mode still forced to None. The smoke ray tracing binding set and
shader now expose the Step 3 RTXDI PT reservoir ABI at b28/u29 and compile
`Rtxdi/PT/Reservoir.hlsli` through `RTXDI_PT_RESERVOIR_BUFFER`, but no RTXDI
shader pass, temporal reuse, or spatial reuse consumes it yet. Step 4 local RAB
type declarations exist under `RtxdiBridge/`, Step 5 smoke-payload-to-RAB
surface helpers plus intentionally invalid screen G-buffer callbacks are present
in `pathtrace_smoke.rt.hlsl`, and Step 6 RAB RNG callbacks wrap RTXDI's own
sampler helpers. Step 7 RAB BRDF callbacks provide only conservative
opaque-diffuse sampling/evaluation; glossy, transmission, decal, GUI, particle,
and volumetric BSDFs remain future work. Step 8 RAB light callbacks expose
current smoke emissive triangles and Doom analytic sphere lights to the bridge,
but do not add traced visibility, environment sampling, previous-frame light
history, or active RTXDI passes. Step 9 compiles the RTXDI initial-sampling
path-tracer context against a local `RAB_PathTrace` skeleton. Step 10 adds the
first gated dispatch path in `r_pathTracingDebugMode 26`: raygen builds the
current primary RAB surface, calls RTXDI `GenerateInitialSamples`, and stores
the result through `RTXDI_StorePTReservoir` into
`bufferIndices.initialPathTracerOutputBufferIndex`. Misses, GUI surfaces, and
unsupported translucent/special surfaces store an empty reservoir. Mode 26 is a
debug reservoir-writer/heat view only. The local path tracer now records one
conservative NEE sample from the current RAB light list per visited path vertex
before BRDF continuation so initial reservoirs are visible near valid light
candidates; the first BRDF-only probe and the first single uniform light pick
were too sparse to validate visually. The current NEE picker uses the existing
weighted emissive-triangle distribution and keeps the best local target from a
small trial set. This is still a debug visibility aid, not final light sampling.
It does not shade from the reservoir, does not enable temporal/spatial reuse,
does not use previous-frame G-buffer history, does not trace final RAB
visibility, and does not change modes 18/20.

Mode 26 uses the same source3 scene-membership baseline as the playable modes
for validation: rigid TLAS routing, rigid residency, and routed rigid
emissive-card inventory are enabled for it. Without this, mode 26 can test
ReSTIR PT against a culled/fallback scene that does not match modes 18/20.

Mode 27 is the paired initial-reservoir shading preview. It uses the same
initial reservoir write path and source3 scene-membership baseline as mode 26,
but visualizes `TargetFunction * WeightSum` instead of the temporary diagnostic
fallback color key. It is intentionally initial-only because temporal
resampling still needs previous primary-surface data and motion vectors.
User validation reported that mode 27 works fairly well, runs well
performance-wise, and has no serious stability or gameplay issues. Remaining
black objects/rough normals should be handled as local RAB material and normal
contract cleanup. `r_pathTracingNormalMapFlipGreen` is available as a targeted
diagnostic for Doom 3's legacy green/Y normal-map convention, but the larger
normal fidelity fix is still to pass the original idTech4 vertex tangent/sign
basis into the PT geometry path instead of reconstructing TBN per triangle in
the hit shader. The RAB surface builder now constrains decoded shading normals
to the oriented geometric-normal hemisphere before BRDF checks, which avoids
fully black reservoir-lit surfaces when a normal map tilts past the valid side.

Mode 28 is the paired initial-reservoir visibility preview. It keeps the same
mode 27 reservoir generation and preview shading, but multiplies the preview by
one conservative visibility ray toward the selected reservoir target. NEE
reservoirs decode the packed RTXDI sampled-light reference and resample the
current RAB light; non-NEE reservoirs use the translated reservoir world
position. This is still initial-only validation, not temporal/spatial ReSTIR PT
or final path replay.
User validation reported no issues. Hard-edged shadows are expected because the
mode traces a single binary visibility ray to the selected reservoir sample;
soft shadows require later multi-sample/temporal/spatial accumulation and final
shading.

Mode 29 validates the primary-surface history buffer contract needed before
temporal/spatial ReSTIR PT. It compares current and previous same-pixel history
records and highlights missing current/previous records, material mismatches,
normal mismatches, and roughness mismatches. It intentionally does not reproject
or use motion vectors yet. Primary-surface history clearing is deliberately
separate from `m_smokeReservoirNeedsClear`: the reservoir reset signature is
camera-sensitive, so using it to clear primary history made every moving-camera
frame look like "current valid, no previous" full-screen blue even though the
previous frame copy was otherwise valid.
After splitting those lifetimes, user validation reports the moving-camera
full-screen blue case is gone. Purple normal mismatches and orange roughness
mismatches while moving are expected same-pixel-history artifacts until motion
vectors and reprojection are wired.

Mode 30 validates a temporary screen-space reprojection path before RTXDI
temporal resampling is enabled. PathTraceSmokeDispatch stores the current camera
basis/tangents after copying current primary history to previous; the next
frame's shader projects each current primary world position into that previous
camera and samples the previous history pixel. It uses the same color key as
mode 29. This is useful for camera-motion validation on static world surfaces,
but it is not a replacement for vertex/object motion vectors.
User validation reports mode 30 is mostly green, with purple/orange outlines on
moving-object and silhouette edges. Treat those outlines as the expected limit
of camera-only screen-space reprojection, not a failure of the primary history
copy or previous-camera state.

Mode 31 is the first temporal-reservoir validation view. PathTraceSmokeDispatch
sets the ReSTIR PT context to temporal mode only for debug mode 31, which gives
RTXDI the alternating initial/temporal input/output reservoir indices. The
shader writes the current initial reservoir, projects the current primary
surface to the previous pixel, rejects missing or mismatched previous
surface/reservoir data, then calls RTXDI_PTTemporalResampling and stores the
temporal output reservoir. This remains a diagnostic color view; it is not final
shading and it does not enable spatial reuse.
User validation reports mode 31 is broadly similar to mode 30; violent camera
motion can leave blue edge bands until history catches up. Treat those bands as
offscreen/disocclusion misses from the temporary reprojection path rather than
reservoir corruption.

Mode 20 has a known device-remove stress risk under heavier/high-FPS testing.
When it fails, the scene/surface buffers can visibly degrade before the driver
removes the device. This resembles the older mode 18 `r_pathTracingLightCount
32` at 120 FPS failure, where brute-force primary plus one-bounce selected-light
shadowing could produce up to 64 shadow rays per pixel, saturate queued GPU
work while the camera was moving, and trigger transient surface corruption
followed by device removal. Keep reservoir experiments conservative until
buffer lifetime, barriers, UAV writes, and per-pixel ray workload have dedicated
validation.

Mode 20 now samples a merged light list: persistent static emissives plus
current-frame dynamic emissives. Semi-static promoted dynamic records are
tracked for identity/history diagnostics, but missing dynamic records are not
injected by default because only retaining the light without the dynamic
occluder geometry produces ghost lighting. Use r_pathTracingLightUniverseDump 1
to inspect persistent/merged counts.

Q2RTX's Ray Reconstruction reference fork handles off-camera emissives by
promoting static emissive BSP surfaces into light polygon records, assigning
them to BSP clusters, and sampling per-cluster light lists rather than the
current screen-visible emissive set. Dynamic/model emissive lights are injected
into those cluster lists each frame and temporal ReSTIR history uses explicit
previous-to-current light identity remapping. If a previous dynamic light cannot
be matched in the current frame, the history is invalidated. For this branch,
that points toward a future light-universe/area-membership contract for static
and stable entity emissives, not stale dynamic candidate retention.

Modes 18, 19, and 20 all honor per-surface evaluated emissive state.
Switchable or dimming lights can have a material that is capable of emissive
output while the current draw surface has its emissive stage disabled or dimmed
by shader registers. Treat material-universe emissive facts as stable
capability only; runtime active/dimmed emissive state belongs to
capture/emissive-candidate records.

Safer places to touch:

- Add emissive inventory stats or dumps.
- Add Doom light identity/candidate stats or dumps in PathTraceDoomLights.
- Improve candidate weighting diagnostics.
- Add dispatch constants for debug-only experiments if shader layout is updated
  carefully.
- Add feature gates/CVars in PathTraceCVars.
- Add contract fields to PathTraceSmokeEmissiveTriangle when they are stable
  light-candidate facts needed by future reservoir sampling, such as identity,
  universe material index, weight, or PDF.

Riskier places to touch:

- PathTraceSmokeEmissiveTriangle layout. It is GPU-facing.
- PathTraceDoomAnalyticLightCandidate layout. It is GPU-facing.
- Dispatch constants layout. It must match shader expectations.
- Accumulation signature inputs in PathTraceSmokeDispatch.cpp.
- Blending future ReSTIR/PT ownership into smoke helper modules without a clear
  boundary.

Suggested future boundary:

Keep emissive inventory generation and Doom analytic light generation as
smoke-path input producers. A future PT or ReSTIR module should consume clear
candidate buffer/metadata contracts rather than reaching back into scene
capture, material discovery, or game entities directly.

Texture binding and fallback behavior
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Primary files involved:

- PathTraceMaterialTextureDiscovery.*
- PathTraceTextureRegistry.*
- PathTraceMaterialUniverse.*
- PathTraceDynamicMaterialState.*
- PathTraceSmokeResources.*
- PathTraceSmokeDispatch.cpp
- PathTraceDebugDumps.*

Current state:

Textures are discovered from visible materials, validated into safe handles, then
stored in the texture registry and reflected into persistent material-universe
records. PathTraceDynamicMaterialState then packs the current active bindless
texture table and patches descriptor indexes into the compact shader-facing
material records. Unsupported or unsafe cases keep fallback reasons for
diagnostics.

Safer places to touch:

- Better fallback names/reasons.
- Additional texture metadata that does not change shader layout.
- More diagnostic dumps around active texture windows.

Riskier places to touch:

- Descriptor table capacity and active texture cap.
- Force-fallback behavior.
- Texture index assignment in material table records.
- Binding set/layout creation in PathTraceSmokeResources.cpp.

Scene build ownership
~~~~~~~~~~~~~~~~~~~~~

Primary files involved:

- PathTracePrimaryPass.*
- PathTraceSmokeSceneBuild.cpp
- PathTraceSmokeResources.*
- PathTraceSmokeDispatch.cpp
- PathTraceSmokeReadback.cpp

Current state:

PathTracePrimaryPass still owns persistent m_smoke* state. Several modules define
private PathTracePrimaryPass methods so they can work with that state without a
large public state object. Static-world geometry cache storage is the first
exception: it now lives behind RtSmokeGeometryUniverse so persistent geometry can
grow without making PathTracePrimaryPass larger.

Safe next architectural step:

For geometry, continue growing PathTraceGeometryUniverse deliberately:

1. Expand the current static surface records into explicit persistent
   surface/range records.
2. Add one-frame history requirements to those records before finalizing the
   range/buffer model: previous transform, previous range, previous vertex data
   or delta data, and invalidation flags.
3. Move static GPU buffer and dirty-range ownership only after the CPU cache
   record model is stable.

For broader smoke pass state, design a PathTraceSmokeState or PathTraceSmokeScene
object first, then move ownership in one dedicated architecture change. Do not
gradually leak pieces of m_smoke* state into unrelated helper modules without a
plan.

What to avoid:

- Moving files into folders before deciding how CMake/project layout should be
  updated.
- Mixing future ReSTIR/PT ownership into the smoke scene builder.
- Reorganizing base RBDoom3 renderer systems as part of RT/PT cleanup.
