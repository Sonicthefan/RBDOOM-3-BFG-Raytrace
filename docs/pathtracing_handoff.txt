Recently tested observations:

- Normal/spec texture previews work.
- Mode 14 normal mapping initially broke lighting because Doom normal maps are
  DXT5-style packed; matching raster G-buffer decode fixed it.
- Mode 14 specular map highlights look good as a diagnostic path.
- Smoke particle dithering is passable as a placeholder, but still visibly
  noisy/card-like. It should be treated as a cheap hack, not final translucency.
- `r_pathTracingSmokeParticleAlphaScale` had little visible impact in one test
  before the texture-mask/edge-fade adjustment; retest if this becomes relevant.
- GUI/SWF surfaces/textures appear safe behind the current opt-in diagnostics,
  but default behavior remains conservative.
- Emissive/glow material capture is broadly useful in mode 14. Important
  exclusions were added for GUI, particle, glass/window, animated/view-dependent
  stages, and skinned/deformed hits to avoid false positives.
- The early emissive pass exposed a texture table limit symptom. The active
  descriptor table has since been raised to 2048 in the notes/workflow, and
  long traversal texture coverage became acceptable.
- Portal/window glass and small object glass have a stochastic transparency
  placeholder. It is "good enough" for debug mode, not physically correct.
- Reflective eyewear/glasses/goggles needed special cutout handling and dark
  fallback albedo. Glasses are currently fixed enough for placeholder mode.
- GUI/SWF panels are visible only via a fragile opt-in direct-quad diagnostic.
  Doom generates many painter-order SWF/GUI polygons around the same physical
  panel plane. Tracing those quads directly causes layer-order artifacts and can
  hide nearby world geometry. The correct long-term approach is to keep Doom's
  GUI raster path, render each panel offscreen, then project the composited
  texture onto the physical panel surface. Do not treat the current GUI TLAS
  path as a viable renderer.
- A repeatable hang/exit when collecting a desk disk item was fixed by skipping
  non-GUI callback render entities that also use a custom shader. A broad
  callback skip hid all skinned/dynamic characters; the current narrow
  custom-shader guard resolved both the crash and the invisible-character
  regression.
- Doom BFG generated image names use #__%02d%02d, where the first two digits are
  textureUsage_t and the second two are cubeFiles_t. Useful 2D codes:
      #__0000 = TD_SPECULAR / CFM_DEFAULT
      #__0100 = TD_DIFFUSE / CFM_YCOCG_DXT5
      #__0200 = TD_DEFAULT / CFM_DEFAULT
      #__0300 = TD_BUMP / CFM_NORMAL_DXT5
      #__1000 = TD_COVERAGE / CFM_GREEN_ALPHA
- Never treat #__0100 / CFM_YCOCG_DXT5 alpha as material alpha in the general
  case. The sampled alpha is Y/luminance for CoCg_Y color reconstruction.
- #__0200 / CFM_DEFAULT is ordinary RGBA; if it has alpha and the material
  stage/blend uses it, that alpha is real.
- #__1000 / CFM_GREEN_ALPHA is the dedicated coverage/cutout path.
- Classic Doom 3 decal intent survives strongly in paths and material stages.
  /decals/ + #__0100/CFM_YCOCG_DXT5 usually means a blend decal using decoded
  RGB as the effect, not an alpha decal. Lighten/additive decals use black as
  the no-effect key; darken/filter decals use white as the no-effect key.
- Emissive stages on characters are often game-logic overlays such as death
  dissolve/fire effects. The RT smoke material path hides those from emissive
  lighting/debug by skipping skinned/deformed hits and sampling emissive
  luminance for the mode 19 overlay.
- Mode 18 is now a surprisingly useful crude PT reference, but it is not the
  intended final integrator. It has low-bounce path tracing and accumulation;
  no MIS, denoiser, ReSTIR, or sampled emissive area lights yet.
- Mode 19 inventory is intentionally non-final lighting work. It provides
  ReSTIR/PT-friendly candidate infrastructure: emissive triangle center, area,
  normal, luminance proxy, material index, instance id, primitive index, flags,
  and emissive texture slot/dimensions. Its dump prints both per-material
  summary rows and per-triangle samples.
- The latest in-game emissive dump from a clustered Mars City scene looked
  healthy: 11 unique emissive materials, real texture slots/sizes for all
  printed materials, no cap pressure, no invalid material refs, and 930
  skinned/game-logic dissolve candidates skipped.

Overall Goal
------------

Evolve the Vulkan/NVRHI renderer in RBDoom3 BFG toward a Quake II RTX style
brute-force path traced renderer. The target hardware is an RTX 4090 class GPU.
Early priorities are correctness, image quality, and learning. Broad hardware
support and performance are secondary for now.

Current work is intentionally debug-only. The "RT smoke" path in
PathTracePrimaryPass is a ladder of small tests that prove primary rays,
geometry capture, BLAS/TLAS, material tables, bindless albedo, alpha cutouts,
and simple lighting before replacing the main 3D renderer.

Current Texture and Material Binding Architecture
-------------------------------------------------

Current RT material flow:

1. During RT smoke scene capture, each captured Doom draw surface is assigned a
   compact RT material table index.

2. Per-triangle metadata stores:
   - surface class
   - material table index
   - flags such as "force geometric normal"
   - translucent subtype data for debug mode 12

3. CPU material discovery looks for safe, already available diffuse/base-color
   textures:
   - fast-path diffuse images
   - explicit diffuse material stages
   - no editor-image fallback
   - no forced image loading during RT capture

4. A bindless NVRHI Texture_SRV descriptor table is used for sampled textures.
   Current active capacity is 512 descriptors.

5. RT material table entries store descriptor indexes for diffuse/albedo and,
   where discovered, alpha/coverage textures.

6. Shader sampling uses Texture.Load by default. SampleLevel is kept as a
   diagnostic path only because it caused device-removed crashes near active
   SWF/GUI screens.

Important Design Decisions
--------------------------

- Keep the experiment behind r_pathTracing.
- Present the RT output as the 3D scene layer only; keep Doom HUD, GUI, and
  menus drawn on top by the existing renderer.
- Use NVRHI RT APIs and bindless descriptor tables rather than raw Vulkan
  descriptors.
- Use a persistent static-world cache and rebuild dynamic-candidate geometry
  every frame.
- Keep static and dynamic geometry in separate buffers so BLAS geometry and
  closest-hit attribute fetches stay matched.
- Rebuild TLAS every frame.
- Prefer conservative classification over unsafe static caching.
- Use RT-only CPU skinning for MD5/skinned surfaces when raster GPU skinning is
  enabled. This is a correctness bridge, not a final performance plan.
- Use Texture.Load for RT texture fetches until sampler/mip/synchronization
  issues near SWF/GUI surfaces are understood.
- Do not try to make translucent/decal/glow semantics perfect yet. Mode 12 is a
  classifier/debug tool, not final rendering.


Failed Approaches to Avoid Repeating
------------------------------------

- Do not call idMaterial::GetEditorImage() during RT texture discovery. It can
  force missing image loads and caused repeatable device-removed crashes.

- Do not force-load missing images during RT scene capture.

- Do not use SampleLevel as the normal RT texture path yet. It caused
  device-removed crashes near active animated SWF/GUI terminals, even when
  sampling fallback textures.

- Do not reuse a cached BLAS with different shader-side vertex/index/attribute
  buffers. The BLAS geometry and closest-hit fetch buffers must match.

- Do not treat Doom draw-surface order as stable identity.

- Do not break the whole capture loop when one surface exceeds a limit. Skip
  suspect/oversized surfaces and keep scanning.

- Do not merge static and dynamic triangle metadata casually. InstanceID is used
  to choose the correct buffer set in closest-hit.

- Do not revive the old Donut/RTXPT CMake integration attempt unless explicitly
  planning a clean integration. It caused dependency/version confusion.


Known Bugs and Unresolved Issues
--------------------------------

- Mode 18 is a crude real-time PT baseline with selected-light shadow rays,
  one diffuse-ish bounce, texture sampling, and accumulation. It is not a final
  integrator: no MIS, no denoiser, no ReSTIR, no real emissive area-light
  sampling, and only partial Doom light/material integration.

- Mode 13 is simple fixed-direction Lambert lighting only.

- Texture.Load has a manual bilinear path, but still does not implement mip
  selection, ray cones, derivatives, or anisotropy.

- SampleLevel/sampler path is unstable near active SWF/GUI surfaces.

- Material support is incomplete:
  - diffuse/albedo
  - alpha/coverage for cutout/perforated materials
  - diagnostic normal/specular/emissive maps
  - no roughness/metalness approximation
  - no final emissive area-light sampling
  - no blended translucent compositing

- Translucent subtype classification is approximate. Some glow/signage/object
  glass classifications are useful diagnostics but not final material behavior.

- The floor alpha issue seen in one spot appears to exist in the base game, not
  the RT path.

- Per-frame dynamic capture and RT-only CPU skinning are not optimized.

- PathTracePrimaryPass.cpp is too large and mixes too many responsibilities.
