Current feature state to preserve
---------------------------------

The current RT smoke path supports:

- Primary camera rays through NVRHI ray tracing.
- Static and dynamic BLAS/TLAS with separate geometry buffers.
- Persistent static-world geometry cache now owned by PathTraceGeometryUniverse,
  with static surface records and universe-owned key lookup/append completion.
- Per-frame dynamic geometry capture.
- RT-only CPU skinning for skinned MD5 characters as a temporary correctness
  bridge.
- Bindless material texture table using Texture.Load.
- Manual bilinear Texture.Load filtering.
- Doom diffuse YCoCg decode.
- Alpha any-hit for cutout/perforated materials.
- Albedo, normal, specular, and emissive texture discovery/diagnostics.
- Mode 14 selected point-light direct lighting with RT shadow rays.
- Mode 14 normal-map lighting using Doom/RBDoom's DXT5-style normal decode:
  normal X from texture alpha, normal Y from green, normal Z reconstructed.
- Mode 14 simple Blinn-style diagnostic specular from Doom spec maps.
- Mode 14 unshadowed emissive/glow preview from safe ambient/additive stages.
- Mode 18 toy one-bounce path tracing with selected Doom lights, texture
  sampling, force-ambient control, simple secondary rays, temporal accumulation,
  and lighting scale CVars.
- Mode 19 emissive triangle inventory/debug visualization. The CPU inventory is
  a broad material/triangle candidate list for future ReSTIR/PT work; the debug
  view samples emissive texture luminance and skips skinned/deformed character
  dissolve overlays.
- GUI/SWF texture and surface diagnostics behind opt-in CVars.
- Smoke particle placeholder dithering behind CVars.
- Portal/window and object-glass stochastic transparency placeholders.
- Debug-only selected-light sprite proxies for strobe/flare-like light shaders.
- Narrow callback-entity skip for custom-shader item shells to avoid the disk
  pickup/delete crash without hiding normal skinned/dynamic characters.
- Persistent material universe records with stable material indexes, material
  facts, emissive facts, and universe-derived active material table enabled by
  default.
- Cached universe-derived active material table enabled by default through
  r_pathTracingMaterialCache.

Current module/refactor state
-----------------------------

PathTracePrimaryPass used to own nearly everything. It is now intentionally
small: feature detection, per-frame entry, output sizing, and present/blit glue.
It still owns most persistent m_smoke* handles/counters. Static-world geometry
cache storage and static surface identity/append bookkeeping now live in
PathTraceGeometryUniverse.

Main module split:

- PathTracePrimaryPass.*: frame-level pass shell and persistent state owner.
- PathTraceSmokeSceneBuild.cpp: high-level per-frame scene build sequence.
- PathTraceSmokeResources.*: NVRHI smoke resource creation, reset, binding, and
  resource commit packaging.
- PathTraceSmokeDispatch.*: constants, resource transitions, dispatch,
  accumulation, and readback queueing.
- PathTraceSmokeReadback.cpp: optional debug readback logging.
- PathTraceAcceleration.*: BLAS/TLAS helpers, upload batch, static BLAS
  signature, and acceleration submit helpers.
- PathTraceSceneCapture.*: Doom draw-surface traversal, validation, bucketing,
  stats, and dynamic geometry append. Static surface cache identity and append
  completion now go through PathTraceGeometryUniverse.
- PathTraceGeometry.*: shared RT smoke vertex layout and pure geometry helpers.
- PathTraceGeometryUniverse.*: persistent static geometry universe scaffold,
  static surface records, static cache vectors, static lookup/append ownership,
  generation/stats, and future home for previous-frame geometry history.
- PathTraceSkinning.*: temporary CPU skinning bridge.
- PathTraceSurfaceClassification.*: surface bucket and translucent subtype
  classification.
- PathTraceDoomMaterialClassifier.*: Doom material/stage classification.
- PathTraceMaterialTextureDiscovery.*: visible material texture discovery and
  metadata registration.
- PathTraceTextureRegistry.*: safe texture metadata registry.
- PathTraceMaterialUniverse.*: persistent per-material records/facts and stable
  material universe indexes.
- PathTraceDynamicMaterialState.*: GPU-facing material table and cache.
- PathTraceGuiSurfaces.*: GUI surface admission gate.
- PathTraceEmissiveCandidates.*: emissive triangle candidate records and dumps.
- PathTraceLightSelection.*: selected debug light ranking.
- PathTraceDebugDumps.* and PathTraceSurfaceDebugDumps.*: terminal diagnostics
  and one-shot CVar trigger/reset behavior.
- PathTraceCVars.*: RT/PT CVar declarations and definitions.

The broad extraction phase is probably complete. Do not keep splitting purely
for line count. A future PathTraceSmokeState / PathTraceSmokeScene ownership
object may be useful, but that should be a deliberate architecture change.

    Copy-Item -LiteralPath E:\prog\rbdoom-3-bfg-rt\build\Debug\RBDoom3BFG.exe -Destination E:\prog\rbdoom-3-BFG-prebuilt\RBDoom3BFG.exe -Force

    cmake --build build --config Debug --target PathTracingShaders

    Copy-Item -LiteralPath E:\prog\rbdoom-3-bfg-rt\base\renderprogs2\spirv\builtin\pathtracing\pathtrace_smoke.rt.bin -Destination E:\prog\rbdoom-3-BFG-prebuilt\base\renderprogs2\spirv\builtin\pathtracing\pathtrace_smoke.rt.bin -Force

    cmake -S neo -B build

needed after adding a source file.

Material universe status
------------------------

The material side is now fairly complete for easy single-threaded wins:

- PathTraceMaterialUniverse owns persistent material records/facts.
- Stable material universe indexes exist and are used by emissive summaries and
  uploaded light candidate records.
- PathTraceDynamicMaterialState builds the active shader-facing material table
  from universe records by default.
- r_pathTracingMaterialCache caches the universe-derived active table by default.
- Static triangle material-index compatibility is preserved by keeping the
  compact active row order compatible with cached static material-index buffers.
- Optional validation can compare universe-derived and legacy active tables.

Do not casually change active material row order. It is tied to static cached
triangle material-index buffers. If changing this, use
r_pathTracingMaterialUniverseTableValidate and test mode 18 in material-heavy
areas.

Geometry universe status
------------------------

Current geometry universe implementation:

- PathTraceGeometryUniverse owns the persistent static-world cache vectors:
  surface keys, vertices, indexes, triangle classes, and triangle material IDs.
- It owns static surface records:
  RtSmokePersistentStaticSurfaceRecord with key, class/material IDs,
  current and previous vertex/index/triangle ranges, per-frame lifecycle flags,
  history validity, dirty flag, and geometry format enum.
- It owns static surface lookup:
  HasStaticSurface / FindStaticSurface.
- It owns append policy entry/completion:
  CanAppendStaticSurface, BeginStaticSurfaceAppend, CompleteStaticSurfaceAppend.
- It owns per-frame update bookkeeping:
  BeginFrame, TouchStaticSurface, and EndFrame mark static records as seen,
  newly created, disappeared, history-valid, invalidated, or dirty.
- Static record validation is reported through existing smoke diagnostics:
  validate=total/range/duplicate-key/history/key-vector. Clean tested frames
  report validate=0/0/0/0/0.
- PathTraceSceneCapture still builds the actual static vertex/index/class/material
  data, but it touches cached static surfaces through the universe and completes
  new appends through the universe.
- PathTraceSmokeSceneBuild still consumes the same shader-facing arrays, so the
  shader/resource contract is unchanged.
- No previous-frame GPU buffers or motion-vector output exists yet.

The next geometry work should be deliberate and behavior-preserving. The current
records are still a bridge: they encode identity, current/previous ranges, and
lifecycle state, but they do not yet own full previous-position buffers or
dynamic/rigid/skinned history.

Critical DLSS Ray Reconstruction / motion-vector requirement
------------------------------------------------------------

The user discovered this requirement during geometry-universe planning:

- Stock Q2RTX does not support DLSS Ray Reconstruction because it cannot produce
  the required motion-vector output.
- A Q2RTX Ray Reconstruction fork exists at:

      E:\prog\references\Q2RTX-rayreconstruction-master

- That fork is useful as an architecture reference. In particular, its shader
  path stores/uses previous triangle positions and previous instance transforms,
  then projects current and previous hit positions to produce motion vectors.

Design rule for this branch:

    PathTraceGeometryUniverse must be designed to retain one previous frame of
    geometry/history data. Geometry-derived motion vectors are a core future
    requirement, not a denoiser/RR afterthought.

Why this matters:

- Camera-only motion vectors are insufficient for moving rigid entities,
  animated/skinned meshes, reflected/refracted surfaces, and disocclusions.
- Frame-local geometry rebuilds lose the identity/history needed for temporal
  reconstruction.
- If the universe only stores "current packed PathTraceSmokeVertex arrays",
  adding DLSS RR later could require rebuilding the geometry ownership model.

The long-term target:

- Each persistent surface/triangle record should be able to map current and
  previous geometry.
- For static world, previous positions can often be identical or share the same
  range.
- For rigid entities, store current and previous transforms or equivalent
  current/previous world-space ranges.
- For skinned/deformed meshes, eventually store previous skinned positions or
  provide a GPU skinning history source.
- Records need invalidation/disocclusion flags when previous-frame mapping is
  missing or untrustworthy.
- Future GPU buffers should allow current/previous position data, position
  deltas, or another explicit geometry-history representation.

Compression/quantization note:

- Static mesh compression or quantization may be useful later for bandwidth/VRAM
  and cache behavior.
- Do not implement it before stable records/history and buffer ownership exist.
- Leave room in records for a geometry format enum/descriptor and bounds or
  quantization basis.
- Avoid making "std::vector<PathTraceSmokeVertex> forever" the permanent
  universe contract.

Geometry universe next-step plan
--------------------------------

Completed current step:

- PathTraceGeometryUniverse now has BeginFrame / TouchStaticSurface / EndFrame.
- Static records track lastSeenFrame, previousSeenFrame, seenThisFrame,
  newlyCreatedThisFrame, disappearedThisFrame, previousRangeValid, historyValid,
  and dirty.
- Static records carry explicit previous vertex/index/triangle offsets and
  counts. For static world, consecutive-frame history promotes the current
  range as the previous range because the cached static geometry is unchanged.
- Existing smoke logs report seen/new/gone, history/previous-range, dirty, and
  validation counters. User-tested mode 18 showed no crashes or visible
  regressions, history=N/N on stable seen surfaces, dirty=0 when standing still,
  and validate=0/0/0/0/0.

Recommended next step:

1. Normalize current/previous range fields into explicit small range structs.

   This is a code-cleanliness step before changing buffers:

   - current vertex/index/triangle range,
   - previous vertex/index/triangle range,
   - helper validation for both ranges,
   - no shader/resource behavior changes.

2. Add a one-shot geometry-universe diagnostic dump if validation fails.

   Useful dump fields:

   - record key,
   - current and previous ranges,
   - seen/new/gone/history/dirty flags,
   - validation failure category.

3. Later, add previous-frame/range history beyond static identity.

   Start with static world:

   - Static world records can mark previous range valid when unchanged.
   - Current and previous ranges may be identical for static surfaces.
   - This is the safest way to validate the history model before dynamic objects.

   Then rigid entities:

   - They need stable identity separate from static world keys.
   - Store current/previous model transforms and range mapping.
   - Do not rely on Doom draw-surface order as identity.

   Then skinned/deformed:

   - The current CPU skinning path is temporary.
   - Long term probably wants GPU-side skinned data or a history source.
   - Do not build a large CPU threading solution for this unless profiling proves
     it is worth it.

4. Later, move static GPU buffer ownership/dirty uploads into the universe.

   Only after CPU records/history are stable:

   - persistent static vertex/index/class/material buffers,
   - dirty ranges,
   - current/previous position or delta buffers,
   - buffer format descriptor/enum,
   - BLAS build/update policy tied to dirty state.

5. Later, generate motion vectors.

   This requires shader/resource contract work:

   - output motion/depth buffers,
   - current and previous view/projection matrices,
   - current/previous hit position reconstruction,
   - sky/background motion,
   - invalid/disoccluded motion handling,
   - eventually reflection/refraction motion if those paths are added.

   Use the Q2RTX Ray Reconstruction fork as a design reference, not as code to
   drop in wholesale.

Immediate next-work suggestions
-------------------------------

The heavy broad-refactor phase is complete. Next work should be focused and
specific.

1. CPU-bound performance investigation

   The user observed that raising mode 18 output resolution to 4K did not
   materially change FPS, strongly suggesting a CPU-side bottleneck.

   First use existing timing logs to identify whether time is in:

   - scene capture
   - CPU skinning
   - material metadata discovery
   - material table build
   - buffer creation/upload
   - BLAS/TLAS submit
   - dispatch preparation

   Low-risk CPU optimizations to consider:

   - Reuse/reserve per-frame scratch vectors.
   - Avoid repeated static-world capture/material work.
   - Strengthen material metadata caching.
   - Add more timing/counters around PathTraceSceneCapture and CPU skinning.

   Avoid building a large CPU job system unless profiling identifies a large,
   isolated, data-parallel hotspot. GPU-side skinning and GPU-side light prep are
   likely better long-term answers than elaborate CPU multithreading.

2. Optick profiling

   Optick is integrated but behind CMake option OPTICK, default OFF. Existing
   evidence:

   - neo/cmake-vs2022-win64-optick-profiling.bat exists.
   - USE_OPTICK controls whether OPTICK_EVENT macros compile in.
   - win_main.cpp has OPTICK_FRAME("MainThread").
   - engine/backend/GPU markers exist.
   - No obvious gameplay CVar for starting/stopping captures was found.

   Useful if doing a small prep pass:

   - Build with -DOPTICK=ON.
   - Add OPTICK_EVENT scopes in the PathTrace* hot paths.
   - Try external Optick capture again.

   If capture remains awkward, the current r_pathTracingTimingLog path is still
   simpler for immediate CPU bottleneck work.

3. ReSTIR/PT bare-minimum vertical slice

   Do not start by trying to integrate all of NVIDIA RTXPT. A good first slice:

   - Promote mode 19 emissive inventory into a real GPU light candidate buffer
     contract.
   - Add reservoir buffers/swap resources in a new PathTraceReservoirs.* or
     PathTraceReSTIR.* module.
   - Add a new debug mode that does single-frame reservoir direct lighting from
     emissive candidates.
   - Add spatial reuse after a single-frame reservoir works.
   - Add temporal reuse/reprojection after spatial/debug data are stable.

   Files likely involved:

   - PathTraceEmissiveCandidates.*
   - PathTraceDynamicMaterialState.*
   - PathTraceSmokeResources.*
   - PathTraceSmokeDispatch.cpp
   - pathtrace_smoke.rt.hlsl
   - PathTraceCVars.*

   Needed data contracts:

   - GPU material record with base color, alpha/cutout, normal/specular/roughness
     approximation, emissive color/texture, and unsupported/fallback flags.
   - Emissive candidate buffer with area, normal, material/emissive texture slot,
     approximate radiance/luminance, weight/pdf fields, and stable-ish identity.
   - Reservoir buffers for current/previous/intermediate data.
   - Reset rules when camera, output size, scene signature, material table, or
     candidate buffer changes.

4. Doom 3 interactive GUI surfaces

   Current direct-GUI-surface tracing is diagnostic only. Long-term direction:

   - Keep Doom's GUI/SWF raster path for panel contents.
   - Render each live panel offscreen or expose its composited texture.
   - Project that composited texture onto the physical panel surface.
   - Keep generated painter-order GUI quads out of the TLAS.

   Start in PathTraceGuiSurfaces.*, PathTraceSceneCapture.*, and
   PathTraceMaterialTextureDiscovery.* if working here. Preserve diagnostics
   before broadening behavior.

Design traps to avoid
---------------------

- Do not call idMaterial::GetEditorImage() during RT texture discovery.
- Do not force-load missing images during frame capture.
- Do not switch default RT texture sampling back to SampleLevel until the
  SWF/GUI device-removal issue is understood.
- Do not reuse a cached BLAS with mismatched shader-side attribute buffers.
- Do not treat Doom draw-surface order as stable identity.
- Do not casually merge static and dynamic triangle metadata. InstanceID matters.
- Do not broaden callback-entity capture without testing the disk pickup/delete
  crash scenario.
- Do not treat dynamic/game-logic-driven materials as static material truth.
  Improve diagnostics and narrow bridges first.
- Do not revive old in-tree RTXPT/Donut/CMake leftovers unless doing a deliberate
  clean integration.

End-of-phase recommendation
---------------------------

This is a good point to end high-reasoning broad refactor work and return to
focused, measured tasks:

- one CPU performance hotspot at a time,
- one Doom material family at a time,
- one GUI-surface experiment at a time,
- one ReSTIR/PT vertical slice at a time.
