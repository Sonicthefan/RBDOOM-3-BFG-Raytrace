Current module/refactor state
-----------------------------

PathTracePrimaryPass used to own nearly everything. It is now intentionally
small: feature detection, per-frame entry, output sizing, and present/blit glue.
It still owns most persistent m_smoke* handles/counters. Static-world geometry
cache storage and static surface identity/append bookkeeping now live in
PathTraceGeometryUniverse.

Main module split:

- PathTracePrimaryPass.*: frame-level pass shell and persistent state owner.
- PathTraceSmokeSceneBuild.cpp: high-level per-frame scene build sequence.
- PathTraceSmokeResources.*: NVRHI smoke resource creation, reset, binding, and
  resource commit packaging.
- PathTraceSmokeDispatch.*: constants, resource transitions, dispatch,
  accumulation, and readback queueing.
- PathTraceSmokeReadback.cpp: optional debug readback logging.
- PathTraceAcceleration.*: BLAS/TLAS helpers, upload batch, static BLAS
  signature, and acceleration submit helpers.
- PathTraceSceneCapture.*: Doom draw-surface traversal, validation, bucketing,
  stats, and dynamic geometry append. Static surface cache identity and append
  completion now go through PathTraceGeometryUniverse.
- PathTraceGeometry.*: shared RT smoke vertex layout and pure geometry helpers.
- PathTraceGeometryUniverse.*: persistent static geometry universe scaffold,
  static surface records, static cache vectors, static lookup/append ownership,
  generation/stats, and future home for previous-frame geometry history.
- PathTraceSkinning.*: temporary CPU skinning bridge.
- PathTraceSurfaceClassification.*: surface bucket and translucent subtype
  classification.
- PathTraceDoomMaterialClassifier.*: Doom material/stage classification.
- PathTraceMaterialTextureDiscovery.*: visible material texture discovery and
  metadata registration.
- PathTraceTextureRegistry.*: safe texture metadata registry.
- PathTraceMaterialUniverse.*: persistent per-material records/facts and stable
  material universe indexes.
- PathTraceDynamicMaterialState.*: GPU-facing material table and cache.
- PathTraceGuiSurfaces.*: GUI surface admission gate.
- PathTraceEmissiveCandidates.*: emissive triangle candidate records and dumps.
- PathTraceLightSelection.*: selected debug light ranking.
- PathTraceDebugDumps.* and PathTraceSurfaceDebugDumps.*: terminal diagnostics
  and one-shot CVar trigger/reset behavior.
- PathTraceCVars.*: RT/PT CVar declarations and definitions.

The broad extraction phase is probably complete. Do not keep splitting purely
for line count. A future PathTraceSmokeState / PathTraceSmokeScene ownership
object may be useful, but that should be a deliberate architecture change.

Build and copy workflow
-----------------------

Preferred Windows Release/PT builds now use the CMake presets in
neo/CMakePresets.json. They keep performance testing on explicit Release build
lanes and avoid accidentally launching the legacy Debug executable.

Configure/build PT retail:

    cmake --preset win64-pt-retail
    cmake --build --preset win64-pt-retail-release

Configure/build PT dev:

    cmake --preset win64-pt-dev
    cmake --build --preset win64-pt-dev-release

Configure/build PT dev with Optick:

    cmake --preset win64-pt-dev-optick
    cmake --build --preset win64-pt-dev-optick-release

Equivalent .bat wrappers in neo:

    cmake-vs2022-win64-pt-retail.bat
    cmake-vs2022-win64-pt-dev.bat
    cmake-vs2022-win64-pt-dev-optick.bat
    cmake-build-vs2022-win64-pt-retail-release.bat
    cmake-build-vs2022-win64-pt-dev-release.bat
    cmake-build-vs2022-win64-pt-dev-optick-release.bat

Copy the preset-built executable from the matching build folder, for example:

    Copy-Item -LiteralPath E:\prog\rbdoom-3-bfg-rt\build-win64-pt-dev\Release\RBDoom3BFG.exe -Destination E:\prog\rbdoom-3-BFG-prebuilt\RBDoom3BFG.exe -Force

Do not use the legacy Debug executable path for PT validation:

    cmake --build build --config Debug --target RBDoom3BFG

This path is kept only as a compile/link sanity check. It is broken for branch
validation: the generated legacy project can be stale until `cmake -S neo -B
build` is rerun after NVRHI source additions, and the resulting
`build\Debug\RBDoom3BFG.exe` is not representative for RT/PT behavior or
performance. Do not copy it into the prepared game folder for normal PT testing.

Avoid this old Debug copy path except when deliberately testing the broken
legacy lane:

    Copy-Item -LiteralPath E:\prog\rbdoom-3-bfg-rt\build\Debug\RBDoom3BFG.exe -Destination E:\prog\rbdoom-3-BFG-prebuilt\RBDoom3BFG.exe -Force

If pathtrace_smoke.rt.hlsl changes, also build/copy the RT shader blob:

    cmake --build build --config Debug --target PathTracingShaders

    Copy-Item -LiteralPath E:\prog\rbdoom-3-bfg-rt\base\renderprogs2\spirv\builtin\pathtracing\pathtrace_smoke.rt.bin -Destination E:\prog\rbdoom-3-BFG-prebuilt\base\renderprogs2\spirv\builtin\pathtracing\pathtrace_smoke.rt.bin -Force

If a new .cpp is added under neo/renderer/NVRHI, rerun:

    cmake -S neo -B build

The legacy Debug project does not reliably pick up new globbed NVRHI
PathTrace sources until this regeneration is done. This is another reason not
to treat `cmake --build build --config Debug --target RBDoom3BFG` as the branch
validation path.

For preset builds, rerun the relevant configure preset instead:

    cmake --preset win64-pt-dev

Material universe status
------------------------

The material side is now fairly complete for easy single-threaded wins:

- PathTraceMaterialUniverse owns persistent material records/facts.
- Persistent records store stable material facts only. Per-frame descriptor
  indexes, visible texture bindings, texture probe state, and compact active
  material table row order stay in PathTraceDynamicMaterialState.
- Stable material universe indexes exist and are used by emissive summaries and
  uploaded light candidate records.
- Persistent lookup is keyed by materialId plus a 64-bit material-name hash.
  The shader-facing materialId remains the Doom material-name hash used by
  triangle material arrays and active material tables, but the stronger
  persistent key reduces collision/oscillation risk.
- BeginSmokeMaterialUniverseFrame resets frame-local churn counters before the
  material table path. GetSmokeMaterialUniverseStats reports cumulative and
  frame-local hits, misses, rebuilds, signature checks, validation checks, and
  validation mismatches.
- ReserveSmokeMaterialUniverse reserves the persistent unordered_map before
  material table builds. Future level-load estimates should reserve closer to
  expected level material count.
- ClearSmokeMaterialUniverse exists for full reset boundaries: level unload,
  renderer shutdown, vid_restart, map changes, or an explicit future PT cache
  reset command.
- PathTraceDynamicMaterialState builds the active shader-facing material table
  from universe records by default.
- r_pathTracingMaterialCache caches the universe-derived active table by default.
- Static triangle material-index compatibility is preserved by keeping the
  compact active row order compatible with cached static material-index buffers.
- Optional validation can compare universe-derived and legacy active tables.
- Persistent-record validation remains gated by
  r_pathTracingMaterialUniverseValidate and defaults to off in normal gameplay.
- Mutation is render/main-thread owned. Worker jobs may prepare immutable
  material candidates, but should not mutate the persistent material universe
  map directly without a deliberate synchronization design.

Do not casually change active material row order. It is tied to static cached
triangle material-index buffers. If changing this, use
r_pathTracingMaterialUniverseTableValidate and test mode 18 in material-heavy
areas.

Geometry universe status
------------------------

Current geometry universe implementation:

- PathTraceGeometryUniverse owns the persistent static-world cache vectors:
  surface keys, vertices, indexes, triangle classes, and triangle material IDs.
- It owns static surface records:
  RtSmokePersistentStaticSurfaceRecord with key, class/material IDs,
  current and previous vertex/index/triangle ranges, per-frame lifecycle flags,
  history validity, dirty flag, and geometry format enum.
- It owns static surface lookup:
  HasStaticSurface / FindStaticSurface through a key-to-record lookup map.
- It owns append policy entry/completion:
  CanAppendStaticSurface, BeginStaticSurfaceAppend, CompleteStaticSurfaceAppend.
- It owns per-frame update bookkeeping:
  BeginFrame, TouchStaticSurface, and EndFrame mark static records as seen,
  newly created, disappeared, history-valid, invalidated, or dirty.
- Static record validation is reported through existing smoke diagnostics:
  validate=total/range/duplicate-key/history/key-vector when
  r_pathTracingGeometryUniverseValidate is enabled. Clean tested frames report
  validate=0/0/0/0/0.
- PathTraceSceneCapture still builds the actual static vertex/index/class/material
  data, but it touches cached static surfaces through the universe and completes
  new appends through the universe.
- PathTraceSmokeSceneBuild still consumes the same shader-facing arrays, so the
  shader/resource contract is unchanged.
- No previous-frame GPU buffers or motion-vector output exists yet.

The next geometry work should be deliberate and behavior-preserving. The current
records are still a bridge: they encode identity, current/previous ranges, and
lifecycle state, but they do not yet own full previous-position buffers or
dynamic/rigid/skinned history.

Important guardrails:

- Keep TouchStaticSurface on the key-to-record lookup path; do not return it to
  per-surface O(n) scans.
- Keep expensive validation and duplicate-key checks behind
  r_pathTracingGeometryUniverseValidate or another explicit diagnostic gate.
- Persistent records are for static/semi-static geometry with stable identity,
  not muzzle flashes, particles, sparks, or one-frame transient effects.
- The universe is single-owner mutable render-thread state unless a deliberate
  synchronization design is added.
- Record materialId is a persistent Doom/material-universe link, not a direct
  shader table row.

Critical DLSS Ray Reconstruction / motion-vector requirement
------------------------------------------------------------

The user discovered this requirement during geometry-universe planning:

- Stock Q2RTX does not support DLSS Ray Reconstruction because it cannot produce
  the required motion-vector output.
- A Q2RTX Ray Reconstruction fork exists at:

      E:\prog\references\Q2RTX-rayreconstruction-master

- That fork is useful as an architecture reference. In particular, its shader
  path stores/uses previous triangle positions and previous instance transforms,
  then projects current and previous hit positions to produce motion vectors.

Design rule for this branch:

    PathTraceGeometryUniverse must be designed to retain one previous frame of
    geometry/history data. Geometry-derived motion vectors are a core future
    requirement, not a denoiser/RR afterthought.

Why this matters:

- Camera-only motion vectors are insufficient for moving rigid entities,
  animated/skinned meshes, reflected/refracted surfaces, and disocclusions.
- Frame-local geometry rebuilds lose the identity/history needed for temporal
  reconstruction.
- If the universe only stores "current packed PathTraceSmokeVertex arrays",
  adding DLSS RR later could require rebuilding the geometry ownership model.

The long-term target:

- Each persistent surface/triangle record should be able to map current and
  previous geometry.
- For static world, previous positions can often be identical or share the same
  range.
- For rigid entities, store current and previous transforms or equivalent
  current/previous world-space ranges.
- For skinned/deformed meshes, eventually store previous skinned positions or
  provide a GPU skinning history source.
- Records need invalidation/disocclusion flags when previous-frame mapping is
  missing or untrustworthy.
- Future GPU buffers should allow current/previous position data, position
  deltas, or another explicit geometry-history representation.

Compression/quantization note:

- Static mesh compression or quantization may be useful later for bandwidth/VRAM
  and cache behavior.
- Do not implement it before stable records/history and buffer ownership exist.
- Leave room in records for a geometry format enum/descriptor and bounds or
  quantization basis.
- Avoid making "std::vector<PathTraceSmokeVertex> forever" the permanent
  universe contract.

Geometry universe next-step plan
--------------------------------

Completed current step:

- PathTraceGeometryUniverse now has BeginFrame / TouchStaticSurface / EndFrame.
- Static records track lastSeenFrame, previousSeenFrame, seenThisFrame,
  newlyCreatedThisFrame, disappearedThisFrame, previousRangeValid, historyValid,
  and dirty.
- Static records carry explicit previous vertex/index/triangle offsets and
  counts. For static world, consecutive-frame history promotes the current
  range as the previous range because the cached static geometry is unchanged.
- Existing smoke logs report seen/new/gone, history/previous-range, dirty, and
  validation counters. User-tested mode 18 showed no crashes or visible
  regressions, history=N/N on stable seen surfaces, dirty=0 when standing still,
  and validate=0/0/0/0/0.

Recommended next step:

1. Normalize current/previous range fields into explicit small range structs.

   This is a code-cleanliness step before changing buffers:

   - current vertex/index/triangle range,
   - previous vertex/index/triangle range,
   - helper validation for both ranges,
   - no shader/resource behavior changes.

2. Add a one-shot geometry-universe diagnostic dump if validation fails.

   Useful dump fields:

   - record key,
   - current and previous ranges,
   - seen/new/gone/history/dirty flags,
   - validation failure category.

3. Later, add previous-frame/range history beyond static identity.

   Start with static world:

   - Static world records can mark previous range valid when unchanged.
   - Current and previous ranges may be identical for static surfaces.
   - This is the safest way to validate the history model before dynamic objects.

   Then rigid entities:

   - They need stable identity separate from static world keys.
   - Store current/previous model transforms and range mapping.
   - Do not rely on Doom draw-surface order as identity.

   Then skinned/deformed:

   - The current CPU skinning path is temporary.
   - Long term probably wants GPU-side skinned data or a history source.
   - Do not build a large CPU threading solution for this unless profiling proves
     it is worth it.

4. Later, move static GPU buffer ownership/dirty uploads into the universe.

   Only after CPU records/history are stable:

   - persistent static vertex/index/class/material buffers,
   - dirty ranges,
   - current/previous position or delta buffers,
   - buffer format descriptor/enum,
   - BLAS build/update policy tied to dirty state.

5. Later, generate motion vectors.

   This requires shader/resource contract work:

   - output motion/depth buffers,
   - current and previous view/projection matrices,
   - current/previous hit position reconstruction,
   - sky/background motion,
   - invalid/disoccluded motion handling,
   - eventually reflection/refraction motion if those paths are added.

   Use the Q2RTX Ray Reconstruction fork as a design reference, not as code to
   drop in wholesale.

Critical Q2RTX / ReSTIR light-list lesson
-----------------------------------------

The Q2RTX Ray Reconstruction reference fork was checked specifically for
off-camera emissive behavior. Its useful lesson is not "keep the last visible
emissive surface alive." Q2RTX promotes emissive surfaces into an explicit light
sampling contract:

- Static emissive BSP surfaces become persistent light polygon records.
- Those light records are assigned to BSP clusters and gathered into
  per-cluster light lists using PVS plus a light-influence test.
- ReSTIR samples the light list for the cluster containing the shaded point,
  not the set of emissive surfaces visible to the current camera.
- Dynamic/model emissive lights are transformed and injected into the same
  cluster light lists each frame.
- Temporal ReSTIR history uses previous-to-current light identity remapping. If
  a previous dynamic light cannot be matched to a current light, that history is
  invalidated.

Direction adjustment for this branch:

- The current reservoir scaffold is still valid and should be kept.
- Do not treat final ReSTIR input as "visible emissive candidates plus a little
  persistence." That will keep producing screen-space-like behavior.
- Static world emissives should become persistent light-universe records that
  can be sampled independently of current camera visibility.
- Stable rigid/entity emissives, such as object-like vending-machine panels,
  need persistent identity plus current transform and some future
  area/portal/cluster-style membership.
- Game-logic emissives, such as switchable or dimming fluorescent lights, need
  persistent identity but per-frame evaluated active/color/intensity state from
  drawSurf->shaderRegisters.
- Truly transient or currently unmatchable dynamic emissives should remain
  current-frame injections until previous-to-current identity and matching
  occluder geometry are trustworthy.
- Temporal reuse should wait for stable light identity, previous-to-current
  remapping, runtime emissive state signatures, and geometry/occluder
  confidence.

The next useful ReSTIR work is therefore light-universe quality and spatial
membership diagnostics before spatial or temporal reuse. A Doom-side equivalent
does not have to clone Q2RTX clusters exactly, but it needs some area/portal or
leaf-style membership so off-camera emissives can affect nearby shaded points
without relying on current screen visibility.

Immediate next-work suggestions
-------------------------------

The heavy broad-refactor phase is complete. Next work should be focused and
specific.

1. CPU-bound performance investigation

   The user observed that raising mode 18 output resolution to 4K did not
   materially change FPS, strongly suggesting a CPU-side bottleneck.

   First use existing timing logs to identify whether time is in:

   - scene capture
   - CPU skinning
   - material metadata discovery
   - material table build
   - buffer creation/upload
   - BLAS/TLAS submit
   - dispatch preparation

   Low-risk CPU optimizations to consider:

   - Reuse/reserve per-frame scratch vectors.
   - Avoid repeated static-world capture/material work.
   - Strengthen material metadata caching.
   - Add more timing/counters around PathTraceSceneCapture and CPU skinning.

   Avoid building a large CPU job system unless profiling identifies a large,
   isolated, data-parallel hotspot. GPU-side skinning and GPU-side light prep are
   likely better long-term answers than elaborate CPU multithreading.

2. Optick profiling

   Optick is integrated but behind CMake option OPTICK, default OFF. Existing
   evidence:

   - neo/cmake-vs2022-win64-optick-profiling.bat exists.
   - USE_OPTICK controls whether OPTICK_EVENT macros compile in.
   - win_main.cpp has OPTICK_FRAME("MainThread").
   - engine/backend/GPU markers exist.
   - PT scene build/dispatch and cinematic decode scopes have been added in the
     hot paths that were useful during the GUI/video investigation.
   - No obvious gameplay CVar for starting/stopping captures was found.

   Useful if doing a small prep pass:

   - Build with -DOPTICK=ON.
   - Try external Optick capture again.
   - Keep new scopes boring and non-behavioral. Avoid broad console breadcrumb
     spam unless actively chasing a non-returning hitch; the temporary
     r_pathTracingPhaseLog/r_pathTracingPhaseSlowThreshold path was removed.

   If capture remains awkward, the current r_pathTracingTimingLog path is still
   simpler for immediate CPU bottleneck work.

3. ReSTIR/PT bare-minimum vertical slice

   Do not start by trying to integrate all of NVIDIA RTXPT or by reviving the
   removed Donut/RTXPT checkout. Current status:

   - Mode 19 emissive inventory now has a real GPU light-candidate contract:
     area, normal, UV bounds, material universe index, emissive texture slot,
     placeholder weight/PDF fields, and stable-ish identity.
   - PathTraceReservoirs.* now owns current/previous/spatial reservoir buffers.
     Reservoir history is cleared when output/camera/FOV, material table,
     emissive candidate records, light-candidate records, or static-BLAS
     signature changes. Use r_pathTracingReservoirDump 1 to inspect the state.
   - Mode 20 now does single-frame reservoir direct lighting from emissive
     candidates and writes current reservoir records. It is intentionally noisy
     and has no spatial/temporal reuse yet.
   - PathTraceLightUniverse.* now keeps static emissive candidates persistent by
     stable-ish identity, can promote stable rigid/object-like emissives into
     semi-static dynamic records, and merges persistent static records with
     current-frame dynamic emissives. Missing dynamic records are tracked but
     not injected by default because their matching occluder geometry is not
     persistent yet. Skinned/particle/non-rigid dynamic emissives remain
     frame-local.
     Use r_pathTracingLightUniverseDump 1 to inspect static seen/new/missing,
     semi-static dynamic promoted/missing/aged counts, dynamic frame count, and
     merged upload count.
   - Relevant mode 20/light-universe CVars:
     r_pathTracingReservoirTwoSidedEmissives,
     r_pathTracingLightUniversePersistDynamic,
     r_pathTracingLightUniverseInjectMissingDynamic,
     r_pathTracingLightUniverseDynamicMinSeenFrames,
     r_pathTracingLightUniverseDynamicMaxMissingFrames.
   - Important material/runtime split: PathTraceMaterialUniverse stores stable
     emissive capability/facts only. Per-frame capture owns evaluated runtime
     emissive state from drawSurf->shaderRegisters, including game-logic driven
     on/off/dimming behavior. Do not regress this back to material-only
     emissive decisions.
   - Mode 20 is a useful scaffold/debug view, not the final ReSTIR input model.
     Q2RTX points toward a proper light universe with spatial membership:
     persistent static lights, current dynamic injections, explicit
     previous-to-current light id remapping, and invalidation when a dynamic
     light cannot be matched.
   - Add candidate-origin diagnostics before reuse: visible/frame candidate,
     persistent static light, current dynamic injection, promoted/semi-static
     rigid light, and runtime/game-driven light state.
   - Start planning Doom-side spatial membership for light candidates before
     spatial or temporal reuse. This may be area/portal/leaf-style rather than
     a direct Q2RTX cluster clone.
   - Add spatial reuse only after the light list/candidate origin model is less
     screen-visible dependent.
   - Add temporal reuse/reprojection only after stable light identity,
     previous-to-current remapping, runtime emissive state signatures, and
     matching geometry/occluder confidence are available.

   Files likely involved:

   - PathTraceEmissiveCandidates.*
   - PathTraceDynamicMaterialState.*
   - PathTraceSmokeResources.*
   - PathTraceSmokeDispatch.cpp
   - pathtrace_smoke.rt.hlsl
   - PathTraceCVars.*

   Needed data contracts:

   - GPU material record with base color, alpha/cutout, normal/specular/roughness
     approximation, emissive color/texture, and unsupported/fallback flags.
   - Emissive candidate buffer with area, normal, material/emissive texture slot,
     approximate radiance/luminance, weight/pdf fields, and stable-ish identity.
   - Reservoir buffers exist for current/previous/intermediate data; shading
     does not consume them yet.
   - Reset rules currently cover camera, FOV, output size, static BLAS,
     material table, emissive candidates, and light candidates.

4. Doom 3 interactive GUI surfaces

   Current direct-GUI-surface tracing is diagnostic only. Long-term direction:

   - Keep Doom's GUI/SWF raster path for panel contents.
   - Render each live panel offscreen or expose its composited texture.
   - Project that composited texture onto the physical panel surface.
   - Keep generated painter-order GUI quads out of the TLAS.

   Start in PathTraceGuiSurfaces.*, PathTraceSceneCapture.*, and
   PathTraceMaterialTextureDiscovery.* if working here. Preserve diagnostics
   before broadening behavior.

   Important: do not reintroduce the old GuiModel.cpp generated-frontEndGeo or
   layer-offset experiment as a normal path. It made Doom's painter-order GUI
   layers become physical geometry, broke raster monitor/video screens, and
   produced repeatable GUI/cinematic hitches. GuiModel.cpp has been restored to
   original behavior: generated GUI drawSurfs have frontEndGeo = NULL. The RT
   path should eventually consume a composited panel texture on the physical
   screen surface, not the generated UI quads themselves.

5. Build/profile configuration notes

   - FFMPEG=ON/BINKDEC=OFF and FFMPEG=OFF/BINKDEC=ON were both tested in Debug
     for the monitor/video route. Decoder choice did not change the remaining
     behavior after GuiModel.cpp was restored.
   - RETAIL=ON only defines ID_RETAIL. It strips/gates shipping-vs-dev behavior
     and is not an optimization switch for Visual Studio multi-config builds.
     It was an early suspect because the first fresh clean RT build was
     retail-like, but a fresh generic Release non-retail control also ran at the
     120 FPS cap. Do not treat ID_RETAIL as the proven cause of the stall.
   - PATH_TRACING_DEV defines ID_PATH_TRACING_DEV on top of ID_RETAIL. Add
     future PT-only diagnostics or commands under this define for the explicit
     PT Release build lanes.
   - OPTICK=ON is available through win64-pt-dev-optick and the matching batch
     files. Keep it off for final performance sanity checks.
   - The legacy Visual Studio Debug compile path is broken for PT validation.
     It can be used as a compile/link sanity check only, and even then the
     generated project may need `cmake -S neo -B build` after NVRHI source
     additions. Do not copy `build\Debug\RBDoom3BFG.exe` into the prepared game
     folder for normal PT testing. Use the preset Release/PT lanes instead.
     Debug executables are expected to be about 30 MB here while Release
     executables are about 14-15 MB. The difference comes from /Od, /Zi, /RTC1,
     debug CRT, and incremental debug linking. The legacy Debug executable
     reproduced the broken raster/PT frame-pacing performance, while fresh
     Release builds were clean.
   - r_useValidationLayers defaults to 1, which enables NVRHI validation.
     Launching Debug with `+set r_useValidationLayers 0` restored raster to the
     120 FPS cap, so validation was the main raster-side Debug multiplier. Use
     validation only for targeted resource/state/binding investigations.
   - PT remains much slower in Debug/RelWithDebInfo even with validation off.
     The PT prototype does substantial CPU staging every frame: Doom draw-surface
     capture, vertex transforms, material metadata/table work, emissive
     inventory, buffer upload setup, and dynamic BLAS/TLAS preparation. Debug
     CRT allocation checks, disabled/reduced inlining, /Od, /RTC1, and
     debug-friendly codegen make that path non-representative. RelWithDebInfo is
     optimized but this generated project still uses MultiThreadedDebugDLL and
     weaker inlining than Release.
   - Clean upstream/master was bisected from v1.6.0 to babd04b3 with matching
     generated renderprogs2 and tested clear. Do not chase upstream again unless
     a clean upstream Release build starts reproducing.
   - r_pathTracing must be enabled before device creation for RT feature support
     to be available. If PT appears dead in a Release build, launch with
     +set r_pathTracing 1 or ensure the config sets it before renderer startup.

Design traps to avoid
---------------------

- Do not call idMaterial::GetEditorImage() during RT texture discovery.
- Do not force-load missing images during frame capture.
- Do not switch default RT texture sampling back to SampleLevel until the
  SWF/GUI device-removal issue is understood.
- Do not reuse a cached BLAS with mismatched shader-side attribute buffers.
- Do not treat Doom draw-surface order as stable identity.
- Do not casually merge static and dynamic triangle metadata. InstanceID matters.
- Do not broaden callback-entity capture without testing the disk pickup/delete
  crash scenario.
- Do not treat dynamic/game-logic-driven materials as static material truth.
  Improve diagnostics and narrow bridges first.
- Do not let generated GUI draw commands become persistent TLAS geometry. They
  are UI painter-order layers, not physical panels.
- Do not revive the removed in-tree RTXPT/Donut/CMake leftovers unless doing a
  deliberate clean integration.

End-of-phase recommendation
---------------------------

This is a good point to end high-reasoning broad refactor work and return to
focused, measured tasks:

- one CPU performance hotspot at a time,
- one Doom material family at a time,
- one GUI-surface experiment at a time,
- one ReSTIR/PT vertical slice at a time.

The module split is now readable enough that future sessions should start from
docs/pathtrace_modules.txt, inspect only the relevant files, and avoid reopening
the entire refactor unless a real ownership problem appears.
