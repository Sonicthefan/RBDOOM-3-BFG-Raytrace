Task 01: GI Reservoir Reuse, Visual-Gated Micro-Slices
Date: 2026-05-16
Project: rbdoom-3-bfg-rt path tracing / mode 56 ReSTIR PT

Task Boundary
-------------
Do not implement this whole file in one pass.

The user must commission exactly one micro-slice at a time:

    01A: audit and map current GI reservoir flow, no code changes
    01B: add visual compare diagnostics only
    01C: prove real GI temporal reuse only
    01D: prove GI spatial reuse only
    01E: switch mode 56 resolve only after user visual approval

Every micro-slice must build, deploy, report, and stop. The user will visually
inspect the result and decide whether to continue.

Reference-First Rule
--------------------
Do not invent or "solve" ReSTIR PT processing in this task. The reservoir math
and processing model must come from the local reference implementations. Agents
must inspect the relevant RTXDI / RTX Remix reference step when temporal reuse,
spatial reuse, neighbor validation, target-function evaluation, normalization,
bias correction, or final shading behavior is unclear.

The implementation goal is to make a rbdoom-local version of the existing
reference idea. Deviate only when rbdoom-specific pass structure, buffer
ownership, surface data, Doom lights, or RAB bridge constraints prevent a
direct mapping. In that case, document the mismatch and keep the deviation
debug-scoped until visually accepted.

Why This Matters
----------------
User testing showed that the GI view contains many black/dark samples on bright
surfaces. Albedo floors and DLSS/RR input brightness changes only turn black
dropout into grey dropout. This points upstream: the reservoir estimate itself
is too sparse or under-reused before it becomes image radiance.

Agents should not decide that a visual result is improved. The correct behavior
is to expose controlled comparison modes and let the user judge.

Current Branch Baseline
-----------------------
- Direct lighting/DI must be treated as raw reservoir output in the normal
  renderer. The current DI work is debug-view only through
  `r_pathTracingRestirPTDiDebugView`; do not describe normal DI as having
  completed temporal/spatial reservoir processing on this branch.
- GI/indirect has an initial-reservoir producer, visible initial views, and a
  debug-only temporal candidate exposed through
  `r_pathTracingRestirPTGiDebugView 2`.
- DI/direct now has debug-only temporal and spatial reservoir experiments
  exposed through `r_pathTracingRestirPTDiDebugView 2` and `3`. They use a
  DI-owned scratch reservoir buffer and do not feed normal mode 56.
  `r_pathTracingRestirPTDiDebugView 5` shows DI spatial/temporal status colors
  for investigation.
- The DI temporal debug view uses the primary-surface producer's exported
  motion-vector mask, xy motion, and reset/disocclusion mask for previous-pixel
  lookup. Same-pixel fallback is only allowed when that exported motion is
  valid and effectively zero, or when an ordinary rigid surface lacks an
  object-motion source but the same-pixel previous surface still matches. This
  is still an unvalidated reservoir-history debug path. View 2 evaluates the
  temporal reservoir in direct-light preview units without multiplying
  brightness by reservoir `M`; do not describe it as completed final DI
  lighting. Matching RTXDI PT temporal structure, an invalid current raw DI
  sample no longer immediately clears history; validated previous DI temporal
  history may still carry the pixel for this debug path.
  The local bridge still differs from the RTXDI PT reference because `u39`
  stores xy pixel motion only; the reference motion vector also carries a depth
  delta used for temporal depth validation.
- The DI spatial debug view first runs a debug-only temporal prepass into the
  DI-owned reservoir buffer, then the visible spatial pass consumes completed
  current-frame temporal DI neighbors. It is a debug-only candidate, not normal
  mode 56 output and not RR input.
- Normal mode 56 still combines the existing raw direct/DI reservoir preview
  with GI initial output.
- The current final resolve still loads the GI initial reservoir. This is bad
  for production/RR, but it is also the known baseline. Do not replace it until
  a new path is visually proven.

Global Do Not Do
----------------
- Do not copy reference source code into this repository.
- Do not implement RTX Remix sample stealing, adaptive history, RR
  compatibility randomization, virtual samples, or boiling filters in Stage 01.
- Do not implement ReSTIR PT Enhanced duplication maps, reciprocal paired
  spatial reuse, vector-valued shading weights, unified reservoirs, or dual
  motion vectors in Stage 01.
- Do not change mode 18 default rendering.
- Do not remove modes 53-55 diagnostics.
- Do not solve the problem by adding a brightness floor after final shading.
- Do not relabel initial reservoirs as temporal output.
- Do not add a side reservoir buffer unless the RTXDI resampling functions are
  also correctly pointed at that buffer and its current/previous slice history.
- Do not switch mode 56 resolve to any new GI output without user visual
  acceptance of the comparison mode.

01A: Audit And Map Current GI Flow
----------------------------------
Default task when no micro-slice is named.

Goal:

    Produce a written map of the current GI initial path and the exact places a
    future temporal/spatial path would connect.

Allowed edits:

    None, unless the user explicitly asks to write the audit into docs.

Required answers:

1. Which dispatch writes GI initial reservoirs in mode 56?
2. Which buffer and buffer index/page contain that output?
3. Which shader loads GI for mode 56 final resolve?
4. Which direct-path helpers already call true temporal/spatial resampling?
5. Which pieces are direct-domain-specific and cannot be reused blindly for GI?
6. What is the smallest diagnostic-only code slice for 01B?

Stop:

    Report the map and recommended 01B change. Do not implement 01B.

01B: Visual Compare Diagnostics Only
------------------------------------
Goal:

    Add a debug comparison mode that lets the user visually compare current GI
    initial output against any future GI candidate output. This slice must not
    change normal mode 56 rendering or RR input behavior.

Allowed edits:

- Add a small CVar or reuse an existing debug-view CVar for comparison.
- Add views such as:

      0 normal mode 56 output
      1 raw GI initial target estimate
      2 reserved GI temporal estimate, black if not implemented
      3 reserved GI spatial estimate, black if not implemented
      4 false-color reservoir validity / M / WeightSum check

- Add pass-plan text that prints which view is active.

Forbidden edits:

- Do not add new temporal/spatial producer passes.
- Do not add new reservoir buffers.
- Do not change `PathTraceRRInputColor`.
- Do not change normal mode 56 output when the debug view is off.

Validation:

    Build and deploy. Tell the user exactly which CVar values to try and what
    to compare against modes 54, 55, and 56.

Stop:

    Wait for user visual feedback. Do not implement 01C.

01C: Prove Real GI Temporal Reuse Only
--------------------------------------
Goal:

    Add a real GI temporal candidate path and expose it through the 01B debug
    view. Do not feed it into normal mode 56 output yet.

Required behavior:

- It must consume the existing GI initial reservoir output.
- It must call true temporal resampling or an explicitly documented equivalent.
- It must use a reservoir history path that actually corresponds to GI, not the
  direct-lighting reservoir history.
- It must preserve the raw GI initial view for side-by-side comparison.
- It must show rejected/invalid cases clearly in a debug view.

Forbidden behavior:

- Do not call `GenerateRestirPTInitialReservoir` and store the result as
  "temporal".
- Do not overwrite or relabel raw direct/DI reservoir state.
- Do not switch final resolve.
- Do not add spatial reuse.

Validation:

    Build and deploy. The user should compare:

      GI initial debug view
      GI temporal debug view
      mode 54
      mode 56 normal output

Stop:

    Report whether the temporal view is denser/brighter/darker, but do not
    claim success without user visual acceptance. Do not implement 01D.

01D: Prove GI Spatial Reuse Only
--------------------------------
Goal:

    Add GI spatial reuse on top of the user-accepted temporal GI output and
    expose it through the debug comparison view. Do not feed it into normal mode
    56 output yet.

Required behavior:

- It must consume the accepted GI temporal output.
- It must use validated current-frame neighbors.
- It must expose validity/rejection information.
- It must preserve initial and temporal comparison views.

Forbidden behavior:

- Do not switch final resolve.
- Do not use generic direct-spatial helpers unless the audit proves their buffer
  indices, domains, and history ownership are valid for GI.
- Do not add Remix hardening.
- Do not add unified DI/GI reservoirs.

Validation:

    Build and deploy. The user should compare:

      GI initial debug view
      GI temporal debug view
      GI spatial debug view
      mode 56 normal output

Stop:

    Wait for user visual acceptance. Do not implement 01E.

01E: Switch Mode 56 Resolve
---------------------------
Goal:

    Only after the user says the GI spatial/temporal debug output is visually
    better than raw GI initial, switch normal mode 56 and RR input to consume
    the accepted GI output.

Required behavior:

- Keep a CVar fallback to raw GI initial.
- Keep debug views for initial/temporal/spatial comparison.
- Keep final visibility/validation policy explicit.
- Keep `PathTraceRRInputColor` behavior documented.

Forbidden behavior:

- Do not remove the comparison views.
- Do not add Remix hardening.
- Do not add unified reservoirs.
- Do not tune brightness floors to hide failure.

Validation:

    Build and deploy. User compares normal mode 56 before/after using the
    fallback CVar, plus RR input preview if requested.

Stop:

    Report final files changed, exact CVar fallback, and remaining limitations.
    Do not start Stage 02.
