Implementation Slices For RTX Remix-Shaped Light Manager
========================================================

Purpose
-------

The old LM-01 through LM-08 sequence is complete enough to serve as scaffold,
but it is no longer the design direction. Do not assign new workers LM-09.

Assign exactly one LU-RM slice per worker. The goal is to rebuild the manager
contract in the RTX Remix shape while preserving Doom-specific adapter logic.

Do not define success as "the old rbdoom reservoir path stops flickering after
one more patch." The old path is too entangled with broad light counts,
portal-domain membership, and reset signatures. Use it as evidence and fallback
only. The implementation direction is to grow a new Remix-compliant chain from
PathTraceRestirLightManager until the manager-owned outputs can replace the old
ReSTIR light path.

Portal RTX / RTX Remix runtime debug toggles are part of planning. Before
claiming a layer is optional or required, compare against the reference runtime
when possible:

    - disable/toggle the reference feature in Remix
    - observe whether the same artifact class appears
    - if the artifact appears, promote that layer earlier in LU-RM ordering
    - if visual impact is low, defer it behind manager/RAB correctness

Every LU-RM layer that connects to a downstream lighting module must start by
checking the RTX Remix producer and immediate consumer pair. For light-manager
work that means:

    LightManager::prepareSceneData outputs
    RtxContext constants and descriptor bindings
    RtxdiApplicationBridge.slangh light reads and translation
    the first RTXDI/ReSTIR pass that consumes those reads

The worker must document the producer outputs and consumer reads before
changing rbdoom behavior. Matching that interface is what keeps later modules
free of invented ReSTIR math.

Initial acceptance is narrow:

    - basic diffuse DI/GI only
    - Remix-shaped temporal reprojection/reuse first
    - stable enough output for DLSS RR to consume
    - defer less important Remix features until that baseline works

Treat temporal reprojection as the single most critical slice family. If the
DI/GI temporal path cannot maintain stable history like Remix, downstream
denoisers and DLSS RR are not expected to fix it.

Use the RTX Remix temporal reuse viewer as the acceptance reference. During
rapid or erratic camera movement, the reference keeps most of the frame
temporally usable. rbdoom does not pass the temporal slice if its equivalent
debug view collapses across most of the screen under similar motion.

LU-RM-01: Manager Contract Records
==================================

    Reshape PathTraceRestirLightManager records around a Remix-style manager
    contract. No shader behavior change.

Owned files:

    neo/renderer/NVRHI/PathTraceRestirLightManager.h
    neo/renderer/NVRHI/PathTraceRestirLightManager.cpp

Steps:

    1. Separate stable identity fields from frame-local source indices.
    2. Keep source indices only as payload-load routing fields.
    3. Add explicit continuity class:

           stable
           current-only

           remap-invalid

       unknown/unsupported/incompatible/deleted.
    5. Add explicit payload-changed flags that do not imply remap invalid.

    7. Do not route new behavior yet.

    no RAB changes
    no reservoir clear changes
    no ReSTIR math changes
    no similarity fallback

LU-RM-03: Manager-Owned Current Previous Domains
================================================

Goal:

    Build manager-owned current and previous domains plus bidirectional maps.
    Stable identity maps even when payload changed.

Owned files:

    neo/renderer/NVRHI/PathTraceRestirLightManager.*
    neo/renderer/NVRHI/PathTraceSmokeSceneBuild.cpp

Steps:

    1. Keep previous manager domain independent of current frame upload order.
    2. Match current observations to previous manager records by stable identity.
    3. Produce currentToPrevious and previousToCurrent maps.
    4. Use invalid sentinel for missing, deleted, temporary, duplicate,
       projectile, unknown, unsupported, or explicitly incompatible records.
    5. Do not make color/intensity/radius/position payload changes invalidate
       a stable identity map.
    6. Dump counts:

           stableMapped
           payloadChangedMapped
           currentOnly
           previousOnly
           invalidByReason
           mapInvalidCurrent
           mapInvalidPrevious

    no shader routing changes

LU-RM-05: Structural-Only Global Clears
=======================================

Goal:

    Stop animated payload changes from causing global reservoir clears after
    LU-RM-04 evidence shows classification is correct.

    neo/renderer/NVRHI/PathTraceSmokeSceneBuild.cpp
    neo/renderer/NVRHI/PathTraceSmokeResources.cpp
    neo/renderer/NVRHI/PathTraceRestirLightManager.*

Steps:

    1. Use only structural incompatibility for global reservoir scene clears.
    2. Keep mapping/identity changes visible but not global clears.
    3. Keep payload changes visible but not global clears.
    4. Deleted/incompatible lights must map invalid locally.
    5. Build/deploy and stop for user visual testing.

Forbidden:

    no fallback lights

Acceptance:

    whole-domain reservoir refreshes. Dynamic lights that actually changed the
    local lighting may reject touched samples/reservoirs, but unrelated
    reservoir history survives.
